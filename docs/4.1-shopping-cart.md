## 1. Bài toán & các quyết định thiết kế then chốt

Giỏ hàng là nơi user "giữ tạm" sản phẩm trước khi đặt hàng. API cần: thêm / cập nhật số lượng / xóa 1 dòng / xóa hết / xem giỏ (kèm tổng tiền).

### Domain model

```
   User (1) ──────── (1) Cart
                          │ 1
                          │ N
                       CartItem ──── (N:1) ──── Product
                                └─── (N:1, optional) ── ProductVariant
```

So với Node.js: Express hay nhét cart vào `req.session.cart` (RAM/Redis, mất khi hết session) hoặc 1 collection Mongo. Bên Spring/JPA ta làm **persistent cart** — lưu DB, gắn cứng vào `user_id`, đăng nhập lại vẫn còn.

### 5 quyết định + chốt

| # | Quyết định | Chốt | Lý do |
|---|-----------|------|-------|
| ① | Persistent cart vs session cart | **Persistent** (DB), mỗi user 1 Cart `@OneToOne` | đã có User + JWT |
| ① | Tạo cart lúc nào | Lúc **register** + service vẫn **get-or-create** phòng thủ | dữ liệu user cũ chưa có cart |
| ② | Giá: snapshot hay live | **Cart = live price**, **Order = snapshot** | giỏ chỉ là "danh sách ý định"; đơn đã chốt thì đóng băng giá |
| ③ | Variant | **optional**; có variant → lấy giá/stock theo variant, không thì theo product | có product không variant |
| ④ | Dòng trùng | **cộng dồn quantity** (không tạo dòng mới) | chuẩn UX e-commerce |
| ⑤ | Check tồn kho | **mềm** lúc addToCart (BE), **cứng** lúc checkout (BE, có locking) | stock đổi giữa add và mua |

### Hệ quả của "live price"
- `CartItem` **KHÔNG** lưu `unitPrice`. Giá tính lúc view từ `product.price`/`variant.price` hiện tại.
- Đến Order (4.2) mới thêm `unitPrice` snapshot ghi xuống DB.

### Pattern get-or-create (chống NPE + lỗ hổng user cũ)
```java
Cart cart = cartRepository.findByUserId(uid)
        .orElseGet(() -> createCartFor(uid));
```

### ⚠️ Đính chính tư duy quan trọng — "check mềm" KHÔNG phải ở Front-end

> **Never trust the client.** FE check chỉ là UX (báo đỏ cho đẹp), KHÔNG bao giờ là lớp bảo vệ.

- **Check mềm (soft)** = vẫn ở **Backend**, lúc `addToCart`: validate `quantity <= availableStock` *tại thời điểm đó*. "Mềm" vì best-effort/advisory — stock có thể đổi giữa lúc add và lúc mua.
- **Check cứng (hard)** = ở **Backend**, lúc `checkout`: validate lại + **trừ tồn kho trong transaction có locking** (4.2). Đây mới là lớp đảm bảo thật.

FE check thêm vào tốt cho UX, nhưng BE **luôn phải tự check cả 2 lần**.

## 2. Entity — Owning side vs Inverse side

### Chốt data model
```
Cart
├── id
├── user        @OneToOne, unique, NOT NULL   (Cart giữ FK user_id)
├── cartItems   @OneToMany(mappedBy="cart", cascade=ALL, orphanRemoval=true)
└── (audit + version từ BaseEntity)

CartItem  — unique (cart_id, product_id, product_variant_id)
├── id
├── quantity        NOT NULL, > 0
├── cart            @ManyToOne, NOT NULL
├── product         @ManyToOne, NOT NULL
├── productVariant  @ManyToOne, NULLABLE (optional)
└── (audit + version từ BaseEntity)
```

### 🐞 Lỗi 1: vừa `mappedBy` vừa `@JoinColumn`

Một quan hệ 2 chiều trong DB chỉ có **đúng 1 cột FK**. JPA bắt chỉ định bên nào quản cột đó:

```
        OWNING side                         INVERSE side
   (bên giữ FK, có @JoinColumn)        (bên "soi gương", có mappedBy)
   @JoinColumn(name="user_id")         @OneToOne(mappedBy = "user")
   → Hibernate ghi cột user_id          → KHÔNG ghi gì vào DB, chỉ đọc ngược
```

- `mappedBy = "x"` = "Tôi là bên phụ (inverse). Bên chính là field tên `x` ở entity kia." → không có cột FK.
- `@JoinColumn` = "Tôi là bên chính (owning), cột FK ở bảng của tôi."
- → Vừa `mappedBy` vừa `@JoinColumn` = mâu thuẫn → Hibernate báo lỗi mapping khi khởi động.

Trong dự án: `User` đã khai `@OneToOne(mappedBy = "user") private Cart cart;` (User = inverse) → **Cart bắt buộc là owning side**:
```java
@OneToOne
@JoinColumn(name = "user_id", unique = true, nullable = false)
private User user;
```

> So với Sequelize: `Cart.belongsTo(User)` (giữ `userId`) + `User.hasOne(Cart)` (đọc ngược). `belongsTo` = owning, `hasOne` = inverse.

### 🐞 Lỗi 2: bỏ `mappedBy` nhưng lỡ xóa luôn `@OneToOne`

`@JoinColumn` chỉ là "phụ kiện" mô tả *cột FK tên gì* — **không tự định nghĩa quan hệ**. Phải có `@OneToOne`/`@ManyToOne` mới nói cho Hibernate biết đây là association. Thiếu nó → Hibernate cố map `User` như cột basic → fail.

### Khởi tạo collection (chống NPE)
```java
private List<CartItem> cartItems = new ArrayList<>();
```
Để `null` thì lúc add item đầu tiên `cart.getCartItems().add(...)` sẽ NPE (bài học từ Product).

### 📝 Note fetch type
`@OneToOne` và `@ManyToOne` mặc định fetch **EAGER**. Load `CartItem` là Hibernate tự kéo luôn `product` + `variant`. Lúc build query "xem giỏ" sẽ bàn `JOIN FETCH` / `@EntityGraph` để tránh N+1.

## 3. Repository — bẫy NULL trong derived query & `@Query`

### Naming của derived query
Spring Data đọc tên method như mini-DSL, tách theo `And`/`Or`, **mỗi token phải khớp đúng tên field** entity. Field tên `productVariant` mà viết `...AndVariantId` → tìm property `variant` → không có → app **fail lúc khởi động** (`PropertyReferenceException`).

### Bẫy `= NULL` (sự thật CS thuần, đúng mọi SQL/JPQL)
> `product_variant_id = NULL` **KHÔNG BAO GIỜ** đúng. So sánh `=` với NULL cho ra `UNKNOWN`, không phải `TRUE`. Muốn match NULL phải dùng `IS NULL`.

Derived query *có* tự đổi `= ?` → `IS NULL` khi tham số null, nhưng đó là hành vi ngầm. Với logic merge quan trọng → viết `@Query` tường minh:

```java
@Query("""
    SELECT ci FROM CartItem ci
    WHERE ci.cart.id = :cartId
      AND ci.product.id = :productId
      AND ( (:variantId IS NULL AND ci.productVariant IS NULL)
            OR ci.productVariant.id = :variantId )
    """)
Optional<CartItem> findExisting(@Param("cartId") Long cartId,
                                @Param("productId") Long productId,
                                @Param("variantId") Long variantId);
```

### 🐞 Bug từng mắc: `:productId IS NULL` thay vì `ci.productVariant IS NULL`
`:productId` là tham số `@NotNull` → không bao giờ null → khi add sản phẩm không variant, điều kiện ra `FALSE` → query không match → service tạo dòng mới mỗi lần = **đẻ dòng trùng**.

> **Quy tắc:** trong `@Query`, `IS NULL` phải áp lên **field của entity** (kiểm tra trạng thái), tham số chỉ dùng để **so giá trị**. Nhầm 2 vai là lỗi hay gặp.

### ⚠️ Unique constraint không cứu được dòng trùng khi variant NULL
MySQL **cho phép nhiều dòng NULL** trong unique constraint → 2 CartItem cùng product mà `product_variant_id = NULL` vẫn lọt. → **Logic merge ở service mới là lớp chặn chính**, constraint chỉ là defense-in-depth.

## 4. Mapper — presentation logic & `@AfterMapping`

### Ranh giới trách nhiệm
| Tầng | Lo việc gì |
|------|-----------|
| **Service** | Business logic: validate, check stock, merge, transaction, quyết định ghi DB |
| **Mapper** | Entity → DTO, gồm cả **field dẫn xuất để hiển thị** (`unitPrice`, `subTotal`, `variantName`) |

Theo convention từ module Product (`primaryImageUrl` dùng `default method` + `@Named`), presentation logic để ở **mapper**, KHÔNG đẩy sang service. `unitPrice`/`subTotal` chỉ là đọc + tính từ entity để render → presentation derivation → thuộc mapper.

> Phân biệt tinh tế: "lấy giá variant hay product" nghe như business rule, nhưng vì Cart dùng **live price** nên chỉ là quy tắc *đọc*. Khi sang Order, `unitPrice` snapshot ghi vào DB → lúc đó việc chọn giá là **business logic ở service**. Cùng phép tính: "tính để hiển thị" (mapper) ≠ "tính để lưu" (service).

### `@AfterMapping` — lifecycle hook
- Field dẫn xuất từ **1 object** (`variantName` ← `productVariant`) → `@Named` + `qualifiedByName`.
- Field cần **cả nhiều field / tính sau** (`unitPrice` cần product + variant; `subTotal`, `totalAmount`, `totalItems` cộng dồn collection) → `@AfterMapping`: chạy **sau** khi code generate map xong các field thường, nhận `@MappingTarget` để set nốt.
- ⚠️ Policy `unmappedTargetPolicy = ERROR`: field set bằng `@AfterMapping` không được tính là đã map → phải `@Mapping(target=..., ignore=true)` cho chúng, rồi mới set trong hook.

File `CartMapper` hoàn chỉnh: xem `api/src/main/java/com/ecommerce/api/mapper/CartMapper.java`.

### 🐞 Bug từng mắc: `totalAmount` cộng `unitPrice` thay vì `subTotal`
`unitPrice` là giá 1 đơn vị, chưa nhân số lượng. Tổng giỏ phải cộng `subTotal` (= `unitPrice × quantity`).

## 5. DEEP DIVE — `reduce` trong Java Stream API (Q&A nguyên văn)

> Đây là phần lý thuyết được hỏi sâu nhất buổi này. Giữ verbatim.

### Q: Tôi chưa có nhiều kinh nghiệm về reduce trong java stream api, giờ tôi đa gặp vấn đề ở fillCartTotals, hãy giảng cho tôi về reduce để tôi triển khai tiếp nhé

**Ý tưởng cốt lõi: "gấp" một stream thành 1 giá trị.** `reduce` = lấy nhiều phần tử → gộp dần → một kết quả. Giống `Array.prototype.reduce` của JS:

```js
const total = items.map(i => i.subTotal).reduce((acc, x) => acc + x, 0);
```

Java tương đương, bắt kiểu chặt hơn:
```java
BigDecimal total = items.stream()
    .map(CartItemResponse::getSubTotal)        // Stream<...> -> Stream<BigDecimal>
    .reduce(BigDecimal.ZERO, BigDecimal::add); // (identity, accumulator)
```

**Mổ xẻ 2 tham số:**
- `identity` (= JS initialValue): giá trị acc ban đầu, VÀ kết quả khi stream rỗng.
- `accumulator` = `BinaryOperator<BigDecimal>` = `(a, b) -> a.add(b)` — gộp 2 phần tử thành 1.

**Cơ chế (fold từ trái):** `[100,200,50]` → `ZERO→100→300→350`.

**`totalItems` dùng `mapToInt().sum()` gọn hơn:**
```java
int totalItems = items.stream().mapToInt(CartItemResponse::getQuantity).sum();
```
`mapToInt` → `IntStream` (tránh boxing) có sẵn `.sum()`/`.average()`/`.max()`. `BigDecimal` là object thường → phải tự `reduce`. **Tuyệt đối không** convert tiền sang `double` (sai số dấu phẩy động).

---

### Q: bạn đang nói về quy tắc này, hãy chỉ ra rõ hơn được không: dạng reduce(identity, accumulator) 2 tham số yêu cầu phần tử stream cùng kiểu với identity...

**3 overload của `reduce`:**
```java
Optional<T>  reduce(BinaryOperator<T> accumulator);                                   // (1) trả Optional
T            reduce(T identity, BinaryOperator<T> accumulator);                        // (2) ← đang dùng
<U> U        reduce(U identity, BiFunction<U,? super T,U> accumulator, BinaryOperator<U> combiner); // (3) đổi kiểu
```

Trong (2), `T` xuất hiện 3 lần và **đều là một kiểu** = `T` của `Stream<T>`.

**Mấu chốt `BinaryOperator<T>`:**
```java
interface BinaryOperator<T> extends BiFunction<T, T, T> {
    T apply(T a, T b);   // T + T -> T   ← cả input lẫn output đều T
}
```
→ accumulator của (2) ép: acc kiểu `T`, phần tử kiểu `T`, kết quả kiểu `T`. **Tất cả cùng một `T`.**

**Ráp vào `Stream<CartItemResponse>`:** `T = CartItemResponse`, nhưng `identity = BigDecimal.ZERO` → lệch kiểu → compile error. Sửa: `.map(...)` biến thành `Stream<BigDecimal>` → `T = BigDecimal` → khớp.

> **Nguyên tắc nhớ:** overload `reduce(identity, accumulator)` ép identity = phần tử stream = kết quả = cùng 1 kiểu `T`. Nên trước khi reduce một field nằm *bên trong* object, luôn `.map()` để kéo field đó ra.

---

### Q: Vậy bạn đọc đoạn code này xem có đúng không... (ví dụ Product/Rating reduce 3 tham số, in ra Point: 3.6)

**Cấu trúc: ĐÚNG** — dùng đúng overload (3) đổi kiểu `Product → Rating`: identity `new Rating()` (U=Rating), accumulator `(Rating, Product)->Rating`, combiner `Rating::average` `(Rating,Rating)->Rating`.

**Nhưng `reduce` có 3 luật:**
| Luật | Yêu cầu |
|------|---------|
| Identity | `combiner(identity, u) == u` |
| Associative | `(a⊕b)⊕c == a⊕(b⊕c)` |
| Consistency | `combiner(u, accumulator(identity, t)) == accumulator(u, t)` |

**"Trung bình" vi phạm associative** — trung bình của các trung bình ≠ trung bình tổng:
```
a:[5,3] avg=4.0   b:[4,2,4] avg=3.333
combiner trung bình: (4.0+3.333)/2 = 3.667   ❌
trung bình đúng: (5+3+4+2+4)/5 = 3.6          ✓
```
`Arrays.asList` là sequential → không gọi combiner → kết quả tùy `Rating` lưu gì: lưu `sum+count` thì 3.6 đúng & associative; lưu một số `points` (trung bình thuần) thì sai trọng số, parallel ra số khác.

> 🔑 Test nhanh associative: đổi `.stream()` → `.parallelStream()`. Kết quả đổi = logic reduce sai bản chất.

### Q: Đây là code trôi lấy trên mạng thôi chứ không phải tôi code

Chốt 2 ý: (1) reduce field trong object → `.map()` kéo ra trước rồi `reduce(identity, accumulator)`; (2) hàm gộp phải associative + identity thật. `fillCartTotals` dùng `BigDecimal::add` (cộng — associative, identity ZERO) → không dính bẫy.

### Q: tôi đã viết totalAmount0 theo cách khác... kiểm tra xem đúng không, và viết tường minh hơn sẽ như nào (nếu không dùng BigDecimal::add cho cả 2 hướng)

`totalAmount0` dùng overload (3) để khỏi cần `.map()` — **đúng 100%**:
```java
.reduce(BigDecimal.ZERO,                        // identity (U=BigDecimal)
        (a, b) -> a.add(b.getSubTotal()),       // accumulator (U,T)->U
        BigDecimal::add);                        // combiner (U,U)->U
```

**accumulator vs combiner:**
```
accumulator:  U + T -> U   gộp MỘT phần tử (CartItemResponse) vào acc
combiner:     U + U -> U   gộp HAI acc bộ phận (của 2 luồng song song)
```
Sequential chỉ gọi accumulator, combiner không chạy (nhưng vẫn bắt buộc khai báo).

**Viết tường minh (combiner thành lambda):**
```java
.reduce(BigDecimal.ZERO,
        (acc, item) -> acc.add(item.getSubTotal()),  // cộng subTotal 1 item vào tổng
        (acc1, acc2) -> acc1.add(acc2));             // ghép 2 tổng bộ phận
```
`BigDecimal::add` ⟺ `(x, y) -> x.add(y)` (instance method gọi trên tham số đầu, tham số sau làm argument).

### Q: reduce này đang thực sự chạy như nào nhỉ... sequential stream và parallel stream là gì, lấy ví dụ cụ thể xong mô phỏng luồng chạy từng bước

```
sequential = 1 thread, xử lý phần tử LẦN LƯỢT theo thứ tự
parallel   = chia dữ liệu nhiều mảnh, NHIỀU thread song song, rồi ghép lại
```
`.stream()` → sequential (mặc định). `.parallelStream()` → parallel.

> So với Node.js: `Array.reduce` của JS **luôn** sequential trên event loop, không có bản parallel sẵn (phải dùng Worker Threads). Java bật parallel chỉ bằng đổi `.stream()` → `.parallelStream()`.

**Ví dụ: subTotal = [100, 200, 50, 80].**

SEQUENTIAL (chỉ accumulator, fold trái→phải):
```
acc=ZERO → +100=100 → +200=300 → +50=350 → +80=430   ← KẾT QUẢ
```
Bản chất = vòng for. Combiner KHÔNG chạy.

PARALLEL (`Spliterator.trySplit()` chẻ đôi, ForkJoinPool):
```
            [100,200,50,80]
          ┌───────┴───────┐
      [100,200]        [50,80]
      thread-A         thread-B
      partialA=300     partialB=130
          └───────┬───────┘
          combiner(300,130) = 430   ← KẾT QUẢ
```
- Mỗi thread bắt đầu từ **identity (ZERO)** → identity phải trung tính thật.
- combiner mới là chỗ ghép kết quả 2 luồng.

**Vì sao phải associative:** `((100+200)+50)+80 = 430` (sequential) = `(100+200)+(50+80) = 430` (parallel). Còn "trung bình" thì 2 cách gom ra 2 số khác → sai.

| | Sequential | Parallel |
|---|---|---|
| Số thread | 1 | nhiều (ForkJoinPool) |
| Chia dữ liệu | không | `Spliterator.trySplit()` |
| accumulator | mọi phần tử | trong từng mảnh |
| combiner | **không gọi** | ghép các mảnh |

> ⚠️ Parallel **không phải lúc nào cũng nhanh hơn** — chia mảnh + quản thread + ghép có overhead. List nhỏ (giỏ hàng vài chục item) → sequential luôn thắng. Parallel chỉ đáng khi data rất lớn + tính nặng + nguồn dễ chẻ (`ArrayList`/mảng, không phải `LinkedList`).

---

### Q: theo như bạn giải thích thì combiner bị thừa dù cho mình có viết logic khi stream ở dạng sequential à?

**Runtime sequential: combiner không chạy (đúng).** Nhưng **thiết kế: không thừa — bắt buộc.** Stream không biết trước lúc compile sẽ chạy sequential hay parallel; ai đó thêm `.parallel()` → combiner sẽ dùng. → Đừng viết bừa `(a,b)->a`; luôn viết combiner đúng.

**"Aha" — vì sao có 2 overload:**
```
2 tham số:  T == U  →  accumulator (BinaryOperator<T>) kiêm luôn combiner  →  KHỎI cần combiner
3 tham số:  T ≠ U   →  accumulator (U,T)→U không ghép được U+U  →  PHẢI có combiner
```
- Overload (2): phần tử và acc cùng kiểu `T` → accumulator `(T,T)->T` vừa fold vừa ghép → Java dùng chính accumulator làm combiner khi parallel.
- Overload (3): accumulator `(U,T)->U` không nhận `(U,U)` → bắt buộc combiner riêng.

| | `.map().reduce(2 args)` | `.reduce(3 args)` |
|---|---|---|
| Có `.map()` riêng | có | không |
| Phải viết combiner | không (accumulator kiêm) | có (bắt buộc) |
| Khi nào hợp | T==U (phổ biến nhất) | T≠U, hoặc tránh stream trung gian |

> 💡 Mặc định ưu tiên `.map(...).reduce(identity, op)` 2 tham số — ngắn, không phải nghĩ combiner. Chỉ dùng 3 tham số khi thật sự cần đổi kiểu mà không muốn `.map()`.

## 6. Service — transaction & các quyết định

### Chốt 2 câu hỏi thiết kế
1. **Check stock khi cộng dồn**: theo **tổng mới** (`old + new`) — chặn sớm, báo cho khách tránh checkout fail sau này.
2. **Transaction**: class-level `@Transactional(readOnly = true)` default + override `@Transactional` (write) cho mọi method. **`getMyCart` cũng phải write** vì get-or-create có thể tạo cart.

### ⚠️ `getMyCart` KHÔNG được là `readOnly`
`@Transactional(readOnly = true)` không chỉ là nhãn:
1. Hibernate set **FlushMode = MANUAL** → thay đổi có thể không ghi xuống DB.
2. Spring gọi `Connection.setReadOnly(true)` → một số DB/driver reject INSERT/UPDATE.

→ Method nào *có khả năng ghi* → không gắn readOnly. (Nuance: `GenerationType.IDENTITY` buộc INSERT ngay để lấy id nên *đôi khi* lọt qua FlushMode, nhưng đừng dựa vào may rủi.)

### ⚠️ `REQUIRES_NEW` + bẫy self-invocation
Ý "tách create ra method `REQUIRES_NEW`" là **sai công cụ**:
- `REQUIRES_NEW` = tạm dừng tx hiện tại, mở tx **độc lập**, commit riêng — dùng cho việc *phải commit dù tx ngoài rollback* (audit log). Việc tạo cart **nên** chung tx với caller (add fail thì rollback cả cart).
- **Bẫy self-invocation**: `@Transactional` hoạt động qua **AOP proxy**. Khi `getMyCart()` gọi `this.createCart()` → gọi TRỰC TIẾP, KHÔNG qua proxy → annotation transaction **bị bỏ qua hoàn toàn**.

```
caller ──► [CartProxy] ──► CartService thật
              │
              └─ proxy mới là chỗ mở/commit transaction

getMyCart ──► this.createCart()   ← internal call, KHÔNG qua proxy → @Transactional vô hiệu
```

> So với Node.js: Express tự `await db.transaction(...)` tường minh, gọi hàm nào cũng được. Spring giấu transaction sau proxy AOP → có bẫy "gọi nội bộ thì mất phép thuật". Muốn proxy can thiệp phải gọi qua bean khác (hoặc self-inject).

→ Giải pháp dùng ở đây: `getOrCreateCart` là **private helper gọi thẳng** trong `getMyCart` → chạy chung tx của `getMyCart` (đúng điều ta muốn), không cần annotation riêng.

### Các quyết định khác trong Service
- **Check ACTIVE**: đẩy vào repository `findByIdAndStatus(id, ACTIVE)` thay vì check status ở service → 1 round-trip, sản phẩm ẩn coi như "không tồn tại" với người mua.

## 8. DEEP DIVE — N+1 Query Problem & cách xử lý

### N+1 là gì
```
1  query lấy danh sách cha (N bản ghi)
+N query lấy con cho TỪNG cha
= N+1 query
```
Trong Cart, map giỏ 10 dòng chưa tối ưu: 1 (carts) + 1 (cart_items) + N (products) + N (variants) ≈ 21 query. Mỗi query 1 round-trip → latency cộng dồn.

> ORM cho truy cập quan hệ như thuộc tính object → **giấu** việc mỗi lần chạm là 1 query. Lazy loading trong vòng lặp càng dễ dính.
> Node.js tương đương: Sequelize/Prisma `include` chính là để gộp query — `@EntityGraph`/`JOIN FETCH` của JPA = `include`.

### 4 cách xử lý
| Cách | Cơ chế | Khi nào dùng |
|------|--------|--------------|
| **`JOIN FETCH`** (JPQL) | dịch thẳng thành SQL có JOIN, kéo cha+con 1 lần | kiểm soát chính xác; nhớ `LEFT JOIN` kẻo mất cha rỗng |
| **`@EntityGraph`** | khai báo path cần fetch, Hibernate tự sinh JOIN (≈ JOIN FETCH) | không muốn viết JPQL; tách "fetch gì" khỏi query ← **Cart dùng cái này** |
| **`@BatchSize`** | gom N query lẻ thành query `IN (...)` → `1 + ceil(N/size)` | khi có `Pageable` (JOIN FETCH collection làm hỏng phân trang); nhiều collection ← **Product.images dùng** |
| **DTO Projection** | SELECT thẳng cột → ra DTO, bỏ qua entity | read-heavy/report, cần tốc độ tối đa |

### 2 bẫy cực quan trọng
- **Bẫy A — `JOIN FETCH collection` + `Pageable` = sai phân trang.** JOIN collection nhân dòng → Hibernate không phân trang ở DB được, kéo HẾT về RAM rồi cắt (`HHH000104`). → Đây là lý do `Product.images` dùng `@BatchSize` chứ không JOIN FETCH.
- **Bẫy B — `MultipleBagFetchException`.** JOIN FETCH 2 collection kiểu `List` cùng lúc → exception (tích Đề-các). Vá: dùng `Set`, tách query, hoặc 1 JOIN FETCH + 1 `@BatchSize`.

### Vì sao `getMyCart` không dính bẫy
- Chỉ **1 collection** (`cartItems`) → không `MultipleBagFetchException`. ✅
- **Không `Pageable`** (giỏ load hết) → JOIN FETCH collection vô tư. ✅
- `cartItems.product` + `cartItems.productVariant` là **to-one** → JOIN không nhân dòng nguy hiểm. ✅

```java
// CartRepository — chống N+1 cho hot path "xem giỏ"
@EntityGraph(attributePaths = {"cartItems", "cartItems.product", "cartItems.productVariant"})
Optional<Cart> findByUserIdWithItems(Long userId);
```

### Trade-off ghép vào `getMyCart`
```java
Cart cart = cartRepository.findByUserIdWithItems(userId)
        .orElseGet(() -> getOrCreateCart(userId));
```
- Cart đã tồn tại (hot path, nhiều lần): 1 query JOIN, không N+1. ✅
- Cart chưa có (cold path, 1 lần/đời user): tốn 2 SELECT nhưng cart rỗng → không có gì để N+1. Chấp nhận.

→ Tối ưu hot path, chịu thiệt nhỏ ở cold path hiếm. Đừng làm phức tạp code chỉ để tối ưu cold path.

### Known minor (để sau)
`addToCart`/`updateItem` cũng map cart ở cuối → nếu giỏ có nhiều dòng cũ thì lazy-load chúng (N+1 nhẹ). Không xử lý ngay vì: sau add/update người dùng thường gọi `GET /api/carts` (đã tối ưu); xử lý sẽ làm 2 method phức tạp (phải reload với fetch).
