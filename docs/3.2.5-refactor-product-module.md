# Bài 3.2.5 — Refactor: Lombok + MapStruct (Phần 4: Module Product + ProductVariant + ProductImage)

> Module cuối cùng và **phức tạp nhất** của bài 3.2.5. Khác User + Category ở 3 điểm:
> - **Nested mapping**: `seller.id → sellerId`, `category.name → categoryName`
> - **Multi-mapper compose**: 1 `ProductMapper` `uses` 1 `ProductImageMapper` riêng
> - **Custom presentation logic**: `primaryImageUrl` không phải 1-1 field, phải lọc + tìm + map URL

---

## 1. Khác biệt vs Category module

| Khía cạnh | Category | Product |
|---|---|---|
| Quan hệ | Self-reference (1 cấp) | `ManyToOne` Category + `ManyToOne` User seller + `OneToMany` Variant + `OneToMany` Image |
| Response | Tree (đệ quy) | Flatten nested (id + name) + embed images |
| Custom logic | Không | `primaryImageUrl` từ lọc list images |
| Số mapper | 1 | **3** mapper riêng |
| Sub-entity API | Không | Variant + Image có service/controller riêng |

→ Product buộc trả lời 4 câu mới không gặp ở Category.

## 2. Q&A trong buổi học (verbatim)

### Câu 1: Scope mapper — 1 hay 3?

**Trả lời người học:** Tôi nghĩ chia thành 3 mapper riêng, tại vì mỗi mapper sẽ làm đúng nhiệm vụ của nó, map từ cái gì về cái gì chứ không phải một file hỗn trộn.

**Đáp đúng — 3 mapper riêng.** Đây là Single Responsibility Principle áp dụng đúng.

#### Cơ chế compose: `uses = {...}`

`ProductMapper` cần map `List<ProductImage> → List<ProductImageResponse>`. Khai báo:
```java
@Mapper(uses = {ProductImageMapper.class})
public interface ProductMapper {
    ProductResponse fromEntity(Product product);
    // MapStruct thấy field images cần map → tự tìm trong ProductImageMapper
    // → có method fromEntity(ProductImage) → gọi để map từng element
}
```

**Pitfall:** Nếu quên `uses` → compile error với policy `ERROR` (mapper không biết map kiểu `ProductImage → ProductImageResponse`). An toàn ở chỗ này.

### Câu 2: MapStruct có null-safe nested không?

**Trả lời người học:** Theo đúng business thì sẽ có seller và category cho một product, vậy việc truy cập nested an toàn. Nhưng nếu seller null thì MapStruct có check null không?

**Đáp đúng — CÓ, MapStruct tự sinh null-check.**

Code MapStruct sinh ra (xem ở `target/generated-sources/annotations/`):
```java
public ProductResponse fromEntity(Product product) {
    if (product == null) return null;

    ProductResponse response = new ProductResponse();

    if (product.getSeller() != null) {            // ← null-check tự sinh
        response.setSellerId(product.getSeller().getId());
        response.setSellerName(product.getSeller().getFullName());
    }
    if (product.getCategory() != null) {           // ← null-check tự sinh
        response.setCategoryId(product.getCategory().getId());
        response.setCategoryName(product.getCategory().getName());
    }
    // ...
}
```

→ **Không bao giờ NPE** vì nested null. Mặc định `nullValueCheckStrategy = ALWAYS` cho nested.

**Đối chiếu Node.js:** Tương đương `product?.seller?.id`. MapStruct làm việc này lúc compile thay vì runtime → nhanh hơn, type-safe hơn.

**Insight thực tế:** Entity DB đang `nullable = false` cho `seller` và `category` → null không xảy ra trừ khi corrupt data → null-check của MapStruct là **defensive programming free**: rẻ + cứu mạng.

### Câu 3: Logic `primaryImageUrl` — để mapper hay service?

**Trả lời người học:** Đúng ra để service làm nhưng để trong mapper cũng hợp lý, vì nhiều chỗ gọi đến mapper sẽ tránh được duplicate code.

**Đáp đúng — để mapper.** Đây là **presentation logic** (cách hiển thị dữ liệu sẵn có), không phải business rule.

#### Khi nào để mapper, khi nào để service?

| Tiêu chí | Để mapper | Để service |
|---|---|---|
| Là **presentation logic** (cách *hiển thị* dữ liệu sẵn có) | ✅ | |
| Là **business rule** (có điều kiện nghiệp vụ, role-based) | | ✅ |
| Cần truy cập **repository / external service** | | ✅ |
| Pure function của data input | ✅ | |
| Có thể có nhiều variant (premium/normal user thấy khác nhau) | | ✅ |

`primaryImageUrl` thoả mãn:
- ✅ Pure function (chỉ phụ thuộc list images)
- ✅ Presentation
- ❌ Không cần repo
- ❌ Không có business rule

→ Để mapper là đúng.

#### Cách viết — `default method` + `@Named` + `qualifiedByName`

```java
@Mapper(uses = ProductImageMapper.class)
public interface ProductMapper {

    @Mapping(source = "images", target = "primaryImageUrl", qualifiedByName = "extractPrimaryUrl")
    ProductResponse fromEntity(Product product);

    @Named("extractPrimaryUrl")
    default String extractPrimaryUrl(List<ProductImage> images) {
        if (images == null) return null;
        return images.stream()
            .filter(img -> Boolean.TRUE.equals(img.getIsPrimary()))
            .findFirst()
            .map(ProductImage::getUrl)
            .orElse(null);
    }
}
```

**Cơ chế:**
- `@Named("extractPrimaryUrl")` đánh dấu method có thể được gọi từ `@Mapping`
- `qualifiedByName = "extractPrimaryUrl"` ở `@Mapping` chỉ định method nào dùng cho field `primaryImageUrl`
- `default method` cho phép viết logic trong interface (Java 8+)

#### ⚠️ Cảnh báo senior — YAGNI risk

Nếu mai mốt logic primary URL có business rule kiểu:
- "Seller gói premium được dùng ảnh primary; gói free thì luôn lấy ảnh đầu tiên"
- "User chưa verify email không thấy primary image"

→ Khi đó **phải chuyển sang service** vì cần check role/state. Nhớ điểm này khi feature evolves.

### Câu 4: Variants — embed trong ProductResponse hay tách API riêng?

**Trả lời người học:** Tôi nghĩ để riêng, tại vì khi nào user thật sự cần vào xem chi tiết sản phẩm thì mới lấy variant về, không thì tránh trả về gây thừa data và phải xử lý nhiều.

**Đáp đúng — tách API riêng.**

#### Quy tắc senior: khi nào embed, khi nào tách

| Child resource | Quyết định | Lý do |
|---|---|---|
| `images` | **Embed** | Nhỏ (5-10 ảnh max), UI luôn cần (thumbnail, gallery) |
| `variants` | **API riêng** | Có thể nhiều (50 màu × 5 size = 250 record), UI chỉ cần khi vào trang chi tiết |

> **Quy tắc:** *Embed khi child resource (1) nhỏ và (2) thường xuyên cần cùng parent. Tách API khi child resource lớn hoặc tuỳ chọn.*

#### Optimize tương lai — list vs detail DTO

API `GET /products` (list) có thể chỉ trả `primaryImageUrl`, KHÔNG trả `images` đầy đủ. API `GET /products/{id}` (detail) trả cả `images`.

→ Hai endpoint dùng **2 response DTO khác nhau** (`ProductListItemResponse` vs `ProductDetailResponse`). Đây là pattern thực tế tránh over-fetching ở list view. **Defer sang cleanup session** vì vượt scope.

### `ProductVariantMapper.java`

```java
@Mapper
public interface ProductVariantMapper {
    ProductVariantResponse fromEntity(ProductVariant productVariant);

    @Mapping(target = "id", ignore = true)
    @Mapping(target = "product", ignore = true)
    ProductVariant fromRequestDto(CreateProductVariantRequest request);

    @Mapping(target = "id", ignore = true)
    @Mapping(target = "product", ignore = true)
    @Mapping(target = "sku", ignore = true)
    @Mapping(target = "size", ignore = true)
    @Mapping(target = "color", ignore = true)
    @BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE)

}
```

**Lưu ý ignore `sku`, `size`, `color` ở update:**
- `sku` unique toàn DB → service tự check duplicate trước khi set
- `size + color` là **composite unique** `(product_id, size, color)` → service phải combine, validate, mới set

### `ProductImageMapper.java`

```java
@Mapper
public interface ProductImageMapper {
    ProductImageResponse fromEntity(ProductImage image);
}
```

**Chỉ 1 chiều entity → response.** Lý do:
- Image upload có logic phức tạp (file validation, storage upload, count limit) — không qua mapper
- Update chỉ là `setPrimary` (toggle boolean) — service tự xử lý

## 4. Bug & bẫy phát hiện trong buổi review

### Bug 1: 🔴 `source = "category.fullName"` — field không tồn tại

```java
@Mapping(target = "categoryName", source = "category.fullName")  // ← SAI
```

`Category` entity chỉ có `name`, không có `fullName`. **Fix:** `source = "category.name"`.

**Học được:** MapStruct check source path lúc compile, KHÔNG bị silent. Nếu compile pass mà field sai → có thể đang dùng Maven cache cũ. Luôn `./mvnw clean compile` sau khi sửa mapper.

### Bug 2: 🔴 `source = "seller.name"` — field không tồn tại

`User` entity có `fullName`. **Fix:** `source = "seller.fullName"`.

### Bug 3: 🔴 `@Mapping` duplicate cho `primaryImageUrl`

```java
@Mapping(target = "primaryImageUrl", ignore = true)                                              // (1) còn sót từ skeleton
@Mapping(source = "images", target = "primaryImageUrl", qualifiedByName = "extractPrimaryUrl")   // (2) thực sự dùng
ProductResponse fromEntity(Product product);
```

2 dòng cùng target → `Property 'primaryImageUrl' is mapped more than once`. **Fix:** xoá dòng (1).

**Học được:** khi chuyển field từ `ignore` sang `qualifiedByName`, phải XOÁ ignore cũ, đừng để cả 2.

### Bẫy 4: Naming method update lệch convention

Lần đầu user đặt `updateProduct` / `updateProductVariant`. User module + Category module đều dùng `updateEntity`. **Fix:** đổi cả 2 mapper sang `updateEntity`. Chữ ký method đã có đầy đủ context type, không cần lặp.

### Bẫy 5: Naming repo lệch convention

```java
private final ProductVariantRepository variantRepository;     // ← rút gọn
private final ProductVariantMapper productVariantMapper;      // ← đầy đủ
```

**Fix:** Rút gọn cả 2 (`variantRepository` + `variantMapper`) hoặc đầy đủ cả 2. Tránh mix.

### Bẫy 6: `deleteProduct` dùng `findById` rồi không dùng entity

```java
// Hiện tại — dư
Product product = productRepository.findById(id).orElseThrow(...);
productRepository.deleteById(id);

// Fix — gọn hơn 1 query
if (!productRepository.existsById(id)) throw new ResourceNotFoundException(...);
productRepository.deleteById(id);
```

`existsById` chỉ query count (rẻ), `findById` load full row (đắt). Chỉ dùng `findById` khi cần object để check business rule.

### Bẫy 7: NPE tiềm ẩn `getCategory()` trong update

```java
if (...&& !product.getCategory().getId().equals(categoryId))  // NPE nếu category null
```

DB constraint `nullable=false` → không xảy ra. Nhưng defensive:
```java
if (CheckData.checkIsNotNull(categoryId)
    && (product.getCategory() == null || !product.getCategory().getId().equals(categoryId))) {
    ...
}
```

> **Senior comment:** "Code thừa check tốt hơn code thiếu check khi rẻ và an toàn" — user áp dụng đúng tinh thần.

### Bẫy 8: Default `private Integer stock = 0;` trong response DTO

Response là output, default sẽ bị mapper overwrite → dư. **Fix:** bỏ `= 0`.

**Ngoại lệ:** `private List<ProductImageResponse> images = Collections.emptyList();` thì NÊN GIỮ — nếu source null/empty, default giúp client không phải null-check.

## 5. Trao đổi pragmatic vs purist — flow `updateProduct`

User dùng cùng pattern Category — mapper map field an toàn, validate-then-set rời cho field cần validate. Senior **đồng ý** vì:
- Mapper đã ignore `slug`, `sku`, `category` → 3 field cần validate vẫn theo validate-then-set
- Field bị mutate sớm (name, description, price, stock) — nếu throw → transaction rollback giữ nguyên DB

**Caveat (chưa có `@Transactional`):** đã ghi cleanup backlog.

## 6. Pattern hay đáng khen

### `ProductImageService.deleteImage` — comment WHY chuẩn senior

```java
@Transactional
public void deleteImage(Long imageId) {
    ...
    imageRepository.delete(image);
    // Note: storage.delete() throws RuntimeException nếu fail → @Transactional rollback DB.
    // An toàn cho local filesystem (atomic). Khi migrate sang S3, cần đổi sang
    // TransactionSynchronization.afterCommit() để tránh broken-link khi network fail.
    storageService.delete(key);
}
```

→ Giải thích trade-off transactional, nêu rõ giả định (local FS), warn future migration (S3). Đây chính là kiểu comment giá trị mà CLAUDE.md mong muốn.

### `ProductImageService.setPrimary` — idempotent + dirty checking

```java
@Transactional
public ProductImageResponse setPrimary(Long imageId) {
    ProductImage image = ...;
    Optional<ProductImage> primaryImage = imageRepository.findByProductIdAndIsPrimaryTrue(...);
    // Đã là primary rồi → no-op
    if (primaryImage.isPresent() && primaryImage.get().getId().equals(image.getId()))
        return imageMapper.fromEntity(image);
    image.setIsPrimary(true);
    primaryImage.ifPresent(old -> old.setIsPrimary(false));
    return imageMapper.fromEntity(image);
}
```

- Idempotent check tốt — gọi 2 lần không gây side effect
- Dùng `@Transactional` + JPA dirty checking → không cần `save()` thủ công
- `Optional.ifPresent` thay vì `if (... != null)` — idiomatic Java 8

### `ProductVariantService.updateVariant` — xử lý composite unique chuẩn

```java
boolean sizeChanged = CheckData.checkIsNotNull(newSize) && !newSize.equals(variant.getSize());
boolean colorChanged = CheckData.checkIsNotNull(newColor) && !newColor.equals(variant.getColor());
if (sizeChanged || colorChanged) {
    String size = sizeChanged ? newSize : variant.getSize();
    String color = colorChanged ? newColor : variant.getColor();
    if (variantRepository.existsByProductIdAndSizeAndColor(variant.getProduct().getId(), size, color)) {
        throw new DuplicateResource("Variant already exists");
    }
    variant.setSize(size);
    variant.setColor(color);
}
```

Logic composite unique `(product_id, size, color)` là **dễ sai nhất** mà user xử lý đúng: tách `sizeChanged`/`colorChanged`, recombine từ current + new, mới check duplicate.

## 7. Production-grade issues defer sang cleanup session

Thêm vào memory `project-production-cleanup-session`:

**Phát hiện ở Product module (2026-05-21):**
- `ProductService.createProduct` dùng `SecurityContextHolder.getContext().getAuthentication().getName()` + `findByEmail` → cleaner pattern là `@AuthenticationPrincipal CustomUserDetails`
