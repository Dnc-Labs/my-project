### Ownership authorization (cốt lõi của e-commerce multi-seller)

Trong ROADMAP có 3 role: `ADMIN`, `SELLER`, `CUSTOMER`. Hình dung tình huống:

```
Seller A tạo product "Nike Air Force"
Seller B tạo product "Adidas Stan Smith"

Seller A có quyền SỬA/XOÁ "Adidas Stan Smith" không?
→ KHÔNG! Chỉ Seller B được sửa/xoá product của mình.
```

**Nếu không lưu `seller_id`**, khi có request `PUT /api/products/123` từ Seller A:
- Hệ thống thấy Seller A có role `SELLER` → cho phép sửa
- **Đây là lỗ hổng bảo mật**: Seller A có thể sửa product của bất kỳ seller khác!

→ Bắt buộc phải lưu **ai là chủ của product đó**.

### Quan hệ đúng

```java
@ManyToOne
@JoinColumn(name = "seller_id", nullable = false)
private User seller;
```

**@ManyToOne đúng** — 1 user (seller) có thể có nhiều product, 1 product thuộc 1 seller.

**Câu hỏi 3: API nào cần cho Product?**

> Các api cơ bản: CRUD. Các api nâng cao: lấy products theo đa điều kiện: search, category, pagination. Các api public: đọc product. Về phân quyền: đọc product → mọi user, tạo, sửa, xoá thì cần quyền SELLER (nếu có triển khai thêm seller) còn không chỉ cần quyền ADMIN.

**Review:** Đúng idea, nhưng cần phân biệt:

| API | Quyền cần có |
|-----|-------------|
| **GET** list/detail | Public (ai cũng xem được) |
| **POST** create | `SELLER` hoặc `ADMIN` |
| **PUT** update | `ADMIN` **HOẶC** (`SELLER` + **là chủ của product**) |
| **DELETE** | `ADMIN` **HOẶC** (`SELLER` + **là chủ của product**) |

Điểm quan trọng: **UPDATE/DELETE có thêm ownership check**, không chỉ check role.

Search/filter/pagination là **3.3**, không phải buổi này.

**Câu hỏi 4: Giá tiền — nên dùng kiểu dữ liệu gì?**

> Giá trị tiền thì sẽ dùng BigDecimal vì tiền có thể sẽ lẻ như 0.3$, nếu thực hiện tính toán tiền trên các số lẻ này có thể bị lỗi.

**Review:** ✅ Đúng 100%.

### Deep dive — tại sao không dùng `double`/`float`?

Máy tính biểu diễn số bằng **binary (cơ số 2)**. Nhưng `0.1` ở hệ 10 khi chuyển sang hệ 2 → **số thập phân vô hạn tuần hoàn** (giống `1/3 = 0.333...` ở hệ 10). CPU **cắt bớt** → mất độ chính xác.

Demo bug kinh điển:
```java
double a = 0.1;
double b = 0.2;
System.out.println(a + b);   // 0.30000000000000004  ❌
```

### Tại sao tiền **không được** phép sai dù 0.0000001

```
Mỗi giao dịch sai 0.00001$ × 1 triệu giao dịch = 10$
× 1 năm × 100 công ty fintech = bankrupt
```

→ Industry standard: **dùng `BigDecimal`** (Java) / `Decimal` (C#) / `decimal.js` (Node.js) cho tiền.

### So sánh Node.js

Bên Node bị cùng vấn đề:
```javascript
0.1 + 0.2 === 0.3   // false
```

Giải pháp bên Node:
- Library `decimal.js`, `big.js`, `bignumber.js`
- Hoặc lưu **cents** (integer): `9900` cent thay vì `99.00$`

Bên Java có **built-in** `BigDecimal`.

### Gotcha khi dùng `BigDecimal`

**KHÔNG** dùng `.equals()` so sánh 2 `BigDecimal`:

```java
BigDecimal a = new BigDecimal("10.00");
BigDecimal b = new BigDecimal("10.0");

a.equals(b)      // false! (khác scale: 2 vs 1 chữ số thập phân)
a.compareTo(b)   // 0 — bằng nhau về giá trị ✅
```

**Luôn dùng `compareTo()` khi so sánh giá trị**.

## Phần 2: Implementation

### Entity

```java
@Entity
@Table(name = "products")
public class Product {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false) private String name;
    @Column(unique = true, nullable = false) private String slug;
    @Column(unique = true, nullable = false) private String sku;
    @Column(nullable = false, precision = 19, scale = 2) private BigDecimal price;
    @Column(columnDefinition = "TEXT") private String description;
    @Column(nullable = false) private Integer stock = 0;

    @Enumerated(EnumType.STRING)
    private ProductStatus status = ProductStatus.ACTIVE;

    private String imageUrl;
    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;

    @ManyToOne
    @JoinColumn(name = "category_id", nullable = false)
    private Category category;

    @ManyToOne
    @JoinColumn(name = "seller_id", nullable = false)
    private User seller;

    @OneToMany(mappedBy = "product", cascade = CascadeType.ALL)
    private List<ProductVariant> productVariants;
}
```

**Điểm đáng chú ý**:
- `@Column(precision = 19, scale = 2)` — khai báo column cho `BigDecimal` trong DB (tối đa 19 chữ số, 2 chữ số thập phân). Đủ để lưu tiền đến ~$170 tỷ.
- `stock` mặc định = 0
- `status` mặc định = ACTIVE

### ProductSecurity — bean cho ownership check

Đặt trong package `security` (không phải `services`) vì **đây là logic BẢO MẬT**, không phải business logic.

```java
@Component("productSecurity")
public class ProductSecurity {

    private final ProductRepository productRepository;
    private final UserRepository userRepository;

    public ProductSecurity(ProductRepository productRepository, UserRepository userRepository) {
        this.productRepository = productRepository;
        this.userRepository = userRepository;
    }

    public boolean isOwner(Long productId, Authentication authentication) {
        if (authentication == null || !authentication.isAuthenticated()) return false;
        String email = authentication.getName();
        Optional<User> seller = userRepository.findByEmail(email);
        Optional<Product> product = productRepository.findById(productId);
        if (seller.isEmpty() || product.isEmpty()) return false;
        return product.get().getSeller().getId().equals(seller.get().getId());
    }
}
```

**Vì sao dùng `@Component("productSecurity")` (có tên)?**
- SpEL gọi bằng `@beanName` → cần đặt tên cho bean
- Nếu không đặt tên, Spring mặc định lấy tên class viết thường ký tự đầu — vẫn gọi được nhưng khai báo rõ tên thì an toàn khi đổi tên class

### Controller với `@PreAuthorize` ownership

```java
@RestController
@RequestMapping("/api/products")
public class ProductController {

    @PreAuthorize("hasAnyRole('SELLER', 'ADMIN')")
    @PostMapping
    public ResponseEntity<BaseResponse<ProductResponse>> create(...)

    @GetMapping
    public ResponseEntity<BaseResponse<List<ProductResponse>>> getAll() { ... }  // public

    @GetMapping("/{id}")
    public ResponseEntity<BaseResponse<ProductResponse>> getById(...)  // public

    @PutMapping("/{id}")
    @PreAuthorize("hasRole('ADMIN') or (hasRole('SELLER') and @productSecurity.isOwner(#id, authentication))")
    public ... update(...)

    @DeleteMapping("/{id}")
    @PreAuthorize("hasRole('ADMIN') or (hasRole('SELLER') and @productSecurity.isOwner(#id, authentication))")
    public ... delete(...)
}
```

### Service — seller lấy từ SecurityContext, KHÔNG từ request

```java
public ProductResponse createProduct(CreateProductRequest request) {
    Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
    String email = authentication.getName();
    User user = userRepository.findByEmail(email)
        .orElseThrow(() -> new ResourceNotFoundException("User not found"));

    Product product = new Product();
    product.setSeller(user);   // ← lấy từ context, không tin request
    // ... set các field khác
    return ProductResponse.fromEntity(productRepository.save(product));
}
```

**Lý do**: Nếu cho client truyền `sellerId` trong request → Seller A có thể gửi `sellerId = B` → gian lận. Luôn lấy seller **từ server-side context**.

### SecurityConfig — whitelist GET public

```java
.requestMatchers(HttpMethod.GET, "/api/products/**").permitAll()
```

Dùng `HttpMethod.GET` để **chỉ GET** public, không phải toàn bộ `/api/products/**`. POST/PUT/DELETE vẫn yêu cầu auth.

## Phần 4: Tích hợp Swagger UI

### Swagger vs OpenAPI — khái niệm

| Khái niệm | Ý nghĩa |
|-----------|---------|
| **OpenAPI Specification (OAS)** | **Chuẩn/spec** mô tả REST API (YAML/JSON). Ban đầu tên là "Swagger Spec" |
| **Swagger** | **Bộ công cụ** (Swagger UI, Swagger Editor...) đọc spec OpenAPI |
| **Swagger UI** | Một **HTML page** render spec thành UI tương tác |

OpenAPI là **ngôn ngữ**, Swagger là **trình đọc và render**.

### So sánh Node.js

| Node.js | Spring Boot |
|---------|-------------|
| `swagger-jsdoc` + `swagger-ui-express` — phải viết JSDoc thủ công | Framework tự scan annotation để sinh spec |
| Phải maintain spec song song với code | Spec **tự động** sinh từ code — luôn đồng bộ |

### Thư viện — `springdoc-openapi`

Trong hệ sinh thái Spring có 2 thư viện:

| Thư viện | Tình trạng |
|----------|-----------|
| `springfox` | **Đã chết** — không support Spring Boot 3+ |
| **`springdoc-openapi`** ✅ | Đang maintain, support Spring Boot 3+ |

### Cơ chế hoạt động

Khi start app, springdoc làm 3 việc:

```
┌─────────────────────────────────────────────────────────┐

                           ▼

         │    Tại: /swagger-ui.html            │

```

### 4 bước tích hợp

**Bước 1: Thêm dependency** (`api/pom.xml`)

```xml
<dependency>
    <groupId>org.springdoc</groupId>
    <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
    <version>2.8.0</version>
</dependency>
```

> ⚠️ Version phải tương thích với Spring Boot version. 2.6.0 với Spring Boot 3.5.x gây "Internal Server Error" ở `/v3/api-docs`. Dùng 2.8.0.

**Bước 2: Whitelist Swagger paths trong SecurityConfig**

```java
.requestMatchers("/swagger-ui/**", "/swagger-ui.html", "/v3/api-docs/**").permitAll()
```

**Bước 3: Config thông tin API + JWT authorization**

```java
@Configuration
public class OpenApiConfig {
    private static final String SECURITY_SCHEME_NAME = "bearerAuth";

    @Bean
    public OpenAPI openAPI() {
        return new OpenAPI()
                .info(new Info()
                        .title("E-commerce API")
                        .version("1.0")
                        .description("REST API cho hệ thống e-commerce")
                        .contact(new Contact().name("CuongDn").email("...")))
                .addSecurityItem(new SecurityRequirement().addList(SECURITY_SCHEME_NAME))
                .components(new Components()
                        .addSecuritySchemes(SECURITY_SCHEME_NAME, new SecurityScheme()
                                .name(SECURITY_SCHEME_NAME)
                                .type(SecurityScheme.Type.HTTP)
                                .scheme("bearer")
                                .bearerFormat("JWT")));
    }
}
```

### 3 khái niệm quan trọng

**`Info`** — metadata hiển thị đầu trang (title, version, description). Không ảnh hưởng logic.

**`SecurityScheme`** — định nghĩa **kiểu** xác thực. Khi khai báo → Swagger UI hiện nút **🔒 Authorize**.

```java
new SecurityScheme()
    .type(SecurityScheme.Type.HTTP)      // loại HTTP auth
    .scheme("bearer")                    // Bearer
    .bearerFormat("JWT");                // format JWT (hiển thị)
```

**`SecurityRequirement`** — **yêu cầu** dùng scheme. `SecurityScheme` chỉ định nghĩa, `SecurityRequirement` mới bắt buộc.

**Hệ quả**: Khi nhập token → UI tự thêm header `Authorization: Bearer <token>` vào **mọi request**.

## Phần 5: GlobalExceptionHandler — các cải thiện

### Issue 1: Handler generic `Exception.class` leak info

```java
@ExceptionHandler(Exception.class)
public ResponseEntity<BaseResponse<Void>> handleException(Exception e) {
    return ResponseEntity.status(INTERNAL_SERVER_ERROR)
        .body(BaseResponse.error(e.getMessage()));   // ⚠️
}
```

**Vấn đề:**
- **Leak info nhạy cảm**: Exception nội bộ (ví dụ `SQLException: Access denied for user 'root'@'localhost'`) bị trả thẳng cho client
- **`e.getMessage()` có thể null**: Ví dụ `NullPointerException` có message null → client nhận `"message": null`

**Production best practice:**
```java
log.error("Unhandled exception", e);   // log chi tiết server
return BaseResponse.error("Internal server error");   // message chung
```

### Issue 2: Thiếu handler cho `AccessDeniedException`

Khi CUSTOMER gọi API `@PreAuthorize("hasRole('ADMIN')")`, Spring Security throw:
- Spring Security 5: `AccessDeniedException`
- **Spring Security 6**: `AuthorizationDeniedException`

Cả 2 đều **không được handle** → rơi vào `handleException` → trả **500** thay vì **403**.

Fix:
```java
@ExceptionHandler({AccessDeniedException.class, AuthorizationDeniedException.class})
public ResponseEntity<BaseResponse<Void>> handleAccessDenied(Exception e) {
    return ResponseEntity.status(HttpStatus.FORBIDDEN)
        .body(BaseResponse.error("You do not have permission to perform this action"));
}
```

**Giải thích HTTP status:**
- **401 Unauthorized**: Chưa xác thực (thiếu token / token sai)
- **403 Forbidden**: Đã xác thực nhưng không đủ quyền

### Issue 3: Fix email enumeration vulnerability

**Flow cũ:**
```java
User user = userRepository.findByEmail(email)
    .orElseThrow(() -> new ResourceNotFoundException("User not found"));   // 404
if (!password.matches) throw new InvalidUserOrPassword("...");              // 401
```

**Attacker có thể enumerate email:**

| Request | Response cũ | Response mới |
|---------|-------------|--------------|
| Email không có trong DB | **404** User not found | **401** Invalid email or password |
| Email đúng + password sai | **401** Invalid email or password | **401** Invalid email or password |

→ Attacker **không còn thông tin** để đoán email tồn tại.

**Fix:**
```java
User user = userRepository.findByEmail(email)
    .orElseThrow(() -> new InvalidUserOrPassword("Invalid email or password"));
boolean validPassword = passwordEncoder.matches(...);
if (!validPassword) throw new InvalidUserOrPassword("Invalid email or password");
```

### Bonus: timing attack (chưa fix, để sau)

Code hiện tại vẫn bị timing attack:
- Email tồn tại → chạy BCrypt `matches()` (~100ms)
- Email không tồn tại → throw ngay (~1ms)

→ Attacker đo thời gian response → đoán được email nào tồn tại.

Fix hoàn hảo (tương lai):
```java
User user = userRepository.findByEmail(email).orElse(null);
String hashToCheck = user != null ? user.getPassword() : DUMMY_BCRYPT_HASH;
boolean match = passwordEncoder.matches(password, hashToCheck);
if (user == null || !match) throw new InvalidUserOrPassword(...);
```

### Chuẩn hoá message lỗi (VN → EN, consistent)

Ví dụ:
| Trước | Sau |
|-------|-----|
| `"This Slug has existed"` | `"Slug already exists"` |
| `"Parent not exits"` (typo) | `"Parent category not found"` |
| `"Can't set category as a parent"` (duplicate) | `"Cannot set category as its own parent"` + `"Cannot set a descendant as parent (circular reference)"` |
| `"You can't delete this parent category"` | `"Cannot delete category that has children"` |
| `"Bạn không có quyền thực hiện hành động này"` | `"You do not have permission to perform this action"` |

## Phần 6: Deep dive `@PreAuthorize`

### Câu hỏi 1: Thứ tự annotation

> `@PreAuthorize("hasRole('ADMIN') or (hasRole('SELLER') and @productSecurity.isOwner(#id, authentication))")` nên đặt lên trước annotation method mapping hay là đặt dưới, việc sắp xếp này có đại diện cho thứ tự chạy không?

**Trả lời:** KHÔNG ảnh hưởng gì.

Java/JVM coi annotation trên cùng 1 method là **một tập hợp không có thứ tự** (unordered set). Thứ tự vị trí chỉ là **style/convention**, không ảnh hưởng runtime.

```java
// Cách 1 — giống hệt cách 2
@PutMapping("/{id}")
@PreAuthorize("hasRole('ADMIN')")
public ... update(...) { ... }

// Cách 2
@PreAuthorize("hasRole('ADMIN')")
@PutMapping("/{id}")
public ... update(...) { ... }
```

### Thứ tự chạy của aspect

Do **`@Order`** annotation hoặc `Ordered` interface trên aspect class, KHÔNG phải thứ tự annotation.

- `@PreAuthorize` có Order = 500 (runs early)
- `@Transactional` có Order = Integer.MAX_VALUE - 1 (runs late)

→ Security aspect **luôn** chạy trước Transaction aspect, bất kể thứ tự annotation.

### Flow runtime

```
  Request đến controller method
         │
         ▼
  ┌────────────────────────────────────┐
  │     Spring proxy chain             │
  │                                    │
  │  @PreAuthorize aspect  (Order 500) │ ← chạy trước
  │         │                          │
  │         ▼                          │
  │  Các aspect khác                   │
  │         │                          │
  │         ▼                          │
  │  Real controller method            │
  │         │                          │
  │         ▼                          │
  │  Dispatch đến URL (@PutMapping)    │
  └────────────────────────────────────┘
```

**Lưu ý**: `@PutMapping` **không** phải aspect — nó là **mapping metadata** được Spring MVC đọc khi build route table.

### Convention

- **Style Spring chính thức**: HTTP mapping trước, security sau
- **Style "security-first"**: `@PreAuthorize` lên trên

→ Chọn 1 style và dùng nhất quán toàn project.

### Câu hỏi 2: `#id` là gì?

> `@productSecurity.isOwner(#id, authentication))` cụ thể là như nào, tại sao lại là `#id`, nếu như tôi sửa thành `#productId` có được không?

**Trả lời:** `#id` = tham chiếu đến **tham số của method**.

Trong SpEL, `#` là cú pháp **tham chiếu biến**. `#id` nghĩa là: "lấy giá trị của tham số có tên là `id` trong method".

```java
@PutMapping("/{id}")
@PreAuthorize("... @productSecurity.isOwner(#id, authentication)")
//                                         ^^^ refer tới @PathVariable Long id bên dưới
public ResponseEntity<...> update(
    @PathVariable Long id,           // ← #id trỏ về đây
    @Valid @RequestBody UpdateProductRequest request
) { ... }
```

### Đổi thành `#productId` — được, nhưng phải đồng bộ

```java
@PutMapping("/{id}")
@PreAuthorize("... @productSecurity.isOwner(#productId, authentication)")
public ResponseEntity<...> update(
    @PathVariable("id") Long productId,   // ← tên JavaParam = productId
    ...
) { ... }
```

**Cả 2 chỗ phải khớp**: SpEL expression `#productId` ↔ tham số method `Long productId`.

### Pitfall — tên param có được giữ sau compile?

Mặc định, Java compiler **xoá** tên tham số → bytecode chỉ còn `arg0`, `arg1`... SpEL không lấy được tên gốc → throw error.

**Giải pháp:** Cần compile với flag `-parameters`. Spring Boot 3+ đã **bật mặc định** trong Maven plugin.

**Cách an toàn hơn:** Dùng `#p0`, `#p1`... để refer theo **vị trí**:
```java
@PreAuthorize("@productSecurity.isOwner(#p0, authentication)")
```

`#p0` = tham số đầu tiên. An toàn với mọi compile flag, nhưng đọc khó hơn.

### Truy cập field của object tham số

```java
public ... update(@RequestBody UpdateRequest request) { ... }

@PreAuthorize("#request.userId == authentication.principal")
// ↑ truy cập field .userId của object request
```

Mạnh hơn Express rất nhiều.

### Câu hỏi 3: `authentication` là gì?

> `authentication` là gì?

**Trả lời:** Là **biến built-in** của SpEL trong Spring Security.

Không phải tham số method, không phải bean, mà là **biến ngầm** do Spring Security inject vào SpEL context khi evaluate `@PreAuthorize`.

`authentication` trỏ về object `Authentication` trong `SecurityContextHolder`:

```java
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
```

### `Authentication` object có gì?

```
┌─────────────────────────────────────────────┐
│  Authentication                             │
│  ├── principal       (ai đang login)         │
│  │   → trong project này = email (String)   │
│  ├── credentials     (password / null)      │
│  ├── authorities     (danh sách role/perm)  │
│  ├── isAuthenticated (đã login chưa)        │
│  └── getName()       (trả principal String) │
└─────────────────────────────────────────────┘
```

### Các helper method như `hasRole()` bản chất là shortcut

```java
hasRole('ADMIN')
// ≡
authentication.getAuthorities().stream()
    .anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"));
```

Đều dựa trên `authentication`.

### Truyền `authentication` vào bean — DI tinh tế

```java
@productSecurity.isOwner(#id, authentication)
//                             │
//                             └── Authentication object — truyền vào
```

Bạn **truyền `authentication` vào bean** thay vì để bean tự lấy từ `SecurityContextHolder`. Lý do:

#### Cách 1 (đang dùng) — truyền authentication
```java
public boolean isOwner(Long productId, Authentication authentication) {
    String email = authentication.getName();
    ...
}
```
✅ Test dễ: mock Authentication, pass vào
✅ Explicit: signature nói rõ cần gì

#### Cách 2 — tự lấy từ SecurityContextHolder
```java
public boolean isOwner(Long productId) {
    Authentication auth = SecurityContextHolder.getContext().getAuthentication();
    ...
}
```
❌ Test khó: phải mock `SecurityContextHolder` (static call)
❌ Implicit: signature không nói gì về security

→ Cách 1 là **best practice**.

## Phần 7: Deep dive Unidirectional vs Bidirectional

### Câu hỏi

> Chúng ta có cần thêm quan hệ vào entity USER không nhỉ, nếu như không thêm vào có khác gì thêm vào không, tại tôi thấy các quan hệ trước có thêm ngược lại vào user?

### Unidirectional (hiện tại — chỉ Product → User)

```java
// Product entity
@ManyToOne
@JoinColumn(name = "seller_id")
private User seller;

// User entity — KHÔNG có field ngược lại
```

Quan hệ 1 chiều.

### Bidirectional (nếu thêm vào User)

```java
// User entity
@OneToMany(mappedBy = "seller")
private List<Product> products;
```

### DB giống hệt nhau

```
products table: seller_id (FK)
users table:    KHÔNG có cột tham chiếu ngược
```

**Foreign key luôn nằm ở phía `@ManyToOne` (Product)**. Thêm hay không `@OneToMany(mappedBy)` bên User **không tạo thêm cột nào** trong DB.

### Quy tắc **Owning side vs Inverse side**

- **Owning side**: Có `@JoinColumn` → **control** FK trong DB
- **Inverse side**: Có `mappedBy` → chỉ là "view ngược lại" trong code, không ảnh hưởng DB

```
      Product (owning)                User (inverse)
   ┌──────────────────┐           ┌──────────────────┐
   │ @ManyToOne       │           │ @OneToMany       │
   │ @JoinColumn      │ ◀────────▶│ mappedBy="seller"│
   │ seller_id 🔑     │           │  (chỉ code view) │
   └──────────────────┘           └──────────────────┘
             │
             ▼
        ┌────────┐
        │   DB   │  ← chỉ 1 FK
        └────────┘
```

### Khác nhau về CODE

**Unidirectional:**
```java
// ✅ Có thể: product → seller
product.getSeller();

// ❌ Không thể:
user.getProducts();   // compile error
```

**Bidirectional:**
```java
user.getProducts();   // ✅ access trực tiếp
```

Muốn lấy products của user (nếu unidirectional) → phải gọi repository:
```java
productRepo.findBySellerId(userId);
```

### Trade-off

| Tiêu chí | Unidirectional | Bidirectional |
|----------|----------------|---------------|
| Truy cập code | Chỉ 1 chiều | Cả 2 chiều |
| Hiệu năng (load user) | ✅ Nhẹ | ⚠️ Có thể nặng nếu EAGER |
| Risk JSON infinite loop | ✅ Không | ⚠️ Cần fix bằng `@JsonIgnore` hoặc DTO |
| `LazyInitializationException` | ✅ Ít | ⚠️ Nhiều hơn |
| Maintain consistency | ✅ Tự động | ⚠️ Cần helper method giữ sync |
| Logic code | Đơn giản | Phức tạp hơn |

### Risk JSON infinite loop

```
User → products → Product → seller (User) → products → Product → ... ♾️
```

→ Backend crash hoặc trả response khổng lồ. Đây là lý do **luôn dùng DTO** khi trả response, không trả entity trực tiếp.

### Vì sao các relationship TRƯỚC ĐÂY là bidirectional?

- `User.orderList` ↔ `Order.user`: cần use case "lấy lịch sử mua hàng" → bidirectional hợp lý
- `User.cart` ↔ `Cart.user`: 1-1 nhẹ, pattern tự nhiên
- `Category.children` ↔ `Category.parent`: bắt buộc để build tree

### Khuyến nghị cho Product-User

**Giữ Unidirectional** — 3 lý do:

1. **User có thể có rất nhiều product** (seller Amazon có hàng triệu) → bidirectional + vô ý gọi `user.getProducts()` → OOM
2. **Không có use case "lấy products từ User object"** — khi cần list, gọi `GET /api/products?sellerId=5` có pagination
3. **Tránh vấn đề JSON/Lazy** — với bidirectional, mỗi lần trả UserResponse phải nhớ đừng include products

### Rule of thumb senior

```
╔════════════════════════════════════════════════════════════╗
║                                                            ║
║  Default: UNIDIRECTIONAL                                   ║
║                                                            ║
║  Chuyển sang BIDIRECTIONAL khi:                            ║
║  1. Có use case cụ thể cần access ngược lại                ║
║  2. Collection ngược lại NHỎ (dưới vài trăm items)         ║
║  3. Chấp nhận chi phí handle JSON & Lazy                   ║
║                                                            ║
║  ❌ Đừng bidirectional chỉ vì "cho đầy đủ"                 ║
║                                                            ║
╚════════════════════════════════════════════════════════════╝
```

## Phần 8: Giảm boilerplate DTO (Lombok / Records / MapStruct)

### Câu hỏi

> Tôi thấy mỗi khi tạo một class DTO, chúng ta đều phải triển khai các method getter/setter, và các hàm convert, có cách nào để làm ngắn gọn đoạn này không?

### Giải pháp 1 — Lombok (phổ biến nhất)

Thư viện đã tồn tại >10 năm, **90% Java project thực tế dùng**.

Dùng **Annotation Processing** — lúc compile, tự động sinh getter/setter/constructor/toString.

**Trước (80 dòng):**
```java
public class CreateProductRequest {
    @NotBlank private String name;
    // ... 6 field
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    // ... ×12 method
}
```

**Sau (10 dòng):**
```java
@Data
public class CreateProductRequest {
    @NotBlank private String name;
    // ... 6 field
}
```

### Các annotation Lombok phổ biến

| Annotation | Sinh ra |
|-----------|---------|
| `@Getter` | getter cho tất cả field |
| `@Setter` | setter |
| `@Data` | `@Getter + @Setter + @ToString + @EqualsAndHashCode + @RequiredArgsConstructor` |
| `@NoArgsConstructor` | constructor rỗng |
| `@AllArgsConstructor` | constructor nhận tất cả field |
| `@Builder` | sinh builder pattern |

### Giải pháp 2 — Java Records (built-in Java 14+)

Java 21 có sẵn. `record` là keyword khai báo class **bất biến** với auto-generate.

```java
public record CategoryResponse(
    Long id, String name, String slug,
    /* ... */
    List<CategoryResponse> children
) {
    public static CategoryResponse fromEntity(Category c) { ... }
}
```

**Lưu ý**: Getter gọi là `name()` chứ không phải `getName()`. Khác convention JavaBean.

### So sánh Record vs Lombok

| | Record | Lombok |
|---|--------|--------|
| Cần dependency | ❌ Built-in | ✅ Có |
| Immutable | ✅ Bắt buộc | ❌ (có setter) |
| JPA Entity | ⚠️ Không dùng được | ✅ |
| Response DTO | ✅ **Rất phù hợp** | ✅ |
| Request DTO | ⚠️ Spring 3+ support, validation hơi khác | ✅ |

### Giải pháp 3 — MapStruct (cho convert Entity ↔ DTO)

Lombok/Record giúp field, MapStruct giúp **mapping**:

```java
@Mapper(componentModel = "spring")
public interface ProductMapper {
    @Mapping(source = "seller.id", target = "sellerId")
    @Mapping(source = "seller.fullName", target = "sellerName")
    ProductResponse toResponse(Product product);

    List<ProductResponse> toResponseList(List<Product> products);
}
```

Tự sinh code mapping, hiệu năng như code tay.

### Khuyến nghị

**Option A (đã chọn)**: Giữ viết tay → xong bài 3.2 Buổi B → **buổi refactor 3.2.5** với Lombok + MapStruct. Lý do:
1. Hiểu sâu boilerplate của Java trước, biết nó làm gì
2. Tự viết vài lần để nắm JavaBean convention
3. Có giá trị so sánh trước/sau rõ rệt

Buổi refactor sẽ dạy thêm: **Annotation Processing** — cơ chế chung của Lombok, MapStruct, thậm chí JPA `@Entity`.
