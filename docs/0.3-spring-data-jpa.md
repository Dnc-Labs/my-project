## 1. ORM là gì?

ORM (Object-Relational Mapping) cho phép thao tác với database thông qua các object trong code, thay vì viết SQL thuần.

Bên Java, hệ sinh thái ORM có 3 tầng:

| Khái niệm | Là gì? | So sánh Node.js |
|---|---|---|
| **JPA** | Một **specification** (bộ quy chuẩn/interface) — chỉ định nghĩa "ORM trong Java nên hoạt động thế nào", **không có code thực thi** | Giống như một bản "tiêu chuẩn" mà ai cũng phải tuân theo |
| **Hibernate** | Một **implementation** của JPA — code thực sự chạy bên dưới | Giống Sequelize/Prisma — thư viện thực thi |
| **Spring Data JPA** | Một **lớp abstraction** của Spring, bọc lên trên JPA/Hibernate, giúp viết code cực kỳ ngắn gọn | Không có tương đương trực tiếp — giống kiểu "auto-generate" repository |

Mối quan hệ:

```
Spring Data JPA  →  sử dụng  →  JPA (spec)  →  được implement bởi  →  Hibernate
```

### Tại sao Java tách specification và implementation?

Đây là tư duy **"program to interface, not implementation"**:

- Hôm nay dùng **Hibernate**, mai muốn đổi sang **EclipseLink** (implementation khác của JPA) → chỉ đổi dependency, **không sửa code business**
- So sánh: bên Prisma nếu ngừng phát triển → phải **rewrite toàn bộ** query. Với JPA chỉ cần swap implementation
- Thực tế 99% dự án Java đều dùng Hibernate, nhưng tư duy tách layer này còn **giúp testing, mocking dễ hơn**

## 2. Kết nối Database

### Dependencies cần thiết (pom.xml)

```xml
<!-- Spring Data JPA + Hibernate + config cần thiết -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>

<!-- JDBC driver để Java nói chuyện với MySQL qua TCP -->
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <scope>runtime</scope>
</dependency>
```

Hai dependency này đảm nhiệm hai việc khác nhau:

| Dependency | Vai trò | So sánh Prisma |
|---|---|---|
| `spring-boot-starter-data-jpa` | Tầng ORM: mapping object ↔ table, generate query | Giống `@prisma/client` |
| `mysql-connector-j` | Driver kết nối MySQL | Prisma tự bundle sẵn, Java phải khai báo riêng |

Luồng hoạt động:

```
Code của bạn  →  Spring Data JPA  →  Hibernate  →  JDBC  →  mysql-connector-j  →  MySQL Server
```

> Nếu đổi sang PostgreSQL: chỉ đổi dependency `mysql-connector-j` → `postgresql` và đổi URL. Code business **không đổi gì**.

### Cấu hình application.yaml

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/ecommerce
    username: root
    password: admin
  jpa:
    hibernate:
      ddl-auto: update
    show-sql: true    # In SQL ra console để xem Hibernate generate query gì
```

### Docker Compose cho MySQL

```yaml
services:
  mysql:
    image: mysql:8.0
    container_name: ecommerce-mysql
    ports:
      - "3306:3306"
    environment:
      MYSQL_ROOT_PASSWORD: admin
      MYSQL_DATABASE: ecommerce
    volumes:
      - mysql_data:/var/lib/mysql

volumes:
  mysql_data:
```

- `MYSQL_DATABASE: ecommerce` → Docker tự tạo database khi khởi động lần đầu
- `volumes: mysql_data` → data lưu lại, không mất khi restart container

## 3. Hibernate DDL Auto Strategy

`ddl-auto` quyết định Hibernate **làm gì với database schema khi app khởi động**:

| Giá trị | Hành vi | Khi nào dùng |
|---|---|---|
| `create` | **Xóa sạch** tất cả table rồi tạo lại từ Entity | Prototype, học tập |
| `create-drop` | Giống `create`, nhưng **xóa table luôn khi app tắt** | Chạy test |
| `update` | So sánh Entity với table hiện tại, **thêm column/table mới** (không xóa gì) | Dev local |
| `validate` | Chỉ **kiểm tra** schema có khớp Entity không, sai thì báo lỗi | **Production** |
| `none` | Không làm gì cả | Production (khi dùng migration tool) |

### Trong production nên dùng gì?

**`validate`** hoặc **`none`**, kết hợp với **migration tool** như **Flyway** hoặc **Liquibase**.

Lý do để Hibernate tự sửa schema trên prod là nguy hiểm:
- Đổi tên field `userName` → `fullName` trong Entity
- Hibernate `update` sẽ **tạo column mới** `full_name` nhưng **không xóa** `user_name`, **không migrate data**
- → Data cũ mất, schema rác

> Migration tool giống `prisma migrate` — viết từng file SQL rõ ràng, có version, có thể rollback.

## 4. Entity

Entity là class Java đại diện cho một bảng trong database.

```java
@Entity
@Table(name = "product")
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    private BigDecimal price;

    @Column(columnDefinition = "TEXT")
    private String description;

    // getter, setter...
}
```

### Các annotation quan trọng

| Annotation | Ý nghĩa |
|---|---|
| `@Entity` | Đánh dấu class này là một Entity, JPA sẽ tạo table tương ứng |
| `@Table(name = "...")` | Chỉ định tên bảng (nếu không có, dùng tên class) |
| `@Id` | Đánh dấu field là primary key |
| `@GeneratedValue(strategy = GenerationType.IDENTITY)` | Tự tăng id (AUTO_INCREMENT trong MySQL) |
| `@Column(columnDefinition = "TEXT")` | Tùy chỉnh column — mặc định String là `VARCHAR(255)` |

### Lưu ý kiểu dữ liệu

- **`Long`** (wrapper) thay vì **`long`** (primitive) cho id — vì khi chưa lưu vào DB, id chưa có giá trị, `Long` có thể là `null`, `long` mặc định là `0` → dễ nhầm
- **`BigDecimal`** cho giá tiền — tránh lỗi làm tròn số thập phân mà `Double` gặp phải

## 5. Repository

Repository là interface để thao tác với database. Chỉ cần extend `JpaRepository`, không cần viết implementation:

```java
@Repository
public interface ProductRepository extends JpaRepository<Product, Long> {
}
```

### Các method có sẵn từ JpaRepository

| Method | Chức năng | So sánh Prisma |
|---|---|---|
| `save(entity)` | Tạo hoặc cập nhật | `prisma.product.create(...)` |
| `findById(id)` | Tìm theo id, trả `Optional` | `prisma.product.findUnique(...)` |
| `findAll()` | Lấy tất cả | `prisma.product.findMany()` |
| `deleteById(id)` | Xóa theo id | `prisma.product.delete(...)` |
| `count()` | Đếm số record | `prisma.product.count()` |

### Query Derivation — tự generate SQL từ tên method

```java
public interface ProductRepository extends JpaRepository<Product, Long> {
    List<Product> findByName(String name);              // WHERE name = ?
    List<Product> findByPriceLessThan(BigDecimal price); // WHERE price < ?
    List<Product> findByNameContaining(String keyword);  // WHERE name LIKE %keyword%
}
```

Spring tự parse tên method và generate query tương ứng — không cần viết SQL.

### findById vs getReferenceById

| Method | Hành vi |
|---|---|
| `findById(id)` | Query DB ngay, trả `Optional<Product>` — xử lý not found rõ ràng |
| `getReferenceById(id)` | Trả proxy object, **không query DB ngay** — lỗi chỉ xảy ra sau khi access field |

→ **Luôn dùng `findById`** cho các trường hợp thông thường:

```java
Product product = productRepository.findById(id)
    .orElseThrow(() -> new RuntimeException("Product not found with id: " + id));
```

## 6. DTO (Data Transfer Object)

### Tại sao dùng DTO mà không dùng thẳng Entity?

**1. Bảo mật — kiểm soát dữ liệu đầu vào:**
Nếu dùng Entity trực tiếp, client có thể gửi thêm field `id`, `role`, `status`... và ghi đè. DTO chỉ nhận đúng những field cho phép.

**2. Entity ≠ API contract:**
Entity gắn với database schema, DTO gắn với API. Hai thứ thay đổi vì lý do khác nhau:
- Thêm column `created_at` vào DB → không muốn client phải gửi lên
- API cần field tổng hợp → không có trong Entity

**3. So sánh Node.js:**
Giống việc validate `req.body` rồi chỉ pick những field cần thiết trước khi lưu DB.

> **Entity** = hình dạng data trong DB. **DTO** = hình dạng data mà API nhận/trả.

### Lưu ý khi viết DTO

DTO cần **no-arg constructor** để Jackson (thư viện JSON) deserialize JSON → object:

```java
public class ProductRequestDto {
    private String name;        // không dùng final
    private BigDecimal price;
    private String description;

    public ProductRequestDto() {}  // bắt buộc cho Jackson

    // getter, setter...
}
```

## 7. Relationship Mapping

### Các loại quan hệ

| Annotation | Ý nghĩa | Ví dụ |
|---|---|---|
| `@ManyToOne` | Nhiều → một | Nhiều Product thuộc 1 Category |
| `@OneToMany` | Một → nhiều | 1 Category có nhiều Product |
| `@ManyToMany` | Nhiều → nhiều | Product ↔ Tag |

### Ví dụ: Category — Product (One to Many)

**Phía Product** (phía sở hữu — giữ foreign key):

```java
@Entity
public class Product {
    // ...

    @ManyToOne
    private Category category;
}
```

**Phía Category** (phía tham chiếu ngược):

```java
@Entity
public class Category {
    // ...

    @OneToMany(mappedBy = "category")
    private List<Product> products;
}
```

### mappedBy là gì?

`mappedBy = "category"` nghĩa là: "quan hệ này đã được map bởi field `category` bên class Product".

- JPA sẽ **không tạo thêm bảng trung gian**, mà dùng foreign key `category_id` trong bảng `product`
- Nếu thiếu `mappedBy`, JPA tạo thêm bảng `category_products` — không đúng ý

```
Product (nhiều) ──── @ManyToOne category ────→ Category (một)
                     ↑ phía này giữ FK

Category (một) ──── @OneToMany(mappedBy="category") ────→ List<Product>
                     ↑ phía này chỉ tham chiếu ngược, không giữ FK
```

## 8. Controller — @RequestBody

Khi nhận JSON từ client, phải dùng `@RequestBody` để Spring parse JSON body thành object:

```java
@PostMapping
public ResponseEntity<Product> createProduct(@RequestBody ProductRequestDto dto) {
    // ...
}
```

Không có `@RequestBody` → tất cả field sẽ là `null`. Tương đương `req.body` bên Express.

## Câu hỏi & Trả lời trong quá trình học

### Q: Tôi đã dùng Prisma, mục đích ORM là gì?
**A:** ORM giúp viết code nhanh hơn, tiện hơn khi làm việc với database — thao tác qua object thay vì SQL thuần. Prisma và Spring Data JPA đều là ORM, chỉ khác cách tiếp cận.

### Q: Tại sao Java tách specification (JPA) và implementation (Hibernate)?
**A:** Để đảm bảo sự nhất quán — code bên ngoài không phải thay đổi khi đổi thư viện bên dưới. Các thư viện muốn hỗ trợ phải tuân theo chuẩn JPA. Đây là nguyên tắc "program to interface, not implementation" rất phổ biến trong Java.

### Q: Hibernate DDL auto nên dùng gì trong production?
**A:** Dùng `validate` hoặc `none`, kết hợp migration tool (Flyway/Liquibase). Không bao giờ để Hibernate tự sửa schema trên production vì nó không biết migrate data, có thể gây mất dữ liệu.

### Q: spring-boot-starter-data-jpa và mysql-connector-j khác nhau thế nào?
**A:** `spring-boot-starter-data-jpa` là tầng ORM (mapping, generate query). `mysql-connector-j` là JDBC driver (kết nối TCP tới MySQL). Hai tầng khác nhau — nếu đổi database chỉ cần đổi driver, không đổi code.

### Q: Tại sao dùng DTO mà không dùng thẳng Entity?
**A:** Ba lý do chính: (1) Bảo mật — kiểm soát field nào client được gửi lên, (2) Entity gắn với DB schema, DTO gắn với API — hai thứ thay đổi độc lập, (3) Giống việc validate và pick field từ `req.body` bên Express.

### Q: findById và getReferenceById khác nhau thế nào?
**A:** `findById` query DB ngay, trả `Optional` — xử lý not found rõ ràng. `getReferenceById` trả proxy, không query ngay, lỗi xảy ra sau → khó debug. Luôn dùng `findById` cho trường hợp thông thường.

### Q: mappedBy để làm gì?
**A:** Báo JPA biết phía nào sở hữu quan hệ (giữ foreign key). Không có `mappedBy`, JPA tạo thêm bảng trung gian không cần thiết. `mappedBy = "category"` nghĩa là field `category` bên Product đã giữ FK rồi.
