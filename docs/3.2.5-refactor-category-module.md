# Bài 3.2.5 — Refactor: Lombok + MapStruct (Phần 3: Module Category)

> Phần 2 (`docs/3.2.5-refactor-user-module.md`) đã làm User module — entity phẳng, không quan hệ phức tạp. Phần 3 sang **Category** — có **self-referencing relationship** (parent + children) → ra mắt khái niệm **recursive mapping** của MapStruct.

---

## 1. Khác biệt vs User module — tại sao Category đáng học riêng

| Khía cạnh | User | Category |
|---|---|---|
| Quan hệ | `OneToMany` đến entity khác (Order, Cart) | **Self-reference**: `parent: Category` + `children: List<Category>` |
| Response | Phẳng | **Tree** — `List<CategoryResponse> children` đệ quy |
| Request → Entity | Field trùng tên/type | `parentId: Long` (request) → `parent: Category` (entity) — **khác type** |
| Logic update | Đơn giản | Nhiều business rule: slug unique, circular parent, self-parent |

→ Category buộc trả lời 3 câu hỏi không gặp ở User:
1. MapStruct có tự đệ quy được không?
2. Field khác type (`Long` → `Category`) xử lý ở đâu?
3. Business rule (validate slug, parent circular) thuộc về mapper hay service?

---

## 2. Q&A trong buổi học (verbatim)

### Câu 1: MapStruct có tự đệ quy được cho self-reference không?

**Trả lời người học:** Tôi nghĩ là MapStruct sẽ không tự đệ quy được.

**Đáp đúng:** **CÓ** — MapStruct tự đệ quy được, với điều kiện có sẵn method map đúng kiểu.

#### Nguyên tắc

> **Khi MapStruct gặp một field cần map mà có sẵn một method trong mapper map đúng kiểu đó → nó sẽ gọi lại method đó.**

Áp dụng vào Category:
```java
@Mapper
public interface CategoryMapper {
    CategoryResponse fromEntity(Category category);   // ← method A
    //                  ↑
    //   MapStruct biết: "Tôi có method map Category → CategoryResponse"
}
```

Khi sinh code cho `fromEntity`, MapStruct gặp field:
```java
private List<CategoryResponse> children;   // từ List<Category> children
```

Nó tự tạo loop:
```java
// Pseudo-code MapStruct sinh ra
List<CategoryResponse> list = new ArrayList<>();
for (Category child : category.getChildren()) {
    list.add(fromEntity(child));   // ← gọi đệ quy chính nó!
}
target.setChildren(list);
```

#### ASCII flow

```
fromEntity(rootCategory)
   │
   ├─ map id, name, slug, description, status      (field cơ bản)
   │
   └─ children: List<Category> → List<CategoryResponse>
        │
        ├─ for each child:
        │     fromEntity(child)  ◄─── đệ quy
        │        │
        │        └─ children: List<Category> → List<CategoryResponse>
        │              └─ for each grandchild:
        │                    fromEntity(grandchild)  ◄─── đệ quy tiếp
        │                       └─ ...
        │
        └─ kết quả: List<CategoryResponse>
```

→ Không cần viết gì thêm cho `children`. Chỉ cần đảm bảo có method `Category → CategoryResponse`, MapStruct tự lo phần đệ quy.

#### Đối chiếu Node.js

```js
function fromEntity(cat) {
    return {
        id: cat.id,
        children: cat.children?.map(fromEntity) ?? []   // ← tự viết .map(fromEntity)
    };
}
```
→ Node.js thì tự gọi `.map(fromEntity)`. MapStruct làm điều tương tự lúc compile-time, không thấy code đó nhưng có.

#### Lưu ý nhỏ — `parent`

Nếu `CategoryResponse` có field `parent` (Category) thì MapStruct đệ quy **cả lên parent** → vòng lặp vô hạn (parent → child → parent → ...). May là `CategoryResponse` chỉ có `children`, không có `parent` → an toàn.

---

### Câu 2: `parentId` (Long) trong request → `parent` (Category) trong entity, xử lý thế nào?

**Trả lời người học:** Tôi nghĩ là Option A (service tự `findById` rồi `setParent`) vì chúng ta sẽ xử lý được một số về object null hay validate.

**Đáp đúng — Option A.** Lý do:

- **`findById` cần `Repository`** → mapper không nên inject repository (mapper là pure transformation, không chạm DB)
- **Validation null/not-found** thuộc business rule → service xử lý
- **Throw exception** chuẩn (`ResourceNotFoundException`) → service quyết định

Pattern y hệt password ở User module: mapper `ignore`, service `set`.

```java
// Mapper
@Mapping(target = "parent", ignore = true)
Category fromRequestDto(CreateCategoryRequest request);

// Service
Category existParent = null;
if (request.getParentId() != null) {
    existParent = categoryRepository.findById(request.getParentId())
        .orElseThrow(() -> new ResourceNotFoundException("Parent category not found"));
}
Category category = categoryMapper.fromRequestDto(request);
if (existParent != null) category.setParent(existParent);
```

---

### Câu 3: Business rule (validate slug unique, circular parent) để ở đâu — mapper hay service?

**Trả lời người học:** Logic này để trong service, còn lại mapper chỉ làm đúng việc mapper.

**Đáp đúng.** Mapper = pure function (input → output). Service = orchestration (call repo, validate, throw, set field cần xử lý đặc biệt).

## 3. Skeleton `CategoryMapper`

```java
@Mapper
public interface CategoryMapper {

    CategoryResponse fromEntity(Category category);

    @Mapping(target = "id", ignore = true)
    @Mapping(target = "status", ignore = true)
    @Mapping(target = "parent", ignore = true)
    @Mapping(target = "children", ignore = true)
    Category fromRequestDto(CreateCategoryRequest request);

    @Mapping(target = "id", ignore = true)
    @Mapping(target = "parent", ignore = true)
    @Mapping(target = "children", ignore = true)
    @Mapping(target = "slug", ignore = true)
    @BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE)
    void updateEntity(UpdateCategoryRequest request, @MappingTarget Category category);
}
```

3 method:
1. **`fromEntity`** — Category → CategoryResponse, MapStruct tự đệ quy field `children`
2. **`fromRequestDto`** — CreateCategoryRequest → Category, ignore: `id`, `status` (có default ACTIVE ở entity), `parent` (service xử lý), `children` (không khởi tạo)
3. **`updateEntity`** — partial update, ignore thêm `slug` (service check unique trước khi set)

## 4. Bug & bẫy phát hiện trong buổi review

### Bug 1: `@BeanMapping(IGNORE)` đặt nhầm chỗ — trên method create

```java
@BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE)
Category fromRequestDto(CreateCategoryRequest request);   // ← method create
```

**Sai semantic.** `IGNORE` chỉ có ý nghĩa khi `@MappingTarget` (entity tồn tại sẵn — partial update). Tạo entity mới từ rỗng → không có gì để "giữ lại" → annotation vô nghĩa, không gây bug nhưng làm reader confused.

**Quy tắc nhớ:** `@BeanMapping(IGNORE)` **chỉ** dùng cho method update có `@MappingTarget`.

### Bug 2: `updateCategory` không gọi mapper update — dead code

Lần đầu refactor, service vẫn dùng `if(CheckData...)` cho từng field, mapper `updateEntity` tồn tại nhưng không ai gọi. Mục tiêu refactor (xoá boilerplate) không đạt cho method quan trọng nhất.

**Fix:** dùng `categoryMapper.updateEntity(request, category)` → giữ logic check business rule (slug unique, circular) rời.

### Bug 3: Mapper update **thiếu `@BeanMapping(IGNORE)`** — bug ngầm

Lần đầu mapper update không có `@BeanMapping(IGNORE)`. Bug không lộ ngay vì service không gọi (bug 2). Nhưng khi fix bug 2 mà quên thêm `@BeanMapping(IGNORE)` → mọi field null trong request sẽ đè entity → mất dữ liệu.

**Fix:** thêm `@BeanMapping(nullValuePropertyMappingStrategy = IGNORE)` cho method update — y hệt User module.

### Bug 4: Typo `fromResquestDto` lan sang Service

```java
Category fromResquestDto(CreateCategoryRequest request);   // ← typo "Resquest"
//          ↑↑↑↑↑↑↑

// Service gọi luôn cái sai
Category category = categoryMapper.fromResquestDto(request);
```

**Học được:** đổi tên method trong mapper → IDE không tự đổi chỗ gọi (vì MapStruct sinh code lúc build). Phải tự rename cả 2 chỗ, hoặc dùng refactor → rename của IDE.

### Bẫy 5: Naming method mapper quá dài

```java
fromCreateCategoryResquest    // dài, lặp "Category"
fromUpdateCategoryRequest     // dài, lặp "Category"
```

→ Đặt theo pattern ngắn User module: `fromEntity` / `fromRequestDto` / `updateEntity`. Mỗi module lặp pattern này → nhìn 1 mapper bất kỳ là đoán được API. Java đã có context type ở chữ ký method (`Category fromRequestDto(CreateCategoryRequest request)`) — không cần nhắc lại tên type trong method name.

### Bẫy 6: `@AllArgsConstructor` còn sót trong `CategoryResponse`

Sau khi xoá `fromEntity` viết tay → constructor `@AllArgsConstructor` không còn ai gọi. MapStruct dùng setter (vì có `@Setter`) → annotation thành dead code. Đề xuất chuyển `@Data` cho gọn (giống `UserResponse`).

### Bẫy 7: Imports + comment chết trong `CategoryResponse`

Sau xoá `fromEntity` viết tay (commented out 30 dòng) → còn rác:
- `import com.ecommerce.api.entity.Category;` không dùng
- `import java.util.ArrayList;` không dùng
- Javadoc lỗi thời: `"fromEntity gọi đệ quy để build tree"` — fromEntity đã chuyển sang MapStruct

**Fix:** xoá imports + comment cũ, sửa Javadoc sang `"Mapping bởi MapStruct (CategoryMapper) — tự đệ quy field children."`

## 5. Trao đổi về flow `updateCategory` — pragmatic vs purist

### Hai cách tiếp cận

**Cách A (purist — em ban đầu đề xuất):** Validate hết → mutate hết → save
```java
// 1. Validate
if (slug duplicated) throw;
if (parent circular) throw;
// 2. Mutate
categoryMapper.updateEntity(request, category);
category.setSlug(...);
category.setParent(...);
// 3. Save
```

**Cách B (pragmatic — anh chọn):** Mapper map các field không cần validate → validate-then-set rời cho field cần validate
```java
categoryMapper.updateEntity(request, category);  // map name, description, status (đã ignore slug/parent)
if (slug khác) {
    if (duplicated) throw;
    category.setSlug(...);
}
if (parent set) {
    ...validate...
    category.setParent(...);
}
```

### Phân tích

**Cách B vẫn an toàn** vì:
- Mapper chỉ chạm vào field không cần validate (`name`, `description`, `status`) — đã ignore `slug`/`parent`
- Field cần validate vẫn theo pattern validate-then-set
- Nếu throw giữa chừng → transaction rollback (Spring default rollback on `RuntimeException`) → state cũ giữ nguyên

**Caveat:**
- Hiện chưa có `@Transactional` → rollback không guarantee (đã ghi cleanup backlog)
- Nếu sau này đổi sang checked exception → rollback không kích hoạt → bug

**Quyết định:** Giữ cách B. Đây là **trade-off**, không phải bug. Cleanup session sẽ add `@Transactional` → cách B trở thành an toàn hoàn toàn.

**Senior lesson:** Đôi khi senior review quá strict — phải nhận diện được khi nào là code smell thật vs khi nào là style preference.

## 6. Bug nghiệp vụ phát hiện — circular check chỉ 1 cấp

```java
if (CheckData.checkIsNotNull(category.getChildren())) {
    boolean check = category.getChildren().stream()
        .anyMatch(c -> c.getId().equals(parentId));   // ← chỉ check children trực tiếp
    if (check) throw new RuntimeException("...circular reference");
}
```

**Vấn đề:** chỉ phát hiện 1 cấp con. Cháu/chắt không detect.

**Scenario lộ bug:**
```
Trước update:
A (root)
 └─ B
     └─ C
         └─ D

Update: A.parent = D

→ Check hiện tại:
  - D.id == A.id? Không
  - A.children = [B], có chứa D không? KHÔNG (D là chắt, không phải con)
  - → Bypass → setParent thành công

→ Sau update:
  D → A → D → A → ...  (vòng lặp vô hạn trong DB!)
```

**Hậu quả:**
- `getAllCategories()` đệ quy build tree → `StackOverflowError`
- `categoryMapper.fromEntity` đệ quy → cũng `StackOverflowError`

**Fix đúng** cần recursive traversal:
```java
private boolean isDescendant(Category root, Long targetId) {
    if (root.getChildren() == null) return false;
    for (Category child : root.getChildren()) {
        if (child.getId().equals(targetId)) return true;
        if (isDescendant(child, targetId)) return true;
    }
    return false;
}
```

Hoặc query DB bằng recursive CTE nếu cây sâu.

**Quyết định:** Defer sang cleanup session — đòi hỏi recursive traversal + cân nhắc performance (nếu tree sâu), vượt scope refactor Lombok+MapStruct. **Đã ghi memory backlog.**

### File `CategoryService.java` sau cùng

```java
@RequiredArgsConstructor
@Service
public class CategoryService {

    private final CategoryRepository categoryRepository;
    private final CategoryMapper categoryMapper;

    public CategoryResponse createCategory(CreateCategoryRequest request) {
        boolean checkExistsSlug = this.categoryRepository.existsBySlug(request.getSlug());
        if (checkExistsSlug) throw new DuplicateResource("Slug already exists");
        Category existParent = null;
        if (request.getParentId() != null) {
            existParent = this.categoryRepository.findById(request.getParentId())
                .orElseThrow(() -> new ResourceNotFoundException("Parent category not found"));
        }

        Category category = categoryMapper.fromRequestDto(request);
        if (existParent != null) category.setParent(existParent);
        Category created = this.categoryRepository.save(category);
        return categoryMapper.fromEntity(created);
    }

    public List<CategoryResponse> getAllCategories() {
        List<Category> categories = this.categoryRepository.findByParentIsNull();
        return categories.stream().map(categoryMapper::fromEntity).toList();
    }

    public CategoryResponse getCategoryById(Long id) {
        Category category = this.categoryRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("Category not found"));
        return categoryMapper.fromEntity(category);
    }

    public CategoryResponse updateCategory(Long id, UpdateCategoryRequest request) {
        Category category = this.categoryRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("Category not found"));
        categoryMapper.updateEntity(request, category);
        String slugUpdate = request.getSlug();
        if (CheckData.checkIsNotNull(slugUpdate) && !slugUpdate.equals(category.getSlug())) {
            boolean checkExistedBySlug = this.categoryRepository.existsBySlug(slugUpdate);
            if (checkExistedBySlug) throw new DuplicateResource("Slug already exists");
            category.setSlug(slugUpdate);
        }
        Long parentId = request.getParentId();
        if (CheckData.checkIsNotNull(parentId)) {
            Category parentCategory = this.categoryRepository.findById(parentId)
                .orElseThrow(() -> new ResourceNotFoundException("Parent category not found"));
            if (parentCategory.getId().equals(category.getId()))
                throw new RuntimeException("Cannot set category as its own parent");
            if (CheckData.checkIsNotNull(category.getChildren())) {
                boolean check = category.getChildren().stream()
                    .anyMatch(c -> c.getId().equals(parentId));
                if (check) throw new RuntimeException("Cannot set a descendant as parent (circular reference)");
            }
            category.setParent(parentCategory);
        }
        return categoryMapper.fromEntity(this.categoryRepository.save(category));
    }

    public void deleteCategory(Long id) {
        Category category = this.categoryRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("Category not found"));
        if (CheckData.checkIsNotNull(category.getChildren()))
            throw new RuntimeException("Cannot delete category that has children");
        this.categoryRepository.deleteById(id);
    }
}
```

## 11. Phát hiện sau test E2E — bug `children = null` trong response

Khi test API `Create Category` qua Postman, response trả `"children": null` thay vì `"children": []`. Đây là bug **layer entity** + **layer mapper**, không phải DTO.

### Root cause

```java
// Service createCategory
Category category = categoryMapper.fromRequestDto(request);  // children = null (mapper ignore)
Category created = this.categoryRepository.save(category);   // save xong children VẪN null
return categoryMapper.fromEntity(created);                    // → response.children = null
```

Entity vừa `new Category()` chưa qua `findById` → field `children` chưa được init bao giờ → null. Khác với case `getAll/getById`: `findById` trả về managed entity, Hibernate inject `PersistentBag` không null cho `@OneToMany`.

### Tại sao default `= Collections.emptyList()` ở DTO không cứu

MapStruct sinh code:
```java
if (source.getChildren() != null) {
    target.setChildren(map(source.getChildren()));
} else {
    target.setChildren(null);   // ← ĐÈ LÊN default = emptyList()
}
```

→ Setter của Lombok bị MapStruct gọi tay với `null` → field default vô tác dụng.

### Fix — 2 layer (defense in depth)

**Layer 1: Init field ở entity (root cause):**
```java
@OneToMany(mappedBy = "parent")
private List<Category> children = new ArrayList<>();   // ← init empty
```

Hibernate sẽ thay reference bằng `PersistentBag` khi entity attach session — đây là **best practice** cho mọi `@OneToMany`/`@ManyToMany`, không có side effect.

**Layer 2: Bảo hiểm ở mapper:**
```java
@BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.SET_TO_DEFAULT)
CategoryResponse fromEntity(Category category);
```

`SET_TO_DEFAULT` cho collection → MapStruct sinh:
```java
target.setChildren(source.getChildren() == null ? new ArrayList<>() : map(...));
```

### Tại sao cần CẢ 2

| Chỉ Layer 1 | Chỉ Layer 2 | Cả 2 |
|---|---|---|
| Entity legacy/raw vẫn có thể null | Service code đụng `category.getChildren()` (vd circular check) vẫn NPE | Entity + response đều an toàn |

→ Pattern này đã apply cho cả `Product.productVariants` + `Product.images`.

## 12. Bước tiếp theo

**Product + ProductVariant + ProductImage** — phức tạp nhất:
- Có quan hệ 2 chiều (`Product` ↔ `ProductVariant`, `Product` ↔ `ProductImage`)
- Cần nested mapping: `@Mapping(source = "seller.id", target = "sellerId")`, `@Mapping(source = "category.id", target = "categoryId")`
- Có khả năng cần list mapping cho variants/images
- Cảnh giác lazy-load (`@ToString` của Lombok có thể trigger N+1 nếu vô tình bật)

→ Sẽ là module test cuối cùng cho pattern Lombok + MapStruct trước khi sang cleanup session production-grade.
