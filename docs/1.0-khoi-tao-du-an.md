## 1. Thiết kế Database Schema

### Các Entity chính

| Entity | Mô tả |
|---|---|
| `User` | Người dùng (khách hàng, seller, admin) |
| `Product` | Sản phẩm |
| `ProductVariant` | Biến thể sản phẩm (size, màu...) |
| `Category` | Danh mục (hỗ trợ nested — cha/con) |
| `Order` | Đơn hàng |
| `OrderItem` | Chi tiết từng sản phẩm trong đơn hàng |
| `Cart` | Giỏ hàng |
| `CartItem` | Chi tiết từng sản phẩm trong giỏ hàng |

### Sơ đồ quan hệ

```
User (1) ──── (n) Order
                    │
                   (n) OrderItem (n) ──── (1) Product
                                              │
                                             (n) ProductVariant
                                              │
                                             (1) Category (self-reference: parent_id)

User (1) ──── (1) Cart
                    │
                   (n) CartItem (n) ──── (1) Product
```

### Tại sao cần OrderItem (bảng trung gian)?

Không thể liên kết trực tiếp `Order → Product` vì mỗi đơn hàng cần lưu thêm:
- **quantity** — số lượng mua
- **price** — giá tại thời điểm mua (giá sản phẩm có thể thay đổi sau)
- **variant** — biến thể nào được chọn

Tương tự cho `CartItem`.

### Tại sao Product vẫn có price và stock khi đã có Variant?

Có 2 trường hợp:

**Trường hợp 1: Sản phẩm KHÔNG có variant** (ví dụ: sách)
→ Dùng `Product.price` và `Product.stock` trực tiếp.

**Trường hợp 2: Sản phẩm CÓ variant** (ví dụ: áo thun có size/màu)
→ `Product.price` = giá hiển thị trên trang danh sách ("từ 199k")
→ `Product.stock` = tổng tồn kho tất cả variant
→ Khi đặt hàng, lấy `Variant.price` và trừ `Variant.stock`

```
Áo thun Basic (Product)
├── price: 199,000 (giá hiển thị)
├── stock: 50 (tổng)
├── Variant "Đỏ - M"  → price: 199,000 | stock: 20
├── Variant "Đỏ - L"  → price: 219,000 | stock: 15
└── Variant "Xanh - M" → price: 199,000 | stock: 15
```

Đây là cách Shopee, Tiki, Shopify đều làm.

### Product - Category: ManyToOne (không phải ManyToMany)

Mỗi sản phẩm thuộc **1 danh mục** (iPhone thuộc "Điện thoại", không thuộc đồng thời "Điện thoại" và "Laptop"). Nếu cần phân loại phức tạp hơn, dùng **nested category** (danh mục cha/con):

```
Điện tử (cha)
├── Điện thoại (con)
└── Laptop (con)
```

## 2. Enum thay vì String

Thay vì dùng String thô cho status/role, dùng **enum** — Java sẽ kiểm tra giá trị lúc compile:

```java
public enum Role {
    CUSTOMER, ADMIN, SELLER
}

public enum Status {
    ACTIVE, INACTIVE, BANNED
}

public enum OrderStatus {
    PENDING, CONFIRMED, SHIPPING, DELIVERED, CANCELLED
}

public enum ProductStatus {
    ACTIVE, INACTIVE
}
```

Trong Entity, dùng `@Enumerated(EnumType.STRING)` để lưu **tên enum** (chuỗi) vào DB thay vì số thứ tự:

```java
@Enumerated(EnumType.STRING)
private Role role = Role.CUSTOMER;
```

Nếu dùng `EnumType.ORDINAL` (mặc định) → lưu số 0, 1, 2... → khi thêm/xóa giá trị enum, thứ tự thay đổi → dữ liệu sai. **Luôn dùng `EnumType.STRING`**.

## 3. `mappedBy` — ai giữ Foreign Key?

### Vấn đề

Trong mọi relationship, **chỉ 1 bên giữ FK trong database**. Bên đó gọi là **owner**. Bên còn lại dùng `mappedBy` để nói "FK nằm ở bên kia, không phải ở tôi".

### Ví dụ: User ↔ Cart (OneToOne)

```java
// Cart — OWNER (giữ FK user_id)
@Entity
public class Cart {
    @OneToOne
    @JoinColumn(name = "user_id")
    private User user;
}

// User — bên được tham chiếu (không giữ FK)
@Entity
public class User {
    @OneToOne(mappedBy = "user")  // "user" = tên field trong Cart
    private Cart cart;
}
```

Trong database:
```
Bảng users:    id | email | ...         ← không có FK
Bảng carts:    id | user_id (FK) | ...  ← Cart giữ FK
```

### Nếu cả 2 bên đều `mappedBy` thì sao?

**Lỗi!** Hibernate throw exception lúc khởi động. Vì `mappedBy` nghĩa là "FK ở bên kia" — nếu cả 2 bên đều nói "FK ở bên kia" thì **không bên nào tạo FK** → không có liên kết thật trong DB.

### Nếu cả 2 bên đều có `@JoinColumn` (không ai `mappedBy`)?

JPA tạo FK ở **cả 2 bảng** → dư thừa, lãng phí.

### Quy tắc chọn owner

| Relationship | Owner (giữ FK) |
|---|---|
| `@ManyToOne` ↔ `@OneToMany` | Bên `@ManyToOne` **luôn** là owner |
| `@OneToOne` | Tùy chọn — thường bên nào **phụ thuộc** vào bên kia thì giữ FK |

### Ví dụ trong project

```java
// Order.java — ManyToOne = OWNER
@ManyToOne
@JoinColumn(name = "user_id", nullable = false)
private User user;

// User.java — OneToMany = mappedBy
@OneToMany(mappedBy = "user")
private List<Order> orderList;
```

## 4. Nested Category (self-reference)

Category tự tham chiếu chính nó để tạo cấu trúc cha/con:

```java
@Entity
public class Category {
    @ManyToOne
    @JoinColumn(name = "parent_id")
    private Category parent;       // danh mục cha (null = root)

    @OneToMany(mappedBy = "parent")
    private List<Category> children;  // các danh mục con
}
```

**Đặt tên rõ nghĩa:** `parent` và `children` dễ hiểu hơn `category` và `categoryList`. Khi đọc code, `category.getParent()` và `category.getChildren()` rõ ràng hơn nhiều.

## 5. Exception Handling — `@RestControllerAdvice`

### Cơ chế hoạt động

Giống error-handling middleware trong Express, nhưng cho phép bắt **từng loại exception riêng**:

```
Request → Controller → Service → throw ResourceNotFoundException
                                        ↓
                          Spring tìm @ExceptionHandler phù hợp
                                        ↓
                          Tìm thấy handler cho ResourceNotFoundException → gọi nó
                          Không tìm thấy → tìm handler cho Exception (fallback)
                          Không có fallback → Spring trả default error page
```

Spring **match từ cụ thể đến chung** — nếu có handler cho cả `ResourceNotFoundException` và `Exception`, exception `ResourceNotFoundException` sẽ vào handler cụ thể, không vào fallback.

### ResourceNotFoundException

```java
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String message) {
        super(message);
    }
}
```

**Tại sao extends `RuntimeException` mà không phải `Exception`?**

Java có 2 loại Exception:

| | Checked Exception | Unchecked Exception |
|---|---|---|
| Extends từ | `Exception` | `RuntimeException` |
| Bắt buộc xử lý? | **Có** — phải try-catch hoặc throws | **Không** — tùy chọn |
| Ví dụ | `IOException`, `SQLException` | `NullPointerException`, `IllegalArgumentException` |

```java
// Checked Exception — compiler BẮT BUỘC phải xử lý
public void readFile() throws IOException {
    FileReader f = new FileReader("file.txt");
}

// Unchecked Exception — compiler không bắt buộc
public User findUser(Long id) {  // không cần khai báo throws
    throw new ResourceNotFoundException("User not found");
}
```

**Liên hệ Node.js:** Bên Node.js mọi exception đều giống unchecked — throw bất kỳ đâu, không ai bắt buộc phải catch. Java có thêm checked exception để compiler nhắc xử lý những lỗi có thể dự đoán (đọc file, kết nối DB...).

Trong Spring Boot, custom exception cho business logic **luôn dùng RuntimeException** vì:
- Không muốn mọi method phải khai báo `throws`
- Spring `@ExceptionHandler` bắt được cả 2 loại
- Code sạch hơn

### GlobalExceptionHandler

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<BaseResponse> handleResourceNotFoundException(ResourceNotFoundException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
                .body(BaseResponse.error(e.getMessage()));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<BaseResponse> handleException(Exception e) {
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(BaseResponse.error("Internal Server Error"));
    }
}
```

**3 điểm quan trọng:**

1. **Method phải nhận Exception làm tham số** — để lấy `e.getMessage()` trả về client. Nếu không nhận → phải hardcode message.

2. **Phải set HTTP status code** — mặc định trả HTTP 200 (dù success=false) → frontend kiểm tra `response.status` sẽ nghĩ thành công. Dùng `ResponseEntity` hoặc `@ResponseStatus` để set đúng status.

3. **Handler fallback (Exception.class) không nên trả `e.getMessage()`** — vì exception chung có thể chứa thông tin nhạy cảm (stack trace, tên bảng DB, SQL...). Chỉ trả message chung, log chi tiết ở server.

## 6. BaseResponse — API Response Format chuẩn

Mọi API đều trả về cùng 1 format:

```json
{
    "success": true,
    "message": "Thành công",
    "data": { ... }
}
```

```java
public class BaseResponse<T> {
    private Boolean success;
    private String message;
    private T data;

    public BaseResponse(Boolean success, String message, T data) {
        this.success = success;
        this.message = message;
        this.data = data;
    }

    public static <T> BaseResponse<T> success(T data, String message) {
        return new BaseResponse<>(true, message, data);
    }

    public static <T> BaseResponse<T> error(String message) {
        return new BaseResponse<>(false, message, null);
    }

    // getter/setter...
}
```

Dùng **Generic `<T>`** để `data` có thể là bất kỳ kiểu nào (`User`, `Product`, `List<Order>`...).

**Static factory method** (`BaseResponse.success()`, `BaseResponse.error()`) — tiện hơn gọi constructor trực tiếp, code đọc rõ ý nghĩa hơn.

## 7. `cascade = CascadeType.ALL`

```java
// Product.java
@OneToMany(mappedBy = "product", cascade = CascadeType.ALL)
private List<ProductVariant> productVariants;
```

`cascade` nghĩa là **lan truyền thao tác** từ entity cha sang con:
- Lưu Product → tự động lưu tất cả Variant của nó
- Xóa Product → tự động xóa tất cả Variant của nó

Phù hợp khi entity con **không tồn tại độc lập** (Variant không có ý nghĩa nếu không có Product).
