## LÝ THUYẾT (đã giảng)

### 1. Vì sao SQL `LIKE` "chết" với search thật sự

4 giới hạn chí mạng:
1. **Không dùng index** — `LIKE '%kw%'` leading `%` → full table scan O(N).
2. **Không hiểu "từ", chỉ so chuỗi ký tự** — "áo thun nam" không match "áo nam thun" (sai thứ tự từ).
3. **Không có relevance** — không biết kết quả nào liên quan nhất, chỉ ORDER BY cứng.
4. **Không typo-tolerance, không normalize** — gõ sai "iphpne" → 0 kết quả; không chuẩn hoá dấu tiếng Việt.

### 2. Inverted Index — trái tim của Elasticsearch

- **B-tree index (SQL):** `giá trị cột → vị trí row`. Tốt cho `=`, `<`, `>`, `LIKE 'prefix%'`. Vô dụng cho `LIKE '%...%'`.
- **Inverted index (ES):** `term → danh sách document chứa term`. Như **mục lục cuối sách**.

Ví dụ:
```
doc1: "Áo thun nam"   doc2: "Áo sơ mi nam"   doc3: "Quần jean nam"

term   → documents
"áo"   → [doc1, doc2]
"nam"  → [doc1, doc2, doc3]
"thun" → [doc1]
...
```
Search "áo nam" → tách ["áo","nam"] → tra term → doc1/doc2 match 2 từ (điểm cao) > doc3 (1 từ). Không quét toàn bảng, hiểu từ, có relevance.

### 3. Analyzer — cách ES xử lý chữ trước khi index

3 bước: **char filter → tokenizer → token filter**.
```
"Áo Thun NAM cao-cấp!"
 → char filter → tokenizer → ["Áo","Thun","NAM","cao","cấp"]
 → token filter (lowercase + asciifolding) → ["ao","thun","nam","cao","cap"]
```
Chạy analyzer cả lúc **index** lẫn **search** → "Iphone"/"iphone"/"IPHONE" đều normalize về "iphone" → match nhau.

Tiếng Việt: `standard` analyzer OK cơ bản; cần `asciifolding` để "áo"↔"ao"; plugin `icu`/`vi_analyzer` cho tách từ thông minh hơn. **Bài học dùng analyzer cơ bản + asciifolding**, không đụng plugin phức tạp.

### 4. ES bên cạnh MySQL — CQRS

> **ES KHÔNG thay thế MySQL. ES là "search layer" đặt BÊN CẠNH MySQL.**

```
MySQL (source of truth) ──sync──> Elasticsearch (search layer)
   ▲ write                            ▲ read/search
```
- MySQL = nguồn chân lý (ACID, không mất data). Mọi write vẫn vào MySQL.
- ES = read replica cho search (nhanh + relevance). Có thể rebuild từ MySQL bất cứ lúc nào.
- Pattern **CQRS**: ghi 1 nơi (MySQL), đọc-search 1 nơi (ES).

| | MySQL | Elasticsearch |
|--|-------|---------------|
| Đơn vị lưu | Row trong table | Document (JSON) trong index |
| Query | SQL | Query DSL (JSON) |
| Mạnh | Transaction, JOIN, ACID | Full-text, relevance, aggregation |
| Yếu | Full-text search | Transaction, JOIN, không phải source of truth |

### 5. Đồng bộ MySQL ↔ ES — 4 chiến lược

| Chiến lược | Mô tả | Ưu / Nhược |
|-----------|-------|-----------|
| A. Sync dual-write | Ghi MySQL xong ghi ES trong cùng service | Đơn giản / coupling chặt (ES down = create fail), không atomic |
| **B. Event-driven** ✅ | `ApplicationEventPublisher` + `@TransactionalEventListener` | Decouple, transaction-aware / eventual consistency, cần retry |
| C. CDC (Debezium) | Đọc binlog MySQL → Kafka → ES | Hoàn toàn decouple / hạ tầng phức tạp, overkill |
| D. Batch reindex | Cron quét MySQL đẩy ES | Robust / trễ, không real-time |

→ **Dự án chọn B (event-driven)** với `@TransactionalEventListener` — dạy pattern production thật mà không cần Kafka.

### 6. Spring Data Elasticsearch — Text vs Keyword

```java
@Document(indexName = "products")
public class ProductDocument {
    @Id private String id;
    @Field(type = FieldType.Text, analyzer = "standard") private String name;
    @Field(type = FieldType.Text) private String description;
    @Field(type = FieldType.Keyword) private String status;   // exact match
    @Field(type = FieldType.Double) private BigDecimal price;
}
```

| Type | Phân tích (analyzer)? | Dùng để |
|------|----------------------|---------|
| `Text` | ✅ Có (tách từ, lowercase) | **Full-text search** (name, description) |
| `Keyword` | ❌ Không (lưu nguyên) | **Filter exact, sort, aggregation** (status, category, sku) |

`ElasticsearchRepository<ProductDocument, String>` — API giống `JpaRepository`.

### 7. `bool` query — ES làm CẢ full-text LẪN filter trong 1 query

Use case: `keyword="iphone" + price 10-20tr + status=ACTIVE`. ES dùng `bool` query gồm 4 clause:

```json
{
  "query": {
    "bool": {
      "must":     [ { "multi_match": { "query": "iphone", "fields": ["name^3", "description"] } } ],
      "filter":   [ { "range": { "price": { "gte": 10000000, "lte": 20000000 } } },
                    { "term":  { "status": "ACTIVE" } } ],
      "should":   [ ],   // tăng điểm nếu match, không bắt buộc
      "must_not": [ ]    // loại trừ
    }
  }
}
```

**Phân biệt cốt lõi — query context vs filter context:**

| Clause | Context | Tính điểm? | Cache? | Dùng cho |
|--------|---------|-----------|--------|----------|
| `must` | query | ✅ có (relevance) | ❌ | Keyword text (cần xếp hạng) |
| `filter` | filter | ❌ không (yes/no) | ✅ | Price range, status, category (exact) |
| `should` | query | ✅ tăng điểm | ❌ | Ưu tiên brand... |
| `must_not` | filter | ❌ | ✅ | Loại trừ hết hàng |

→ **Quy tắc vàng:** keyword text → `must` (cần relevance); price/status/category → `filter` (nhanh hơn + cache được). `name^3` = boost field name gấp 3.

## Q&A (verbatim)

**Câu hỏi: Vậy nếu như tôi là một người dùng khi vào trang web, muốn tìm kiếm sản phẩm có chứa text là "Iphone" và cùng với khoảng giá, màu sắc thì bạn sẽ thiết kế api này như nào.**

**Trả lời:** ES làm **CẢ full-text search LẪN filter trong 1 query** qua cấu trúc `bool` query — KHÔNG cần tách "text thì ES, price/color thì SQL".

Thiết kế: **1 endpoint duy nhất**, ES xử lý toàn bộ:
```
GET /api/products/search?keyword=iphone&minPrice=...&maxPrice=...&categoryId=...&status=ACTIVE&page=0&size=20
```
Backend build 1 `bool` query: keyword → `must` (multi_match name^3 + description); minPrice/maxPrice → `filter` (range); category/status → `filter` (term); page/size → from/size của ES.

→ **Không đụng MySQL khi search.** ES trả về kết quả đã filter + xếp hạng relevance + phân trang trong 1 query.

So với cách "lai" (ES keyword + SQL filter): cách lai phải chạy 2 query rồi giao kết quả → **pagination rất khó, mất relevance, dễ bug**. Cách "ES toàn bộ" chỉ 1 query, ES lo hết.

**Phân vai rõ ràng:**
- **Có keyword** (user gõ thanh search) → `/api/products/search` → **Elasticsearch** (relevance quan trọng).
- **Không keyword** (duyệt category, lọc thuần) → `/api/products` → **MySQL Specification** (đã có ở 3.3.3, chính xác, source of truth).

ES không JOIN như SQL → phải **denormalize** (làm phẳng) data lúc index (vd gom color của variants vào 1 field array trong document). Tư duy: **SQL normalize, ES denormalize**.

> Ghi chú: `color` trong câu hỏi chỉ là ví dụ minh hoạ để hiểu — KHÔNG thêm vào implementation thật (Product không có field color, và đã thống nhất bỏ).
