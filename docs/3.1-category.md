## Phần 1: Thiết kế Category Entity

### Bối cảnh

Bước vào Giai đoạn 3 của ROADMAP — Quản lý sản phẩm. Bắt đầu với Category vì Product phụ thuộc vào nó. Đặc thù của Category trong e-commerce: cần hỗ trợ **nested category** (danh mục cha/con).

Ví dụ:
```
Thời trang
├── Thời trang nam
│   ├── Áo nam
│   └── Quần nam
├── Thời trang nữ
Điện tử
├── Điện thoại
├── Laptop
```

### Lý thuyết nền — self-referencing relationship

Cách phổ biến nhất trong JPA là dùng **self-referencing relationship** — một `Category` entity có quan hệ `@ManyToOne` trỏ về chính nó (parent), và `@OneToMany` chứa danh sách children.

So sánh với Node.js: giống kiểu bạn có một document/record mà field `parentId` trỏ về chính collection/table của nó.

### Câu hỏi thiết kế (Q&A)

**Câu hỏi 1: Category entity cần field gì?**

> Tôi nghĩ chỉ cần id, name, parent/children là đủ.

**Review:** Gần đúng, nhưng thiếu một số field quan trọng trong e-commerce thực tế:

- **`slug`** — URL-friendly string (ví dụ: `thoi-trang-nam`). Dùng cho SEO và đường dẫn trên FE (`/category/thoi-trang-nam` thay vì `/category/3`). Bên Node.js chắc cũng từng dùng pattern này.
- **`description`** — mô tả danh mục (không bắt buộc nhưng hay có)
- **`status`** (ACTIVE/INACTIVE) — để admin có thể ẩn danh mục mà không cần xóa

Còn các field như `image`, `displayOrder` thì tuỳ yêu cầu, giai đoạn này chưa cần.

**Câu hỏi 2: API nào cần cho Category?**

> Về admin quản lý danh mục, tôi nghĩ sẽ có API CRUD cho category.

**Review:** CRUD là đủ cho giai đoạn này. Cụ thể:
- **Create** — tạo category (có thể chỉ định parent)
- **Read** — lấy danh sách / lấy chi tiết
- **Update** — sửa tên, đổi parent...
- **Delete** — xoá category

**Lưu ý quan trọng:** khi xoá category cha, cần xử lý children. Có 3 chiến lược:
- Xoá luôn children (cascade delete)
- Chuyển children lên cấp trên
- Chặn không cho xoá nếu còn children

Trong project này mình chọn **chặn nếu còn children** — an toàn nhất. Admin phải xoá con trước.

**Câu hỏi 3: Trả list category dạng flat hay tree?**

> Tôi nghĩ nên trả về tree structure, bởi vì khi đó chúng thể hiện rõ mối quan hệ, dễ dàng cho việc FE render.

**Review:** Hoàn toàn đúng. Tree structure giúp FE dễ render menu, sidebar, breadcrumb... Tuy nhiên cần lưu ý:

- API **lấy tất cả categories** → trả về tree (cho sidebar, menu)
- API **lấy chi tiết 1 category** → trả về flat (thông tin category đó + parent info)

Không phải lúc nào cũng cần tree, tuỳ use case.

## Phần 2: Implementation

### Entity — self-referencing

```java
@Entity
@Table(name = "categories")
public class Category {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String name;

    @Column(unique = true, nullable = false)
    private String slug;

    @Column(columnDefinition = "TEXT")
    private String description;

    @Enumerated(EnumType.STRING)
    private CategoryStatus status = CategoryStatus.ACTIVE;

    @ManyToOne
    @JoinColumn(name = "parent_id")
    private Category parent;

    @OneToMany(mappedBy = "parent")
    private List<Category> children;

    // getter/setter...
}
```

### Repository

```java
public interface CategoryRepository extends JpaRepository<Category, Long> {
    List<Category> findByParentIsNull();                // root categories
    Optional<Category> findBySlug(String slug);
    boolean existsBySlug(String slug);

    @Query("SELECT c FROM Category c WHERE c.parent IS NULL")
    List<Category> findByParentIsNullV2();              // JPQL version
}
```

Hai cách để tìm root:
- **Derived query**: `findByParentIsNull()` — Spring Data tự sinh SQL từ tên method
- **JPQL query**: dùng `@Query` khi logic phức tạp

### Response DTO với đệ quy

Điểm thú vị nhất của Category. `fromEntity` cần gọi đệ quy để build tree:

```java
public static CategoryResponse fromEntity(Category category) {
    List<CategoryResponse> childResponses = new ArrayList<>();

    if (category.getChildren() != null && !category.getChildren().isEmpty()) {
        childResponses = category.getChildren().stream()
                .map(CategoryResponse::fromEntity)    // ← đệ quy
                .toList();
    }

    return new CategoryResponse(
            category.getId(),
            category.getName(),
            category.getSlug(),
            category.getDescription(),
            category.getStatus(),
            childResponses
    );
}
```

**Lưu ý về LazyInitializationException:** Vì `@OneToMany(mappedBy = "parent")` mặc định là **LAZY fetch**, nếu `fromEntity` được gọi **ngoài transaction** thì sẽ throw exception. Hiện tại gọi trong service (có transaction ngầm) nên OK. Sau này gặp lỗi này sẽ giải thích sâu về Open Session In View.

## Phần 3: Review Code — các bug đã fix

### Controller — 3 bug critical

**Bug 1: Sai cú pháp path variable**

```java
@PutMapping("/${id}")    // ❌ SAI
@DeleteMapping("/${id}") // ❌ SAI
```

`${}` là **property placeholder** của Spring (dùng đọc config từ `application.properties`). Còn **path variable** dùng **`{}`**.

```java
@PutMapping("/{id}")     // ✅ ĐÚNG
```

**Bug 2: Dùng sai annotation cho path variable**

```java
public ResponseEntity<...> getById(@RequestParam Long id) {   // ❌
```

`@RequestParam` là cho **query string** (`?id=1`). `@PathVariable` là cho **path variable** (`/categories/1`).

**Bug 3: BaseResponse.success(data) không tồn tại**

Ban đầu chỉ có overload `success(data, message)`. Đã thêm overload 1 tham số sau đó:

```java
public static <T> BaseResponse<T> success(T data) {
    return new BaseResponse<T>(true, "Success", data);
}
```

### Service — các bug logic

**Bug 1: NPE khi parentId == null**

```java
if(request.getParentId() > 0 || request.getParentId() == null) {  // ❌ NPE!
```

Nếu `parentId == null` → `getParentId() > 0` sẽ throw NullPointerException vì auto-unboxing. Đúng ra phải là:

```java
if (request.getParentId() != null) {
    existParent = categoryRepository.findById(request.getParentId())
        .orElseThrow(() -> new ResourceNotFoundException("Parent not exists"));
}
```

**Bug 2: Trả null cho empty list**

```java
if(categories.isEmpty()) return null;  // ❌ anti-pattern
```

Nên trả empty list. Lý do:
- FE phải handle 2 trường hợp (null + empty) → dễ sót bug
- Collection trả null là "code smell" phổ biến (Effective Java, Item 54)

Chỉ cần xoá dòng check. `stream().map().toList()` trên empty list vẫn trả empty list.

**Bug 3: Check slug chưa loại trừ chính nó khi update**

Nếu user gửi slug **y hệt slug hiện tại** → sẽ báo "đã tồn tại". Sửa:

```java
if (CheckData.checkIsNotNull(slugUpdate) && !slugUpdate.equals(category.getSlug())) {
    if (categoryRepository.existsBySlug(slugUpdate)) {
        throw new RuntimeException("The slug updated has already existed");
    }
    category.setSlug(slugUpdate);
}
```

**Bug 4: Gọi save() 2 lần**

```java
this.categoryRepository.save(category);
return CategoryResponse.fromEntity(this.categoryRepository.save(category));
```

Không sai nghiêm trọng (Hibernate không INSERT 2 lần vì cùng entity managed), nhưng thừa. Chỉ cần 1 lần.

**Bug 5: Circular reference check chưa đầy đủ**

Đã check parent = chính nó, và parent = direct children. Nhưng chưa check descendant sâu hơn:

```
A
└── B
    └── C
```

Set parent của A = C → vòng lặp `A → C → B → A`. Case này phức tạp, cần đệ quy xuống toàn bộ subtree. Tạm để TODO.

### Bài học về utility design

`CheckData.checkIsNotNull` ban đầu chỉ support `null`, `Long`, `String`. Khi mở rộng sang `Collection`, ý nghĩa "not null" thay đổi:

- Với `Long`, `String`: "not null" đi kèm "có giá trị hợp lệ"
- Với `Collection`: "not null" và "không rỗng" là **2 khái niệm khác nhau** — list rỗng vẫn là list hợp lệ

```java
public static boolean checkIsNotNull(Object o) {
    return switch (o) {
        case null -> false;
        case Long l -> l > 0;
        case String castStringType -> !castStringType.isBlank();
        case List<?> listData -> !listData.isEmpty();   // ← thêm case này
        default -> true;
    };
}
```

**Lesson senior:** Tên method phải phản ánh đúng behavior. Nếu utility làm nhiều hơn tên gọi của nó, nó sẽ trở thành "landmine" — có ngày người khác dùng sai và tạo ra bug.

## Phần 4: Deep dive — `@PreAuthorize`

### 1. Định nghĩa ngắn

`@PreAuthorize` là annotation của Spring Security, đặt trên method (hoặc class). **Trước khi method được gọi**, Spring sẽ evaluate biểu thức trong dấu ngoặc kép — nếu `true` thì method chạy, nếu `false` thì throw `AccessDeniedException` (HTTP 403).

```java
@PreAuthorize("hasRole('ADMIN')")
public void deleteCategory(Long id) { ... }
```

### 2. So sánh với Express

Bên Node.js/Express, bạn thường viết **middleware**:

```javascript
// Express
function requireAdmin(req, res, next) {
  if (req.user.role !== 'admin') return res.status(403).send('Forbidden');
  next();
}

router.delete('/categories/:id', requireAdmin, deleteController);
```

Spring làm tương tự, nhưng **khác về kỹ thuật**:

| | Express | Spring `@PreAuthorize` |
|---|---------|----------------------|
| Vị trí | Gắn vào route | Gắn vào method (bất kỳ layer nào) |
| Trigger | HTTP request đến | **Mỗi lần method được gọi** (kể cả gọi từ service khác) |
| Cách hoạt động | Function call chain | **AOP proxy** |
| Language | JavaScript function | **SpEL** (Spring Expression Language) |

**Điểm khác biệt quan trọng:** Express middleware chỉ chạy khi có HTTP request. `@PreAuthorize` chạy **mọi khi method được gọi** — kể cả một service khác gọi method đó nội bộ. Điều này có lợi vì bảo mật không phụ thuộc vào tầng HTTP.

### 3. Cơ chế bên trong — AOP Proxy

Spring không "magic" — nó dùng **Proxy**.

Khi bạn có class:

```java
@Service
public class CategoryService {
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteCategory(Long id) { ... }
}
```

Spring **KHÔNG trực tiếp inject** instance `CategoryService` vào controller. Thay vào đó, Spring tạo ra một **proxy class** wrapping CategoryService. Khi controller gọi `categoryService.deleteCategory(id)`, thực ra nó gọi vào proxy trước.

#### Sơ đồ luồng chạy

```
┌──────────────────┐
│   Controller     │
│                  │
│ categoryService  │ ──── gọi delete(id) ────┐
│   .delete(id)    │                         │
└──────────────────┘                         ▼
                          ┌─────────────────────────────────┐
                          │     Proxy (do Spring tạo)       │
                          │                                 │
                          │  1. Lấy Authentication từ       │
                          │     SecurityContext             │
                          │  2. Evaluate "hasRole('ADMIN')" │
                          │  3. Check quyền                 │
                          │     ├─ OK → gọi real method     │
                          │     └─ FAIL → throw             │
                          │        AccessDeniedException    │
                          └──────────────┬──────────────────┘
                                         │
                                         │ (nếu pass)
                                         ▼
                          ┌─────────────────────────────────┐
                          │   Real CategoryService          │
                          │   deleteCategory(id) {          │
                          │     // business logic           │
                          │   }                             │
                          └─────────────────────────────────┘
```

#### Proxy được tạo thế nào?

Spring dùng 2 kỹ thuật:
- **JDK Dynamic Proxy** — nếu class implement interface
- **CGLIB** — nếu class không implement interface (tạo subclass runtime)

`CategoryService` không implement interface → Spring dùng **CGLIB**, tạo một class con extends `CategoryService` và override method `deleteCategory` để chèn logic security check trước.

#### Pitfall — self-invocation bypass

```java
@Service
public class CategoryService {
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteCategory(Long id) { ... }
    
    public void deleteBatch(List<Long> ids) {
        for (Long id : ids) {
            this.deleteCategory(id);  // ⚠️ KHÔNG chạy @PreAuthorize!
        }
    }
}
```

**Tại sao?** Vì `this` là reference tới **real object**, không qua proxy. Để `@PreAuthorize` chạy, phải gọi qua **bean tiêm từ bên ngoài**. Đây cũng là lý do `@Transactional` hay có bug tương tự.

### 4. SpEL (Spring Expression Language)

`"hasRole('ADMIN')"` là một **string biểu thức**, không phải code Java. Spring parse và evaluate nó khi chạy.

Các helper method:

| Expression | Ý nghĩa |
|------------|---------|
| `hasRole('ADMIN')` | User có role ADMIN |
| `hasAnyRole('ADMIN', 'SELLER')` | Có 1 trong các role |
| `hasAuthority('READ_PRIVILEGE')` | Có authority cụ thể |
| `hasAnyAuthority(...)` | Có 1 trong các authority |
| `isAuthenticated()` | Đã login (không cần check role) |
| `isAnonymous()` | Chưa login |
| `permitAll()` | Cho tất cả |
| `denyAll()` | Chặn tất cả |
| `#id == authentication.principal.id` | So sánh tham số với user hiện tại |

#### SpEL mạnh: truy cập tham số method

```java
@PreAuthorize("#userId == authentication.principal.id or hasRole('ADMIN')")
public User getUser(Long userId) { ... }
```

Dịch: "Chỉ cho xem user nếu userId bằng id của user đang login **hoặc** user đang login là ADMIN".

Express middleware không làm được dễ dàng — phải viết tay trong mỗi handler.

### 5. `hasRole` vs `hasAuthority` — pitfall kinh điển

- Spring Security **ngầm định** role có prefix `ROLE_`
- Khi dùng `hasRole('ADMIN')` → Spring **tự động thêm `ROLE_`** → so với `ROLE_ADMIN`
- Khi dùng `hasAuthority('ADMIN')` → Spring **KHÔNG** thêm prefix → so với `ADMIN` y nguyên

| Authority trong DB | `hasRole('ADMIN')` | `hasAuthority('ADMIN')` | `hasAuthority('ROLE_ADMIN')` |
|-------------------|:-:|:-:|:-:|
| `ADMIN` | ❌ | ✅ | ❌ |
| `ROLE_ADMIN` | ✅ | ❌ | ✅ |

### 6. `@PreAuthorize` vs anh em

| Annotation | Nguồn gốc | Sức mạnh |
|-----------|----------|----------|
| `@Secured` | Spring cũ | Chỉ check role đơn giản |
| `@RolesAllowed` | Chuẩn JSR-250 | Giống `@Secured` |
| **`@PreAuthorize`** | Spring Security | **Mạnh nhất** — hỗ trợ SpEL đầy đủ |
| `@PostAuthorize` | Spring Security | Check **sau khi** method chạy |

**Khuyến nghị dùng `@PreAuthorize`** — linh hoạt nhất, là chuẩn hiện đại.

### 7. Điều kiện để `@PreAuthorize` hoạt động

Cần bật:

```java
@Configuration
@EnableMethodSecurity  // ⚠️ BẮT BUỘC
public class SecurityConfig { ... }
```

Không có → annotation bị ignore hoàn toàn.

### 8. Quick cheat sheet

```java
// Chỉ ADMIN
@PreAuthorize("hasRole('ADMIN')")

// ADMIN hoặc SELLER
@PreAuthorize("hasAnyRole('ADMIN', 'SELLER')")

// Đã login là được
@PreAuthorize("isAuthenticated()")

// Check theo tham số
@PreAuthorize("#userId == authentication.principal.id")

// Tổ hợp điều kiện
@PreAuthorize("hasRole('ADMIN') or #userId == authentication.principal.id")
```

## Phần 5: Custom expression cho `@PreAuthorize`

### Câu hỏi

> Bây giờ chúng ta bắt buộc phải dùng các string như "hasRole('ADMIN')" mà không thể thay thế được bằng string mình tự custom như "phai_la()"?

**Trả lời:** KHÔNG bắt buộc. Có 3 cách tạo biểu thức custom.

### Cách 1: Gọi method từ Bean (đơn giản, khuyên dùng)

SpEL có cú pháp **`@tenBean.tenMethod(args)`** — tham chiếu trực tiếp đến một Spring bean.

**Tạo bean chứa logic:**

```java
@Component("categorySecurity")
public class CategorySecurity {
    
    public boolean phaiLa(String roleRequired) {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth == null || !auth.isAuthenticated()) return false;
        
        return auth.getAuthorities().stream()
            .anyMatch(a -> a.getAuthority().equals(roleRequired));
    }
    
    public boolean canDelete(Long categoryId) {
        // Logic phức tạp: query DB, check ownership...
        return true;
    }
}
```

**Dùng trong `@PreAuthorize`:**

```java
@DeleteMapping("/{id}")
@PreAuthorize("@categorySecurity.phaiLa('ADMIN')")
public void delete(@PathVariable Long id) { ... }

@PreAuthorize("@categorySecurity.canDelete(#id)")
public void delete(@PathVariable Long id) { ... }
```

**Vì sao `@` lại ref được bean?** Cú pháp `@beanName` là **SpEL built-in** (không phải Java annotation `@`). SpEL có **BeanResolver** — khi gặp `@tenBean`, nó dùng `ApplicationContext` để resolve bean theo tên. Đó là lý do cần đặt **tên bean** trong `@Component("categorySecurity")`.

**Tương đương Express:** Cách này 1:1 với pattern middleware. `@categorySecurity.phaiLa('ADMIN')` = `requireAdmin` middleware.

### Cách 2: Custom `SecurityExpressionRoot`

Nếu muốn viết **`phaiLa('ADMIN')`** mà không cần prefix `@beanName.`, tức là thêm method mới vào "root namespace" của SpEL security.

Khi gõ `hasRole(...)`, `isAuthenticated()` — đó là method của class `SecurityExpressionRoot`. Spring set object này làm **root object** khi evaluate biểu thức → gọi method không cần prefix.

**Bước 1: Kế thừa `SecurityExpressionRoot`**

```java
public class CustomMethodSecurityExpressionRoot 
        extends SecurityExpressionRoot 
        implements MethodSecurityExpressionOperations {
    
    public CustomMethodSecurityExpressionRoot(Authentication authentication) {
        super(authentication);
    }
    
    public boolean phaiLa(String role) {
        return this.authentication.getAuthorities().stream()
            .anyMatch(a -> a.getAuthority().equals("ROLE_" + role));
    }
    
    // Các method required bởi MethodSecurityExpressionOperations...
}
```

**Bước 2: Tạo custom `ExpressionHandler` + đăng ký bean**

**Bước 3: Dùng trực tiếp, không cần prefix**

```java
@PreAuthorize("phaiLa('ADMIN')")           // ← Gọn đúng ý!
@PreAuthorize("phaiLa('ADMIN') or laChuSoHuu(#id)")
```

### Cách 3: `PermissionEvaluator` — dành cho ACL

Dùng `hasPermission(targetObject, permission)` với custom `PermissionEvaluator`. Dành cho hệ thống **ACL-style** (Access Control List) phức tạp.

```java
@PreAuthorize("hasPermission(#id, 'Category', 'DELETE')")
```

### Khi nào dùng cách nào?

| Tình huống | Nên dùng |
|-----------|---------|
| Check role đơn giản (ADMIN, SELLER...) | Built-in `hasRole()` |
| Logic custom, query DB, check ownership | **Cách 1: `@beanName.method(args)`** ✅ |
| Dùng rất nhiều chỗ, muốn cú pháp ngắn | Cách 2: Custom `SecurityExpressionRoot` |
| Hệ thống ACL phức tạp | Cách 3: `PermissionEvaluator` |

**Cho project e-commerce:** Cách 1 là đủ. Ví dụ: "chỉ seller sở hữu product mới được sửa":

```java
@PreAuthorize("hasRole('ADMIN') or @productSecurity.isOwner(#id, authentication)")
public Product updateProduct(@PathVariable Long id, ...) { ... }
```

## Phần 6: Custom annotation với tên ngữ nghĩa — AOP

### Câu hỏi

> `@PreAuthorize` nếu dùng cho các logic phức tạp thì cũng không đúng với tên lắm nhỉ, ví dụ như các user phải có trên 100 hoá đơn thì mới có thể chạy method này. Lúc đấy tôi nghĩ mình phải tạo ra một annotation mới cũng chạy trước method giống như `@PreAuthorize` này nhưng nó đúng với logic xử lý hơn ví dụ: `@CheckTotalOrderByUser` chẳng hạn.

**Trả lời:** Tư duy rất đúng. `@PreAuthorize` thuộc tầng **security**, không phải **business rule**. Việc tách concern và đặt tên đúng ngữ nghĩa là nền tảng của clean architecture.

Giải pháp: Tạo **custom annotation** riêng cho business rule, chạy trước method — y hệt `@PreAuthorize` về cơ chế, nhưng **khác về ngữ nghĩa**. Đây chính là **AOP** (Aspect-Oriented Programming).

### 1. AOP là gì?

**Aspect-Oriented Programming** là kỹ thuật tách **cross-cutting concern** (quan tâm xuyên suốt) ra khỏi business logic.

**Cross-cutting concern** là logic xuất hiện ở **nhiều chỗ** nhưng **không phải core business**:
- Logging
- Security
- Transaction
- Caching
- Validation
- Business rule check ← case của câu hỏi

#### Sơ đồ tư duy

```
   ┌────────────────────────────────────────────┐
   │                                            │
   │         Business Logic (core)              │
   │   createOrder(), updateUser(), ...         │

   ───┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴───

   ─────────────────────────────────────────── cross-cutting
                     Security              
   ───────────────────────────────────────────
                     Transaction
   ───────────────────────────────────────────
                 Business Rule Check
```

### 2. Key terms cần thuộc

| Thuật ngữ | Ý nghĩa |
|-----------|---------|
| **Aspect** | Module chứa cross-cutting logic |
| **Join Point** | Một điểm trong chương trình có thể chèn aspect (thường là method call) |
| **Pointcut** | Biểu thức chọn xem chèn aspect vào **những** join point nào |
| **Advice** | Logic được chèn (Before, After, Around...) |
| **Target** | Object bị chèn aspect vào |
| **Weaving** | Quá trình "dệt" aspect vào target (Spring làm runtime qua proxy) |

### 3. Hiện thực — Custom annotation + AOP Aspect

**Bước 1: Thêm dependency**

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
```

**Bước 2: Tạo custom annotation**

```java
@Target(ElementType.METHOD)               // gắn trên method
@Retention(RetentionPolicy.RUNTIME)       // có mặt lúc runtime để AOP đọc được
public @interface CheckTotalOrderByUser {
    int min() default 100;
    String message() default "Bạn cần có ít nhất 100 đơn hàng để dùng tính năng này";
}
```

Giải thích:
- `@interface` → tạo annotation (không phải interface thường)
- `@Target(ElementType.METHOD)` → giới hạn chỉ gắn được trên method
- `@Retention(RUNTIME)` → giữ thông tin annotation lúc runtime (mặc định Java là CLASS — bị xoá khi load)
- Annotation có thể có **params** (như `min`, `message`)

**Bước 3: Tạo Aspect — nơi chứa logic**

```java
@Aspect
@Component
public class OrderRuleAspect {
    
    private final OrderRepository orderRepository;
    
    public OrderRuleAspect(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
    
    @Before("@annotation(checkTotalOrderByUser)")
    public void checkOrderCount(JoinPoint joinPoint, 
                                CheckTotalOrderByUser checkTotalOrderByUser) {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        Long userId = ((User) auth.getPrincipal()).getId();
        
        long orderCount = orderRepository.countByUserId(userId);
        
        if (orderCount < checkTotalOrderByUser.min()) {
            throw new BusinessRuleViolationException(checkTotalOrderByUser.message());
        }
    }
}
```

**Bước 4: Dùng annotation**

```java
@PostMapping("/vip-deal")
@CheckTotalOrderByUser(min = 100)
public void claimVipDeal() {
    // logic tặng deal
}
```

### 4. Các pointcut phổ biến

| Pointcut | Chọn những method |
|----------|-------------------|
| `@annotation(CheckTotalOrderByUser)` | Có gắn annotation này |
| `execution(* com.ecommerce.api.services.*.*(..))` | Thuộc package `services` |
| `execution(public * *(..))` | Mọi method public |
| `within(com.ecommerce.api.controllers.*)` | Trong package controllers |
| `args(Long, String)` | Có signature 2 tham số Long + String |
| `target(CategoryService)` | Của class CategoryService |

Tổ hợp bằng `&&`, `||`, `!`.

### 5. Các loại advice

| Advice | Khi chạy |
|--------|---------|
| `@Before` | **Trước** method — như `@PreAuthorize` |
| `@After` | **Sau** method (dù thành công hay lỗi) |
| `@AfterReturning` | Sau khi method return thành công — có thể lấy return value |
| `@AfterThrowing` | Sau khi method throw exception |
| `@Around` | **Bao quanh** method — cực mạnh, có quyền quyết định gọi/không gọi method gốc |

#### `@Around` — anh cả

```java
@Around("@annotation(checkTotalOrderByUser)")
public Object around(ProceedingJoinPoint pjp, 
                     CheckTotalOrderByUser checkTotalOrderByUser) throws Throwable {
    long start = System.currentTimeMillis();
    
    Object result = pjp.proceed();    // gọi method gốc (hoặc không gọi!)
    
    log.info("Took {}ms", System.currentTimeMillis() - start);
    return result;
}
```

`@Around` hiếm dùng vì phức tạp, nhưng cần cho **đo thời gian**, **cache**, **retry**.

### 6. Cơ chế — giống `@PreAuthorize`

**Giống hệt.** Cả `@PreAuthorize` và `@Transactional` cũng là **AOP aspect** do Spring cung cấp sẵn. Tự viết aspect là làm điều tương tự với annotation tự tạo.

#### Sơ đồ runtime

```
   Controller gọi service.claimVipDeal()
                    │
                    ▼
        ┌───────────────────────┐
        │       Proxy           │
        │  (Spring tạo ngầm)    │
        └────────┬──────────────┘
                 │
                 ▼
      ┌────────────────────────┐
      │  Chain of Aspects      │
      │  ┌──────────────────┐  │
      │  │ @Before:         │  │  ← OrderRuleAspect
      │  │ checkOrderCount  │  │     chạy ở đây
      │  └────────┬─────────┘  │
      │           │            │
      │   (pass) OK            │
      │           ▼            │
      │  ┌──────────────────┐  │
      │  │ Real method:     │  │  ← method gốc
      │  │ claimVipDeal()   │  │
      │  └──────────────────┘  │
      └────────────────────────┘
```

Khi có **nhiều aspect** (ví dụ: `@PreAuthorize` + `@CheckTotalOrderByUser` + `@Transactional`), Spring sắp xếp thành **chain** — control thứ tự bằng `@Order`.

**Pitfall giống `@PreAuthorize`:** Self-invocation bypass — gọi `this.method()` từ method khác cùng class → aspect **không chạy**. Vì cùng lý do proxy.

### 7. Cách 2: Meta-annotation — wrap `@PreAuthorize`

Nếu logic **có thể biểu diễn bằng SpEL** — tạo "alias" cho `@PreAuthorize`:

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
@PreAuthorize("@orderRule.userHasMoreThan(authentication, 100)")
public @interface CheckTotalOrderByUser {
}
```

Dùng:
```java
@CheckTotalOrderByUser
public void claimVipDeal() { ... }
```

So sánh:

| | Custom Aspect | Meta-annotation |
|---|---|---|
| Flexibility | Cao — logic Java thuần | Thấp — bị giới hạn SpEL |
| Tham số annotation | ✅ Đọc được (min, message...) | ❌ Khó truyền vào SpEL |
| Boilerplate | Nhiều | Ít |
| Readable | Rõ, debug dễ | Gọn, nhưng "ẩn" |

**Recommend:** Custom Aspect cho business rule phức tạp.

### 8. So sánh với Node.js — NestJS decorator

```typescript
// NestJS
@UseGuards(CheckTotalOrderGuard)
claimVipDeal() { ... }

// Custom decorator:
export function CheckTotalOrderByUser(min: number) {
  return applyDecorators(
    UseGuards(CheckTotalOrderGuard),
    SetMetadata('minOrders', min)
  );
}
```

NestJS lấy cảm hứng nhiều từ Spring. **Tư duy AOP là chung**. Express thuần không có AOP — phải dùng middleware. Đó là lý do Spring mạnh hơn Express trong việc xử lý concern lớn (transaction, security, cache...).

### 9. Khi nào dùng custom annotation AOP?

**✅ Nên dùng:**
- Logic lặp lại ở **≥3 method**
- Tách business rule ra khỏi business logic
- Rule có tên ngữ nghĩa rõ ràng (`@RequiresMembership`, `@OnlyForVipUser`, `@AuditLog`...)
- Cần config qua annotation params

**❌ Không nên dùng:**
- Logic chỉ xuất hiện ở **1 method** → viết thẳng trong method
- Logic cực kỳ đơn giản (if/else ngắn) → AOP overkill
- Logic phụ thuộc nhiều vào state local

**⚠️ Cảnh báo over-engineering:** Mỗi aspect là một lớp indirect — người đọc code phải nhớ "à method này còn annotation X chạy trước". Debug khó hơn. **Rule of thumb:** Nếu logic chưa lặp 3 lần → chưa cần AOP.
