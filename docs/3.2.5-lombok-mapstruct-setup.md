# Bài 3.2.5 — Refactor: Lombok + MapStruct (Phần 1: Setup)

> Phần này chỉ cover **giai đoạn setup dependency + cấu hình build**. Phần refactor code thực tế (entity, DTO, service, mapper) sẽ ở phần 2.

## 1. Vấn đề & lý do refactor

### Vấn đề hiện tại
- Mở `Product.java` hoặc `User.java`: ~70% file là **boilerplate** — getter/setter/constructor/equals/hashCode/toString. Mỗi lần thêm field phải sửa nhiều chỗ.
- Trong `ProductService`, các method `fromEntity()` / `toResponse()` viết tay rất dài, dễ sai khi thêm field (quên map → bug âm thầm).

### Hai thư viện sẽ giải quyết
- **Lombok** — xoá boilerplate trong entity/DTO bằng annotation.
- **MapStruct** — auto generate code mapping Entity ↔ DTO.

## 2. Lombok — internal mechanism (deep dive)

### Lombok làm gì?
Bạn viết:
```java
@Getter @Setter
public class Product {
    private String name;
    private BigDecimal price;
}
```

Sau khi compile, **bytecode** của `Product.class` sẽ có đầy đủ `getName()`, `setName()`, `getPrice()`, `setPrice()` — y hệt như tự viết tay.

### Cơ chế: Annotation Processing (compile-time)

Lombok **KHÔNG** dùng reflection runtime, **KHÔNG** dùng proxy. Nó hook vào **trình biên dịch javac** thông qua API `javax.annotation.processing.Processor`.

```
┌─────────────────────────────────────────────────────────────┐
│  Quy trình compile bình thường:                              │
│                                                              │
│  Product.java ──[javac]──> Product.class (bytecode)          │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│  Quy trình có Lombok:                                        │
│                                                              │
│  Product.java                                                │
│       │                                                      │
│       ▼                                                      │
│  [javac parse → AST (Abstract Syntax Tree)]                  │
│       │                                                      │
│       ▼                                                      │
│  [Lombok Annotation Processor]                               │
│       │  - Quét @Getter/@Setter trong AST                   │
│       │  - Inject thêm node "method getName()" vào AST      │
│       ▼                                                      │
│  [javac generate bytecode từ AST đã modify]                  │
│       │                                                      │
│       ▼                                                      │
│  Product.class (đã có getter/setter)                         │
└─────────────────────────────────────────────────────────────┘
```

→ Vì sửa AST **trước khi** sinh bytecode, nên trong `.class` cuối cùng method có sẵn → IDE, debugger, JVM đều "thấy" như code thật.

### So sánh với Node.js
- Node.js không có khái niệm tương đương trực tiếp. Gần nhất là **Babel plugin** hoặc **TypeScript transformer** — cũng modify AST trước khi emit JS.
- Hoặc `Object.defineProperty` runtime, nhưng đó là reflection-style (như Java reflection), khác hoàn toàn với Lombok.

### Annotations Lombok hay dùng

| Annotation | Sinh ra |
|---|---|
| `@Getter` / `@Setter` | getter/setter cho mọi field |
| `@NoArgsConstructor` | constructor rỗng (JPA cần) |
| `@AllArgsConstructor` | constructor đủ args |
| `@RequiredArgsConstructor` | constructor cho mọi field `final` (constructor injection) |
| `@Builder` | builder pattern |
| `@Data` | combo: `@Getter + @Setter + @ToString + @EqualsAndHashCode + @RequiredArgsConstructor` |
| `@Slf4j` | sinh `private static final Logger log = ...` |

### ⚠️ Lưu ý với JPA Entity
KHÔNG dùng `@Data` cho entity vì:
- `@EqualsAndHashCode` mặc định include tất cả field → có thể gây vòng lặp vô hạn khi entity có quan hệ 2 chiều (Product ↔ ProductVariant).
- `@ToString` mặc định cũng include tất cả field → khi log entity có collection lazy-loaded, sẽ trigger query DB lúc không mong muốn (N+1 problem).

Best practice với JPA entity:
```java
@Getter
@Setter
@NoArgsConstructor   // JPA cần constructor rỗng
public class Product { ... }
```

## 3. MapStruct — internal mechanism

### Vấn đề
Code hiện tại có method kiểu:
```java
ProductResponse toResponse(Product p) {
    ProductResponse r = new ProductResponse();
    r.setId(p.getId());
    r.setName(p.getName());
    r.setPrice(p.getPrice());
    // ... 10 dòng nữa
    return r;
}
```
→ Viết tay, dễ quên field, khó maintain.

### MapStruct làm gì?
Bạn chỉ khai báo **interface**:
```java
@Mapper(componentModel = "spring")
public interface ProductMapper {
    ProductResponse toResponse(Product product);
}
```

MapStruct tự generate **class implementation** `ProductMapperImpl.java` lúc compile, với code y hệt viết tay.

### Cơ chế: cũng là Annotation Processing
Khác Lombok ở chỗ Lombok **modify AST của file gốc**, còn MapStruct **sinh ra file Java mới**.

```
┌───────────────────────────────────────────────────┐
│  ProductMapper.java (interface)                    │
│           │                                        │
│           ▼                                        │
│  [MapStruct Processor đọc method signature]       │
│  - Source type: Product                            │
│  - Target type: ProductResponse                    │
│  - Match field theo TÊN (id → id, name → name)    │
│           │                                        │
│           ▼                                        │
│  Sinh file ProductMapperImpl.java vào             │
│  target/generated-sources/annotations/            │
│           │                                        │
│           ▼                                        │
│  [javac] compile cả 2 file                        │
└───────────────────────────────────────────────────┘
```

→ Khác hoàn toàn với reflection-based mapper (như ModelMapper). MapStruct generate **code tĩnh, type-safe, không reflection runtime** → nhanh ngang viết tay.

### So sánh với Node.js
Không có thứ tương đương 1-1. Closest: tự viết function map, hoặc dùng `class-transformer` (NestJS) — nhưng `class-transformer` dùng reflection runtime, chậm hơn MapStruct.

## 4. Q&A — Hỏi học viên trước khi triển khai

### Câu 1: File nào nên refactor đầu tiên (entity, DTO, service, hay tất cả)?

**Trả lời học viên:** "tôi nghỉ các file cần refactore là dto, entity"

**Review (senior):**
Đúng một phần, nhưng thiếu. Lombok không chỉ giúp DTO/Entity. Bạn quên mất **Service** và **Component** — đây là nơi Lombok phát huy mạnh nhất:

- **`@RequiredArgsConstructor`** trên Service → xoá constructor injection viết tay:
  ```java
  // Trước
  private final ProductRepository productRepository;
  private final UserRepository userRepository;
  public ProductService(ProductRepository pr, UserRepository ur) {
      this.productRepository = pr;
      this.userRepository = ur;
  }
  // Sau (chỉ cần @RequiredArgsConstructor trên class)
  private final ProductRepository productRepository;
  private final UserRepository userRepository;
  ```
  Lombok sẽ tự sinh constructor cho **tất cả field `final`** → đây chính là cách Spring khuyến nghị làm constructor injection (immutable, dễ test).

- **`@Slf4j`** → thay cho `private static final Logger log = LoggerFactory.getLogger(...)`.

→ Nên list cuối cùng: **Entity + DTO + Service + Filter/Component**.

### Câu 2: JPA Entity có quan hệ 2 chiều — chọn `@Data` hay `@Getter @Setter` rời?

**Trả lời học viên:** "JPA Entity có quan hệ 2 chiều sẽ chỉ dùng @Getter với @Setter, vì @EqualsAndHashCode mặc định include tất cả field → có thể gây vòng lặp vô hạn khi entity có quan hệ 2 chiều"

**Review (senior):** Đúng hoàn toàn. Bonus thêm 1 lý do nữa: `@ToString` mặc định cũng include tất cả field → khi log entity có collection lazy-loaded, sẽ trigger query DB lúc không mong muốn (N+1 problem). Cho nên với JPA entity, best practice là dùng `@Getter @Setter @NoArgsConstructor` rời.

### Câu 3: MapStruct sẽ replace những method nào?

**Trả lời học viên:** "Mapstruct sẽ replace các method convert giữa Dto request -> entity và entity -> DTO response"

**Review (senior):** Đúng. Cụ thể trong code hiện tại sẽ có các method kiểu:
- `Product.toEntity(ProductCreateRequest)` hoặc `ProductService.toEntity(...)`
- `ProductResponse.fromEntity(Product)`
- Tương tự cho `ProductVariant`, `ProductImage`, `Category`, `User`...

→ Tất cả gom về `*Mapper` interface, MapStruct generate impl. Lưu ý: với field cần custom logic (ví dụ map `seller.id` từ entity sang `sellerId` trong DTO), MapStruct có annotation `@Mapping(source = "seller.id", target = "sellerId")`.

### Câu 4: Refactor toàn bộ Lombok trước hay làm gọn từng module?

**Trả lời học viên:** "Làm gọn tưng module"

**Review (senior):** Tốt. Đây là cách an toàn vì mỗi module refactor xong là chạy được luôn, dễ rollback nếu lỗi. Thứ tự đề xuất:
```
1. User module       (đơn giản nhất, ít quan hệ)
2. Category module   (ít field, không quan hệ 2 chiều)
3. Product + ProductVariant + ProductImage  (phức tạp nhất, để cuối)
```

## 5. Cấu hình `pom.xml`

### Dependencies cần thêm

```xml
<!-- Lombok: optional=true vì runtime không cần, đã nướng vào bytecode -->
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <version>1.18.42</version>
    <optional>true</optional>
</dependency>

<!-- MapStruct artifact: runtime cần class @Mapper, @Mapping, MappingTarget... -->
<dependency>
    <groupId>org.mapstruct</groupId>
    <artifactId>mapstruct</artifactId>
    <version>1.6.3</version>
</dependency>

<!-- MapStruct processor: chỉ chạy lúc compile để sinh *MapperImpl, runtime không dùng -->
<dependency>
    <groupId>org.mapstruct</groupId>
    <artifactId>mapstruct-processor</artifactId>
    <version>1.6.3</version>
    <scope>provided</scope>
</dependency>

<!-- Bridge: cho MapStruct processor "thấy" được getter/setter mà Lombok generate -->
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok-mapstruct-binding</artifactId>
    <version>0.2.0</version>
    <scope>provided</scope>
</dependency>
```

### `lombok-mapstruct-binding` — tại sao BẮT BUỘC?

Khi dùng Lombok + MapStruct chung, không có dependency này thì:
1. javac đọc `User.java`
2. Annotation Processing chạy theo round:
   - Lombok processor: thêm getter/setter vào AST
   - MapStruct processor: đọc getter/setter để sinh `UserMapperImpl`

Vấn đề: ở **round 1**, Lombok chưa kịp "commit" getter/setter vào model — MapStruct sẽ **không nhìn thấy** getter/setter → sinh ra `UserMapperImpl` rỗng (method trả về `null`) → bug âm thầm, không có lỗi compile.

`lombok-mapstruct-binding` là **bridge plugin** báo cho MapStruct biết "đợi Lombok xử lý xong rồi mới đọc model" → đảm bảo MapStruct thấy đầy đủ getter/setter.

→ Đây là lỗi "kinh điển" mà người mới dùng Lombok+MapStruct hay mắc.

### Cấu hình `maven-compiler-plugin`

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <source>21</source>
        <target>21</target>
        <annotationProcessorPaths>
            <path>
                <groupId>org.projectlombok</groupId>
                <artifactId>lombok</artifactId>
                <version>1.18.42</version>
            </path>
            <path>
                <groupId>org.projectlombok</groupId>
                <artifactId>lombok-mapstruct-binding</artifactId>
                <version>0.2.0</version>
            </path>
            <path>
                <groupId>org.mapstruct</groupId>
                <artifactId>mapstruct-processor</artifactId>
                <version>1.6.3</version>
            </path>
        </annotationProcessorPaths>
        <compilerArgs>
            <arg>-Amapstruct.defaultComponentModel=spring</arg>
            <arg>-Amapstruct.unmappedTargetPolicy=ERROR</arg>
        </compilerArgs>
    </configuration>
</plugin>
```

Giải thích 2 compiler args:
- `mapstruct.defaultComponentModel=spring` → mọi `@Mapper` mặc định là Spring Bean (có thể `@Autowired` hoặc inject qua constructor) — không cần ghi `@Mapper(componentModel = "spring")` trên từng interface.
- `mapstruct.unmappedTargetPolicy=ERROR` → nếu DTO có field mà MapStruct không biết map từ đâu, **build fail** thay vì silent set null. Đây là safety net cực mạnh — bug "quên map field" sẽ bị phát hiện ngay lúc compile.

## 6. Q&A — Khái niệm Maven scope

### Câu hỏi của học viên: "trong scope là provided bạn đang để là 'runtime KHÔNG đóng vào JAR', điều này có nghĩa là gì nhỉ"

### Trả lời (verbatim từ buổi học)

#### "Đóng vào JAR" nghĩa là gì?

Khi bạn chạy `./mvnw package`, Maven sẽ:
1. Compile code → `.class`
2. **Gom tất cả `.class` + dependencies cần ở runtime** vào 1 file `.jar`
3. File JAR cuối nằm ở `target/api-0.0.1-SNAPSHOT.jar`

Với Spring Boot, đây là **fat JAR** (còn gọi uber JAR / executable JAR) — chứa **toàn bộ thư viện** bên trong, để chỉ cần `java -jar api.jar` là chạy được, không cần cài gì thêm.

```
target/api-0.0.1-SNAPSHOT.jar
├── BOOT-INF/
│   ├── classes/           ← code của bạn
│   │   └── com/ecommerce/api/...
│   └── lib/               ← TẤT CẢ dependency JAR
│       ├── spring-core-6.x.jar
│       ├── hibernate-core-6.x.jar
│       ├── mysql-connector-j-8.x.jar
│       ├── tika-core-2.9.2.jar
│       └── ...
├── META-INF/
└── org/springframework/boot/loader/  ← classloader của Spring Boot
```

→ "Đóng vào JAR" = **dependency JAR đó nằm trong `BOOT-INF/lib/`** của file JAR cuối.

#### `scope=provided` → KHÔNG đóng vào JAR

Có nghĩa: dependency đó **KHÔNG xuất hiện trong `BOOT-INF/lib/`** của fat JAR cuối.

Tại sao Maven cho phép vậy? Có 2 nhóm trường hợp:

**Trường hợp 1 — Container đã cung cấp sẵn**
Lịch sử: ngày xưa dự án Java thường deploy lên Tomcat/WildFly dạng `.war`. Tomcat đã có sẵn `servlet-api.jar` trong thư mục lib. Nếu đóng `servlet-api.jar` vào file `.war` → **conflict 2 version** với Tomcat → lỗi `LinkageError`.
→ Giải pháp: khai báo `servlet-api` là `scope=provided` → "tao cần nó lúc compile để gọi `HttpServletRequest`, nhưng đừng đóng vào, container tự lo".
→ **provided** = "tao tin tưởng môi trường runtime đã cung cấp sẵn".

**Trường hợp 2 — Chỉ cần lúc build, runtime không dùng**
Đây là case của **`mapstruct-processor`**. Processor là plugin javac, chạy lúc compile để **sinh ra `UserMapperImpl.java`**. Sau khi sinh xong → nhiệm vụ kết thúc. Lúc app chạy, JVM chỉ cần `UserMapperImpl.class` (đã compile rồi), KHÔNG cần processor nữa.

→ Nếu để `scope=compile` → processor JAR (vài MB) bị nhồi vào fat JAR vô ích → app size phình to, startup chậm hơn.

```
Compile time:                      Runtime:
┌──────────────────────┐          ┌────────────────────┐
│ mapstruct-processor  │  ──×──>  │  KHÔNG cần         │
│ (sinh code)          │          │  (code đã sinh xong) │
└──────────────────────┘          └────────────────────┘
```

#### Demo trực quan

```bash
./mvnw package -DskipTests
unzip -l target/api-0.0.1-SNAPSHOT.jar | grep -E "(lombok|mapstruct)"
```

Sẽ thấy:
```
BOOT-INF/lib/mapstruct-1.6.3.jar              ← CÓ (scope=compile)
                                               ← KHÔNG có lombok (optional=true)
                                               ← KHÔNG có mapstruct-processor (provided)
```

#### So sánh `scope=provided` với `optional=true`

Cùng kết quả "không vào JAR cuối", nhưng **lý do khác**:

| | `scope=provided` | `optional=true` |
|---|---|---|
| Nghĩa | "Môi trường runtime cung cấp" | "Tao dùng nội bộ, ai dùng artifact của tao thì không cần" |
| Runtime | Maven kỳ vọng container có sẵn | Maven KHÔNG load (Lombok đã nướng vào bytecode rồi) |
| Transitive | Không transitive sang downstream | Không transitive sang downstream |
| Use case điển hình | servlet-api, annotation processor | Lombok, JaCoCo |

→ Với Lombok đặc biệt, dùng cái nào cũng chạy được, nhưng `optional=true` **biểu đạt ý đồ chuẩn hơn**: "Lombok không phải requirement của runtime, vì sau compile nó biến mất hoàn toàn khỏi bytecode".

#### So sánh với Node.js

| Node.js (`package.json`) | Maven (`pom.xml`) |
|---|---|
| `dependencies` | `<scope>compile</scope>` (default) |
| `devDependencies` | `<scope>provided</scope>` hoặc `<scope>test</scope>` |
| `peerDependencies` | gần với `<optional>true</optional>` |

Cùng tư tưởng: **"thứ chỉ dùng lúc build/develop thì không phát hành"**.

#### Tóm tắt

> `scope=provided` = "Maven, mày cần thằng này để giúp tao compile (cho IDE autocomplete, cho javac chạy processor), nhưng đừng đóng nó vào JAR cuối — runtime không cần."

## 7. Q&A — Cách tự tìm config chuẩn

### Câu hỏi của học viên: "với lại là một người mới như tôi, làm sao để tôi biết được các làm như nào cho chuẩn giống bạn"

### Trả lời (verbatim — kinh nghiệm thực tế của senior dev)

**1. Đọc official docs trước, không Google trước**
- Lombok: https://projectlombok.org/setup/maven
- MapStruct: https://mapstruct.org/documentation/installation/

Mọi thư viện tử tế đều có **section "Setup"** hoặc **"Installation"** với snippet chuẩn. Copy từ đó **luôn đúng nhất**, vì maintainer chính viết.

→ Câu chuyện `lombok-mapstruct-binding` — chính xác có trong MapStruct docs phần "Lombok integration". Không đọc docs sẽ không biết.

**2. Dùng Spring Initializr làm benchmark**
https://start.spring.io → tích Lombok → click "Explore" → xem `pom.xml` mẫu.
→ Spring team đã chọn sẵn scope `optional` cho Lombok. Đây là **ground truth** — không cần đoán.

**3. mvnrepository.com — không chỉ để tìm version**
Khi search 1 artifact, bên cạnh tab "Files" có tab **"Usage"** — hiển thị **top open-source project nào đang dùng dependency này**, click vào pom của họ để xem cách họ config.
→ Cách này hiếm beginner biết, nhưng cực mạnh: thay vì đọc StackOverflow câu trả lời 5 năm trước, bạn xem trực tiếp **production code** của Apache, Spring, Netflix...

**4. Hiểu 3 khái niệm scope/optional/processor một lần cho xong**

| Khái niệm | Ý nghĩa | Khi nào dùng |
|---|---|---|
| `scope=compile` (default) | Có ở compile + test + runtime | Library dùng trong code business |
| `scope=runtime` | Chỉ runtime, không compile | JDBC driver (mysql-connector), không import class của nó |
| `scope=provided` | Compile + test, runtime KHÔNG đóng vào JAR | Thứ container tự cung cấp (servlet-api), hoặc annotation processor |
| `scope=test` | Chỉ test | JUnit, Mockito |
| `optional=true` | Có lúc compile, KHÔNG leak sang dự án khác dùng artifact này | Lombok (downstream không cần biết) |

→ Học 1 lần, hiểu mọi pom.xml sau này.

**5. Khi sai, đọc stacktrace từ trên xuống**
Maven báo lỗi cực dài, đa số beginner scroll xuống cuối tìm "BUILD FAILURE" rồi copy error message lên Google. Thực tế: cause thật ở **vài dòng đầu** của stacktrace, ví dụ `Could not resolve dependencies for project ...`. Đọc đủ → 80% lỗi tự fix được không cần Google.

**6. Nguyên tắc lâu dài**
> **Đừng copy config bạn không hiểu.** Mỗi tag XML phải biết tại sao có. Nếu copy mà không hiểu, sau này đổi version sẽ break không biết fix.

Cụ thể: sau buổi học này, học viên nên **tự giải thích được**:
- Vì sao Lombok cần `<optional>true</optional>` mà không phải `<scope>provided</scope>`?
- Vì sao MapStruct cần khai báo processor 2 lần (1 lần ở `<dependencies>`, 1 lần ở `<annotationProcessorPaths>`)?
- Vì sao thứ tự processor (Lombok → binding → MapStruct) lại quan trọng?

Nếu trả lời được → đã hấp thụ đủ. Không trả lời được → đọc lại docs trước khi tiếp tục.
