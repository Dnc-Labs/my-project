## 1. So sánh mô hình xử lý: Node.js vs Spring Boot

### Node.js — Single-threaded + Event Loop
- Chỉ có **1 thread chính** xử lý tất cả request
- Khi gặp I/O (đọc file, query DB...), Node.js **không chờ** mà đẩy xuống OS/libuv, rồi tiếp tục xử lý request khác
- Khi I/O xong, callback/promise được đưa vào event loop để xử lý tiếp
- Nếu viết code CPU-intensive (vòng lặp nặng), nó **block cả server**

### Spring Boot — Multi-threaded (Tomcat)
- Mỗi request đến, Tomcat **lấy 1 thread từ thread pool** để xử lý
- Thread đó chạy xuyên suốt từ Controller → Service → Repository → trả response
- Thread đó **bị block** khi chờ I/O — nhưng **không sao**, vì các request khác đang chạy trên thread khác
- Mặc định Tomcat có **200 threads** → xử lý 200 request đồng thời

### Bảng so sánh

| | Node.js | Spring Boot |
|---|---|---|
| Mô hình | 1 thread + event loop | Thread pool (200 threads mặc định) |
| Xử lý request | Tất cả trên 1 thread, non-blocking I/O | Mỗi request 1 thread, blocking I/O |
| CPU-intensive | Block cả server | Chỉ block 1 thread |
| Concurrency | Event loop + async/await | Multiple threads chạy song song |

**Ví dụ hình dung:**
- **Node.js** = nhà hàng có **1 phục vụ siêu nhanh**. Nhận order bàn 1, đưa vào bếp, **không đứng chờ** mà chạy sang nhận order bàn 2 luôn.
- **Spring Boot** = nhà hàng có **200 phục vụ**. Mỗi khách vào, 1 phục vụ **đi theo từ đầu đến cuối**.

## 2. Thread là gì?

Thread là **một luồng thực thi độc lập** trong cùng một process (tiến trình). Mỗi thread có:
- **Stack riêng** — biến cục bộ, lời gọi hàm
- **Dùng chung Heap** — các object, dữ liệu chung của ứng dụng

Các thread chạy **song song thật sự** trên các CPU core khác nhau (nếu máy có nhiều core). Node.js **không chạy song song**, chỉ chuyển đổi nhanh giữa các task trên 1 thread.

```
Process (ứng dụng Spring Boot)
├── Thread-1: đang xử lý GET /products     ← chạy trên CPU core 1
├── Thread-2: đang xử lý POST /orders      ← chạy trên CPU core 2
├── Thread-3: đang xử lý GET /users/1      ← chạy trên CPU core 3
└── ... (lên đến 200 threads)
```

## 3. Race Condition

Vì các thread **dùng chung Heap**, khi nhiều thread cùng đọc/ghi một biến sẽ xảy ra **race condition**.

### Ví dụ: trừ tồn kho

```java
int stock = 10;
// Thread-1: Khách A mua 1 sản phẩm
// Thread-2: Khách B mua 1 sản phẩm
// Cả 2 chạy ĐỒNG THỜI: stock = stock - 1
```

Phép `stock = stock - 1` CPU thực hiện qua **3 bước**:
```
Bước 1: READ  — đọc giá trị stock từ bộ nhớ
Bước 2: CALC  — tính stock - 1
Bước 3: WRITE — ghi giá trị mới vào bộ nhớ
```

**Trường hợp đúng (stock = 8):**
```
Thread-1: READ stock = 10
Thread-1: CALC 10 - 1 = 9
Thread-1: WRITE stock = 9
                                Thread-2: READ stock = 9
                                Thread-2: CALC 9 - 1 = 8
                                Thread-2: WRITE stock = 8  ✅
```

**Trường hợp lỗi (stock = 9):**
```
Thread-1: READ stock = 10
                                Thread-2: READ stock = 10  ← cũng thấy 10!
Thread-1: CALC 10 - 1 = 9
                                Thread-2: CALC 10 - 1 = 9
Thread-1: WRITE stock = 9
                                Thread-2: WRITE stock = 9  ❌ mất 1 lần mua!
```

### Cách giải quyết

| Cách | Mô tả |
|---|---|
| `synchronized` | Khóa method/block, chỉ 1 thread vào tại 1 thời điểm |
| `AtomicInteger` | Biến đặc biệt, phép toán atomic (không bị xen kẽ) |
| `ReentrantLock` | Khóa linh hoạt hơn synchronized |
| `volatile` | Đảm bảo thread luôn đọc giá trị mới nhất từ bộ nhớ chính |
| Optimistic Locking (`@Version`) | Dùng trong JPA — sẽ học ở giai đoạn 4 |

## 4. Thread, Runnable, Callable

### Runnable — công việc không trả kết quả (giống callback)

```java
Runnable task = () -> {
    System.out.println("Chạy trên thread: " + Thread.currentThread().getName());
};
Thread thread = new Thread(task);
thread.start(); // phải dùng start(), KHÔNG phải run()
```

### Callable — công việc có trả kết quả (giống Promise)

```java
Callable<Integer> task = () -> {
    Thread.sleep(2000);
    return 42;
};
```

### So sánh

| | Runnable | Callable |
|---|---|---|
| Return | `void` — không trả gì | Trả về giá trị |
| Exception | Không throw checked exception | Có thể throw Exception |
| Node.js tương đương | `setTimeout(callback)` | `Promise` |

## 5. ExecutorService — Thread Pool

Trong thực tế **không bao giờ** tự tạo `new Thread()` cho mỗi task. Thay vào đó dùng **Thread Pool** — tạo sẵn một nhóm thread, rồi tái sử dụng:

```java
// Tạo pool có 3 threads
ExecutorService pool = Executors.newFixedThreadPool(3);

// Gửi Runnable — không cần kết quả
pool.submit(() -> {
    System.out.println("Task 1 - " + Thread.currentThread().getName());
});

// Gửi Callable — cần kết quả
Future<Integer> future = pool.submit(() -> {
    Thread.sleep(2000);
    return 42;
});

// Lấy kết quả (block cho đến khi có — giống await)
Integer result = future.get();

// Tắt pool khi xong
pool.shutdown();
```

**`Future`** chính là **Promise** của Java:
- `future.get()` giống `await promise`
- `future.isDone()` — kiểm tra đã xong chưa

### Tóm tắt mapping

```
Runnable        → công việc không trả kết quả    → giống callback
Callable        → công việc có trả kết quả       → giống Promise
Future          → "lời hứa" sẽ có kết quả       → giống Promise object
ExecutorService → thread pool quản lý threads    → giống worker pool
```

## 6. Lambda và effectively final

Lambda trong Java chỉ capture được biến **effectively final** (không bị gán lại):

```java
// ❌ Lỗi compile — i thay đổi mỗi vòng lặp
for (int i = 0; i < 5; i++) {
    pool.submit(() -> System.out.println("Task " + i));
}

// ✅ Tạo biến final để lambda capture
for (int i = 0; i < 5; i++) {
    int taskNumber = i; // gán 1 lần duy nhất → effectively final
    pool.submit(() -> System.out.println("Task " + taskNumber));
}
```

**Lý do:** Lambda copy giá trị biến. Nếu biến thay đổi sau khi copy, giá trị trong lambda sẽ sai. Java cấm luôn từ lúc compile để tránh bug.

**So sánh JS:** Giống vấn đề `var` vs `let`:
```javascript
// ❌ var — 1 biến dùng chung → in ra 5,5,5,5,5
for (var i = 0; i < 5; i++) {
    setTimeout(() => console.log(i));
}
// ✅ let — mỗi vòng biến riêng → in ra 0,1,2,3,4
for (let i = 0; i < 5; i++) {
    setTimeout(() => console.log(i));
}
```

> **Lưu ý:** Đây là quy tắc của **lambda**, không phải của thread. Lambda ở bất kỳ đâu đều yêu cầu effectively final.

## 7. Bài thực hành: ThreadDemo

Tạo `ExecutorService` với 3 threads, submit 5 Callable task, lấy kết quả qua Future:

```java
public class ThreadDemo {
    public static void main(String[] args) {
        ExecutorService pool = Executors.newFixedThreadPool(3);
        List<Future<String>> futures = new ArrayList<>();

        // Submit 5 task
        for (int i = 0; i <= 4; i++) {
            int taskNumber = i;
            Future<String> future = pool.submit(() -> {
                try {
                    Thread.sleep(2000);
                    System.out.println("Task " + taskNumber + " - " + Thread.currentThread().getName());
                    return "Kết quả task " + taskNumber;
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            });
            futures.add(future);
        }

        // Lấy kết quả SAU khi submit hết (để các task chạy song song)
        for (Future<String> future : futures) {
            try {
                String result = future.get();
                System.out.println(result);
            } catch (InterruptedException | ExecutionException e) {
                throw new RuntimeException(e);
            }
        }
        pool.shutdown();
    }
}
```

**Lưu ý quan trọng:** Phải tách 2 vòng lặp (submit và get). Nếu gọi `future.get()` ngay trong vòng submit → các task chạy tuần tự (block chờ từng task xong mới submit tiếp).

## 8. `@Async` trong Spring Boot

Thay vì tự tạo `ExecutorService`, Spring cho phép đánh dấu method để tự động chạy trên thread khác:

```java
@Service
public class EmailService {
    @Async
    public void sendEmail(String to) {
        Thread.sleep(3000);
        System.out.println("Đã gửi email đến " + to);
    }
}
```

```java
@RestController
public class OrderController {
    @Autowired
    private EmailService emailService;

    @PostMapping("/orders")
    public String createOrder() {
        System.out.println("Đơn hàng đã tạo");
        emailService.sendEmail("khach@gmail.com"); // chạy trên thread khác, không block
        return "Đặt hàng thành công"; // trả response NGAY
    }
}
```

### 2 điều kiện bắt buộc

**1. Phải bật `@EnableAsync`:**
```java
@SpringBootApplication
@EnableAsync
public class ApiApplication { ... }
```

**2. Phải gọi từ BÊN NGOÀI class:**
```java
// ❌ Gọi nội bộ → @Async bị bỏ qua
@Service
public class OrderService {
    @Async
    public void sendEmail() { ... }

    public void createOrder() {
        sendEmail(); // gọi this.sendEmail() → bypass proxy
    }
}

// ✅ Gọi từ class khác → @Async hoạt động
@RestController
public class OrderController {
    @Autowired
    private EmailService emailService;

    public void createOrder() {
        emailService.sendEmail(); // gọi qua proxy → @Async hoạt động
    }
}
```

**Lý do:** `@Async` hoạt động qua **Spring Proxy**. Khi `@Autowired`, Spring đưa proxy bọc ngoài. Proxy chặn lời gọi, thấy `@Async` thì đẩy sang thread khác. Gọi nội bộ (`this.method()`) bypass proxy → `@Async` không tác dụng.

### Return type

| Trường hợp | Return type | Ví dụ |
|---|---|---|
| Không cần kết quả | `void` | Gửi email, ghi log |
| Cần kết quả | `CompletableFuture<T>` | Tính báo cáo, gọi API |

```java
@Async
public CompletableFuture<String> calculateReport() {
    Thread.sleep(5000);
    return CompletableFuture.completedFuture("Báo cáo hoàn thành");
}
```

### Tùy chỉnh Thread Pool cho production

```java
@Configuration
@EnableAsync
public class AsyncConfig {
    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);       // số thread tối thiểu
        executor.setMaxPoolSize(10);       // số thread tối đa
        executor.setQueueCapacity(25);     // hàng đợi khi tất cả thread bận
        executor.setThreadNamePrefix("Async-");
        executor.initialize();
        return executor;
    }
}

// Chỉ định pool
@Async("taskExecutor")
public void sendEmail(String to) { ... }
```

## 9. Deep dive: Spring Proxy

### Q: Proxy chỉ đúng khi dùng `@Autowired` hay đúng cho tất cả cách inject?

**Đúng cho TẤT CẢ các cách inject** — `@Autowired`, constructor injection, setter injection đều nhận proxy.

Vì cơ chế proxy xảy ra ở **tầng Spring Container**, không phải ở tầng inject:

```
1. Spring khởi động → tạo EmailService object thật
2. Thấy class có @Async → bọc proxy bên ngoài
3. Đăng ký proxy này vào Container (thay vì object thật)
4. Khi ai đó cần EmailService → Container đưa proxy ra
```

Bước 4 mới là inject — lúc đó dù inject bằng cách nào thì Container **chỉ có proxy để đưa**, kết quả giống nhau.

**Proxy được tạo trước, inject xảy ra sau.** Cách inject chỉ là cách "nhận hàng", còn "hàng" đã là proxy từ trước rồi.

### Q: Proxy thực chất là gì? Spring làm gì dưới code?

Proxy là một **object giả mạo**, có cùng interface/class với object thật, nhưng **chen thêm logic trước/sau** khi gọi method.

**Liên hệ Node.js — proxy thủ công:**

```javascript
// Object thật
const emailService = {
    sendEmail(to) { console.log("Gửi email đến " + to); }
};

// "Proxy" thủ công
const emailServiceProxy = {
    sendEmail(to) {
        console.log(">>> Chuyển sang worker thread");
        setTimeout(() => emailService.sendEmail(to), 0); // chạy async
        console.log(">>> Trả về ngay");
    }
};
```

Spring cũng làm y vậy, nhưng **tự động generate lúc runtime**:

```java
// Spring TỰ ĐỘNG sinh ra class này — bạn không bao giờ thấy code này
public class EmailService$$Proxy extends EmailService {
    private EmailService realObject;
    private Executor threadPool;

    @Override
    public void sendEmail(String to) {
        // Đẩy vào thread pool thay vì gọi trực tiếp
        threadPool.execute(() -> realObject.sendEmail(to));
        // return ngay, không chờ
    }
}
```

Container đăng ký proxy:
```
Container registry:
  "emailService" → EmailService$$Proxy (KHÔNG PHẢI EmailService thật)
```

### Tại sao gọi nội bộ bypass proxy?

```
Bên ngoài gọi vào:
  OrderController → EmailService$$Proxy.sendEmail() → proxy chặn → thread khác ✅

Bên trong gọi nội bộ:
  OrderService.createOrder() → this.sendEmail() → gọi thẳng method → cùng thread ❌
                                ↑ this = object thật, không phải proxy
```

### Kỹ thuật tạo proxy

| Kỹ thuật | Khi nào dùng |
|---|---|
| **JDK Dynamic Proxy** | Class **implement interface** — tạo proxy implement cùng interface |
| **CGLIB** | Class **không có interface** — tạo proxy bằng cách **extends class gốc** |

Có thể chứng minh proxy tồn tại:

```java
@GetMapping("/debug")
public String debug() {
    return emailService.getClass().getName();
    // Kết quả: com.ecommerce.api.service.EmailService$$SpringCGLIB$$0
}
```

### Mỗi annotation thì proxy chen logic gì?

| Annotation | Proxy chen ngang làm gì |
|---|---|
| `@Async` | Đẩy sang thread khác → rồi gọi logic gốc |
| `@Transactional` | Mở transaction **trước** → gọi logic gốc → thành công thì commit, lỗi thì rollback |
| `@Cacheable` | Kiểm tra cache **trước** → có thì trả luôn, không có thì gọi logic gốc → lưu cache |

Bean bình thường không có các annotation này thì **không bị bọc proxy**, nhận object thật.

### Q: Chúng ta có học AOP không?

Cơ chế proxy này chính là nền tảng của **AOP (Aspect-Oriented Programming)** — "lập trình hướng khía cạnh". Mỗi khi dùng `@Async`, `@Transactional`, `@Cacheable`... đều là AOP đang hoạt động ngầm.

Theo ROADMAP, không có mục AOP riêng. Nhưng đến **Giai đoạn 7** khi làm Logging, có thể sẽ giới thiệu cách viết **custom AOP** để log tự động cho tất cả API. Tạm thời hiểu proxy hoạt động thế nào là đủ dùng.

> **Tóm lại:** Proxy giống **middleware** trong Express — chen thêm xử lý trước/sau mà không thay đổi logic gốc. Khác ở chỗ middleware chặn HTTP request, còn proxy chặn **lời gọi method**.

### OrderController — gọi song song 3 service bằng `CompletableFuture.allOf()`

```java
@GetMapping("/{id}/detail")
public ResponseEntity<OrderDetailResponse> getOrder(@PathVariable String id) {
    long start = System.currentTimeMillis();

    CompletableFuture<String> orderFuture = orderService.getOrder(id);
    CompletableFuture<Integer> feeFuture = shippingService.calculateFee(id);
    CompletableFuture<Boolean> stockFuture = inventoryService.checkStock(id);
    CompletableFuture.allOf(orderFuture, feeFuture, stockFuture).join();

    String order = orderFuture.join();
    Integer fee = feeFuture.join();
    Boolean inStock = stockFuture.join();

    long duration = System.currentTimeMillis() - start;
    logger.info("Tổng thời gian: {} ms (song song thay vì ~4000ms tuần tự)", duration);

    return ResponseEntity.ok(new OrderDetailResponse(order, fee, inStock));
}
```

**Kết quả:** 3 service chạy song song trên 3 thread khác nhau, tổng thời gian ~2000ms (thay vì ~4000ms nếu tuần tự).

## 11. Thread-safety & `synchronized` chi tiết

### Vấn đề thực tế

Khi `InventoryService` có biến `stock` (state), nhiều request đến cùng lúc → nhiều thread cùng gọi `decreaseStock()` → race condition:

```java
@Service
public class InventoryService {
    private int stock = 100;

    public void decreaseStock(int quantity) {
        if (stock >= quantity) {
            stock -= quantity;  // ← race condition ở đây!
        }
    }
}
```

`stock -= quantity` thực chất là 3 bước (READ → CALC → WRITE). Nếu 2 thread xen kẽ giữa các bước → mất dữ liệu.

### Giải pháp 1: `synchronized`

`synchronized` giống như **khóa cửa phòng** — chỉ 1 thread được vào tại 1 thời điểm, các thread khác phải xếp hàng chờ.

**Cách 1: Khóa cả method**
```java
public synchronized void decreaseStock(int quantity) {
    if (stock >= quantity) {
        stock -= quantity;
    }
}
```

**Cách 2: Khóa 1 đoạn code (synchronized block)** — linh hoạt hơn
```java
public void decreaseStock(int quantity) {
    synchronized (this) {
        if (stock >= quantity) {
            stock -= quantity;
        }
    }
    // code ngoài block này thì các thread vẫn chạy song song
}
```

**So sánh với Node.js:** Bên Node.js không cần lo race condition trong code synchronous vì chỉ có 1 thread. Nhưng nếu từng gặp bug khi 2 request cùng đọc DB rồi update lại (kiểu check-then-act), thì đó cũng chính là race condition — chỉ là xảy ra ở tầng database thay vì tầng memory.

### Giải pháp 2: `AtomicInteger`

Java cung cấp các class **Atomic** — thực hiện phép toán trong 1 bước không thể bị xen kẽ:

```java
private AtomicInteger stock = new AtomicInteger(100);

public void decreaseStock(int quantity) {
    stock.addAndGet(-quantity); // atomic — không cần synchronized
}
```

Nhưng nếu cần **check rồi mới trừ** (check-then-act), `AtomicInteger` đơn giản không đủ — vì check và trừ vẫn là 2 bước tách rời. Lúc đó phải dùng `compareAndSet` hoặc quay lại `synchronized`.

### Giải pháp 3: `ReentrantLock`

Giống `synchronized` nhưng linh hoạt hơn — có thể tryLock (thử khóa, không chờ), lock timeout, v.v.

```java
private final ReentrantLock lock = new ReentrantLock();

public void decreaseStock(int quantity) {
    lock.lock();
    try {
        if (stock >= quantity) {
            stock -= quantity;
        }
    } finally {
        lock.unlock(); // PHẢI unlock trong finally
    }
}
```

### Khi nào dùng cái nào?

| Giải pháp | Dùng khi |
|---|---|
| `synchronized` | Đơn giản, đủ dùng cho hầu hết trường hợp |
| `AtomicInteger` | Chỉ cần phép toán đơn lẻ (tăng/giảm/set) |
| `ReentrantLock` | Cần tryLock, timeout, hoặc lock/unlock ở khác method |
| `@Version` (Optimistic Locking) | Trong JPA — sẽ học ở Giai đoạn 4 |

### Best practice: Service nên stateless

Trong Spring Boot, Bean mặc định là **Singleton** — tất cả request dùng chung 1 instance. Nếu service có **state** (biến instance như `stock`), nhiều thread cùng truy cập → race condition.

**Best practice:** Service nên là **stateless** (không giữ state). Dữ liệu nên nằm ở database. Khi cần xử lý concurrent ở tầng DB thì dùng **Optimistic/Pessimistic Locking** (sẽ học ở Giai đoạn 4).

## 12. Bài thực hành: RaceConditionDemo

Tạo demo chứng minh race condition xảy ra và fix bằng `synchronized`:

```java
public class RaceConditionDemo {
    private int stock = 100;

    // Lần 1: Không có synchronized → stock ra kết quả SAI (ví dụ: 1 thay vì 0)
    // Lần 2: Thêm synchronized → stock luôn ra 0
    public void decreaseStock(int quantity) {
        if (this.stock >= quantity) {
            this.stock -= quantity;
        }
        System.out.println("thread name :" + Thread.currentThread().getName() + ", Current Stock " + this.stock);
    }

    public int getStock() {
        return stock;
    }

    public static void main(String[] args) throws InterruptedException {
        RaceConditionDemo demo = new RaceConditionDemo();
        ExecutorService pool = Executors.newFixedThreadPool(10);

        for (int i = 0; i < 100; i++) {
            pool.submit(() -> {
                demo.decreaseStock(1);
            });
        }

        pool.shutdown();
        pool.awaitTermination(10, TimeUnit.SECONDS);

        System.out.println("=== Kết quả ===");
        System.out.println("Stock còn lại: " + demo.getStock());
        System.out.println("Kết quả đúng phải là: 0");
    }
}
```

**Kết quả thực tế:**
- Không có `synchronized` → stock = 1 (race condition xảy ra, mất 1 lần trừ)
- Có `synchronized` → stock = 0 (chạy 2 lần đều đúng)

**Lưu ý:** `pool.shutdown()` chỉ báo pool ngừng nhận task mới, nhưng không chờ task đang chạy hoàn thành. Phải thêm `pool.awaitTermination()` để chờ tất cả task xong rồi mới in kết quả.

## 13. Câu hỏi phát sinh

### Q: `synchronized` giải quyết được vấn đề, nhưng có nhược điểm gì?

**Nhược điểm về performance.** `synchronized` buộc các thread chạy **tuần tự** thay vì song song → chậm hơn. Đó là lý do không nên synchronized bừa bãi, chỉ khóa đoạn code thật sự cần thiết.

Đây là sự **đánh đổi performance lấy correctness**.

### Q: Nếu tồn kho nằm trong database thay vì biến, thì `synchronized` còn hoạt động không?

**Chỉ đúng khi chạy 1 server (1 JVM).** 

Trong production thực tế, ứng dụng thường chạy **nhiều instance** (2-3 server cùng chạy app giống nhau, đứng sau load balancer):

```
Load Balancer
├── Server A (JVM 1) → synchronized khóa trong JVM này
├── Server B (JVM 2) → synchronized khóa trong JVM này
└── Server C (JVM 3) → synchronized khóa trong JVM này
```

Request khách A vào Server A, request khách B vào Server B — **2 synchronized khác nhau**, không biết nhau → race condition vẫn xảy ra ở tầng database!

**Vì vậy**, khi dữ liệu nằm trong database, phải dùng **cơ chế khóa ở tầng database**:
- **Optimistic Locking** (`@Version`) — kiểm tra version trước khi update, nếu bị thay đổi rồi thì báo lỗi
- **Pessimistic Locking** — khóa row trong DB, thread khác phải chờ

**Tóm lại:** `synchronized` → dùng cho shared state **trong memory, 1 JVM**. Database → dùng locking ở tầng DB.

### Q: Khi đánh dấu method là `synchronized`, JVM kiểm tra có thread nào đang chạy method đó không, nếu có thì chờ đúng không?

**Đúng về ý tưởng, nhưng cần làm rõ:** `synchronized` **khóa theo object, không phải theo method**.

Khi viết:

```java
public synchronized void decreaseStock(int quantity) { ... }
public synchronized void addStock(int quantity) { ... }
```

Thực chất nó khóa trên `this` (object hiện tại). Nghĩa là:

- Thread A đang chạy `decreaseStock()` → Thread B gọi `addStock()` trên **cùng object** → **cũng phải chờ**
- Vì cả 2 method đều khóa trên cùng `this`

```
Thread A → this.decreaseStock()  → lấy khóa "this" ✅ đang chạy
Thread B → this.addStock()       → cần khóa "this" ⏳ chờ (dù khác method!)
Thread C → this.decreaseStock()  → cần khóa "this" ⏳ chờ
```

Nếu là **2 object khác nhau** thì không ảnh hưởng nhau:

```
Thread A → objectA.decreaseStock() → khóa objectA ✅
Thread B → objectB.decreaseStock() → khóa objectB ✅ (chạy song song)
```

Nhưng vì Spring Bean mặc định là **Singleton** — chỉ có 1 instance — nên tất cả request đều dùng chung 1 object → `synchronized` có tác dụng với mọi thread.

**Tóm lại:** JVM không kiểm tra "method này có thread nào đang chạy không", mà kiểm tra "**object này** có thread nào đang giữ khóa không". Mọi `synchronized` method trên cùng 1 object đều dùng chung 1 khóa.

| Chủ đề | Nội dung chính |
|---|---|
| Node.js vs Spring Boot | Single-thread + event loop vs Multi-thread pool (200 threads) |

| Future | "Lời hứa" có kết quả (giống Promise object) |

| Lambda & effectively final | Lambda chỉ capture biến không bị gán lại |

| Spring Proxy | Object giả mạo, chen logic trước/sau khi gọi method (giống middleware) |
| `synchronized` | Khóa theo object, chỉ 1 thread vào tại 1 thời điểm |
| `AtomicInteger` | Phép toán atomic, không cần khóa |
| `ReentrantLock` | Khóa linh hoạt hơn synchronized |
| Thread-safety best practice | Service nên stateless, dữ liệu ở DB, dùng DB locking |
