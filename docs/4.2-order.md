## 2. Chốt thiết kế (Q&A)

**1. Trừ stock theo product hay variant?** → giống Cart: có variant trừ variant, không thì product.
**2. Đặt xong xóa giỏ?** → xóa hết giỏ. All-or-nothing: 1 item hết hàng → rollback toàn bộ, **giỏ giữ nguyên**.
**3. Snapshot `totalAmount` — sao không tính live?** → Lý do CHÍNH là **correctness/hợp đồng**: đơn là cam kết, tiền phải đóng băng đúng giá lúc mua (mai seller đổi giá, đơn cũ không được nhảy giá). Không thể tính live vì giá product hiện tại đã khác. Performance (tránh N+1 ở list) chỉ là bonus.
**4. Phân quyền status?** → CUSTOMER chỉ `CANCELLED` khi chưa SHIPPING; SELLER/ADMIN `confirm/ship/deliver` + cũng cancel được. **Quy tắc lớn: CANCEL phải HOÀN STOCK.**
**5. Chống oversell chọn cách nào?** → Pessimistic Lock.

### State machine
```
PENDING ──confirm──> CONFIRMED ──ship──> SHIPPING ──deliver──> DELIVERED
   │                     │
   └─────cancel──────────┴──> CANCELLED   (sau SHIPPING không hủy được)
```

### Luồng checkout (1 @Transactional — all-or-nothing)
```
1. lấy cart (rỗng → BusinessRuleException)
2. mỗi cartItem: lock product/variant → check stock đủ → TRỪ stock → tạo OrderItem (snapshot price)
3. tính totalAmount
4. tạo Order (PENDING)
5. xóa sạch cart
```

## 3. ⭐ Bài toán oversell (race condition)

Sản phẩm còn 1, 2 người checkout cùng lúc:
```
Thread A            Thread B
đọc stock=1         đọc stock=1
1>=1 ✓              1>=1 ✓
stock=0             stock=0
COMMIT              COMMIT      → bán 2 dù còn 1 (oversell = LOST UPDATE)
```
Gốc rễ: **read-then-write không nguyên tử**. Giữa đọc và ghi, thread khác chen vào.

## 4. 3 cách chống oversell

| Cách | Cơ chế | Ưu | Nhược | Hợp khi |
|------|--------|----|-------|---------|
| **Optimistic** (`@Version`) | check version lúc commit, lệch → `OptimisticLockException` | không khóa, throughput cao | tranh chấp cao → retry storm; phải code retry | đụng độ hiếm |
| **Pessimistic** (`FOR UPDATE`) | khóa row khi đọc, ai sau **chờ** | chắc chắn, không retry | giảm throughput, nguy cơ deadlock | tranh chấp cao, cần chắc |
| **Atomic update** (`UPDATE stock=stock-? WHERE stock>=?`) | check+trừ trong 1 SQL, DB lo nguyên tử | nhanh nhất, không deadlock, gọn | logic ở SQL, không qua entity | counter +/- |

> ⚠️ Hiểu nhầm phổ biến: "optimistic lock không chống được oversell vì think-time". SAI. Việc trừ stock xảy ra trong transaction NGẮN lúc bấm tạo đơn, server **đọc lại stock tươi** — think-time không ảnh hưởng. Optimistic **CHỐNG được** oversell (người thua nhận exception); điểm yếu thật chỉ là **retry storm khi tranh chấp cao**.

**Chọn: Pessimistic Lock** — JPA-native, chắc chắn cho hot path.

## 5. DEEP DIVE — `@Lock` syntax (Q&A nguyên văn)

### Q: Trước khi dựng skeleton thì hãy nói thêm cho tôi một chút về annotation @Lock nhé, tôi đã hiểu về cách hoạt động rồi nhưng một số syntax thì chưa nên hãy nói thêm cho tôi

```java
import org.springframework.data.jpa.repository.Lock;
import jakarta.persistence.LockModeType;

@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Optional<Product> findByIdForUpdate(@Param("id") Long id);
```

- `@Lock` của Spring Data JPA, tham số là `LockModeType` (enum JPA). Đặt trên method repository, dùng được cả `@Query` lẫn derived query.

**`LockModeType` — nhóm Pessimistic:**
| Giá trị | SQL | Ý nghĩa |
|---------|-----|---------|
| `PESSIMISTIC_WRITE` | `SELECT ... FOR UPDATE` | khóa ghi độc quyền ← **ta dùng** |
| `PESSIMISTIC_READ` | `SELECT ... FOR SHARE` | khóa chia sẻ, nhiều người đọc-khóa, không ai ghi |
| `PESSIMISTIC_FORCE_INCREMENT` | `FOR UPDATE` + bump version | |

**Nhóm Optimistic:** `OPTIMISTIC`, `OPTIMISTIC_FORCE_INCREMENT`, `NONE` (liên quan `@Version`).

**⚠️ Bắt buộc nằm trong transaction** — `FOR UPDATE` chỉ giữ khóa trong phạm vi transaction; gọi ngoài `@Transactional` là vô nghĩa.

**Lock timeout (tùy chọn):**
```java
@QueryHints({ @QueryHint(name = "jakarta.persistence.lock.timeout", value = "3000") })
```
- `0` → NOWAIT (fail ngay), `-2` → SKIP LOCKED, `>0` → chờ N ms. Hỗ trợ tùy DB (Postgres tốt; MySQL 8+). MVP không cần đặt.

> So Node.js: `@Lock(PESSIMISTIC_WRITE)` ⟺ Sequelize `findByPk(id, { lock: t.LOCK.UPDATE, transaction: t })`.

## 6. DEEP DIVE — Isolation Level vs Locking (Q&A nguyên văn)

### Q: ok đã rõ, nhưng khi đọc xong lý thuyết thì tôi lại nhớ tới một số khái niệm về transactional trong database như Read_commit hay read_uncommit, chúng có mối liên hệ gì đến phần này không, nếu là các khái niệm riêng biệt thì các giá trị này trong database giải quyết những vấn đề gì, có xử lý được bài toán này không

**2 trục khác nhau:**
- **Isolation level** = hành vi MẶC ĐỊNH: "tx của tôi THẤY gì từ tx khác" → chống bất thường khi **ĐỌC**.
- **Explicit lock** (`FOR UPDATE`/`@Version`) = chủ động giữ 1 row, nằm ĐÈ lên isolation.

**3 read phenomena:**
| Hiện tượng | Là gì |
|-----------|-------|
| Dirty read | đọc dữ liệu chưa commit |
| Non-repeatable read | đọc 1 row 2 lần ra 2 giá trị |
| Phantom read | query 2 lần ra số dòng khác |

| Isolation | Dirty | Non-repeatable | Phantom |
|-----------|:---:|:---:|:---:|
| READ UNCOMMITTED | ❌ | ❌ | ❌ |
| READ COMMITTED | ✅ | ❌ | ❌ |
| REPEATABLE READ | ✅ | ✅ | ❌* |
| SERIALIZABLE | ✅ | ✅ | ✅ |

Default: **MySQL = REPEATABLE READ**, Postgres = READ COMMITTED.

**Mấu chốt: oversell KHÔNG nằm trong 3 hiện tượng đó** — nó là bất thường thứ tư: **LOST UPDATE** (không có trong ANSI gốc). Read-modify-write đua nhau, giá trị bê vào biến Java rồi ghi đè.

**Isolation có cứu lost update?**
- READ UNCOMMITTED/COMMITTED: ❌
- REPEATABLE READ: ❌ (chống non-repeatable read, không chống "tính trên số cũ rồi set tuyệt đối")
- SERIALIZABLE: ✅ nhưng búa tạ (throughput sụp, deadlock)

**Vì sao 3 cách của ta đúng ở isolation mặc định:** `FOR UPDATE` là "current read" + khóa → tx B chờ, chạy tiếp đọc lại giá trị committed mới → đúng kể cả READ COMMITTED. Atomic update tính trong DB → không có khe hở. Optimistic phát hiện lúc commit.

> Quy tắc vàng: **lost update là bài toán GHI, không giải bằng isolation (trừ SERIALIZABLE). Dùng lock/atomic/version.**

### Q: bạn nói level isolation SERIALIZABLE sẽ khoá trong suốt thời gian transaction nên các transaction khác bị block, nhưng nếu dùng select for update thì cũng đang khoá row trong cả transaction mà, vậy 2 cái này có giống nhau không

**Giống**: cùng giữ lock tới commit (cùng họ lock-based).

**Khác — 3 điểm:**

**① Phạm vi:**
- `FOR UPDATE` → khóa ĐÚNG row chỉ định. Row/query khác chạy tự do.
- `SERIALIZABLE` → khóa MỌI row tx ĐỌC tới (InnoDB ngầm biến mọi plain SELECT thành locking read).

**② Ai bị chặn:**
- `FOR UPDATE`: chỉ ai cũng muốn ghi/FOR UPDATE cùng row. Người đọc thường (snapshot) không bị chặn.
- `SERIALIZABLE`: người đọc chặn người ghi và ngược lại, trên mọi row chạm tới.

**③ Tự động vs chủ động:**
- `FOR UPDATE` = per-query, bạn chọn chính xác row + lúc nào.
- `SERIALIZABLE` = per-transaction, bao trùm tự động.

> Ẩn dụ rạp phim: `FOR UPDATE` = đặt đúng 1 ghế cần; `SERIALIZABLE` = thuê nguyên rạp, ghế nào liếc qua cũng coi như giữ.

→ Trừ kho: `FOR UPDATE` đúng và rẻ (chặn đúng chỗ); `SERIALIZABLE` là búa tạ giết ruồi (chặn cả chỗ vô can → chậm, dễ deadlock). **Không giống nhau dù cùng họ.**

---

### Q: tôi thấy các level của isolation chỉ đang tập trung vào việc read data làm sao cho chuẩn đúng không, vậy ứng dụng của nó trong các case thực tế là như nào, có ví dụ không, và trong bài học này có chỗ nào chúng ta động đến isolation không

**Xác nhận:** isolation = độ nhất quán khi ĐỌC, không quản ghi-đè (đó là việc của lock).

**Ví dụ thực tế:**
| Level | Dùng khi | Ví dụ |
|---|---|---|
| READ UNCOMMITTED | số xấp xỉ, 0 khóa | đếm view gần đúng |
| READ COMMITTED | OLTP thường (default Postgres) | xem số dư, list sản phẩm |
| REPEATABLE READ | đọc nhiều lần trong 1 tx (default MySQL) | báo cáo/hóa đơn, **checkout** |
| SERIALIZABLE | invariant phức tạp / check-then-insert | đặt phòng/ghế (chống phantom double-booking), chuyển tiền |

**Dự án ta có động tới isolation?**
- **CHƯA BAO GIỜ set tường minh** — không chỗ nào dùng `@Transactional(isolation=...)`. Chạy ở default MySQL = REPEATABLE READ.
- Nơi isolation ngầm có lợi: `checkout`/`getMyCart` đọc nhiều row trong 1 tx → RR cho snapshot nhất quán (miễn phí).
- Bài "check-then-insert" (phantom/duplicate) ta giải bằng **UNIQUE CONSTRAINT** (`cart_items`, `product_variants`, `slug`/`sku`) + bắt `DataIntegrityViolationException` → 409 (3.2.6 Cụm 2.4), **KHÔNG** dùng SERIALIZABLE. Đây là pattern được ưa hơn trong thực chiến.
- `@Transactional(readOnly=true)` ≠ isolation — nó là hint (FlushMode + connection read-only), khác trục.

## 7. ⭐ Multi-node & stale-read gotcha (Q&A nguyên văn)

### Q: việc xử lý đồng thời này chỉ trong 1 node server thôi đúng không, nếu chạy nhiều node server thì cả 3 không xử lý được đâu đúng không

**Ngược lại: cả 3 cách ĐỀU hoạt động đa node.** Mấu chốt là lock ở TẦNG nào:
```
TẦNG JVM (synchronized, ReentrantLock)   → chỉ trong 1 JVM → NHIỀU NODE = VỠ ❌
TẦNG DATABASE (@Version, FOR UPDATE, atomic) → DB là điểm chung mọi node → ĐA NODE OK ✅
```
3 cách ta dùng đều enforce ở **database**. Dù 10 node Spring Boot, tất cả trỏ về cùng 1 DB, cùng 1 row → DB là trọng tài chung, serialize đúng bất kể request từ node nào.

> Cái KHÔNG chạy đa node là **lock trong RAM Java** (`synchronized`/`ReentrantLock`) — mỗi node có `this` riêng. → Hệ phân tán: đẩy serialize xuống tầng dùng chung (DB lock, hoặc distributed lock Redis/Zookeeper khi cần khóa thứ ngoài DB).
>
> Liên hệ 3.2.6: lúc đó học `synchronized` (tầng JVM, 1 node). Giờ lên bài thật (nhiều node) dùng tầng DB. Nâng cấp tư duy "thread-safe trong 1 process" → "correct trong hệ phân tán".

### 🐞 Gotcha: pessimistic lock + entity pre-loaded → đọc state CŨ (stale)

`checkout` load cart bằng `findWithItemsByUserId` (`@EntityGraph` fetch product+variant) + `@ManyToOne` mặc định EAGER → **product/variant đã ở persistence context TRƯỚC khi** trừ stock.

> Hành vi Hibernate ít biết: nếu entity ĐÃ ở persistence context, query có lock sẽ **khóa được row** nhưng **KHÔNG refresh** state — trả instance cache cũ.

→ Lock đúng row nhưng `getStock()` là **giá trị cũ** → vẫn oversell:
```
T0 Tx_A: load cart → product stock=5 vào cache
T1 Tx_B: khóa, mua 5, stock→0, COMMIT
T2 Tx_A: findByIdForUpdate → khóa OK, DB thật=0, nhưng cache trả 5 ❌
         → check 5>=3 ✓ → set 2 → COMMIT → oversell
```
Lỗi này **không lộ khi test 1 luồng** — cực nguy hiểm.

**Fix: `EntityManager.refresh(entity, PESSIMISTIC_WRITE)`** — đọc lại state TƯƠI từ DB (đè state cũ) **+** khóa row, trong 1 lệnh:
```java
private final EntityManager entityManager;

entityManager.refresh(variant, LockModeType.PESSIMISTIC_WRITE);
if (variant.getStock() < quantity) throw new BusinessRuleException("Out of stock: ...");
variant.setStock(variant.getStock() - quantity);
```
Áp dụng cả `restock`. Sau đó 2 method `findByIdForUpdate` ở repository thành dead code → đã xóa.

> 💡 Bài học cốt lõi: **pessimistic lock chỉ an toàn khi state đọc cũng tươi.** Entity load trước → `findById...ForUpdate` khóa được row nhưng giữ state cũ → phải `refresh`. Lý do nhiều dev "dùng FOR UPDATE rồi mà vẫn oversell".

### Q: nếu như dùng isolation level READ_COMMITTED hay REPEATABLE_READ thì có giải quyết được không

**KHÔNG** — cả hai không cứu pattern read-modify-write set-tuyệt-đối. Chỉ SERIALIZABLE hoặc lock/atomic/version.

- READ_COMMITTED: không khóa → cả 2 đọc 5, ghi 2, lost update.
- REPEATABLE_READ: snapshot nhất quán nhưng vẫn "tính trên số cũ rồi set" → lost update.

> 🔗 Liên hệ gotcha trên: MySQL default **REPEATABLE READ** chính là lý do Hibernate giữ stock cũ trong cache (khớp ngữ nghĩa RR snapshot). `em.refresh` ép **current read** (đọc số mới nhất + khóa), vượt qua snapshot RR. → RR là lý do bug tồn tại, `refresh` là cách phá snapshot lấy số tươi.

| Cách | Cứu oversell? |
|------|:---:|
| READ_COMMITTED / REPEATABLE_READ | ❌ |
| SERIALIZABLE | ✅ (chậm) |
| Pessimistic lock (ta dùng) | ✅ mọi isolation |
| Atomic update / Optimistic @Version | ✅ |

## 8. Controller & Security
