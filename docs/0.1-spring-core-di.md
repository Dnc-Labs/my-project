## 1. Vấn đề bắt đầu từ đâu?

Bên Node.js, khi cần dùng một module:

```javascript
// userService.js
const userRepository = require('./userRepository');

function getUser(id) {
  return userRepository.findById(id);
}
```

Bạn **tự tay** tạo và quản lý dependency. Service cần Repository? Tự `require` vào. Controller cần Service? Tự `import` vào.

**Vấn đề**: Khi có 50 service, mỗi service phụ thuộc 3-4 thứ khác, có thứ cần **dùng chung** (Singleton — ví dụ database connection pool), có thứ cần **tạo mới mỗi lần**. Bạn phải tự quản lý hết — ai tạo trước, ai tạo sau, ai dùng chung, ai dùng riêng.

## 2. Spring giải quyết bằng IoC Container

Spring đảo ngược quyền kiểm soát (**Inversion of Control**): thay vì **bạn tự tạo dependency**, bạn chỉ cần **khai báo "tôi cần cái này"**, rồi Spring lo việc tạo và "tiêm" (inject) vào cho bạn.

| Khái niệm | Ý nghĩa |
|---|---|
| **IoC Container** | "Nhà kho" của Spring — nơi quản lý vòng đời của tất cả các object |
| **Bean** | Một object được Spring quản lý trong container |
| **Dependency Injection (DI)** | Kỹ thuật Spring dùng để "tiêm" Bean này vào Bean khác |

Ví dụ:

```java
@Service
public class UserService {

    private final UserRepository userRepository;

    // Constructor Injection — Spring tự tìm Bean UserRepository và truyền vào
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public User getUser(Long id) {
        return userRepository.findById(id);
    }
}
```

Bạn **không bao giờ** viết `new UserService(new UserRepository(...))`. Spring Container tự:
1. Tạo `UserRepository` Bean
2. Tạo `UserService` Bean, thấy constructor cần `UserRepository` → tiêm vào
3. Quản lý vòng đời (mặc định chỉ tạo **1 instance duy nhất** — Singleton)

## 3. Các annotation đánh dấu Bean

| Annotation | Dùng cho |
|---|---|
| `@Component` | Bean chung chung, không thuộc layer cụ thể |
| `@Service` | Business logic layer |
| `@Repository` | Data access layer (tương tác database) |
| `@Controller` / `@RestController` | API layer (nhận HTTP request) |
| `@Configuration` | Class chứa cấu hình, thường đi kèm `@Bean` method |

Về bản chất `@Service`, `@Repository`, `@Controller` đều **kế thừa từ `@Component`** — Spring đều coi chúng là Bean. Nhưng dùng đúng annotation giúp code rõ ràng về vai trò và Spring có thể xử lý đặc biệt (ví dụ `@Repository` tự động translate database exception).

**Không nên hoán đổi annotation cho nhau** vì một số mang chức năng đặc biệt:

| Annotation | DI hoạt động? | Chức năng đặc biệt? |
|---|---|---|
| `@Component` | Yes | Không có |
| `@Service` | Yes | Không có (chỉ semantic) |
| `@Repository` | Yes | Exception translation |
| `@Controller/@RestController` | Yes | Xử lý HTTP request |

## 4. Constructor Injection vs Field Injection

```java
// Constructor Injection — KHUYẾN KHÍCH
@Service
public class OrderService {
    private final ProductService productService;

    public OrderService(ProductService productService) {
        this.productService = productService;
    }
}

// Field Injection — KHÔNG NÊN
@Service
public class OrderService {
    @Autowired
    private ProductService productService;
}
```

**Tại sao Constructor Injection tốt hơn?**
- Dependency rõ ràng, nhìn constructor biết ngay class cần gì
- Dùng `final` được → đảm bảo không bị thay đổi sau khi khởi tạo
- Dễ viết unit test — truyền mock qua constructor, không cần Spring

## 5. Bean Scope — Các loại Bean

### Nhóm 1: Core Scope

#### `singleton` (mặc định)
- **1 instance duy nhất** trong toàn bộ Spring Container
- Tạo khi ứng dụng khởi động, sống đến khi ứng dụng tắt
- **Dùng khi**: Service, Repository, Controller — những class **stateless**

**Tại sao Singleton là mặc định?**
1. **Tiết kiệm bộ nhớ** — 1000 request không tạo hàng nghìn object
2. **Stateless by design** — Service chỉ chứa logic xử lý, không giữ state riêng
3. **Lưu ý multi-threading**: Bean Singleton bị nhiều thread truy cập đồng thời → **không được giữ mutable state** trong Bean

> So sánh Node.js: `require('module')` trả về cùng một instance — cũng là Singleton pattern!

#### `prototype`
- **Tạo instance mới mỗi lần** được yêu cầu
- Spring chỉ tạo, không quản lý vòng đời
- **Dùng khi**: Object có state riêng mỗi lần dùng

```java
@Component
@Scope("prototype")
public class EmailBuilder {
    private String to;
    private String subject;
    private String body;
}
```

### Nhóm 2: Web Scope

| Scope | Số instance | Vòng đời | Ứng dụng thực tế |
|---|---|---|---|
| **request** | 1 / HTTP request | Bắt đầu → kết thúc request | Trace ID, request-scoped context |
| **session** | 1 / HTTP session | Đăng nhập → hết phiên | Shopping cart (server-side session) |
| **application** | 1 / ServletContext | Toàn bộ app | Hiếm dùng |
| **websocket** | 1 / WS connection | Mở → đóng connection | Chat state, notification |

### Bẫy: Inject Prototype vào Singleton

```java
@Service  // singleton
public class OrderService {
    private final EmailBuilder emailBuilder;  // prototype — CHỈ INJECT 1 LẦN!
}
```

**Giải quyết** bằng `ObjectProvider`:

```java
@Service
public class OrderService {
    private final ObjectProvider<EmailBuilder> emailBuilderProvider;

    public OrderService(ObjectProvider<EmailBuilder> emailBuilderProvider) {
        this.emailBuilderProvider = emailBuilderProvider;
    }

    public void sendEmail() {
        EmailBuilder builder = emailBuilderProvider.getObject(); // instance MỚI mỗi lần
    }
}
```

## 6. `@SpringBootApplication` — 3 trong 1

```java
@SpringBootApplication
// tương đương với:
@SpringBootConfiguration  // Đánh dấu class cấu hình chính
@EnableAutoConfiguration  // Tự động cấu hình dựa trên dependency trong pom.xml
@ComponentScan            // Quét tất cả package con để tìm Bean
```

| Annotation | Tác dụng |
|---|---|
| `@SpringBootConfiguration` | Đánh dấu class cấu hình gốc |
| `@EnableAutoConfiguration` | Thấy `spring-boot-starter-web` → tự cấu hình Tomcat, JSON converter... |
| `@ComponentScan` | Quét từ package chứa class này **trở xuống** để tìm Bean |

> **Lưu ý**: `@ComponentScan` quét từ package hiện tại trở xuống. Class ở package ngang hàng sẽ **không được tìm thấy**.

### `SpringApplication.run(...)` làm gì?

```
1. Tạo IoC Container (ApplicationContext)
        ↓
2. ComponentScan — quét tìm tất cả Bean
        ↓
3. AutoConfiguration — cấu hình tự động
        ↓
4. Tạo tất cả Bean Singleton, inject dependency
        ↓
5. Khởi động Embedded Tomcat server (mặc định port 8080)
        ↓
6. Ứng dụng sẵn sàng nhận HTTP request!
```

### `@Controller` vs `@RestController`
- `@Controller` — trả về **tên view** (server-side rendering)
- `@RestController` = `@Controller` + `@ResponseBody` — trả về **JSON trực tiếp**

## 7. Circular Dependency (phụ thuộc vòng)

### Q: Nếu ServiceA cần ServiceB, ServiceB cần ServiceC, ServiceC lại cần ServiceA thì sao?

Spring **báo lỗi ngay khi khởi động** (với Constructor Injection):

```
BeanCurrentlyInCreationException:
Error creating bean 'serviceA': Requested bean is currently in creation
```

### Cách giải quyết đúng:

Circular Dependency là dấu hiệu **thiết kế sai**. Cách fix:

1. **Tách logic chung** ra Service mới:
```
Trước:  A → B → C → A  (lỗi)
Sau:    A → B → C
        A → D ← C      (D chứa logic chung)
```

2. **Dùng Event** — thay vì C gọi A, C phát event, A lắng nghe (Spring `ApplicationEventPublisher`)

> **Tư duy senior**: Đừng tìm cách hack để Spring chấp nhận circular dependency. Hãy coi đó là tín hiệu để refactor lại thiết kế.

## 8. Lưu ý khi tạo project

- `package` declaration chỉ tính từ `com` trở đi, **không bao gồm** `src/main/java/`
  - Sai: `package src.main.java.com.ecommerce.api.services;`
  - Đúng: `package com.ecommerce.api.services;`
- Nên dùng Spring Boot bản **stable** (ví dụ `3.4.4`), tránh bản SNAPSHOT

## 9. Câu hỏi thực hành (buổi 2 — 2026-03-30)

### Q: Tại sao không dùng `System.out.println` trong production?

`System.out.println` không phân loại mức độ (DEBUG/INFO/WARN/ERROR), không tắt/bật theo môi trường, không ghi ra file, không hiển thị thời gian/class/thread, và performance kém (blocking I/O).

Spring Boot tích hợp sẵn **SLF4J + Logback**, không cần cài thêm gì:

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Service
public class GreetingService {
    private static final Logger log = LoggerFactory.getLogger(GreetingService.class);

    public String handleGetGreeting(String name) {
        log.info("Greeting requested for name: {}", name);
        return name == null || name.isEmpty() ? "Hello, Spring Boot" : "Hello, " + name;
    }
}
```

> So sánh Node.js: tương tự việc dùng **Winston/Pino** thay vì `console.log`.

### Q: `@Autowired` là gì? Ưu nhược điểm?

`@Autowired` là annotation bảo Spring inject dependency. Có 3 cách inject:

```java
// 1. Field Injection — KHÔNG NÊN
@Autowired
private GreetingService greetingService;

// 2. Constructor Injection — KHUYẾN KHÍCH ✅
private final GreetingService greetingService;
GreetingController(GreetingService greetingService) {
    this.greetingService = greetingService;
}

// 3. Setter Injection — ÍT DÙNG
@Autowired
public void setGreetingService(GreetingService greetingService) { ... }
```

| | Field Injection | Constructor Injection |
|---|---|---|
| Dùng `final`? | ❌ | ✅ → immutable |
| Test dễ? | ❌ Cần reflection | ✅ `new Controller(mockService)` |
| Biết thiếu dependency? | Runtime | Compile time |
| Dependency rõ ràng? | Ẩn | Rõ — nhìn constructor là biết |

> **Lưu ý**: Khi class chỉ có 1 constructor, Spring tự động inject mà **không cần** `@Autowired`.

### Q: `@RequestParam` — nhận query parameter

```java
// Dùng required = false thay vì @Nullable — cách "Spring-native" hơn
@GetMapping
ResponseEntity<String> getGreeting(@RequestParam(required = false) String name) {
    ...
}
```

### Q: `@GetMapping("/")` vs `@GetMapping`

- `@RequestMapping("/greeting")` + `@GetMapping("/")` → match `/greeting/` (có slash cuối)
- `@RequestMapping("/greeting")` + `@GetMapping` → match cả `/greeting` lẫn `/greeting/`
