## Phần 1: JPA Entity Lifecycle — nền tảng quan trọng nhất

Đây là concept mà 80% Java dev mới không hiểu, dẫn đến các bug khó debug. Hiểu rồi thì mọi thứ về JPA sáng tỏ.

### 4 trạng thái của 1 entity

```
                     ┌──────────────┐
        new Object() │  Transient   │
       ─────────────▶│  (chưa có ID,│
                     │   chưa quản  │
                     │   lý bởi JPA)│
                     └───────┬──────┘
                             │ persist() / save()
                             ▼
            ┌───────────────────────────────┐
   ┌───────▶│         Managed                │◀───────┐
   │        │  (trong Persistence Context,   │        │
   │        │   thay đổi được auto-flush     │        │
   │  merge │   xuống DB)                    │        │  refresh
   │        └────┬──────────┬────────┬───────┘        │
   │             │          │        │                │
   │   close()/  │  remove()│        │  detach()/     │
   │   tx end    │          │        │  evict()       │
   │             ▼          ▼        ▼                │
   │     ┌──────────┐  ┌──────────┐  ┌───────────┐    │
   └─────│ Detached │  │ Removed  │  │ Detached  │────┘
         └──────────┘  └────┬─────┘  └───────────┘
                            │ tx commit
                            ▼
                       (DELETE row)
```

### 1. Transient — "vô danh"

```java
Product p = new Product();
p.setName("Nike");
// Lúc này: Transient
// - p tồn tại trong RAM
// - JPA KHÔNG biết về nó
// - Chưa có row trong DB
// - Set field gì cũng không ảnh hưởng DB
```

Tương đương: object Java thuần.

### 2. Managed (Persistent) — "đang được JPA chăm sóc"

```java
Product p = new Product();
p.setName("Nike");

productRepository.save(p);
// Sau dòng này: p chuyển sang Managed

p.setName("Adidas");  // KHÔNG cần gọi save() lần 2!
// Khi transaction commit, Hibernate auto SQL UPDATE
```

**Hibernate có cơ chế "dirty checking"** — snapshot state lúc load entity, so sánh khi commit, sinh UPDATE cho field thay đổi.

So sánh Node.js: Sequelize cũng có pattern tương tự với `instance.save()` thủ công, hoặc Mongoose với `doc.save()`. Nhưng **Hibernate auto-flush không cần gọi save()**.

### 3. Detached — "đã từng managed nhưng giờ thì không"

```java
@Transactional
public void demo(Long id) {
    Product p = productRepo.findById(id).get();
    // p đang Managed
    p.setName("Updated");  // dirty checking sẽ trigger UPDATE
}
// Hết @Transactional → p chuyển sang Detached

p.setName("Updated again");  // KHÔNG còn dirty checking, KHÔNG UPDATE DB
```

Detached = "JPA không còn track nữa". Object vẫn có data, nhưng thay đổi không sync xuống DB.

Đây là nguyên nhân **`LazyInitializationException`** — khi truy cập lazy collection trên detached entity → Hibernate không còn kết nối → throw exception.

### 4. Removed — "đánh dấu để xoá"

```java
@Transactional
public void demo(Long id) {
    Product p = productRepo.findById(id).get();
    productRepo.delete(p);
    // p chuyển sang Removed (chưa thực sự DELETE DB)
}
// commit transaction → SQL DELETE thực sự chạy
```

Trạng thái trung gian — entity bị mark là "sẽ xoá" nhưng chỉ khi commit mới xoá thật.

## Phần 2: Cascade — "lan truyền lifecycle"

`cascade` = "khi parent chuyển trạng thái, **lan truyền** thao tác đó xuống children".

### Các CascadeType

```java
@OneToMany(mappedBy = "product", cascade = CascadeType.PERSIST)
private List<ProductVariant> variants;
```

| CascadeType | Khi parent... | Thì children... |
|-------------|---------------|-----------------|
| `PERSIST` | `persist()` | cũng được persist (Transient → Managed) |
| `MERGE` | `merge()` | cũng được merge (Detached → Managed) |
| `REMOVE` | `remove()` | cũng bị remove (Managed → Removed) |
| `DETACH` | `detach()` | cũng detach |
| `REFRESH` | `refresh()` | cũng refresh từ DB |
| `ALL` | bất kỳ | đều cascade |

### Demo cụ thể với Product + Variants

```java
Product product = new Product();   // Transient
ProductVariant v1 = new ProductVariant();
v1.setProduct(product);
ProductVariant v2 = new ProductVariant();
v2.setProduct(product);
product.setVariants(List.of(v1, v2));

// Trường hợp 1: KHÔNG có cascade
productRepo.save(product);
// ❌ ERROR: variants tham chiếu Product nhưng v1, v2 vẫn Transient
//    Hibernate không tự save chúng → throw TransientObjectException

// Trường hợp 2: cascade = PERSIST
productRepo.save(product);
// ✅ Save product → cascade sang variants → INSERT cả 3 row
```

→ Đây là lý do Product entity cần:

```java
@OneToMany(mappedBy = "product", cascade = CascadeType.ALL)
private List<ProductVariant> productVariants;
```

## Phần 3: `orphanRemoval` — phần tinh tế nhất

Đây là chỗ **bug khó debug**. Cần phân biệt rõ với `CascadeType.REMOVE`.

### `CascadeType.REMOVE`

Trigger khi gọi `productRepo.delete(product)`:
- product chuyển Removed
- TẤT CẢ variants cũng Removed
- Commit → DELETE cả product + variants

### `orphanRemoval = true`

Trigger khi 1 variant **bị tách khỏi parent collection**:

```java
Product p = productRepo.findById(1L).get();
// p có 3 variants
p.getVariants().remove(0);  // Xoá variant[0] khỏi list
productRepo.save(p);

// Với orphanRemoval = true:
//   → variant[0] bị DELETE khỏi DB
// Với orphanRemoval = false (default):
//   → Hibernate cố set product_id của variant[0] = null
//   → Vì FK constraint (nullable=false) → SQL ERROR
```

### Tại sao cần cả 2?

Hai case khác nhau:
- `CascadeType.REMOVE`: xoá **toàn bộ** parent + children
- `orphanRemoval`: parent vẫn sống, chỉ xoá **những child bị remove khỏi list**

```java
// Case 1: xoá nguyên cây
productRepo.delete(product);   // CascadeType.REMOVE handle

// Case 2: chỉ xoá 1 variant
product.getVariants().removeIf(v -> v.getId() == 5L);   // orphanRemoval handle
productRepo.save(product);
```

### Bug kinh điển — `= null` vs `.clear()`

```java
// Cách 1: gán = null
product.setVariants(null);
productRepo.save(product);
// Với orphanRemoval = true → có thể throw exception
// "A collection with cascade=all-delete-orphan was no longer
//  referenced by the owning entity instance"

// Cách 2: clear list
product.getVariants().clear();
productRepo.save(product);
// ✅ OK — Hibernate detect collection rỗng → DELETE tất cả variant cũ
```

→ **Rule**: Với `orphanRemoval = true`, **luôn dùng `clear()` thay vì `= null`**.

## Phần 4: Composition vs Aggregation — khái niệm OOP cốt lõi

### `nullable` × `orphanRemoval` — 4 combo

| `nullable` | `orphanRemoval` | Ý nghĩa | Use case |
|-----------|----------------|---------|---------|
| `false` | `true` | **Strong parent-child (Composition)** — variant LUÔN có product, removed khỏi list = xoá DB | ✅ ProductVariant — best practice |
| `false` | `false` | Variant LUÔN có product, không cho remove khỏi list (sẽ SQL error) | ❌ Bug-prone, không nên |
| `true` | `true` | Variant có thể không có product, removed khỏi list = xoá DB | ⚠️ Conflict logic |
| `true` | `false` | **Aggregation** — variant tồn tại độc lập | Comment, audit log... |

### Khi nào dùng Composition vs Aggregation?

- **Composition** (mạnh): Child không tồn tại nếu parent không tồn tại. Ví dụ: Variant ↔ Product, Order ↔ OrderItem. → `nullable=false` + `orphanRemoval=true`
- **Aggregation** (yếu): Child tồn tại độc lập, có thể link với parent này hoặc parent khác. Ví dụ: Comment ↔ Post (nếu cho phép comment ngoài post). → `nullable=true` + `orphanRemoval=false`

ProductVariant rõ ràng là **composition** với Product.

## Phần 6: Q&A trong quá trình code

### Câu hỏi: Không có @Transactional, save() rồi setName() có update DB không?

> Trong phần lifecycle, nếu như trong 1 api tạo product, tôi không đánh dấu là 1 transaction thì khi mà tạo product xong productRepository.save(p) thì tôi lại dùng p.setName("Adidas") thì record trong db có được update không?

**Câu trả lời ngắn: KHÔNG update.**

#### Bên trong save()

Spring Data `save()` được `@Transactional` mặc định ở method level (xem `SimpleJpaRepository`):

```java
@Transactional
public <S extends T> S save(S entity) {
    if (entityInformation.isNew(entity)) {
        em.persist(entity);
        return entity;
    } else {
        return em.merge(entity);
    }
}
```

→ `save()` **TỰ mở transaction**, commit, close — chỉ tồn tại trong scope của method `save()`.

#### Sequence chi tiết

```
┌─────────────────────────────────────────────────────────┐
│  Controller method (KHÔNG @Transactional)               │
│                                                          │
│  ┌────────────────────────────────────────────────┐     │
│  │  productRepository.save(p)                      │     │
│  │  ┌──────────────────────────────────────┐       │     │
│  │  │ @Transactional opens                  │       │     │
│  │  │   em.persist(p)  → p Managed          │       │     │
│  │  │   commit  → SQL INSERT                │       │     │
│  │  │ Transaction CLOSED ←──────────┐       │       │     │
│  │  └────────────────────────────────│──────┘       │     │
│  │                                   │              │     │
│  │  p chuyển sang DETACHED ◀────────┘              │     │
│  └────────────────────────────────────────────────┘     │
│                                                          │
│  p.setName("Adidas")   ← p đã DETACHED                   │
│  → KHÔNG có dirty checking                               │
│  → KHÔNG có transaction → KHÔNG commit → KHÔNG UPDATE   │
└─────────────────────────────────────────────────────────┘
```

#### Nếu CÓ @Transactional

```java
@Transactional   // Outer transaction
public ProductResponse createProduct(...) {
    Product p = new Product();
    productRepository.save(p);
    // save() join vào outer transaction (Propagation.REQUIRED mặc định)
    // p vẫn Managed sau khi save() return

    p.setName("Adidas");
    // Vẫn trong active transaction → Hibernate dirty checking tracking
}  // Outer transaction commit ở đây → INSERT + UPDATE
```

#### OSIV (OpenSessionInView)

Spring Boot mặc định bật `spring.jpa.open-in-view=true`. Nhưng:
- **Session ≠ Transaction**: Session = "scope của Persistence Context", Transaction = "scope của ACID"
- OSIV chỉ ngăn `LazyInitializationException` khi access lazy collection
- **Không tự động save changes** — dirty checking chỉ trigger khi commit transaction

#### Bài học

1. `@Transactional` ở service layer — nguyên tắc vàng
2. Nếu thao tác nhiều bước (load → modify → multiple saves) → **PHẢI có @Transactional**

### Câu hỏi: PERSIST và save() khác gì?

> PERSIST và Save khác gì nhau nhỉ?

**Câu trả lời ngắn: `save()` thông minh hơn — wrap cả `persist()` lẫn `merge()`.**

#### `persist()` — JPA EntityManager

- Dùng cho entity **mới hoàn toàn** (Transient, chưa có id)
- Chuyển entity từ Transient → Managed
- **Throw `EntityExistsException`** nếu entity đã có trong context

#### `merge()` — JPA EntityManager

- Dùng cho entity **đã từng tồn tại** (Detached state)
- Hibernate **tìm entity trong context với id đó**
- Nếu có → copy fields từ argument sang managed entity
- Nếu không có → load từ DB, copy fields, return managed entity
- ⚠️ **Trả về MỘT INSTANCE KHÁC** — không phải instance bạn truyền vào

#### `save()` — Spring Data

```java
public <S extends T> S save(S entity) {
    if (entityInformation.isNew(entity)) {
        em.persist(entity);
        return entity;
    } else {
        return em.merge(entity);
    }
}
```

- Entity `id == null` → `persist()`
- Entity `id != null` → `merge()`

#### Pitfall — merge() trả về instance khác

```java
Product p = new Product();
p.setId(1L);          // entity đã có id → save() sẽ gọi merge()
p.setName("Updated");

Product result = productRepository.save(p);

// ⚠️ p VẪN LÀ DETACHED!
// result là managed entity (instance khác hoàn toàn)

p.setName("X");        // → KHÔNG ảnh hưởng DB
result.setName("Y");   // → ảnh hưởng DB (nếu trong @Transactional)
```

**Bài học**: Sau `save()`, **luôn dùng return value**, không tin reference cũ.

```java
// ❌ Sai
productRepository.save(p);
p.setName("X");

// ✅ Đúng
p = productRepository.save(p);
p.setName("X");
```

### Câu hỏi: orphanRemoval = false thì nullable = true có đúng không?

> Về phần orphanRemoval thì để true hay false tuỳ thuộc vào ý đồ thiết kết đúng không, nếu như để là false thì trong ProductVariant mục relation chỗ product_id có thể để là nullable = true đúng không, còn nếu để là true thì xoá luôn cho đỡ rác database

**Câu trả lời ngắn: Đúng về mặt logic, nhưng thường không phải pattern muốn dùng.**

#### Phân tích logic của bạn — đúng

Khi user xoá variant khỏi list, Hibernate generate `UPDATE product_variants SET product_id = NULL`:
- `nullable = false` → SQL error
- `nullable = true` → variant trở thành "mồ côi" (product_id = NULL)

#### Nhưng đây có phải pattern muốn dùng?

**Variant mồ côi (product_id = NULL)** = Là 1 variant không thuộc product nào → vô nghĩa trong e-commerce → **rác trong DB**.

Đây chính là lý do "xoá luôn cho đỡ rác" — quá đúng.

#### Khi nào nullable = true hợp lý?

Có những entity có **lifecycle độc lập với parent**:
- **Comment** — có thể "uncategorized"
- **Article tags** — tag tồn tại độc lập
- **Audit log entries** — log vẫn cần lưu dù entity gốc đã xoá

Với ProductVariant thì **không có ý nghĩa** để tồn tại độc lập với Product → Combo `nullable=false` + `orphanRemoval=true` là chuẩn nhất.

## Phần 7: Implementation

### Entity với composite unique constraint

```java
@Entity
@Table(
    name = "product_variants",
    uniqueConstraints = @UniqueConstraint(
        name = "uk_variant_product_size_color",
        columnNames = {"product_id", "size", "color"}
    )
)
public class ProductVariant {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false) private String size;
    @Column(nullable = false) private String color;
    @Column(unique = true, nullable = false) private String sku;
    @Column(nullable = false, precision = 19, scale = 2) private BigDecimal price;
    @Column(nullable = false) private Integer stock = 0;
    private String imageUrl;

    @ManyToOne
    @JoinColumn(name = "product_id", nullable = false)
    private Product product;
}
```

### ProductVariantSecurity — pattern y hệt ProductSecurity

```java
@Component("productVariantSecurity")
public class ProductVariantSecurity {

    public boolean isOwner(Long variantId, Authentication authentication) {
        if (authentication == null || !authentication.isAuthenticated()) return false;
        String email = authentication.getName();
        Optional<User> seller = userRepository.findByEmail(email);
        Optional<ProductVariant> variant = productVariantRepository.findById(variantId);
        if (seller.isEmpty() || variant.isEmpty()) return false;
        return variant.get().getProduct().getSeller().getId().equals(seller.get().getId());
    }
}
```

Variant ownership = ownership của Product chứa nó.

### Hybrid REST API

```java
POST   /api/products/{productId}/variants  — tạo (cần biết product)
GET    /api/products/{productId}/variants  — list của 1 product
GET    /api/variants/{id}                   — detail
PUT    /api/variants/{id}                   — update
DELETE /api/variants/{id}                   — delete
```

`@PreAuthorize` 2 cấp:
- **CREATE variant**: check ownership của **PRODUCT** (vì variant chưa tồn tại)
- **UPDATE/DELETE variant**: check ownership của **VARIANT** (đi qua product)

## Phần 8: Code Review — các bug đã fix

### 🔴 Critical bugs

**Bug 1**: DELETE variant không có `@PreAuthorize` → bất kỳ user nào có JWT đều xoá được. Lỗ hổng bảo mật nghiêm trọng.

**Bug 2**: `@ExceptionHandler()` rỗng → Spring infer từ param type → catch `Exception.class` → conflict với handler khác. Phải `@ExceptionHandler(DuplicateResource.class)`.

**Bug 3**: `updateVariant` không update `price` field → user không thể đổi giá variant qua API.

### 🟡 Medium issues

- `ProductId` (capital P) vi phạm JavaBean convention
- `setColor()` được gọi 2 lần trong `fromEntity`
- SpEL update endpoint thiếu parens

### Refactor đáng chú ý

#### `DuplicateResource` exception
Chuyển từ `RuntimeException` chung sang exception cụ thể với HTTP 409 Conflict:

```java
public class DuplicateResource extends RuntimeException {
    public DuplicateResource(String message) { super(message); }
}

// Handler
@ExceptionHandler(DuplicateResource.class)
public ResponseEntity<BaseResponse<Void>> handleDuplicateResource(DuplicateResource e) {
    return ResponseEntity.status(HttpStatus.CONFLICT).body(BaseResponse.error(e.getMessage()));
}
```

#### Refactor sizeChanged/colorChanged

```java
if(checkIsNotNull(size) && !size.equals(variant.getSize())

// Sau (2 boolean rõ nghĩa)

if (sizeChanged || colorChanged) {
```

→ Pattern senior: **đặt tên cho ý định**, không chỉ logic.

## Phần 9: Naming Convention Deep Dive

### Câu hỏi
> Bạn hãy kiểm tra những file tôi vừa làm, có đoạn nào tôi đặt tên biến không phù hợp không, tôi muốn học thật chỉn chu

### 5 vấn đề naming chính

#### 1. Typo trong tên biến

```java
boolean checkExitsBySku = ...;     // THIẾU CHỮ "S"
//           ^^^^^
```
Phải là `checkExistsBySku`. IDE thường không detect → để lâu thành "convention sai".

#### 2. Vietnamese word order

| Hiện tại (theo lối nói tiếng Việt) | Vấn đề | Sửa |
|--------------|--------|-----|
| `createVariantProduct` | Đọc thành "create-variant product" — confusing | `createVariant` |
| `getListVariants` | "getList" thừa — `get` đã trả list rồi | `getVariants` |
| `getDetailVariant` | "getDetail" lai từ "get chi tiết" | `getById` |

Tham khảo framework lớn (Spring REST docs, AWS SDK):
- `listUsers()` (không phải `getListUsers`)
- `getUser(id)` (không phải `getDetailUser`)
- `createUser(req)` (không phải `createUserAccount`)

→ **Rule of thumb**: Method name = verb + object. Object thường **không lặp lại context của class**.

#### 3. "Object" suffix anti-pattern

```java
Optional<User> currentUserObject = ...
Optional<ProductVariant> productVariantObject = ...
```

Anti-pattern vì:
- Mọi thứ trong Java đều extends `Object` → suffix vô nghĩa
- Thói quen từ ngôn ngữ khác (PHP cũ, JavaScript)

**Sửa**: `currentUserObject` → `userOpt` hoặc dùng `orElse(null)` để tránh phải khai Optional variable.

#### 4. Word order trong local variable

```java
String sizeUpdate = ...
String colorUpdate = ...
```

`sizeUpdate` đọc thành "**update of size**". Ý là "**size để update**" → `newSize` đúng tiếng Anh.

#### 5. Verbose local variable

```java
ProductVariant productVariant = new ProductVariant();
productVariant.setColor(...);
ProductVariant newProductVariant = variantRepository.save(productVariant);
```

`productVariant`, `newProductVariant` lặp class name. Trong scope rõ context:

```java
ProductVariant variant = new ProductVariant();
variant.setColor(...);
ProductVariant saved = variantRepository.save(variant);
```

#### Bonus: `check` prefix cho boolean

```java
boolean checkExistsBySku = variantRepository.existsBySku(sku);
boolean checkVariantDuplicate = ...;
```

`check` là **action verb** — không phù hợp cho boolean variable. Java JavaBean convention: dùng `is`, `has`, `should`, `can`, `exists`:

```java
boolean skuExists = variantRepository.existsBySku(sku);
```

Hoặc inline luôn nếu dùng 1 lần:
```java
if (variantRepository.existsBySku(sku)) {
    throw new DuplicateResource("SKU already exists");
}
```

### Triết lý senior về naming

**Code được đọc nhiều hơn được viết 10 lần.** Naming tốt không tốn thời gian khi viết — chỉ cần thói quen.

| Loại | Pattern | Ví dụ |
|------|---------|-------|
| Boolean | `is*`, `has*`, `should*`, `can*`, `*Exists` | `isOwner`, `hasChildren`, `skuExists` |
| Method action | `verb + object` (object thường ngắn) | `createVariant`, `findById`, `delete` |
| Method query | `get*`, `find*`, `list*` | `getVariant`, `findByEmail`, `listVariants` |
| Variable | Danh từ, ngắn nhất có thể trong scope | `variant`, `user`, `email` |
| Constant | UPPER_SNAKE_CASE | `MAX_FILE_SIZE`, `JWT_SECRET` |

**Nguyên tắc tổng**: Mỗi từ trong tên biến/method phải **mang thông tin mới**. Lặp lại class name, dùng từ generic ("Object", "Data") là noise.

### Helper với throwing semantics

```java
// Trước: tên giống method public, nhầm lẫn
private ProductVariant getProductVariantById(Long id) { ... }

// Sau: tên khác hẳn, rõ throws
private ProductVariant findVariantOrThrow(Long id) { ... }
```

Pattern **"throwing semantics in name"** — nhìn tên biết ngay method có throw exception không.

## Phần 10: .editorconfig — đồng bộ format giữa các máy

### Câu hỏi
> Làm sao để config tự động format multi-line cho method signature dài nhỉ

### 3 cấp độ giải pháp

| Cấp | Công cụ | Khi dùng |
|-----|---------|----------|
| 1 | IntelliJ built-in formatter (Cmd+Option+L) | Đang học, 1 máy |
| 2 | `.editorconfig` (cross-IDE, commit theo git) | Nhiều máy, team |
| 3 | Spotless (enforce ở build time) | Production, CI/CD |

### .editorconfig cho project

Tạo file `.editorconfig` ở root:

```ini
root = true

[*]
charset = utf-8
indent_style = space
indent_size = 4
end_of_line = lf
insert_final_newline = true
trim_trailing_whitespace = true

[*.java]
max_line_length = 120
ij_continuation_indent_size = 8
ij_java_method_parameters_wrap = on_every_item
ij_java_method_parameters_new_line_after_left_paren = true
ij_java_method_parameters_right_paren_on_new_line = false
ij_java_call_parameters_wrap = on_every_item
ij_java_imports_layout = $*,|,java.**,|,javax.**,|,jakarta.**,|,org.**,|,com.**,|,*

[*.{xml,yml,yaml}]
indent_size = 2

[*.md]
trim_trailing_whitespace = false
```

**Lợi ích**:
- Setting commit vào git → sync giữa máy nhà/máy work
- Cross-IDE: ai dùng VS Code/Eclipse cũng follow rule
- Pattern bắt đầu với `ij_` là **IntelliJ-specific extension** của EditorConfig — IDE khác bỏ qua

### IntelliJ Auto-format trên save

`Settings → Tools → Actions on Save`:
- ✅ **Reformat code**
- ✅ **Optimize imports**

Mỗi khi `Cmd+S`, code tự format.

### Phím tắt format

- **Mac**: `Cmd + Option + L`
- **Windows/Linux**: `Ctrl + Alt + L`
