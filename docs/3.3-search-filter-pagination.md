```

```

# 3.3.1 — Pageable + Sort foundation

## Lý thuyết cốt lõi

### 1. `Pageable` interface — Strategy pattern

`Pageable` chỉ **mô tả ý định** phân trang (trang X, kích thước Y, sort Z). Implementation cụ thể là `PageRequest`:

```java
Pageable pageable = PageRequest.of(0, 20, Sort.by("price").ascending());
```

`JpaRepository` builtin nhận `Pageable` → tự thêm `LIMIT/OFFSET` + `ORDER BY` + chạy COUNT query → trả `Page<T>`.

So với Node.js (Express + Sequelize):

| Aspect | Node.js | Spring Data |
|--------|---------|-------------|
| Parse query | Tự `parseInt`, validate | Framework parse → tạo `Pageable` |
| Default size | Tự set | `@PageableDefault(size = 20)` hoặc config global |
| Max size guard | Tự code | `spring.data.web.pageable.max-page-size` |
| Response format | Tự build | `Page<T>` có sẵn `totalElements`, `totalPages`... |

### 2. `Page<T>` vs `Slice<T>` vs `List<T>`

```
List<T>     → chỉ data, không metadata               (1 query)
Slice<T>    → data + hasNext                          (1 query, lấy size+1 element)
Page<T>     → data + totalElements + totalPages       (2 query: SELECT + COUNT)
```

| Return type | Khi nào dùng | Cost |
|-------------|--------------|------|
| `Page<T>` | UI paginator hiển thị tổng trang (1,2,3...50) | 2 query |
| `Slice<T>` | Infinite scroll / "Load more" | 1 query |
| `List<T>` | API internal, không cần metadata | 1 query |

**Quyết định dự án:** dùng `Page<T>` cho cả khách + admin/seller (paginator cổ điển, UX nhất quán). Giá phải trả: COUNT query luôn chạy → khi bảng lớn (1M+ rows) cần cân nhắc cache count / approximate count.

### 3. `Sort` object

- Property name là **field entity** (`createdAt`), KHÔNG phải column DB (`created_at`) → parse qua JPA metamodel.
- Hỗ trợ nested (`category.name` → Hibernate tự JOIN) + multi-sort.
- **An toàn SQL injection**: property validate qua metamodel, direction là enum `ASC/DESC`, không string concat.
- Spring parse `?sort=price,desc&sort=name,asc` → `Sort.by(desc("price"), asc("name"))`.

### 4. `HandlerMethodArgumentResolver` — phép màu inject `Pageable`

Spring MVC dùng `PageableHandlerMethodArgumentResolver` (auto-register bởi `SpringDataWebAutoConfiguration` khi có `spring-data-jpa` trên classpath) để convert URL → `Pageable` object. Express không có cơ chế này → phải tự viết middleware.

Customize 3 cách:
1. Per-method: `@PageableDefault(size = 20, sort = "createdAt", direction = DESC)`
2. Global qua `application.yml`: `spring.data.web.pageable.*`
3. Programmatic qua `WebMvcConfigurer.addArgumentResolvers`

**Dự án dùng cách 3** — file `WebConfig.java`:

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        PageableHandlerMethodArgumentResolver pageable = new PageableHandlerMethodArgumentResolver();
        pageable.setMaxPageSize(100);
        pageable.setFallbackPageable(PageRequest.of(0, 20, Sort.by(Sort.Direction.DESC, "createdAt")));
        resolvers.add(pageable);
    }
}
```

→ Global default: page 0, size 20, sort `createdAt DESC`, max page size 100.

## Q&A 3.3.1 (verbatim)

**Câu hỏi: Pageable này đang dùng limit - offset để query database, nhưng mà với mỗi loại db thì sẽ có cách query khác nhau, ví dụ như oracle thì có áp dụng được không.**

**Trả lời:** Có, dùng được cho Oracle (và mọi DB Hibernate support). Không cần thay đổi gì trong code Java — Hibernate tự sinh SQL đúng syntax cho DB đang dùng, qua cơ chế **`Dialect`**.

Mỗi RDBMS có syntax LIMIT khác nhau:
```sql
-- MySQL / PostgreSQL
SELECT * FROM products ORDER BY price LIMIT 20 OFFSET 40;

-- Oracle 12c+ (SQL:2008 standard)
SELECT * FROM products ORDER BY price OFFSET 40 ROWS FETCH NEXT 20 ROWS ONLY;

-- Oracle 11g trở xuống (KHÔNG có OFFSET/FETCH → wrap ROWNUM 2 lớp)
SELECT * FROM (
  SELECT inner_query.*, ROWNUM rownum_ FROM (
    SELECT * FROM products ORDER BY price
  ) inner_query WHERE ROWNUM <= 60
) WHERE rownum_ > 40;
```

Hibernate có abstract class `Dialect` (Strategy pattern), mỗi DB 1 subclass (`MySQLDialect`, `OracleDialect`, `PostgreSQLDialect`...). Mỗi Dialect có `getLimitHandler()` trả về `LimitHandler` — nhận SQL gốc + offset/limit → append phần phân trang đúng syntax.

```
JPQL (DB-agnostic) → Hibernate Engine → Dialect (tuỳ DB) → SQL native đúng cho DB đó
```

Hibernate 6 (Spring Boot 3.x) **tự detect dialect** từ JDBC metadata. Cùng 1 code Java → SQL khác nhau theo DB, không đổi 1 dòng.

**Caveat** (Dialect không magic hết): data type khác nhau (JSONB vs CLOB), sequence vs identity, locking syntax, và **native query** (`nativeQuery = true`) thì khoá vào DB đó. Rule: JPQL/Criteria/Specification → portable; native query → DB-specific.

Sequelize bên Node.js cũng có cơ chế dialect y hệt (`dialect: 'mysql' | 'postgres' | 'oracle'...`).

## Fix warning "Serializing PageImpl instances as-is is not supported"

Spring Data 3.x deprecate việc serialize `Page<T>` trực tiếp ra JSON (PageImpl là internal class, structure không stable). 3 cách fix:

1. `@EnableSpringDataWebSupport(pageSerializationMode = VIA_DTO)` — Spring tự wrap thành `PagedModel`
2. **Tự tạo `PageResponse<T>` DTO** ← **dự án chọn cách này** (tự chủ contract)
3. Kệ warning (sẽ break ở Spring Data 4.x) — không khuyến nghị

`PageResponse<T>` cuối cùng:

```java
@Getter
@Builder
public class PageResponse<T> {
    private List<T> content;
    private PageMetaData page;     // nested metadata (tách data vs metadata)

    @Getter @Builder
    public static class PageMetaData {
        private int number;
        private int size;
        private long totalElements;   // long primitive, KHÔNG Long boxed
        private int totalPages;
        private boolean first;
        private boolean last;
    }

    public static <T> PageResponse<T> from(Page<T> page) {
        return PageResponse.<T>builder()
                .content(page.getContent())
                .page(PageMetaData.builder()
                        .number(page.getNumber())
                        .size(page.getSize())
                        .totalElements(page.getTotalElements())
                        .totalPages(page.getTotalPages())
                        .first(page.isFirst())
                        .last(page.isLast())
                        .build())
                .build();
    }
}
```

JSON output:
```json
{
  "success": true,
  "data": {
    "content": [ {...} ],
    "page": { "number": 0, "size": 20, "totalElements": 100, "totalPages": 5, "first": true, "last": false }
  }
}
```

`Page<T>.map(productMapper::fromEntity)` — convert `Page<Product>` → `Page<ProductResponse>` giữ nguyên metadata. KHÔNG dùng `.stream().toList()` + `new PageImpl<>(...)` (dài + dễ mất total).

# 3.3.2 — Search text (name + description)

## Lý thuyết cốt lõi

### Spring Data Query Derivation

Đặt tên method đúng convention → Spring tự sinh query (parse bởi `PartTree`, validate field lúc app start):

```
findBy + Name + Containing + IgnoreCase
└──┬──┘   └─┬─┘  └────┬───┘    └────┬───┘
 subject  field  predicate    modifier
```

Keyword phổ biến: `Containing` (LIKE %?%), `StartingWith`, `IgnoreCase`, `And`/`Or`, `GreaterThan`/`LessThan`, `Between`, `In`, `OrderBy...`, `Top<N>`.

**Bẫy:** search OR trên cả name + description → tên derived method dài kinh khủng (`findByNameContainingIgnoreCaseOrDescriptionContainingIgnoreCase`, phải truyền keyword 2 lần) → chuyển sang `@Query`.

### `@Query` JPQL

```java
@Query("""
        SELECT p FROM Product p
        WHERE (:keyword IS NULL OR :keyword = ''
               OR LOWER(p.name) LIKE LOWER(CONCAT('%', :keyword, '%'))
               OR LOWER(p.description) LIKE LOWER(CONCAT('%', :keyword, '%')))
        """)
Page<Product> search(@Param("keyword") String keyword, Pageable pageable);
```

Điểm chính:
- **JPQL** reference entity + field (`FROM Product p`, `p.name`), KHÔNG phải table + column.
- `@Query` hỗ trợ `Pageable`: tự thêm ORDER BY + LIMIT/OFFSET + tự derive COUNT query.
- Optional keyword qua trick `(:keyword IS NULL OR :keyword = '' OR ...)`: khi keyword null → `NULL IS NULL` = TRUE → cả mệnh đề TRUE → trả tất cả (list-all).

### LIKE + Collation + Index (deep)

```sql
WHERE name LIKE 'iphone%'    -- ✅ dùng B-tree index (prefix match)
WHERE name LIKE '%iphone%'   -- ❌ KHÔNG dùng index, full scan O(N)
```

- `LOWER(name) LIKE LOWER(...)` → function apply mọi row → index trên `name` không dùng được.
- Giải pháp: collation case-insensitive (`utf8mb4_unicode_ci`) → không cần `LOWER()`, vẫn dùng index.
- Search "contains" (`%kw%`) inherently chậm → data lớn cần FULLTEXT (MySQL) / GIN (Postgres) / **Elasticsearch** (3.3.4).

| Approach | Speed | Khi nào |
|----------|-------|---------|
| `LIKE '%kw%'` | O(N) | Data < 100k rows |
| FULLTEXT index | O(log N) | Data lớn, cùng DB |
| Elasticsearch | O(log N) + scoring + analyzer | Production search nghiêm túc |

## Q&A 3.3.2 (verbatim)

**Câu hỏi: hãy giải thích kỹ hơn @Query và Specification động, trong Prod, chúng ta sẽ dùng cái nào.**

**Trả lời:**

`@Query` là **JPQL string cố định**, Hibernate parse 1 lần lúc startup → cache query plan. Validate lúc start (sai field → app crash khi start). Ưu: đọc dễ, validate sớm, plan cache nhanh, dễ optimize tay. Nhược: cố định, mỗi tổ hợp filter = 1 method mới, optional param phải hack `(:p IS NULL OR ...)`.

Specification là wrapper quanh **JPA Criteria API** — build query bằng code Java (object) thay vì string. Compose động qua `.and()`/`.or()`. Ưu: compose động đúng filter user truyền, tái sử dụng mảnh, không 2^N method. Nhược: verbose, `root.get("field")` là String (typo lỗi runtime), build cây tốn CPU hơn.

**Quan trọng — `@Query` KHÔNG động:** nó là string literal compile vào bytecode, KHÔNG nối thêm điều kiện runtime được. "Query động" thực sự = Specification / QueryDSL. Cảm giác "cần động" của người học đúng, nhưng `@Query` không phải công cụ đó.

**Production dùng cái nào — câu trả lời thật:** dùng **cả hai**, đúng chỗ:

| Tình huống | Chọn |
|-----------|------|
| Query cố định, biết trước (findByEmail, report) | `@Query` / derived |
| 0-1 optional param | `@Query` + `(:p IS NULL OR...)` |
| ≥ 4 optional filter, user tổ hợp tùy ý | **Specification / QueryDSL** |
| Full-text search, relevance, typo-tolerance | **Elasticsearch** |

QueryDSL: nhiều team production dùng thay Specification vì type-safe (`product.price.goe(min)` thay `root.get("price")` String). Đổi tên field → compile error. Nhược: thêm dependency + build step sinh Q-class.

Lộ trình dự án: 3.3.2 dùng `@Query` (1 param) → 3.3.3 refactor sang Specification (5 filter) → 3.3.4 Elasticsearch. Học bằng cách "đau" qua giới hạn, không nghe lý thuyết suông.

## Bài học rút ra 3.3.2

- `@Query` là string **tĩnh**, optional param dùng trick `IS NULL OR`.
- **Luôn bọc nhóm OR trong ngoặc** — phòng bug precedence `AND > OR` khi mở rộng filter. (Người học ban đầu quên ngoặc → đã fix.)
- Controller: `@RequestParam(required = false) String keyword` để optional. `@Nullable` chỉ là doc, KHÔNG làm optional; phải `required = false`.

# 3.3.3 — Filter động bằng JPA Specification

## Lý thuyết cốt lõi

### Vì sao `@Query` "chết" khi nhiều filter

5 filter optional (keyword + categoryId + minPrice + maxPrice + status), user tổ hợp tùy ý. Dùng `@Query` + `(:p IS NULL OR ...)`:
1. **DB optimizer mù**: không biết runtime param null hay không → không chọn được index plan tối ưu → fallback full scan.
2. **Không tái sử dụng + không mở rộng**: mỗi query động = 1 string khổng lồ, thêm filter phải sửa string dễ sai ngoặc.

→ Ngưỡng chuyển sang build query bằng code (Specification).

### Criteria API — nền tảng

Build query bằng Java object:
```java
CriteriaBuilder cb       // "nhà máy" tạo predicate
CriteriaQuery<Product> q // cả câu query
Root<Product> root       // bảng gốc (FROM Product) — truy cập field
Predicate                // 1 mệnh đề WHERE
```

Mỗi `Predicate` là object độc lập → tạo có điều kiện, bỏ vào List, ghép sau = "động".

### `Specification<T>` — functional interface

```java
@FunctionalInterface
public interface Specification<T> {
    Predicate toPredicate(Root<T> root, CriteriaQuery<?> query, CriteriaBuilder cb);
}
```

1 Specification = 1 mảnh WHERE, viết bằng lambda. Compose qua `.and()`/`.or()`.

Repository extends `JpaSpecificationExecutor<Product>` → có `findAll(spec, pageable)`, `count(spec)`.

### N+1 problem khi paginate có collection — BẪY LỚN NHẤT

`Product` có `images` lazy, mapper đụng `images` để lấy `primaryImageUrl`. Load 20 product → 1 page query + 20 query images = **21 query** (N+1).

| Giải pháp | Query | Pagination | Khi nào |
|-----------|-------|------------|---------|
| Lazy mặc định | 1 + 2N | ✅ | ❌ N+1 |
| `JOIN FETCH` collection + Pageable | 1 | ❌ **SAI/OOM** | ❌ tránh với *To-Many |
| **`@BatchSize`** | 1 + ceil(N/batch) | ✅ | ✅ list cần collection |
| List DTO không có collection | 1 | ✅ | ✅✅ tốt nhất |
| `JOIN FETCH` + Pageable cho *To-One* | 1 | ✅ | ✅ category, seller (ManyToOne an toàn) |

**Quy tắc vàng:** `JOIN FETCH` + Pageable **an toàn với `@ManyToOne`** (không nhân dòng), **nguy hiểm với `@OneToMany`** (nhân dòng → LIMIT cắt sai → Hibernate warning `HHH000104` → paginate in-memory → OOM).

**Dự án chọn `@BatchSize(20)`** trên `images`:
```java
@OneToMany(mappedBy = "product", cascade = CascadeType.ALL, orphanRemoval = true)
@BatchSize(size = 20)
private List<ProductImage> images = new ArrayList<>();
```

## Code cuối cùng

**ProductSpecification** (5 static method):
```java
public class ProductSpecification {
    private ProductSpecification() {}   // utility class, chặn new

    public static Specification<Product> hasKeyword(String keyword) {
        String pattern = "%" + keyword.toLowerCase() + "%";
        return (root, query, cb) -> cb.or(
                cb.like(cb.lower(root.get("name")), pattern),
                cb.like(cb.lower(root.get("description")), pattern));
    }
    public static Specification<Product> hasCategory(Long categoryId) {
        return (root, query, cb) -> cb.equal(root.get("category").get("id"), categoryId);
    }
    public static Specification<Product> priceGoe(BigDecimal minPrice) {
        return (root, query, cb) -> cb.greaterThanOrEqualTo(root.get("price"), minPrice);
    }
    public static Specification<Product> priceLoe(BigDecimal maxPrice) {
        return (root, query, cb) -> cb.lessThanOrEqualTo(root.get("price"), maxPrice);
    }
    public static Specification<Product> hasStatus(ProductStatus status) {
        return (root, query, cb) -> cb.equal(root.get("status"), status);
    }
}
```

**ProductService** — build spec động:
```java
Specification<Product> spec = Specification.unrestricted();   // Spring Boot 3.x (thay where(null) deprecated)
if (CheckData.checkIsNotNull(filter.getKeyword()))    spec = spec.and(ProductSpecification.hasKeyword(filter.getKeyword()));
if (CheckData.checkIsNotNull(filter.getCategoryId())) spec = spec.and(ProductSpecification.hasCategory(filter.getCategoryId()));
if (CheckData.checkIsNotNull(filter.getMinPrice()))   spec = spec.and(ProductSpecification.priceGoe(filter.getMinPrice()));
if (CheckData.checkIsNotNull(filter.getMaxPrice()))   spec = spec.and(ProductSpecification.priceLoe(filter.getMaxPrice()));
if (CheckData.checkIsNotNull(filter.getStatus()))     spec = spec.and(ProductSpecification.hasStatus(filter.getStatus()));

Page<ProductResponse> products = productRepository.findAll(spec, pageable).map(productMapper::fromEntity);
return PageResponse.from(products);
```

> `CheckData.checkIsNotNull` xử lý thông minh: `null → false`, `Long l → l > 0`, `String → !isBlank()`, `List → !isEmpty()`. Nên không cần check `isBlank()` riêng cho keyword.

**ProductController** — bind object:
```java
@GetMapping
public ResponseEntity<BaseResponse<PageResponse<ProductResponse>>> getAll(
        ProductFilterRequest filter,   // Spring auto-bind query param qua @ModelAttribute ngầm
        Pageable pageable
) { ... }
```

## BUG kinh điển gặp phải: Specification IMMUTABLE

Người học viết ban đầu:
```java
if (...) spec.and(ProductSpecification.hasKeyword(...));   // ❌ vứt kết quả!
```

`spec.and(...)` KHÔNG sửa `spec` — nó trả về **Specification MỚI**. Không gán lại → object mới bị vứt, `spec` vẫn rỗng → filter chết, mọi request trả tất cả. **Compile pass nhưng logic sai** — chỉ test data thật mới lộ.

Fix: `spec = spec.and(...)` (gán lại cả 5 dòng).

Giống hệt `String` immutable:
```java
s.toUpperCase();        // ❌ vứt, s không đổi
s = s.toUpperCase();    // ✅
```

`String`, `Specification`, `BigDecimal`, `LocalDate`, Stream... đều immutable → mọi method "biến đổi" trả object mới. **Quên gán lại = bug âm thầm.** IntelliJ có inspection "Result of `.and()` is ignored" → bật để bắt sớm.

## Q&A 3.3.3 (verbatim)

**Câu hỏi: trong Product bạn có thêm một annotation là @BatchSize(size = 20), điều gì sẽ xảy ra nếu số lượng product lớn hơn hoặc nhỏ hơn 20 thì sao.**

**Trả lời:** `size = 20` KHÔNG có nghĩa "chỉ xử lý 20 product". Nó là **kích thước tối đa mỗi lô (batch)** khi Hibernate gom `product_id` để load `images`: "mỗi lần cần load lazy collection, gom tối đa 20 parent id chưa khởi tạo vào 1 câu `WHERE product_id IN (?, ..., ?)`". Giống `chunk(array, 20)` bên Node.js.

3 trường hợp (N = số product page query trả về):

- **N = 20** (đúng batch): 1 page query + 1 image query (`IN (1..20)`) = **2 query**.
- **N < 20** (vd 5): gom hết 5 id vào 1 lô = **2 query**.
- **N > 20** (vd 50): chia `ceil(N/batchSize)` lô:
  ```
  Lô 1: IN (1..20)   → 1 query
  Lô 2: IN (21..40)  → 1 query
  Lô 3: IN (41..50)  → 1 query
  Tổng: 1 + 3 = 4 query
  ```

Công thức: **số query con = `ceil(N / batchSize)`**. Vẫn tốt hơn rất nhiều so với không có `@BatchSize` (N=50 → 51 query).

**Chọn size bao nhiêu:** khớp page size phổ biến nhất (dự án = 20). Trade-off: nhỏ quá → nhiều round-trip; lớn quá → câu IN dài (Oracle giới hạn 1000 phần tử, ORA-01795). Sweet spot e-commerce: 20-50.

**Đặt ở đâu:** field-level (`@BatchSize` trên collection — ưu tiên cao, rõ ràng) hoặc entity-level (`@BatchSize` trên class `ProductImage`). Hoặc global: `spring.jpa.properties.hibernate.default_batch_fetch_size: 20` (áp dụng mọi lazy association — tiện nhưng kém tường minh).

## Bài học rút ra 3.3.3

1. `@Query` "IS NULL OR" làm optimizer mù → nhiều filter dùng **Specification** để query sạch, index tốt.
2. **`Specification` IMMUTABLE** → `spec = spec.and(...)` bắt buộc gán lại. Cùng họ `String`, `BigDecimal`, `LocalDate`.
3. **Compile pass ≠ logic đúng** — bug immutable chỉ lộ khi test data thật → luôn test filter thực tế.
4. **N+1 + Pagination:** `JOIN FETCH` collection làm SAI pagination → dùng `@BatchSize`; `JOIN FETCH` chỉ an toàn với `@ManyToOne`.
5. `root.get("category").get("id")` đi qua quan hệ — Hibernate tự JOIN.
6. Field name String trong Criteria → typo lỗi **runtime** (nâng cao: Metamodel/QueryDSL).
7. `Specification.unrestricted()` thay `Specification.where(null)` (deprecated) ở Spring Boot 3.x.
