## 2. Skeleton `UserMapper`

```java
@Mapper
public interface UserMapper {

    UserResponse fromEntity(User user);

    @Mapping(target = "password", ignore = true)
    @Mapping(target = "id", ignore = true)
    @Mapping(target = "role", ignore = true)
    @Mapping(target = "status", ignore = true)
    @Mapping(target = "createdAt", ignore = true)
    @Mapping(target = "updatedAt", ignore = true)
    @Mapping(target = "orderList", ignore = true)
    @Mapping(target = "cart", ignore = true)
    User fromRequestDto(CreateUserRequest request);

    @BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE)
    @Mapping(target = "id", ignore = true)
    @Mapping(target = "email", ignore = true)
    @Mapping(target = "password", ignore = true)
    @Mapping(target = "role", ignore = true)
    @Mapping(target = "status", ignore = true)
    @Mapping(target = "createdAt", ignore = true)
    @Mapping(target = "updatedAt", ignore = true)
    @Mapping(target = "orderList", ignore = true)
    @Mapping(target = "cart", ignore = true)
    void updateEntity(UpdateUserRequest request, @MappingTarget User user);
}
```

3 method:
1. **`fromEntity`** — User → UserResponse, không cần `@Mapping` vì field cùng tên auto-match
2. **`fromRequestDto`** — CreateUserRequest → User, cần ignore tất cả field entity không có trong request (vì `unmappedTargetPolicy=ERROR`)
3. **`updateEntity`** — partial update, mutate vào entity sẵn có qua `@MappingTarget`, dùng `NullValuePropertyMappingStrategy.IGNORE` để field null trong request không đè entity

## 3. MapStruct deep dive — cơ chế bên trong

### So sánh Node.js
Bên Node.js, mapping object thường viết tay hoặc dùng `class-transformer`:
```typescript
plainToClass(UserResponse, user);  // reflection runtime → chậm
```

MapStruct khác hẳn: là **annotation processor** chạy ở **compile-time**. Bạn viết interface rỗng, MapStruct sinh ra file `.java` thật chứa code mapping tay — như tự viết, nhưng máy viết hộ.

### Flow internal

```
                    ┌─────────────────────────┐
   Bạn viết:        │  UserMapper.java        │  ← interface, rỗng
                    │  @Mapper                │
                    │  UserResponse           │
                    │    fromEntity(User u);  │
                    └────────────┬────────────┘
                                 │ javac chạy
                                 ▼
                    ┌─────────────────────────┐
   MapStruct sinh:  │  UserMapperImpl.java    │  ← code thật, sinh tự động
                    │  @Component             │     trong target/generated-sources/
                    │  public UserResponse    │
                    │    fromEntity(User u) { │
                    │      var r = new ...;   │
                    │      r.setId(u.getId());│
                    │      r.setEmail(...);   │
                    │      return r;          │
                    │    }                    │
                    └────────────┬────────────┘
                                 │ Spring scan
                                 ▼
                    ┌─────────────────────────┐
                    │  Spring container       │
                    │  bean UserMapperImpl    │  → @Autowired bình thường
                    └─────────────────────────┘
```

**Key insight:**
- Không có reflection lúc chạy → nhanh ngang code tay
- Lỗi mapping bị bắt ở **compile-time** (đổi tên field mà quên update → build fail)
- File sinh ra ở `api/target/generated-sources/annotations/...` — mở xem code thật MapStruct viết để hiểu sâu

### Quy tắc match field

MapStruct match field theo **tên**. Field tên giống nhau giữa source/target → auto map. Field tên khác → dùng `@Mapping(source=..., target=...)`. Field ignore → `@Mapping(target=..., ignore=true)`.

### `componentModel = "spring"`

Mặc định MapStruct sinh class có constructor `new UserMapperImpl()` → không phải bean. Cần báo "đánh `@Component`":
```java
@Mapper(componentModel = "spring")
```

Trong dự án này đã set global qua `compilerArgs` (`mapstruct.defaultComponentModel=spring`) → mọi `@Mapper` đều mặc định là Spring bean, không cần ghi lại.

### Pattern partial update

`UserService.updateUser` cũ phải check `null/blank` từng field rồi mới set:
```java
if (request.getFullName() != null && !request.getFullName().isBlank())
    user.setFullName(request.getFullName());
if (request.getAddress() != null && ...) ...
if (request.getPhone() != null && ...) ...
```

→ MapStruct gom thành 1 dòng:
```java
userMapper.updateEntity(request, user);
```

Cơ chế:
- `@MappingTarget User user` → "đừng tạo entity mới, set vào entity có sẵn"
- `NullValuePropertyMappingStrategy.IGNORE` → "field nào null thì bỏ qua, không set null đè lên"

→ Đây là **chính giá trị** lớn nhất của MapStruct trong service.

## 4. Q&A trong buổi học

### Câu 1 (review skeleton): `@RequiredArgsConstructor` sẽ tạo ra một constructor với đầy đủ fields à, thế còn `@AllArgsConstructor` thì sao nhỉ?

**Trả lời (verbatim):**

Anh hiểu nhầm: `@RequiredArgsConstructor` **KHÔNG** sinh constructor với "đầy đủ fields". Nó chỉ sinh constructor với các field "required". Cụ thể:

#### Bảng so sánh

| Annotation | Sinh constructor với những field nào? |
|---|---|
| `@NoArgsConstructor` | **Không field nào** — constructor rỗng `new User()` |
| `@RequiredArgsConstructor` | Chỉ field **`final`** hoặc field có **`@NonNull`** mà CHƯA được khởi tạo |
| `@AllArgsConstructor` | **Tất cả** field (final hay không, có `@NonNull` hay không) |

#### Ví dụ minh hoạ

```java
public class Foo {
    private final String a;        // final
    private String b;              // không final
    @NonNull private String c;     // có @NonNull
    private final String d = "x";  // final NHƯNG đã init
}
```

| Annotation | Constructor sinh ra |
|---|---|
| `@NoArgsConstructor` | `Foo()` |
| `@RequiredArgsConstructor` | `Foo(String a, String c)` ← chỉ `a` (final, chưa init) + `c` (@NonNull). `d` final nhưng đã init → bỏ. `b` không final, không @NonNull → bỏ. |
| `@AllArgsConstructor` | `Foo(String a, String b, String c, String d)` |

#### Tại sao Spring DI nên dùng `@RequiredArgsConstructor`?

Pattern chuẩn Spring constructor injection:
```java
@Service
@RequiredArgsConstructor
public class UserService {
    private final UserRepository userRepository;  // ← inject
    private final UserMapper userMapper;           // ← inject
    private String someConfigValue;                // ← KHÔNG inject, set sau
}
```

- `@RequiredArgsConstructor` → constructor có **2 tham số** (`userRepository`, `userMapper`) → Spring inject 2 cái này, không động vào `someConfigValue`. ✅
- `@AllArgsConstructor` → constructor có **3 tham số** → Spring phải tìm bean cho `someConfigValue` (String) → không có bean → app fail khi khởi động. ❌

Đó là lý do **mọi Spring bean nên dùng `@RequiredArgsConstructor` + khai báo dependency là `final`**, không bao giờ dùng `@AllArgsConstructor` trên Spring bean.

#### Khi nào dùng `@AllArgsConstructor`?

Hợp với:
- **DTO test data**: `new UserResponse(1L, "a@b.com", "An", ...)` — viết test gọn
- **Value object / immutable record-like class** (Java 17+ thì dùng `record` còn gọn hơn)

#### So sánh Node.js

Bên TypeScript anh viết `constructor(private foo: Foo, private bar: Bar) {}` — đây tương đương `@RequiredArgsConstructor` (chỉ những param được khai báo). Không có khái niệm "all args".

### Câu 2: Tôi đang set nếu không map được thì sẽ lỗi mà nhỉ: `<arg>-Amapstruct.unmappedTargetPolicy=ERROR</arg>`

**Trả lời (verbatim):**

Anh đúng, em sai chỗ đó. Trước đó em nói "có thể xoá hết `@Mapping(ignore=true)` vì global IGNORE" — nhưng anh set `ERROR` global → mapper phải khai báo **MỌI** field không map. Build output:

```
UserMapper.java:[40] Unmapped target properties: "updatedAt, orderList, cart"
UserMapper.java:[48] Unmapped target properties: "id, email, password, role, status,
                                                  createdAt, updatedAt, orderList, cart"
```

#### Trade-off `ERROR` vs `IGNORE` — bàn nghiêm túc

| Tiêu chí | `ERROR` (đang dùng) | `IGNORE` |
|---|---|---|
| An toàn khi đổi schema | ✅ Thêm field mới vào entity → build fail → buộc xử lý ngay | ❌ Field mới bị silent skip → có thể quên |
| Verbose | ❌ Rất verbose, mapper to | ✅ Mapper gọn |
| Tự documenting | ✅ Đọc mapper biết ngay field nào skip | ❌ Phải đọc cả entity lẫn DTO mới biết |
| Rủi ro với entity quan hệ | ❌ Mỗi lần thêm `@OneToMany` mới → mọi mapper liên quan phải thêm ignore | ✅ Tự bỏ qua |

**Quan điểm senior:** với dự án học/portfolio thì `ERROR` rất tốt vì rèn thói quen "nghĩ về mọi field". Nhưng với dự án production thực tế, đa số team chọn `IGNORE` cho `unmappedTargetPolicy` kết hợp `ERROR` cho `unmappedSourcePolicy` (chiều ngược lại):
- DTO có field mà entity không có → ERROR (typo nguy hiểm)
- Entity có field mà DTO không có → IGNORE (chuyện thường)

Pattern này cân bằng tốt giữa an toàn và verbose. Dự án giữ `ERROR` để học cho kỹ hết bài 3.2.5, đến Product module xem có quá verbose không rồi quyết.

## 5. Bug & bẫy thực tế phát hiện trong buổi review

### Bug 1: Dùng cả `@AllArgsConstructor` + `@RequiredArgsConstructor` trên cùng class

```java
@AllArgsConstructor
@RequiredArgsConstructor  // ← cả 2 cùng lúc
@Service
public class UserService {
    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;
    private final UserMapper userMapper;
}
```

Tất cả field đều `final` → 2 annotation sinh ra constructor **cùng signature** → compile error "duplicate constructor" (hoặc Lombok skip 1 cái nhưng vẫn là code smell).

**Fix:** xoá `@AllArgsConstructor`, giữ `@RequiredArgsConstructor`.

### Bug 2: Bẫy bảo mật trong `fromRequestDto` — quên ignore password

Lần đầu viết mapper:
```java
@Mapping(target = "id", ignore = true)
@Mapping(target = "role", ignore = true)
// ... thiếu @Mapping(target = "password", ignore = true)
User fromRequestDto(CreateUserRequest request);
```

`CreateUserRequest` có field `password` plain, entity `User` cũng có field `password` → MapStruct **map thẳng plain password sang entity**. Service sau đó mới `setPassword(passwordEncoder.encode(...))` đè lên — kết quả cuối cùng đúng, NHƯNG:
- Plain password tồn tại trong memory một khoảnh khắc
- Nếu mai mốt ai refactor service mà quên `setPassword(hash)` → **plain password ngấm vào DB**. Lỗi loại này rất khó phát hiện qua test.

**Fix:** thêm `@Mapping(target = "password", ignore = true)` để mapper khai báo rõ "password KHÔNG đi qua đây, service tự xử lý".

### Bug 3: `@AllArgsConstructor` trên Controller — pattern không an toàn

```java
@AllArgsConstructor
@RestController
public class UserController {
    private final UserService userService;
}
```

Hiện tại chạy được vì chỉ có 1 field final. Nhưng mai mốt thêm field non-final (`@Value String appVersion` hoặc field cấu hình) → `@AllArgsConstructor` ép luôn field đó vào constructor → Spring không inject được → app fail.

**Fix:** đổi sang `@RequiredArgsConstructor` cho **mọi** Spring bean (Service/Controller/Component). Đây là quy ước chuẩn.

### Bẫy 4: Javadoc đặt SAU annotation

```java
@Getter
@Setter
/**
 * DTO cho API cập nhật profile...   ← Javadoc đặt SAU annotation
 */
public class UpdateUserRequest { ... }
```

Javadoc phải đứng **trước** annotation thì IDE/tool mới gắn được vào class. Vị trí hiện tại bị coi là comment lơ lửng, không xuất hiện trong Javadoc generation.

## 6. Pattern comment WHY (không phải WHAT)

Theo `CLAUDE.md` "Comment chỉ nên nói WHY, code đã nói WHAT rồi". Sau khi clean TODO scaffolding, chỉ giữ 3 comment WHY ở chỗ "tại sao lại làm ngược/lạ":

**`UserMapper.fromRequestDto`:**
```java
// password: ignore cố ý để service hash trước khi set, tránh plain password lọt qua mapper
@Mapping(target = "password", ignore = true)
```

**`UserMapper.updateEntity`:**
```java
// NullValuePropertyMappingStrategy.IGNORE: partial update — field null trong request không đè entity
@BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE)
```

**`UserService.createUser`:**
```java
// hash password sau khi map vì mapper cố ý ignore field password
user.setPassword(this.passwordEncoder.encode(request.getPassword()));
```

→ 3 comment này kết nối 2 điểm cách xa nhau (mapper ↔ service) bằng giải thích "tại sao". Đọc 1 trong 2 chỗ là hiểu cả cụm.

## 7. Production-grade issues đã defer

Buổi review đã chỉ ra **nhiều vấn đề production** vượt scope bài 3.2.5. Đã chốt tách riêng **một buổi cleanup chuyên sâu** sau ROADMAP để xử lý tập trung. Memory `project-production-cleanup-session` đã ghi lại đầy đủ.

Tóm tắt:
- Thiếu `@Transactional` ở Service layer → race condition, lost update
- Thiếu `@Version` (optimistic locking)
- TOCTOU race khi `existsByEmail` + `save` ở 2 transaction tách biệt → cần catch `DataIntegrityViolationException`
- Timestamp set tay trong service → nên dùng `@CreationTimestamp` + `@UpdateTimestamp` hoặc Spring Data JPA Auditing
- Service chưa có `@Slf4j` + log audit
- Validation password yếu (chỉ `@Size(min=6)`, thiếu regex complexity)
- Cân nhắc đổi Response DTO sang Java `record`
- Naming MapStruct: convention industry là `toResponse` / `toEntity`, không phải `fromEntity` / `fromRequestDto`

## 8. Kết quả cuối

### Build
```bash
./mvnw clean compile
# BUILD SUCCESS — không còn warning mapstruct.unmappedTargetPolicy
```

### File `UserService.java` sau cùng

```java
@RequiredArgsConstructor
@Service
public class UserService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;
    private final UserMapper userMapper;

    public UserResponse createUser(CreateUserRequest request) {
        boolean existUser = userRepository.existsByEmail(request.getEmail());
        if (existUser) throw new DuplicateResource("Email is already registered");
        User user = userMapper.fromRequestDto(request);
        // hash password sau khi map vì mapper cố ý ignore field password
        user.setPassword(this.passwordEncoder.encode(request.getPassword()));
        user.setCreatedAt(LocalDateTime.now());
        User createdUser = this.userRepository.save(user);
        return userMapper.fromEntity(createdUser);
    }

    public UserResponse getUserById(Long id) {
        User user = this.userRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("User not found"));
        return userMapper.fromEntity(user);
    }

    public UserResponse updateUser(Long id, UpdateUserRequest request) {
        User user = this.userRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException("User not found"));
        userMapper.updateEntity(request, user);
        user.setUpdatedAt(LocalDateTime.now());
        User userUpdate = this.userRepository.save(user);
        return userMapper.fromEntity(userUpdate);
    }
}
```

So với bản trước refactor: ngắn hơn ~30%, constructor injection biến mất, logic `updateUser` từ 10 dòng if-null xuống 1 dòng `userMapper.updateEntity`.
