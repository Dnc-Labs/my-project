## 1. Cấu trúc project Maven

```
api/src/
├── main/           ← Code chính của ứng dụng
│   ├── java/       ← Viết code Java ở đây
│   │   └── com/ecommerce/api/
│   │       ├── ApiApplication.java
│   │       ├── controllers/
│   │       ├── services/
│   │       └── dto/
│   └── resources/  ← File cấu hình (application.yaml, templates...)
│
└── test/           ← Code test (JUnit)
    └── java/
        └── com/ecommerce/api/
```

| Thư mục | Mục đích | Đóng gói vào .jar? |
|---|---|---|
| `src/main/java` | Code ứng dụng chạy thật | Có |
| `src/main/resources` | File cấu hình | Có |
| `src/test/java` | Code test (JUnit, Mockito) | Không |

> So sánh Node.js: `src/` ↔ `src/main/java/`, `test/` hoặc `__tests__/` ↔ `src/test/java/`

**Lưu ý**: `src/main/java/` chỉ là đường dẫn vật lý của Maven. Package Java bắt đầu **sau** `java/`:

```
Đường dẫn:  src/main/java/com/ecommerce/api/services/MyService.java
Package:    com.ecommerce.api.services   ✅
KHÔNG phải: src.main.java.com.ecommerce.api.services   ❌
```

## 2. Layered Architecture

```
Client (HTTP Request)
        ↓
Controller  — Nhận request, gọi Service, trả response
        ↓
Service     — Business logic
        ↓
Repository  — Truy cập database
        ↓
Entity      — Đại diện bảng trong database
```

Ngoài ra còn có **DTO** (Data Transfer Object) — object dùng để truyền dữ liệu giữa các layer hoặc trả về cho client, tách biệt khỏi Entity.

## 3. REST API trong Spring Boot

```java
@RestController                    // Đánh dấu class xử lý HTTP, trả JSON
@RequestMapping("/greeting")       // Base path
public class GreetingController {

    private final GreetingService greetingService;

    // Constructor Injection
    GreetingController(GreetingService greetingService) {
        this.greetingService = greetingService;
    }

    @GetMapping                    // GET /greeting
    ResponseEntity<String> getGreeting(@RequestParam(required = false) String name) {
        String result = this.greetingService.handleGetGreeting(name);
        return ResponseEntity.ok(result);
    }
}
```

### Các annotation HTTP mapping

| Annotation | HTTP Method |
|---|---|
| `@GetMapping` | GET |
| `@PostMapping` | POST |
| `@PutMapping` | PUT |
| `@DeleteMapping` | DELETE |
| `@PatchMapping` | PATCH |

### `@GetMapping` vs `@GetMapping("/")`

- `@RequestMapping("/greeting")` + `@GetMapping("/")` → chỉ match `/greeting/` (có slash cuối)
- `@RequestMapping("/greeting")` + `@GetMapping` → match cả `/greeting` lẫn `/greeting/`

### `@RequestParam` — nhận query parameter

```java
// GET /greeting?name=Avada
@GetMapping
ResponseEntity<String> getGreeting(@RequestParam(required = false) String name) {
    // required = false → name có thể null
}
```

### `ResponseEntity` — kiểm soát HTTP response

```java
// Cách đầy đủ
return new ResponseEntity<>(body, HttpStatus.OK);

// Cách gọn
return ResponseEntity.ok(body);
```

Khi trả về object, Spring Boot (nhờ Jackson) **tự động convert thành JSON**.

## 4. `application.yml` — File cấu hình

### Config chuẩn của Spring vs config tự định nghĩa

```yaml
# Config chuẩn Spring — phải đúng key, Spring tự đọc
server:
  port: 8081
spring:
  application:
    name: my-app
  datasource:
    url: jdbc:mysql://localhost:3306/mydb

# Config tự định nghĩa — đặt tên tùy ý, tự đọc bằng @Value
app:
  name: my-app
  version: "1.0"
  description: "This is learning app"
```

### Tra cứu config chuẩn của Spring

1. **Official docs**: https://docs.spring.io/spring-boot/appendix/application-properties/
2. **IDE auto-complete**: IntelliJ gợi ý khi gõ trong `application.yml`
3. **Đọc class `*Properties`**: mỗi starter có class chứa các config (ví dụ `DataSourceProperties`)

### `.properties` vs `.yml`

```properties
# application.properties — phẳng, lặp prefix
server.port=8081
spring.datasource.url=jdbc:mysql://localhost:3306/mydb
```

```yaml
# application.yml — phân cấp, dễ đọc
server:
  port: 8081
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/mydb
```

Cả 2 hoạt động như nhau. Dự án thực tế phần lớn dùng `.yml`.

### Profiles — cấu hình theo môi trường

```
src/main/resources/
├── application.yml           # Config chung
├── application-dev.yml       # Môi trường dev
└── application-prod.yml      # Môi trường production
```

Kích hoạt profile:

```yaml
# application.yml
spring:
  profiles:
    active: dev    # hoặc qua command line: --spring.profiles.active=prod
```

### Quản lý secret trên production

**Không bao giờ đẩy password, secret key lên Git!**

**Cách phổ biến nhất — Environment Variables:**

```yaml
# application-prod.yml — commit được vì không chứa secret thật
spring:
  datasource:
    url: ${DB_URL}               # Đọc từ biến môi trường
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
```

```bash
# Trên server production
export DB_URL=jdbc:mysql://prod-server:3306/ecommerce
export DB_PASSWORD=super_secret_123
```

> So sánh Node.js: `.env` + `dotenv` ↔ `application.yml` + env var. `process.env.DB_URL` ↔ `${DB_URL}` trong yml.

### Đọc config trong code

```java
@Service
public class AppInfoService {

    private final String appName;

    public AppInfoService(@Value("${app.name}") String appName) {
        this.appName = appName;
    }
}
```

## 5. Bean Lifecycle — Vòng đời của Bean

```
1. Instantiation — Spring tạo object (gọi constructor)
        ↓
2. Dependency Injection — Tiêm dependency vào
        ↓
3. @PostConstruct — Chạy method khởi tạo sau khi inject xong
        ↓
4. Bean sẵn sàng sử dụng ✅
        ↓
   ... (ứng dụng chạy) ...
        ↓
5. @PreDestroy — Chạy method dọn dẹp trước khi Bean bị hủy
        ↓
6. Bean bị hủy (khi ứng dụng tắt)
```

### Ví dụ

```java
@Service
public class CacheService {

    private Map<String, String> cache;

    @PostConstruct
    public void init() {
        // Chạy SAU khi dependency đã inject xong
        // Dùng để: load dữ liệu ban đầu, warm up cache, check connection...
        cache = new HashMap<>();
        log.info("CacheService initialized!");
    }

    @PreDestroy
    public void cleanup() {
        // Chạy TRƯỚC khi Bean bị hủy
        // Dùng để: đóng connection, giải phóng resource, flush data...
        cache.clear();
        log.info("CacheService destroyed!");
    }
}
```

### Constructor vs `@PostConstruct`

- **Constructor**: chỉ nên gán dependency
- **`@PostConstruct`**: logic khởi tạo phức tạp — lúc này tất cả dependency đã sẵn sàng 100%

## 6. Câu hỏi thực hành

### Q: `private` vs `private final` trong field

```java
private String name;        // có thể bị thay đổi sau khi tạo
private final String name;  // gán 1 lần, không đổi được — compiler bảo vệ
```

**Khi nào cần `final`?**

| Trường hợp | Cần `final`? | Lý do |
|---|---|---|
| Bean field (Service, Controller) | **Nên** | Singleton, multi-thread, gán 1 lần |
| DTO trả về API | Tùy | Không bắt buộc, nhưng rõ ý đồ hơn |
| Entity (JPA) | **Không** | JPA cần setter để map dữ liệu từ database |

> So sánh Node.js: `final` ↔ `const`, không có `final` ↔ `let`. Mặc định dùng `const`, chỉ dùng `let` khi cần thay đổi.

### Q: DTO là gì? Tại sao tạo class riêng thay vì Map?

- **Type-safe** — compiler kiểm tra, không sợ typo key
- **Rõ ràng** — nhìn class biết ngay API trả về gì
- **Dễ mở rộng** — thêm validation, annotation sau này dễ dàng

`Map` nhanh cho prototype, nhưng dự án thật luôn dùng DTO.
