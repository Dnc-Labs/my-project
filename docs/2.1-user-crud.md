## 1. Validation — kiểm tra dữ liệu đầu vào

### So sánh với Node.js

Bên Node.js dùng Joi, Yup, hoặc `express-validator`. Bên Spring Boot dùng **Bean Validation** — annotation trực tiếp trên field của DTO.

Cần thêm dependency:
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
```

### Các annotation thường dùng

| Annotation | Ý nghĩa |
|---|---|
| `@NotNull` | Không được null |
| `@NotBlank` | Không được null, rỗng, hoặc chỉ có khoảng trắng (dùng cho String) |
| `@Email` | Phải đúng format email |
| `@Size(min, max)` | Độ dài tối thiểu/tối đa |
| `@Min`, `@Max` | Giá trị số tối thiểu/tối đa |
| `@Pattern(regexp)` | Phải match regex |

### Cách dùng

Đặt annotation trên field của DTO:

```java
public class CreateUserRequest {
    @NotBlank
    @Email
    private String email;

    @NotBlank
    @Size(min = 6)
    private String password;

    @NotBlank
    private String fullName;

    private String phone;    // không bắt buộc
    private String address;  // không bắt buộc
}
```

Trong Controller, thêm `@Valid` trước `@RequestBody` để kích hoạt validation:

```java
@PostMapping
public ResponseEntity<BaseResponse<UserResponse>> createUser(
        @Valid @RequestBody CreateUserRequest request) {
    // Nếu validation fail → Spring tự throw MethodArgumentNotValidException
    // Nếu pass → code chạy bình thường
}
```

## 2. Xử lý MethodArgumentNotValidException

Khi `@Valid` fail, Spring throw `MethodArgumentNotValidException`. Cần thêm handler trong `GlobalExceptionHandler`:

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<BaseResponse> handleValidation(MethodArgumentNotValidException e) {
    String message = e.getBindingResult().getFieldErrors().stream()
            .map(error -> error.getField() + ": " + error.getDefaultMessage())
            .collect(Collectors.joining(", "));
    return ResponseEntity.status(HttpStatus.BAD_REQUEST)
            .body(BaseResponse.error(message));
}
```

Kết quả trả về dễ đọc: `"email: must not be blank, password: size must be between 6 and 2147483647"`

### Thứ tự khai báo @ExceptionHandler

Đặt handler **cụ thể trước, chung sau**:

```java
@ExceptionHandler(ResourceNotFoundException.class)     // 1. Cụ thể nhất
@ExceptionHandler(MethodArgumentNotValidException.class) // 2. Cụ thể
@ExceptionHandler(Exception.class)                      // 3. Fallback — luôn đặt cuối
```

Nếu đặt `Exception.class` trước → nó "nuốt" hết tất cả exception, các handler cụ thể bên dưới không bao giờ được gọi.

## 3. DTO Pattern — tại sao không trả Entity trực tiếp?

Khi trả về User cho client, **không nên trả trực tiếp Entity** vì:
- Lộ **password** (dù đã hash)
- Lộ các field nội bộ không cần thiết
- Entity thay đổi → API response thay đổi theo (coupling)

Dùng **UserResponse** (DTO đầu ra) — chỉ chứa các field an toàn:

```java
public class UserResponse {
    private Long id;
    private String email;
    private String fullName;
    private String phone;
    private String address;
    private Role role;
    private Status status;

    public static UserResponse fromEntity(User user) {
        return new UserResponse(user.getId(), user.getEmail(),
                user.getFullName(), user.getPhone(), user.getAddress(),
                user.getRole(), user.getStatus());
    }
}
```

**`fromEntity()` static method:** Convert Entity → DTO ở 1 chỗ duy nhất. Nếu sau này thêm field, chỉ cần sửa 1 chỗ thay vì sửa mọi nơi tạo `UserResponse`.

## 4. Mã hóa Password — BCrypt

### Tại sao không lưu password thô?

Nếu database bị hack → hacker có toàn bộ password → dùng được luôn. Phải **hash password** trước khi lưu.

### BCrypt

BCrypt là thuật toán hash **1 chiều** — không thể giải ngược từ hash ra password gốc. Mỗi lần hash cùng 1 password sẽ cho kết quả khác nhau (vì có **salt** ngẫu nhiên).

```java
// Cấu hình Bean
@Configuration
public class PasswordEncoderConfig {
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}

// Sử dụng trong Service
String passwordHash = passwordEncoder.encode(request.getPassword());
user.setPassword(passwordHash);
```

### Dependency

Dùng `spring-security-crypto` — chỉ lấy phần mã hóa, không kéo theo Spring Security filter chain (tránh bị block tất cả API).

## 5. Câu hỏi phát sinh

### Q: @Configurable vs @Configuration?

| | `@Configuration` | `@Configurable` |
|---|---|---|
| Package | `org.springframework.context.annotation` | `org.springframework.beans.factory.annotation` |
| Mục đích | **Khai báo class chứa @Bean** — Spring quét và đăng ký Bean | Dùng cho AspectJ — inject dependency vào object tạo bằng `new` |
| Dùng khi | Cấu hình Bean, DataSource, Security... | Rất hiếm khi dùng |

**Nhầm lẫn phổ biến:** IDE gợi ý `@Configurable` khi gõ `@Config...` → chọn nhầm → Bean không được đăng ký → app lỗi khi khởi động.

### Q: spring-security-crypto vs spring-boot-starter-security?

| | `spring-security-crypto` | `spring-boot-starter-security` |
|---|---|---|
| Bao gồm | Chỉ BCrypt, SCrypt, Argon2... | **Toàn bộ** Spring Security (filter chain, authentication, authorization...) |
| Ảnh hưởng | Không ảnh hưởng API | **Tự động bảo vệ tất cả API** — yêu cầu login |
| Dùng khi | Chỉ cần hash password | Cần authentication + authorization đầy đủ |

Giai đoạn hiện tại dùng `spring-security-crypto` là hợp lý. Đến bài 2.2 sẽ chuyển sang `spring-boot-starter-security` để làm JWT authentication.

## 6. Luồng hoạt động API Create User

```
Client gửi POST /api/users
    ↓
UserController.createUser(@Valid @RequestBody CreateUserRequest)
    ↓
Spring kiểm tra @Valid
    ├── Fail → throw MethodArgumentNotValidException → GlobalExceptionHandler → 400
    └── Pass ↓
UserService.createUser(request)
    ├── Kiểm tra email trùng → isPresent() → throw RuntimeException
    ├── Hash password bằng BCryptPasswordEncoder
    ├── Tạo User entity, set createdAt
    ├── Lưu vào DB qua UserRepository.save()
    └── Return UserResponse.fromEntity(createdUser)
    ↓
Controller trả ResponseEntity 201 + BaseResponse.success()
```
