# Phần 1 — Deep dive: `@Transactional` & AOP Proxy

## 1.1 "Phép màu" của một annotation — so sánh Node.js

**Node.js + Sequelize** (explicit, không có magic):
```javascript
async createOrder(data) {
  const t = await sequelize.transaction();
  try {
    const order = await Order.create(data, { transaction: t });
    await Inventory.decrement(..., { transaction: t });
    await t.commit();              // gọi tay
    return order;
  } catch (err) {
    await t.rollback();            // gọi tay
    throw err;
  }
}
```

**Spring** (chỉ cần annotation):
```java
@Transactional
public Order createOrder(CreateOrderRequest req) {
    Order order = orderRepository.save(...);
    inventoryRepository.decrement(...);
    return order;
    // KHÔNG có commit/rollback ở đâu
}
```

**Câu hỏi:** Ai chạy `BEGIN` / `COMMIT` thay mình?
**Trả lời:** AOP proxy.

## 1.2 AOP — giải quyết "cross-cutting concerns"

Có những logic không thuộc business nhưng xuất hiện ở rất nhiều method:
- Transaction (begin/commit/rollback)
- Logging (log trước/sau method)
- Security (check quyền trước khi gọi)
- Caching
- Metrics

Được gọi là **cross-cutting concerns** vì cắt ngang qua nhiều layer.

**Trước AOP:** viết tay ở mọi method, 90% boilerplate, 10% business.

**Với AOP:** tách "concern" thành **aspect** riêng, "weave" (đan) vào method runtime thông qua **proxy**.

> So sánh Express: middleware (`app.use(logger)`) là AOP rất thô — chuỗi function explicit. Spring AOP implicit hơn — chỉ cần annotation, không thấy chuỗi đâu.

---

## 1.3 Proxy pattern — cách Spring implement AOP

### Khi Spring khởi tạo bean có `@Transactional`:

```
┌─────────────────────────────────────────────────┐
│  Bạn viết:                                      │
│  @Service                                       │
│  @Transactional                                 │
│  public class UserService {                     │
│      public User createUser(...) { ... }        │
│  }                                              │
└─────────────────────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────────┐
│  Spring KHÔNG inject thẳng UserService          │
│  Spring tạo class CON kế thừa UserService       │
│  (CGLIB proxy):                                 │
│                                                 │
│  UserService$$EnhancerBySpringCGLIB$$abc123     │
│                                                 │
│  → OVERRIDE TẤT CẢ method public                │
└─────────────────────────────────────────────────┘
                  │
                  ▼
   @Autowired UserService userService;
   → biến userService TRỎ VÀO PROXY
     không phải UserService thật
```

### Flow khi gọi method:

```
controller.createUser(req)
        │
        ▼
    [PROXY].createUser(req)
        │
        ├─► TransactionInterceptor.invoke()
        │      │
        │      ├─► txManager.getTransaction()   ← BEGIN
        │      │
        │      ├─► [TARGET REAL].createUser(req)   ← code mình viết
        │      │      │
        │      │      └─► userRepository.save(...)
        │      │
        │      └─► return / catch
        │            ├─► success → commit()
        │            └─► RuntimeException → rollback()
        │
        └─► return result lên controller
```

### 2 loại proxy:

| | **JDK Dynamic Proxy** | **CGLIB Proxy** |
|---|---|---|
| Cơ chế | `Proxy.newProxyInstance()` | Bytecode generation — subclass runtime |
| Yêu cầu | Class phải có interface | Không cần interface |
| Proxy là | Instance của interface | Instance của subclass |
| `final` method | OK | **Không proxy được** |
| `private` method | Không | Không |
| `static` method | Không | Không |
| Spring Boot default | (cũ) | **CGLIB từ 2.0** |

## 1.4 Pitfall huyền thoại — Self-invocation

```java
@Service
public class UserService {

    public void methodA() {
        this.methodB();   // gọi cùng class
    }

    @Transactional
    public void methodB() {
        userRepository.save(...);
    }
}
```

**Khi controller gọi `userService.methodA()`, transaction của `methodB` KHÔNG chạy.**

```
controller.methodA()
        │
        ▼
   [PROXY].methodA()
        │
        ├─► [TARGET REAL].methodA()           ← code mình viết
        │       │
        │       └─► this.methodB()             ← THIS Ở ĐÂY = TARGET REAL
        │                                        KHÔNG PHẢI PROXY
        │            └─► userRepository.save(...)   ← KHÔNG có transaction
```

→ `this` bên trong method là instance **THẬT**, không phải proxy.
→ Bypass proxy → bypass `TransactionInterceptor` → `@Transactional` vô hiệu.

**Fix:** tách method sang class khác → inject lại.

**Quy tắc senior:** Thấy `this.methodCó@Transactional()` trong cùng class → red flag, refactor.

---

## 1.5 Pitfall thứ 2 — Checked vs RuntimeException

Default rollback rule của Spring:
- **Rollback** khi throw `RuntimeException` hoặc `Error`
- **KHÔNG rollback** khi throw checked `Exception` (vd `IOException`)

```java
@Transactional
public void doWork() throws IOException {
    repository.save(...);
    throw new IOException("oops");   // ← KHÔNG rollback !!!
}
```

→ Muốn rollback cả checked: `@Transactional(rollbackFor = Exception.class)`
→ **Best practice:** Custom exception đều extend `RuntimeException`. Tránh checked exception trong business layer.

## 1.6 4 quy tắc vàng

1. **Annotation `@Transactional` chỉ ăn khi method được gọi QUA proxy** → method phải `public`, gọi từ ngoài class.
2. **`this.method()` trong cùng class luôn bypass proxy** → tách class nếu cần.
3. **Mặc định chỉ rollback `RuntimeException`** → custom exception nên extend `RuntimeException`.
4. **Đọc method nào cũng đặt câu hỏi: "Method này đang được gọi qua proxy hay qua `this`?"** → trả lời được là hiểu AOP.

## 1.7 Q&A check kiến thức — verbatim

### Câu hỏi 1: `@Transactional` ở method `private`?

**Trả lời của học viên:**
> Nếu viết @Transactional ở một metho private thì nó sẽ không hoạt động, bản chất của @Transactional là đang tạo ra một proxy là một class mới, extend từ class cũ và sẽ ghi đè các method để quản lý. do đó private thì sẽ không ghi đè được để quản lý.

**Review:** ✅ Đúng hoàn toàn. CGLIB tạo subclass → override method để thêm logic transaction. `private` không thể override (Java spec) → proxy không "chạm" được.

**Bổ sung:** Tương tự với `static` method (thuộc class, không thuộc instance → không có khái niệm override) và `final` method (Java cấm override).

**Quy tắc nhớ:** Bất cứ thứ gì proxy KHÔNG override được → `@Transactional` vô hiệu **im lặng** (không error, không warning). Nguy hiểm nhất — bug âm thầm.

---

### Câu hỏi 2: `@Transactional` lên `final` method?

**Trả lời của học viên:**
> Tôi không hiểu việc class UserService không implemnt interface nào có ý nghĩa gì không, nhưng nếu đặt @Transaction lên 1 mehtod final thì cũng giống như private thì cũng không được, không hoạt động được.

**Review:** ✅ Đúng về kết quả (không hoạt động).

**Làm rõ ý "không implement interface":**
- Class **CÓ** interface → Spring có thể chọn JDK Dynamic Proxy (proxy tạo từ interface, không cần subclass)
- Class **KHÔNG** có interface → Spring bắt buộc CGLIB

| | JDK Proxy | CGLIB |
|---|---|---|
| Cơ chế | Implement lại interface | Subclass extend class gốc |
| `final` method có sao không? | **Không sao** | **Hỏng** (không override) |
| `final` class có sao không? | **Không sao** | **Hỏng** (không extend) |

→ Spring Boot default CGLIB → `final` method/class = `@Transactional` chết.
→ **Quy tắc:** Đừng dùng `final` trên Spring bean và method của nó. Lombok `@Value` (sinh class final) là trap với JPA Entity.

---

### Câu hỏi 3: Transaction qua đệ quy `this.buildTree(child)`?

**Trả lời của học viên:**
> Tôi nghĩ là @Transactional chỉ stop ở lần đầu, không dùng được cho các lần sau vì this vẫn đang chỉ đến object hiện tại.

**Review:** ❌ Sai một chút — đây là detail tinh tế nhất của AOP transaction.

Sự thật: transaction **kéo dài** qua tất cả lần đệ quy. Nhưng KHÔNG phải vì proxy — mà vì cơ chế khác:

#### Cơ chế: `TransactionSynchronizationManager` + `ThreadLocal`

Khi proxy gọi `TransactionInterceptor.invoke()`:
1. Lấy `Connection` JDBC từ pool
2. `connection.setAutoCommit(false)` + `BEGIN`
3. **Lưu Connection vào `ThreadLocal`** (key = DataSource, value = ConnectionHolder)
4. Gọi method thật

Trong method thật, mỗi khi Hibernate cần Connection:
```java
DataSourceUtils.getConnection(dataSource)
   ↓
   Check ThreadLocal trước → có Connection đang mở không?
   → CÓ → dùng lại Connection đó (cùng transaction)
   → KHÔNG → mở Connection mới (transaction mới)
```

**Vì cùng 1 thread chạy toàn bộ stack** → cùng `ThreadLocal` → cùng 1 Connection → **cùng 1 transaction**.

```
Thread #42:
  ─ [PROXY].getCategoryTree()
      │  ThreadLocal: { ds → Connection#7 (BEGIN) }
      │
      ├─ [REAL].getCategoryTree()
      │   │
      │   └─ this.buildTree(rootCat)             ← bypass proxy
      │       │  → DataSourceUtils.getConnection()
      │       │  → ThreadLocal có Connection#7 → DÙNG LẠI
      │       │  → vẫn TRONG transaction
      │       │
      │       └─ this.buildTree(child)           ← đệ quy tiếp
      │           │  vẫn Connection#7
      │
      └─ commit Connection#7 → clear ThreadLocal
```

#### Phân biệt 2 thứ:

| | Thông qua proxy? | Có transaction? |
|---|---|---|
| Caller ngoài gọi `getCategoryTree()` | ✅ | ✅ Transaction MỚI start |
| `this.buildTree(child)` đệ quy | ❌ | ✅ Transaction CŨ kéo dài (qua ThreadLocal) |
| `this.buildTree(child)` mà `buildTree` cũng có `@Transactional(REQUIRES_NEW)` | ❌ | ✅ Vẫn transaction CŨ (annotation bị bypass) |

**Insight:** `@Transactional` annotation chỉ ăn khi qua proxy, NHƯNG transaction context đã start rồi thì cứ kéo dài tự nhiên qua ThreadLocal.

→ **Quy tắc:** Đặt `@Transactional` ở Service-level method được Controller gọi vào. Method helper bên trong KHÔNG cần annotation.

## 1.8 Q&A — `@Transactional(readOnly = true)` deep dive

### Câu hỏi của học viên:
> bạn có đề cập đến @Transactional(readOnly = true), nó thực sự khác gì so với không có tham số, hãy giải thích kỹ hơn cho tôi, Hibernate sẽ skip dirty checking có nghĩ là sẽ làm gì.

### A. Hibernate Session — "Persistence Context"

Phải hiểu Persistence Context trước (Hibernate gọi là Session, JPA gọi là EntityManager — về cơ bản là một).

```
Khi @Transactional bắt đầu:
  ┌──────────────────────────────────────────┐
  │  Spring tạo 1 Session (EntityManager)    │
  │  + bind vào ThreadLocal                  │
  │                                          │
  │  Session = "1st-level cache"             │
  │           + "snapshot store"             │
  └──────────────────────────────────────────┘

Khi: User u = userRepository.findById(1L);
  ┌──────────────────────────────────────────┐
  │  Hibernate: SELECT * FROM users WHERE id=1│
  │  → load thành object u                    │
  │                                          │
  │  Session lưu LẠI 2 thứ:                  │
  │  ┌──────────────────────────────────┐    │
  │  │ Managed entities:                │    │
  │  │   { id=1 → User u (current) }    │    │
  │  ├──────────────────────────────────┤    │
  │  │ Snapshot (state lúc load):       │    │
  │  │   { id=1 → [Tung, t@x.com, ...]} │    │
  │  └──────────────────────────────────┘    │
  └──────────────────────────────────────────┘
```

→ Session lưu **2 bản** mỗi entity: bản hiện tại (managed) + bản snapshot (lúc load).

### B. Dirty Checking là gì?

**Dirty checking** = quá trình Hibernate **so sánh** state hiện tại với snapshot, phát hiện field nào bị sửa → tự sinh UPDATE.

```java
@Transactional
public void changeUserName(Long id, String newName) {
    User u = userRepository.findById(id).orElseThrow();
    u.setName(newName);
    // KHÔNG có save() / update() / persist()
}
```

**Sequelize phải làm:**
```javascript
const user = await User.findByPk(id);
user.name = newName;
await user.save();   // gọi explicit
```

**Hibernate KHÔNG cần `save()`** — tới cuối transaction (lúc commit):

```
Flush phase (chạy trước commit):
  for each managed entity:
    compare(current, snapshot)
    if (current.name != snapshot.name) → UPDATE users SET name=? WHERE id=?
```

### C. Cái giá của Dirty Checking

1. **Bộ nhớ:** Session giữ 2 bản (current + snapshot) → 10,000 entity = 20,000 object.
2. **CPU:** Lúc flush, lặp qua tất cả managed entity, so sánh từng field.
3. **Auto-flush bất ngờ:** Mỗi query trigger flush trước để query thấy state mới.

### D. `readOnly = true` làm 3 việc:

#### Việc 1: Đổi `FlushMode` sang `MANUAL`
```java
// Bình thường: FlushMode.AUTO
//   → trước mỗi query: flush
//   → trước commit: flush

// readOnly = true: FlushMode.MANUAL
//   → KHÔNG bao giờ tự flush
```
→ Dù cậu lỡ `entity.setName(...)`, Hibernate KHÔNG so sánh snapshot, KHÔNG sinh UPDATE.

#### Việc 2: Hibernate skip lưu snapshot (một số version)
Tuỳ Hibernate version, có thể skip allocate snapshot → tiết kiệm bộ nhớ ngay từ load.

#### Việc 3: Hint xuống JDBC Connection
```java
connection.setReadOnly(true);
```
- **PostgreSQL:** session vào read-only mode → reject mọi `INSERT/UPDATE/DELETE` ngay tại DB
- **MySQL:** routing sang read replica (nếu config)
- Driver có thể skip cấp transaction id

### E. So sánh head-to-head

| Khía cạnh | `@Transactional` | `@Transactional(readOnly = true)` |
|---|---|---|
| Hibernate FlushMode | `AUTO` | `MANUAL` |
| Lưu snapshot? | Có | Tuỳ version, thường không |
| Dirty check khi flush? | Có | Không (vì không flush) |
| Bộ nhớ Session | Cao | Thấp hơn ~50% |
| CPU lúc commit | Lặp so sánh | Skip phase |
| Routing read replica | Không | Có thể |
| Tự document | "Có thể write" | "Chắc chắn không write" |
| Lỡ sửa entity | UPDATE chạy ngầm | Lặng lẽ mất |

### F. Pattern senior — class-level default

```java
@Service
@Transactional(readOnly = true)  // ← class-level default
public class UserService {

    public List<User> findAll() { ... }      // thừa hưởng readOnly
    public User findById(Long id) { ... }    // thừa hưởng readOnly

    @Transactional  // ← override cho write method
    public User createUser(...) { ... }
}
```

→ Default đã read-only; quên override write method = error sớm thay vì corrupt data.

## 1.9 Q&A — `save()` không sinh SQL ngay

### Câu hỏi check kiến thức:

> Trong method `@Transactional(readOnly = true)`, nếu lỡ làm:
> ```java
> User u = userRepository.findById(1L).get();
> u.setName("Hacked");
> // không gọi save()
> ```
> Database có thay đổi không? Nếu cố tình gọi `save()` thì sao?

### Trả lời của học viên:
> Sau khi method return, database sẽ không thay đổi vì nó không tự động flush data xuống dưới database. Tôi nghĩ nếu có tình gọi save thì vẫn lưu được db xuống dượi nhưng chúng ta phải chủ động save

**Review:**
- Phần 1 (DB không thay đổi): ✅ Đúng — `FlushMode = MANUAL` → không auto-flush.
- Phần 2 (chủ động `save()` sẽ lưu được): ❌ Sai.

### Chỗ hiểu nhầm: `save()` KHÔNG sinh SQL

Trực giác Sequelize: `await user.save()` = DB đã update.
Trực giác này SAI với Hibernate. `save()` và "execute SQL" là **2 phase tách biệt**.

#### `save()` thực sự làm gì?

| Entity state | `save()` làm | Có sinh SQL không? |
|---|---|---|
| **Transient** (object mới, chưa có ID hoặc chưa trong Session) | `EntityManager.persist()` → mark MANAGED | **KHÔNG** (trừ IDENTITY strategy mới INSERT ngay) |
| **Managed** (đã trong Session, vd vừa load qua `findById`) | **Không làm gì** — entity đã managed, dirty checking sẽ lo | **KHÔNG** |
| **Detached** (object có ID nhưng không trong Session) | `EntityManager.merge()` → load DB version, copy field | **SELECT** (load), nhưng **không UPDATE ngay** |

→ 3 case đều KHÔNG sinh UPDATE ngay. UPDATE chỉ chạy ở **flush phase**.

#### SQL UPDATE chạy lúc nào?

3 thời điểm:
1. Auto-flush trước query (chỉ khi `FlushMode = AUTO`)
2. Auto-flush trước commit (chỉ khi `FlushMode = AUTO`)
3. Manual flush — `entityManager.flush()` hoặc `repository.saveAndFlush()`

#### Áp dụng vào câu hỏi:
```java
@Transactional(readOnly = true)
public void hackedMethod() {
    User u = userRepository.findById(1L).get();  // u là MANAGED
    u.setName("Hacked");
    userRepository.save(u);                      // KHÔNG làm gì (đã managed)
    // commit time: FlushMode=MANUAL → KHÔNG flush
}
```
→ **Database vẫn KHÔNG thay đổi.** Dù gọi `save()` rồi.

#### Defense in depth (nếu cố ý `saveAndFlush`):
```
Layer 1 — FlushMode=MANUAL → save() không flush → chặn được
Layer 2 — saveAndFlush() bypass FlushMode → force sinh UPDATE → thoát layer 1
Layer 3 — JDBC Connection.setReadOnly(true) → Hibernate ép Connection read-only
Layer 4 — Database level:
  PostgreSQL: "SET TRANSACTION READ ONLY" → reject thẳng
    ERROR: cannot execute UPDATE in a read-only transaction
  MySQL: tuỳ engine, có thể không strict
```

→ PostgreSQL + Spring đúng config → chặn cả 4 layer.
→ MySQL → Layer 4 không strict, nhưng Layer 1 + 3 vẫn chặn 99% case.

**Quy tắc senior:** `save()` không phải `INSERT/UPDATE`. Trong Hibernate, mọi thứ là **deferred** đến flush time. Khác biệt mindset lớn nhất so với Sequelize.

## 1.10 Q&A — Flush là gì? Tại sao read method cần `readOnly`?

### Câu hỏi của học viên:
> hãy cho tôi biết xem là flush này cụ thể là gì? thêm nữa là tại sao các method getAll, get data lại cần transaction(read only=true) nếu như không dùng transaction thì sao

### A. Flush — đào sâu cơ chế

#### A.1 Flush ≠ Commit

```
Bạn viết Java                            Cái gì xảy ra ở DB
────────────────────────                ────────────────────────
@Transactional                           
public void doSomething() {              
   // begin                               BEGIN;
                                          
   user1.setName("A");                    (chưa có SQL nào gửi xuống)
   user2.setEmail("b@x.com");             (chưa có SQL nào gửi xuống)
   userRepository.save(newUser);          (chưa có INSERT)
                                          
   entityManager.flush();                 UPDATE users SET name='A' WHERE id=1;
                                          UPDATE users SET email='b@x.com' WHERE id=2;
                                          INSERT INTO users (...) VALUES (...);
                                          
                                          ← SQL đã chạy ở DB
                                          ← NHƯNG transaction CHƯA commit
                                          ← Connection khác: KHÔNG thấy thay đổi
                                          
   // method return → Spring commit       COMMIT;
                                          ← Data thực sự permanent
}
```

#### A.2 Flush làm gì cụ thể?

**Flush = đồng bộ Persistence Context (in-memory) xuống DB bằng cách sinh SQL.**

```
┌─────────────────────────────────────────────────────────────────┐
│ Step 1 — DIRTY CHECKING                                          │
│   Lặp qua tất cả managed entity.                                │
│   So sánh từng field current vs snapshot.                       │
│   Tạo list actions: [INSERT, UPDATE, DELETE]                    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ Step 2 — SẮP XẾP ACTIONS THEO THỨ TỰ CỐ ĐỊNH                    │
│   1. INSERT (tất cả entity mới)                                 │
│   2. UPDATE (entity đã sửa)                                     │
│   3. Collection removal                                          │
│   4. Collection update                                           │
│   5. Collection creation                                         │
│   6. DELETE                                                      │
│   → Đảm bảo FK constraint không vi phạm.                       │
│   → Lý do thứ tự code đôi khi không khớp thứ tự SQL log.        │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ Step 3 — GỬI SQL QUA JDBC                                       │
│   PreparedStatement.executeBatch() nếu batching bật             │
│   → Data XUỐNG DB, nhưng còn trong transaction                  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ Step 4 — RESET SNAPSHOT                                          │
│   Current = mới nhất → cập nhật snapshot = current              │
│   → Lần flush sau, không sửa gì → không sinh SQL                │
└─────────────────────────────────────────────────────────────────┘
```

#### A.3 Khi nào flush xảy ra?

**Trigger 1 — Trước query (auto-flush)**
```java
@Transactional
public void example() {
    User u = userRepository.findById(1L).get();
    u.setAge(99);  // chưa có SQL
    
    List<User> adults = userRepository.findByAgeGreaterThan(18);
    // ← TRƯỚC query này, auto-flush:
    //   UPDATE users SET age=99 WHERE id=1
    // → Query thấy u trong kết quả vì age=99 > 18
}
```
→ Đảm bảo "read-your-own-writes" trong cùng transaction.

**Trigger 2 — Trước commit**

**Trigger 3 — Manual flush**
```java
entityManager.flush();
userRepository.saveAndFlush(user);
```
Khi dùng:
- Cần lấy DB-generated value ngay (vd ID với SEQUENCE)
- Cần test constraint violation sớm
- Batch processing chunk lớn — flush + clear định kỳ

#### A.4 Phân biệt 3 thuật ngữ

```
┌──────────┬─────────────────────────────────────────────────────┐
│ persist  │ Đưa entity vào Session, mark MANAGED.               │
│          │ KHÔNG sinh SQL ngay (trừ IDENTITY).                 │
├──────────┼─────────────────────────────────────────────────────┤
│ flush    │ Sinh SQL từ pending changes, gửi xuống DB.          │
│          │ KHÔNG commit. Có thể rollback sau.                  │
├──────────┼─────────────────────────────────────────────────────┤
│ commit   │ Báo DB "transaction xong, persist permanent".       │
└──────────┴─────────────────────────────────────────────────────┘
```

| Hibernate | Sequelize | Ý nghĩa |
|---|---|---|
| `entityManager.persist(u)` | (không có) | Đăng ký với Session |
| Setter rồi để Hibernate track | `u.name='X'; await u.save()` | Sequelize SQL ngay; Hibernate defer |
| `entityManager.flush()` | (không có) | Force gửi pending SQL |
| `transaction.commit()` | `await t.commit()` | Giống nhau |

→ Hibernate = "tích luỹ → flush 1 lần" (batch). Sequelize = "mỗi `save()` = 1 SQL ngay".

#### A.5 Rollback sau flush
```java
@Transactional
public void demo() {
    userRepository.save(u1);
    entityManager.flush();   // INSERT đã chạy ở DB
    
    if (someCondition) {
        throw new RuntimeException("...");
        // ← rollback → INSERT vừa rồi BIẾN MẤT
    }
}
```
→ Flush ≠ permanent. Chỉ commit mới permanent.

### B. Tại sao read method cần `@Transactional(readOnly = true)`?

#### B.1 Sự thật bất ngờ: repository method MẶC ĐỊNH đã có transaction

Spring Data JPA `SimpleJpaRepository` (parent của mọi repository) annotate sẵn:
```java
@Transactional(readOnly = true)
public class SimpleJpaRepository<T, ID> implements JpaRepository<T, ID> {

    @Override
    public Optional<T> findById(ID id) { ... }
    
    @Override
    public List<T> findAll() { ... }
    
    @Override
    @Transactional   // ← write method override
    public <S extends T> S save(S entity) { ... }
}
```

→ Nếu chỉ gọi 1 repository method từ controller, vẫn có transaction (nhỏ, readOnly).

#### B.2 Vậy tại sao Service vẫn cần?

**Lý do 1 — `LazyInitializationException` (quan trọng nhất)**

```java
// KHÔNG có @Transactional
public CategoryResponse getCategoryById(Long id) {
    Category category = categoryRepository.findById(id).orElseThrow(...);
    // ← Repository transaction kết thúc, SESSION ĐÓNG
    
    return categoryMapper.fromEntity(category);
    // ← Mapper truy cập category.getChildren() (lazy)
    // ← Session đã đóng → throw LazyInitializationException
}
```

**Cơ chế:**
```
[Service.getCategoryById]
       │
       ├─► [Repository.findById]
       │       ├─ Spring mở Transaction A (readOnly)
       │       ├─ Mở Session A
       │       ├─ SELECT * FROM categories WHERE id=?
       │       │  → Category load, children = LAZY proxy
       │       ├─ Commit Transaction A
       │       └─ ĐÓNG Session A
       │       
       ▼ (quay về Service, category DETACHED)
       
[Mapper.fromEntity(category)]
       └─ category.getChildren()
              └─ Hibernate proxy: "Session đâu?"
                 → throw LazyInitializationException
```

**Fix:**
```java
@Transactional(readOnly = true)
public CategoryResponse getCategoryById(Long id) {
    Category category = categoryRepository.findById(id).orElseThrow(...);
    return categoryMapper.fromEntity(category);
    //     ← children lazy load OK vì Session vẫn mở
}
```

**Cơ chế sau fix:**
```
[Service.getCategoryById]  ← Spring proxy mở Transaction B + Session B
       │
       ├─► [Repository.findById]
       │       └─ Spring thấy Transaction B → JOIN vào (không tạo mới)
       │          → SELECT ...
       │
       ▼ (Session B VẪN MỞ)
       │
       ├─► [Mapper.fromEntity(category)]
       │       └─ category.getChildren()
       │              └─ Session B đang mở
       │                 → SELECT * FROM categories WHERE parent_id=?
       │                 → Lazy load OK ✅
       │
       ▼ Method return → close Transaction B + Session B
```

→ Service-level `@Transactional` **mở rộng vòng đời Session** ra toàn bộ method.

**Lý do 2 — Atomicity của reads**
```java
// KHÔNG có @Transactional
public DashboardResponse getDashboard() {

    long totalProducts = productRepository.count();   // Transaction B
    long totalOrders = orderRepository.count();       // Transaction C
    // ← Giữa A và B có thể có insert mới → số liệu không nhất quán
}

// CÓ @Transactional(readOnly = true)
@Transactional(readOnly = true)

    // ← Tất cả query trong CÙNG 1 transaction
```

### C. Bonus — `OpenSessionInView` (OSIV): cái bẫy ẩn

**Spring Boot mặc định bật `spring.jpa.open-in-view=true`.**

```
HTTP request → [OpenEntityManagerInViewFilter] → Controller → Service → Repository
                       │                                                      │
                       └─ Mở Session ngay từ ĐẦU request                    │
                       └─ Đóng Session sau KHI response gửi xong  ────────┘
```

→ Session sống suốt request → lazy loading hoạt động mọi nơi.

**Tại sao là anti-pattern?**
1. **N+1 query ẩn:** Serializer truy cập lazy field → query phụ không thấy trong Service.
2. **Transaction boundary mờ:** Bug khó debug.
3. **Connection giữ lâu:** Pool cạn dưới tải cao.
4. **Sai khi audit/security context** đã đóng nhưng query vẫn chạy.

**Best practice production:** Tắt OSIV — `spring.jpa.open-in-view=false`. Bắt buộc lazy load phải trong `@Transactional` Service. Bug lộ ra sớm.

→ Trong Cụm 2, sẽ apply `@Transactional` cho mọi service, sau đó tắt OSIV, chạy lại E2E test.

# Phần 1.5 — Transaction Propagation (intro)

## 1.5.1 Vấn đề — service-to-service

Khi `@Transactional` đứng một mình thì dễ. Real-world: service luôn gọi service khác.

```java
@Service
public class OrderService {
    @Transactional
    public Order createOrder(...) {
        Order order = orderRepository.save(...);
        inventoryService.decrement(...);        // service KHÁC, cũng @Transactional
        auditService.log("ORDER_CREATED", ...); // service KHÁC nữa
        return order;
    }
}
```

**Câu hỏi:** Đây là 3 transaction tách biệt hay 1 transaction chung?
**Trả lời:** Phụ thuộc **propagation** của từng `@Transactional`.

## 1.5.2 7 mode propagation — chỉ giảng 3 mode chính

| Mode | Tóm tắt |
|---|---|
| **`REQUIRED`** (default) | Có tx → join. Không có → tạo mới. |
| **`REQUIRES_NEW`** | Luôn tạo tx mới. Tx ngoài (nếu có) bị SUSPEND. |
| **`NESTED`** | Có tx → tạo savepoint trong tx đó. Không có → tạo mới. |
| `SUPPORTS` | Có tx → join. Không có → chạy không tx. |
| `NOT_SUPPORTED` | Có tx → SUSPEND. Chạy không tx. |
| `MANDATORY` | Có tx → join. Không có → throw. |
| `NEVER` | Có tx → throw. Không có → chạy không tx. |

## 1.5.3 `REQUIRED` — mặc định (99% case dùng)

**Luật:** Đang có tx → join. Chưa có → tạo mới.

```
HTTP request → createOrder
[PROXY OrderService.createOrder]
        │  Spring: chưa có tx → BEGIN tx#1, Connection#7 → ThreadLocal
        │
        ├─► orderRepository.save(...)              → Connection#7 (tx#1)
        │
        ├─► [PROXY InventoryService.decrement]
        │       │  Spring: ĐANG có tx#1 → JOIN
        │       └─► decrement → Connection#7 (vẫn tx#1)
        │
        ├─► [PROXY AuditService.log]
        │       │  ĐANG có tx#1 → JOIN
        │       └─► log → Connection#7 (vẫn tx#1)
        │
        └─ return → COMMIT tx#1
```

→ 3 method = 1 transaction duy nhất. Atomic. Nếu `auditService.log()` throw → rollback toàn bộ.

## 1.5.4 `REQUIRES_NEW` — audit/notification độc lập

**Luật:** Luôn tạo tx mới. Tx ngoài bị SUSPEND, lấy Connection MỚI từ pool.

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void log(...) { ... }
```

```
[OrderService.createOrder] ─ tx#1 mở, Connection#7
        │
        ├─► orderRepository.save(...)              → Connection#7 (tx#1)
        ├─► inventoryService.decrement(...)        → Connection#7 (tx#1, REQUIRED)
        │
        ├─► [AuditService.log] (REQUIRES_NEW)
        │       │  Spring: SUSPEND tx#1
        │       │            mượn Connection#9 mới
        │       │            BEGIN tx#2 trên Connection#9
        │       │
        │       ├─► log → INSERT (Connection#9, tx#2)
        │       │
        │       └─ tx#2 COMMIT ngay → Connection#9 trả pool
        │          Unsuspend tx#1 → tiếp Connection#7
        │
        └─ tx#1 COMMIT cuối
```

**Hệ quả:** Audit log COMMIT độc lập. Nếu order rollback → audit log VẪN CÒN.

**Use case:** Audit logging, notification, fire-and-forget counter.
**Cảnh báo:** 2 transaction = 2 Connection → connection pool nhỏ dễ deadlock.

## 1.5.5 `NESTED` — savepoint trong tx ngoài

**Luật:** Đang có tx → tạo savepoint. Nested fail → rollback CHỈ đến savepoint, tx ngoài còn sống.

**Khác `REQUIRES_NEW`:** 1 transaction, 1 Connection, dùng cơ chế savepoint của DB.

```java
@Transactional
public void processOrders(List<OrderRequest> orders) {
    for (OrderRequest req : orders) {
        try {
            processOneOrder(req);  // @Transactional(NESTED)
        } catch (Exception e) {
            log.warn("Order {} failed", req.getId());
        }
    }
}
```

→ "Fail per item, continue with rest" cho batch processing.
**Yêu cầu:** JDBC driver + DB hỗ trợ savepoint (PostgreSQL, MySQL InnoDB OK).

## 1.5.6 Pitfall — `REQUIRES_NEW` + self-invocation = bypass

```java
@Service
public class OrderService {
    @Transactional
    public Order createOrder(...) {
        this.logAudit(...);   // ← cùng class, this trỏ TARGET REAL
    }
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logAudit(...) { ... }
}
```

**Tưởng:** logAudit chạy tx mới.
**Thực tế:** bypass proxy → `REQUIRES_NEW` vô hiệu → chạy cùng tx#1.

**Fix:** Tách `AuditService` riêng, inject vào → call qua proxy.

## 1.5.7 So sánh Node.js

Sequelize không có khái niệm propagation automatic — phải truyền transaction object explicit:
```javascript
async function createOrder(data, t = null) {
  const newTx = t || await sequelize.transaction();
  try {
    await Order.create(data, { transaction: newTx });
    await decrementInventory(data, newTx);   // phải truyền tay
    if (!t) await newTx.commit();
  } catch (e) {
    if (!t) await newTx.rollback();
    throw e;
  }
}
```

→ Bản chất = `REQUIRED`. Spring làm tự động qua ThreadLocal → clean hơn, nhưng phải hiểu propagation + pitfall proxy.

## 1.5.8 Q&A check kiến thức — verbatim

### Câu hỏi 1: `REQUIRED → REQUIRED` throw RuntimeException

> Trong `OrderService.createOrder` (REQUIRED) gọi `inventoryService.decrement` (REQUIRED). Nếu `decrement` throw RuntimeException thì:
> a) Chỉ rollback decrement, save đã commit
> b) Rollback toàn bộ
> c) Commit cả hai

**Trả lời học viên:** `b`
**Review:** ✅ Đúng. `REQUIRED` join → cùng tx#1. `decrement` throw → toàn bộ tx#1 rollback. Atomic.

### Câu hỏi 2: `REQUIRES_NEW` audit + inventory throw

> save → audit(REQUIRES_NEW) → inventory throw → kết quả?

**Trả lời học viên:** `b` (Order rollback, Audit vẫn lưu, Inventory rollback)
**Review:** ✅ Đúng.
```
save()              → tx#1 (chưa commit)
auditService.log()  → SUSPEND tx#1, tạo tx#2, COMMIT tx#2 NGAY, unsuspend
decrement()         → tx#1 → throw → ROLLBACK tx#1
```
**Bonus pitfall:** Sau đó nếu query audit log từ trong tx#1 (đang còn sống) — có thể KHÔNG thấy do isolation level `READ_COMMITTED` chỉ thấy data committed TRƯỚC khi tx mình mở.

### Câu hỏi 3: 2 vấn đề với `REQUIRES_NEW` self-invoke + private

```java
@Transactional
public void deleteProduct(Long id) {
    this.cleanupImages(id);  // cùng class
}
@Transactional(propagation = Propagation.REQUIRES_NEW)
private void cleanupImages(Long id) { ... }
```

**Trả lời học viên:**
> this.cleanupImages(id) sẽ không được áp dụng REQUIRED_NEW. Nếu cleanupImages bị lỗi thì vẫn rollback logic trong deleteProduct

**Review:** ⚠️ Đúng 1/2 vấn đề.

**Vấn đề 1 (đã chỉ ra):** Self-invocation bypass proxy → `REQUIRES_NEW` vô hiệu.

**Vấn đề 2 (MISS):** Method `private`. CGLIB không override được `private` → `@Transactional` bất kỳ propagation gì cũng chết.

→ **2 vấn đề INDEPENDENT** — fix 1 cái không đủ:

| Fix self-invoke | Fix private | Kết quả |
|---|---|---|
| ❌ | ❌ | Chết (2 lý do) |
| ✅ | ❌ | Chết (vì private) |
| ❌ | ✅ | Chết (vì self-invoke) |
| ✅ | ✅ | Hoạt động ✅ |

→ **Quy tắc senior:** Đặt `@Transactional` xong, đặt 3 câu hỏi:
1. Method có `public` không?
2. Method có được gọi từ NGOÀI class (qua bean inject) không?
3. Class/method có `final` không?

3 câu trả lời "có-không-không" → đúng. Khác là bug âm thầm.

## 1.5.9 Áp dụng vào codebase

Hôm nay tất cả case dùng `REQUIRED` mặc định:

| Case | Propagation | Lý do |
|---|---|---|
| `ProductService.deleteProduct` cascade xoá images | `REQUIRED` | Rollback toàn bộ nếu fail |
| `createProduct` gọi `categoryRepository.findById` | `REQUIRED` | Cùng business tx |
| `ProductImageService.uploadImage` (storage + DB) | `REQUIRED` | Atomic |
| Nếu sau này có `AuditLogService` | `REQUIRES_NEW` | Persist log kể cả khi business rollback |
| Nếu sau này có bulk import sản phẩm | `NESTED` | Continue khi 1 item fail |

# Phần 2 — Cụm 1: Fix Category Circular Reference

## 2.1 Bug nghiệp vụ

`CategoryService.updateCategory` cũ check 1 cấp con:
```java
boolean check = category.getChildren().stream()
    .anyMatch(c -> c.getId().equals(parentId));   // chỉ children TRỰC TIẾP
```

**Kịch bản gây bug:**
```
Tree:  A(1) → B(2) → C(3) → D(4)
PUT /categories/1  body { "parentId": 4 }   (set D làm parent của A)

Bug cũ check:
  A.getChildren() = [B]
  B.id (2) == 4? NO → PASS ✗
  category.setParent(D) → DB ghi xuống cycle

Sau update:
  A.parent = D
  B.parent = A
  C.parent = B
  D.parent = C
  → cycle: A → D → C → B → A → ...
```

**Hậu quả runtime:**
- `getAllCategories()` đệ quy build tree → `StackOverflowError`
- MapStruct map nested `parent` → cũng `StackOverflowError`

**Root cause:** Phải chặn việc set parent thành **bất kỳ DESCENDANT** (con/cháu/chắt/...) nào của node đang sửa.

## 2.3 Q&A — Chọn hướng giải

### Câu hỏi học viên:
> Tôi cũng chưa rõ, bạn có đề xuất hướng đi nào và trade off như thế nào cho từng phương án

### 3 hướng tiếp cận

#### Hướng A — Walk down (duyệt xuống descendants)
```java
private boolean isDescendant(Category root, Long targetId) {
    if (root.getId().equals(targetId)) return true;
    for (Category child : root.getChildren()) {    // LAZY mỗi child
        if (isDescendant(child, targetId)) return true;
    }
    return false;
}
```

| | Đánh giá |
|---|---|
| Worst case queries | **O(N)** với N = descendants |
| Memory | Load subtree |
| Code | Đệ quy idiomatic |
| Portable | ✅ |

**Vấn đề:** Update "Electronics" (5000 con) → 5000 query.

#### Hướng B — Walk up (đi ngược từ parent mới)
```java
private boolean isAncestorOf(Category newParent, Long categoryId) {
    Category current = newParent;
    while (current != null) {
        if (current.getId().equals(categoryId)) return true;
        current = current.getParent();    // LAZY 1 parent
    }
    return false;
}
```

| | Đánh giá |
|---|---|
| Worst case queries | **O(H)** với H = depth |
| Memory | 1 node tại 1 thời điểm |
| Code | While đơn giản |
| Portable | ✅ |

**Insight:** "Parent mới có phải descendant của category?" flip thành "Có ancestor nào của parent mới chính là category?". Cùng bài toán, chiều khác.

Cây thực tế **rộng-nông** → H << N → walk-up rẻ hơn nhiều.

#### Hướng C — Recursive CTE
```sql
WITH RECURSIVE descendants AS (
    SELECT id FROM categories WHERE id = :categoryId
    UNION ALL
    SELECT c.id FROM categories c
    JOIN descendants d ON c.parent_id = d.id
)
SELECT EXISTS(SELECT 1 FROM descendants WHERE id = :parentId)
```

| | Đánh giá |
|---|---|
| Worst case queries | **O(1)** |
| Code | Khó cho Java dev |
| Portable | ⚠️ Vendor-specific (MySQL >= 8.0, PostgreSQL OK, Oracle cú pháp khác) |
| Test | Cần DB thật |

### Bảng so sánh tổng

| | A (walk down) | B (walk up) | C (CTE) |
|---|---|---|---|
| Worst case | O(N) | **O(H)** | **O(1)** |
| Cây 5 cấp 5000 node | 5000 query 😱 | 5 query | 1 query |
| Cây 20 cấp 1M node | 1M query 💀 | 20 query | 1 query |
| Code complexity | Trung bình | Đơn giản nhất | Khó hơn |
| Portable | ✅ | ✅ | ⚠️ |
| Over-engineering | Không | Vừa đủ | Có |

### Đề xuất: Hướng B

1. **Domain fit** — Category e-commerce 3-5 cấp, H rất nhỏ
2. **Code clarity** — vòng while đơn giản, không cần SQL CTE
3. **Test dễ** — chỉ cần mock `getParent()` chain
4. **Không lock-in DB**
5. **Migrate path** — nếu sau này depth > 20 cấp hoặc trở thành hotspot → migrate sang C, code service không đổi nhiều

### Bonus — insight algorithm design

A check trong **tập rộng** (cây con), B check trong **tập hẹp** (chuỗi tổ tiên).
→ Khi có 2 chiều duyệt graph, chọn chiều search space NHỎ HƠN.

## 2.4 Code implementation

```java
private static final int MAX_TREE_DEPTH = 100;

private boolean isAncestorOf(Category newParent, Long categoryId) {
    Category current = newParent;
    int depth = 0;
    while (current != null) {
        if (++depth > MAX_TREE_DEPTH) {
            throw new IllegalStateException(
                "Category tree exceeds max depth " + MAX_TREE_DEPTH
                    + " — possible pre-existing cycle starting at id=" + newParent.getId());
        }
        if (current.getId().equals(categoryId)) return true;
        current = current.getParent();
    }
    return false;
}
```

**3 fix trong version cuối:**

1. **Start `current = newParent`** thay vì `newParent.getParent()` → cover case self-parent → method standalone không phụ thuộc caller pre-check.

2. **Bỏ double null check** → vòng while chỉ check `null` 1 lần.

3. **Depth limit (circuit breaker)** → chống infinite loop nếu DB lỡ có cycle.

**Refactor caller `updateCategory`:**
```java
// Cũ
if (CheckData.checkIsNotNull(category.getChildren())) {
    boolean check = category.getChildren().stream()
        .anyMatch(c -> c.getId().equals(parentId));
    if (check) throw new RuntimeException("...");
}

// Mới
if (isAncestorOf(parentCategory, category.getId())) {
    throw new BusinessRuleException("...");
}
```

## 2.5 Q&A — Infinite loop cụ thể

### Câu hỏi học viên:
> trước khi refactor, tôi vẫn chưa biết case nào gây infiniti loop, hãy cho tôi biết case cụ thể gây infiniti loop

### Case 1 — DB có cycle PREEXIST (từ bug cũ chưa fix)

Trước khi fix, bug cũ cho phép tạo cycle. Giả sử admin đã trigger:
```
Trước: A(1) → B(2) → C(3) → D(4)
PUT /categories/1 body {"parentId": 4}
Bug cũ pass → DB hỏng:
   A.parent = D, B.parent = A, C.parent = B, D.parent = C
   → cycle vô tận
```

**`getAllCategories()` lập tức broken** (`findByParentIsNull()` trả `[]`). Admin deploy fix mới, NHƯNG **DB vẫn còn cycle đó**. Fix code chỉ chặn cycle MỚI.

Trigger infinite loop:
```
Update node 5 với parentId = 4 (D trong cycle):
isAncestorOf(D, 5):
  current = D → C.id ≠ 5, continue
  current = C → B.id ≠ 5, continue
  current = B → A.id ≠ 5, continue
  current = A → D (QUAY VỀ D, cycle)
  current = D → C → B → A → D → ... ∞
```

**Hậu quả thực tế:**
- CPU 100% (hot loop)
- Mỗi iteration = 1 SELECT (lazy load `getParent`)
- DB connection bị giữ vô thời hạn
- ~60s: HTTP timeout, client thấy 504
- Nhưng THREAD VẪN CHẠY (HTTP timeout không kill thread)
- Hibernate Session phình → Heap tăng
- Connection pool dần cạn (request mới cũng dính loop)
- OutOfMemoryError → JVM crash → service down

→ **1 admin click sai → cả service sập.**

### Case 2 — Race condition tạo cycle dù logic ĐÚNG

```
DB: A(1), B(2)→A, C(3)→A
T=0ms:  Admin1: PUT /categories/2 body {"parentId": 3}   (B làm con C)
T=0ms:  Admin2: PUT /categories/3 body {"parentId": 2}   (C làm con B)
```

**Thread 1:**
```
T=10: load B (parent=A)
T=20: load C (parent=A)
T=30: isAncestorOf(C, 2): C→A→null → false (pass)
T=40: B.setParent(C); save; COMMIT
```

**Thread 2 (song song):**
```
T=10: load C (parent=A)
T=20: load B (parent=A)   ← VẪN LÀ A (T1 chưa commit)
T=30: isAncestorOf(B, 3): B→A→null → false (pass)
T=40: C.setParent(B); save; COMMIT
```

**DB sau:** B.parent=C, C.parent=B → cycle B↔C.

→ Cả 2 check ĐÚNG tại thời điểm check, race xảy ra giữa snapshot read.

### So sánh 2 case

| | Case 1 — DB preexist | Case 2 — Race |
|---|---|---|
| Cycle do? | Bug cũ chưa fix | Concurrency, code đúng |
| Fix logic chặn được? | ❌ (cycle đã ở DB) | ❌ (race ở DB layer) |
| Cần thêm gì? | Depth limit | `@Version` (hoặc lock) |
| Tần suất | 1 lần sau migration | Theo traffic |
| Mức độ | 🔴 Service crash | 🔴 Service crash |

→ **Defense in depth** — fix logic check chặn được layer 1 (cycle mới qua API). Layer 2 (DB hỏng) cần depth limit. Layer 3 (race) cần optimistic locking.

## 2.6 Q&A — Tại sao isolation level KHÔNG cứu race

### Câu hỏi học viên:
> Refactor isAncestorOf với depth limit trước, sau đó giả thích cho tôi thêm và case 2 race vẫn pass với mọi isolation level

### Setup recap

```
DB: A(1), B(2)→A, C(3)→A
2 thread song song, mỗi thread: load + check + save commit
```

### Phân tích từng level

#### READ_UNCOMMITTED — vô tác dụng
Thread 2 đọc "dirty" version của Thread 1 → **có thể** chặn được nếu timing đúng. Nhưng:
- Thread 1 còn quyền rollback
- Race timing: T2 đọc TRƯỚC khi T1 sửa → vẫn fail
→ Hên xui, không phải solution.

#### READ_COMMITTED — không cứu
T2 chỉ đọc data đã commit. T1 chưa commit → T2 thấy old state → cả 2 pass check.

→ Cùng đáp án cho **REPEATABLE_READ** (snapshot càng cứng hơn).

#### SERIALIZABLE — tuỳ DB engine

**PostgreSQL (SSI — Serializable Snapshot Isolation):**
Track "predicate conflict". 2 tx đọc/ghi conflict cross-row → abort 1 với `SerializationFailure`.

**Cứu được? Có, nhưng:**
1. Throw lúc commit, không phải lúc check → app phải catch + retry. Không retry → 500 error
2. False positive cao — SSI heuristic abort cả khi không thật sự cycle
3. Lazy load mỗi cấp = 1 SELECT riêng → graph dependency phức tạp → false positive thêm
4. Performance kinh khủng toàn DB (mọi tx khác cũng SERIALIZABLE)

**MySQL InnoDB:**
SERIALIZABLE = mọi SELECT thành `SELECT ... FOR SHARE` (lock S row đọc).
```
T1: SELECT B FOR SHARE, SELECT C FOR SHARE, UPDATE B (upgrade S→X)
T2: SELECT C FOR SHARE, SELECT B FOR SHARE, UPDATE C (cần X trên C, T1 có S)
→ DEADLOCK → InnoDB kill 1 tx → DeadlockLoserDataAccessException
```

Cũng "cứu" được nhưng:
- Deadlock = exception ngẫu nhiên, không deterministic
- Throughput kinh khủng (concurrent read cũng phải wait)
- App phải retry

### Tổng kết

| Level | Chặn? | Cost |
|---|---|---|
| READ_UNCOMMITTED | Hên xui | Dirty read khắp nơi |
| READ_COMMITTED | ❌ | Default OK 99% case khác |
| REPEATABLE_READ | ❌ | Snapshot cứng hơn → càng không thấy thay đổi |
| SERIALIZABLE PG | ⚠️ | False positive, retry, toàn DB chậm |
| SERIALIZABLE MySQL | ⚠️ | Deadlock, lock S read, retry |

→ Không có "miễn phí". Đổi performance toàn cục cho 1 race rất hẹp = over-engineering.

### Tại sao `@Version` cũng không đủ

```
DB: B(v=0), C(v=0)
T1: load B(v=0), C(v=0), check pass
    UPDATE B SET parent=C, version=1 WHERE id=2 AND version=0 → 1 row ✅
T2: load C(v=0), B(v=0)   ← T1 chưa commit
    check pass
    UPDATE C SET parent=B, version=1 WHERE id=3 AND version=0 → 1 row ✅
→ Vẫn cycle B↔C
```

**`@Version` chỉ chặn khi 2 thread sửa CÙNG row.** 2 thread sửa 2 row khác nhau nhưng ràng buộc cross-row → `@Version` không đủ.

### Solution thực tế cho Case 2

**Option A — Pessimistic lock root subtree:**
```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT c FROM Category c WHERE c.parent IS NULL")
List<Category> findRootsForUpdate();
```
→ Serialize update tree. Nặng nề.

**Option B — Application-level distributed lock (Redis):**
```java
redisLock.acquire("category-tree-mutation");
try { ... } finally { redisLock.release(); }
```
→ Cần infrastructure.

**Option C — DB constraint** (PostgreSQL không hỗ trợ CHECK + recursive CTE chuẩn).

**Option D — Chấp nhận + repair job định kỳ:**
```java
@Scheduled(cron = "0 0 * * * *")
public void detectAndRepairCycles() { ... }
```
→ Pragmatic. Category update là admin action, race cực hiếm.

### Insight cuối

**Concurrency bug đôi khi KHÔNG có "đáp án đúng kỹ thuật", chỉ có trade-off.**

Cleanup session áp dụng:
1. **Cụm 2** apply `@Version` → chặn được "lost update" thông thường (80% case)
2. **Chấp nhận** Case 2 race rất hiếm
3. **Circuit breaker** (depth limit) chống service sập khi xảy ra
4. Note memory: nếu category update thành hot operation → migrate sang lock pattern

→ Tư duy senior: **biết bug tồn tại, hiểu cost của fix, chọn defensive layer phù hợp với risk thật.**

## 3.6 Q&A — Production có thực sự làm vậy?

### Câu hỏi học viên:
> hãy nói tóm tắt lại cho tôi biết việc làm này có nghĩa gì nhỉ, và trên product có tắt osiv và set mặc định Transaction ở class-level không hay vẫn làm như code cũ

### Ý nghĩa việc đã làm

**Trước:**
- Không `@Transactional` → mỗi repository call mở/đóng tx riêng → không atomic
- OSIV bật → Session sống suốt request → lazy bug ẩn, N+1 không ai biết
- Mỗi business op tốn nhiều round-trip connection

**Sau:**
- 1 business op = 1 transaction = 1 Connection xuyên suốt → atomic, rollback đúng
- `readOnly` ở read method → skip dirty checking, hint DB optimize
- Lazy load TRONG Session do `@Transactional` mở → bug lazy lộ ngay tầng Service
- Connection pool xài hiệu quả hơn

**Ngầm hơn:**
- Đã chuẩn để scale write throughput sau `@Version` (Cụm 2.3)
- Đã chuẩn cho Auditing (Cụm 3) — entity listener chạy trong tx scope
- Đã chuẩn cho custom exception (Cụm 4) — rollback rule đúng

### Class-level readOnly default — production có dùng?

**✅ Phổ biến trong production.** Vlad Mihalcea (Hibernate champion) và nhiều thought leader Java recommend. Lý do:
- Safe by-default — quên override = không lưu, KHÔNG corrupt ngầm
- Self-documenting
- Performance tốt hơn

Một số team KHÔNG dùng vì:
- Teach junior pattern "annotate explicit từng method"
- Service mix read-heavy + write-heavy → class-level không fit

→ Cả 2 cách đều OK. Class-level phổ biến hơn enterprise team.

### Tắt OSIV — production-grade?

**✅ Nên tắt thực sự.**
- Spring team warn ngầm khi OSIV bật (log entry "spring.jpa.open-in-view is enabled by default")
- Vlad Mihalcea viết bài "The Open Session In View Anti-Pattern"
- Production checklist enterprise template (JHipster, Spring Initializr) đều disable

**Tại sao Spring Boot vẫn default `true`?** Backward compat. Không phải best practice.

**Khi nào còn giữ OSIV `true`?**
1. Templating engine truyền entity vào view (Thymeleaf/JSP) — codebase mình dùng REST + JSON qua DTO, không dính
2. Legacy code chưa kịp audit
3. Quick prototype/MVP

→ Codebase này dùng DTO + Mapper + audit hết → **tắt OSIV là natural fit**.

# Phần 4 — Cụm 2.3: Optimistic Locking (giảng lý thuyết, DEFER apply)

## 4.1 Trạng thái

- **Đã giảng lý thuyết đầy đủ** trong session 2026-05-25
- **Chưa apply code** — user mệt sau ngày học nhiều lý thuyết, defer cho buổi tiếp theo
- Plan apply vẫn còn nguyên trong backlog

## 4.2 Vấn đề "Lost Update"

```
T=0ms:   DB: Product(stock=10)
T=10ms:  Admin1 load → stock=10
T=15ms:  Admin2 load → stock=10
T=30ms:  Admin1 set stock=5 → save → DB: stock=5
T=40ms:  Admin2 set stock=7 → save → DB: stock=7  ← Admin2 ĐÈ Admin1
```

→ **Lost Update** — silent data loss, không exception, không log.

## 4.3 2 chiến lược

### Pessimistic — "Bi quan, lo xa"
- Giả định conflict sẽ xảy ra → lock ngay
- `SELECT ... FOR UPDATE`
- Pros: chắc chắn. Cons: chậm, deadlock risk
- Use case: bank transfer, inventory critical

### Optimistic — "Lạc quan"
- Giả định conflict hiếm → check lúc commit
- Không lock. Field `version` + `WHERE version=?`
- Pros: throughput cao. Cons: app code retry logic
- Use case: CRUD admin, user profile (95% case)

## 4.4 Cơ chế `@Version` của Hibernate

```java
@Entity
public class Product {
    @Id ...
    private Long id;
    private String name;
    private BigDecimal price;
    
    @Version              // ← THÊM DÒNG NÀY
    private Long version;
}
```

Khi UPDATE, Hibernate auto sinh:
```sql
UPDATE products
SET name=?, price=?, version = 6              -- TỰ ĐỘNG +1
WHERE id=1 AND version=5                       -- TỰ ĐỘNG check version cũ
```

- Row affected = 1 → OK
- Row affected = 0 → `ObjectOptimisticLockingFailureException`

## 4.5 Q&A — Optimistic/Pessimistic là khái niệm chung

### Câu hỏi học viên:
> khái niệm Optimistic Lock và Pessimistic Lock là 2 khái niệm như nào. Chúng chỉ thuộc trong phạm vi của spring boot thôi hay là các khái niệm chung trong ngành lập trình.

### Trả lời tóm tắt

**Là khái niệm GENERAL của computer science / concurrency**, KHÔNG phải Spring-specific.

Xuất hiện ở:
- **DB engines:** MySQL, PostgreSQL, Oracle, MongoDB, DynamoDB, Redis WATCH

```javascript
await sequelize.transaction(async (t) => {

```

## 4.6 Trade-off quan trọng — Include version trong Response DTO?

**Trường phái A — KHÔNG include (đơn giản):**
- Server-side optimistic only — chặn được race **2 request gần đồng thời tới server**
- KHÔNG chặn được "stale form" (admin để form 2 giờ mới submit)

**Trường phái B — Include version, client gửi lại:**
- True optimistic locking — chặn cả stale form bao lâu cũng được
- DTO có field "kỹ thuật" lạ
- Use case: app collaboration realtime

→ Cho admin panel e-commerce → đi **A** đủ. Note memory nếu sau cần B.

### Review: ~85% đúng, 3 điểm tinh chỉnh

**Tinh chỉnh 1 — Phân biệt Session ≠ Connection ≠ Transaction:**

| Khái niệm | Là gì | Ai quản lý |
|---|---|---|
| Session / EntityManager (Persistence Context) | Vùng nhớ chứa managed entity + snapshot + 1st-level cache | Hibernate/JPA |
| JDBC Connection | Kết nối vật lý tới DB, từ pool (HikariCP) | DataSource pool |
| Transaction | Đơn vị atomic BEGIN...COMMIT | Spring `@Transactional` |

→ 1 Session dùng 1 Connection chạy SQL, 1 Transaction bao quanh. OSIV giữ **Session** lâu → kéo theo **Connection** bị giữ lâu (hệ quả).

**Tinh chỉnh 2 — "Mỗi request 1 session" KHÔNG phải điểm khác biệt:**
- CẢ HAI trường hợp đều có session per request
- Khác biệt thật là **DURATION**: OSIV giữ session suốt request (kể cả lúc serialize JSON); tắt OSIV + tx thì session chỉ sống trong method service

**Tinh chỉnh 3 — OSIV có 2 tác hại, không chỉ cạn pool:**
- (A) Cạn connection pool (học viên đã nêu)
- (B) **N+1 query ẩn lúc serialize** — Jackson đụng lazy field khi session còn sống → Hibernate bắn thêm SELECT ngầm, không thấy trong code service. Tắt OSIV làm lazy load lúc serialize vỡ ngay (`LazyInitializationException`) → bug lộ sớm để fix bằng `JOIN FETCH`/`@EntityGraph`
- Codebase học dùng DTO + Mapper → Jackson chỉ serialize DTO (map xong trong session) → tắt OSIV an toàn

## A.5 Bản tóm tắt đã sửa (verbatim cho học viên)

> **OSIV bật:** Spring mở **Session** (persistence context) từ đầu request, giữ đến khi render xong response. Session giữ lâu → kéo theo **Connection** bị giữ lâu → cạn pool, VÀ lazy load lúc serialize gây N+1 ẩn.
>
> **OSIV tắt + `@Transactional` ở service:** Khi request đến, controller gọi service. Nếu method service có `@Transactional` → Session + Transaction mở lúc vào method, đóng lúc ra method (lazy load phải xảy ra TRONG đây). Nếu method KHÔNG có `@Transactional` → mỗi repository call tự mở/đóng tx riêng (qua `@Transactional` built-in của Spring Data) → nhiều tx tách biệt, không atomic, và lazy load sau khi repository return sẽ vỡ.

# Phần 5 — Cụm 3: BaseEntity + JPA Auditing (2026-06-06)

## 5.1 Vấn đề — set timestamp thủ công rải rác

Trước cleanup, mỗi service tự set timestamp bằng tay:
```java
user.setCreatedAt(LocalDateTime.now());     // UserService.createUser
product.setCreatedAt(LocalDateTime.now());  // ProductService.createProduct
image.setCreatedAt(LocalDateTime.now());    // ProductImageService.uploadImage
```

4 vấn đề:
1. Lặp code — mỗi create/update phải nhớ set tay, quên = null
2. Dễ sai — copy nhầm `setCreatedAt` vào update method
3. Không nhất quán — Category/ProductVariant trước đây KHÔNG có timestamp
4. Thiếu "ai làm" — chỉ có "khi nào" (`createdAt`), thiếu `createdBy`/`updatedBy`

## 5.2 Ba cách làm — so sánh

### Cách A — Hibernate `@CreationTimestamp` / `@UpdateTimestamp`
```java
@CreationTimestamp private LocalDateTime createdAt;
@UpdateTimestamp  private LocalDateTime updatedAt;
```
- Của Hibernate (không phải JPA chuẩn). Tự set timestamp khi INSERT/UPDATE
- **Chỉ làm timestamp** — KHÔNG làm được `createdBy`/`updatedBy`

### Cách B — JPA `@PrePersist` / `@PreUpdate` (lifecycle callback)
```java
@PrePersist protected void onCreate() { createdAt = LocalDateTime.now(); }
@PreUpdate  protected void onUpdate() { updatedAt = LocalDateTime.now(); }
```
- JPA chuẩn. Để lấy `createdBy` phải tự gọi `SecurityContextHolder` trong callback → coupling entity với Security (xấu)

### Cách C — Spring Data JPA Auditing ⭐ (CHỌN)
```java
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {
    @CreatedDate      private LocalDateTime createdAt;
    @LastModifiedDate private LocalDateTime updatedAt;
    @CreatedBy        private String createdBy;
    @LastModifiedBy   private String updatedBy;
}
```
- Của Spring Data. Làm CẢ timestamp VÀ user (`createdBy` qua `AuditorAware`)
- Entity KHÔNG coupling Security (tách qua bean). Chuẩn enterprise

### Bảng so sánh

| | A: Hibernate | B: JPA callback | C: Spring Data ⭐ |
|---|---|---|---|
| Timestamp tự động | ✅ | ✅ | ✅ |
| `createdBy`/`updatedBy` | ❌ | ⚠️ Tự code, coupling | ✅ Qua `AuditorAware` |
| Entity coupling Security | Không | Có (xấu) | Không |
| Tái sử dụng (BaseEntity) | Khó | Được | ✅ Dễ |
| Chuẩn enterprise | Không hẳn | Không | ✅ |

## 5.3 Cơ chế bên trong — `AuditingEntityListener` hook vào đâu

```
save(entity)
   │
   ▼ Hibernate chuẩn bị INSERT/UPDATE
   │
   ▼ JPA Lifecycle Events trigger: @PrePersist (trước INSERT) / @PreUpdate (trước UPDATE)
   │
   │  AuditingEntityListener đăng ký nghe các event này
   ▼
   AuditingEntityListener.touchForCreate() / touchForUpdate():
     1. Set @CreatedDate/@LastModifiedDate = now() (từ DateTimeProvider)
     2. Set @CreatedBy/@LastModifiedBy = gọi AuditorAware.getCurrentAuditor()
            → đọc SecurityContextHolder → username user đang login
   │
   ▼ INSERT/UPDATE chạy với field audit đã set
```

→ So sánh Node.js: gần nhất là Mongoose middleware `schema.pre('save', fn)` — cũng hook lifecycle. Nhưng Mongoose không có "auditor aware" tự lấy user; Spring tách concern này qua `AuditorAware` bean.

## 5.4 Các thành phần đã tạo

| Thành phần | Vai trò |
|---|---|
| `BaseEntity` (abstract, `@MappedSuperclass`, `@EntityListeners`) | Chứa 4 field audit |
| `@EnableJpaAuditing` (file `JpaAuditingConfig`) | Bật auditing |
| `SpringSecurityAuditorAware implements AuditorAware<String>` | Cung cấp "user hiện tại" |
| 5 entity `extends BaseEntity` | User, Category, Product, ProductVariant, ProductImage |
| Service | Xoá các dòng `setCreatedAt`/`setUpdatedAt` thủ công |

Code `BaseEntity`:
```java

@EntityListeners(AuditingEntityListener.class)

    @LastModifiedDate private LocalDateTime updatedAt;
```

Code `SpringSecurityAuditorAware`:
```java
@Component
public class SpringSecurityAuditorAware implements AuditorAware<String> {
    private static final String SYSTEM = "SYSTEM";
    @Override
    public Optional<String> getCurrentAuditor() {
        Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
        if (authentication == null || !authentication.isAuthenticated()
                || authentication.getName().equals("anonymousUser")) {
            return Optional.of(SYSTEM);
        }
        return Optional.of(authentication.getName());
    }
}
```

## 5.5 Q&A — `@MappedSuperclass`

### Câu hỏi học viên:
> @MappedSupperclass là gì? AuditorAware là gì? (câu 3 về DB cột cũ)

### @MappedSuperclass — kế thừa field mà KHÔNG tạo bảng riêng

JPA có 4 chiến lược inheritance:

| Chiến lược | Tổ chức bảng |
|---|---|
| `@MappedSuperclass` | KHÔNG có bảng parent. Field parent **copy xuống** mỗi bảng con |
| `SINGLE_TABLE` | 1 bảng chung + cột `dtype` phân biệt |
| `JOINED` | Mỗi class 1 bảng, JOIN khi query |
| `TABLE_PER_CLASS` | Mỗi class con 1 bảng đầy đủ |

→ Với `@MappedSuperclass`, DB KHÔNG có bảng `base_entity`. Field `createdAt`... copy thẳng vào bảng `users`, `products`...

→ KHÔNG dùng `@Entity` cho BaseEntity vì sẽ tạo bảng `base_entity` riêng + áp inheritance strategy → phức tạp hoá DB không cần thiết. BaseEntity chỉ là "khuôn mẫu chia sẻ field", không phải thực thể nghiệp vụ.

→ So sánh Node: giống base Mongoose schema rồi `.add()` vào schema con — share definition, mỗi collection riêng.

### AuditorAware — "ai là người đang thao tác?"

`@CreatedBy` cần biết user nào, nhưng Spring không tự biết. `AuditorAware` là interface ta implement để trả lời "user hiện tại là ai".

```
save(user) → AuditingEntityListener (@PrePersist)
   → cần điền @CreatedBy
   → gọi auditorAware.getCurrentAuditor()
   → đọc SecurityContextHolder → "tung@example.com"
   → set createdBy = "tung@example.com"
```

→ Là **cầu nối** Spring Data Auditing ↔ Spring Security. Tách biệt: entity không biết Security, AuditorAware là nơi DUY NHẤT đọc SecurityContext. Đổi cách xác định user → chỉ sửa 1 chỗ.

→ Handle chưa login: return `Optional.of("SYSTEM")` cho seed/background/public endpoint. KHÔNG return `Optional.empty()` (sẽ để `createdBy` null, mất audit trail).

### Câu 3 — DB cột cũ với `ddl-auto: update`

**`ddl-auto: update` CHỈ THÊM, không bao giờ XOÁ/SỬA:**
```
✅ Tạo bảng mới, thêm cột mới, thêm constraint mới
❌ Xoá cột (dù entity bỏ field), đổi kiểu, đổi tên, xoá bảng
```

→ Áp dụng: giữ NGUYÊN tên `createdAt`/`updatedAt` → cột DB trùng tên → Hibernate dùng lại cột cũ, không vấn đề. Chỉ thêm 2 cột mới `created_by`/`updated_by`.

→ Bài học chung: production KHÔNG dùng `ddl-auto: update` (rủi ro schema drift) → dùng **Flyway/Liquibase** (migration script có version). Note item production-grade này cho sau.

## 5.6 Bug review — `ProductImage` còn sót field `createdAt`

Học viên `extends BaseEntity` nhưng QUÊN xoá `private LocalDateTime createdAt;` ở `ProductImage` → entity có **2 field `createdAt`** (1 kế thừa từ BaseEntity có `@CreatedDate`, 1 field riêng) → cả 2 cùng map cột `created_at`:

```

ProductImage column: created_at
```

→ KHÔNG phải compile error (javac pass) mà là **runtime MappingException** lúc Hibernate build metamodel → app crash lúc khởi động.

→ **Bài học:** Khi gom field lên BaseEntity, phải xoá HẾT field trùng ở entity con. Sót lại = bug khó vì javac không bắt được, chỉ lộ lúc chạy.

## 5.7 Q&A — Tại sao `BaseEntity` là `abstract`?

### Câu hỏi học viên:
> Tại sao BaseEntity lại là abstract class mà không là class bình thường? Nếu dùng class bình thường thì sao?

### Trả lời

**Không bắt buộc** — class thường vẫn chạy. Nhưng `abstract` đúng thiết kế:

1. **BaseEntity không bao giờ tồn tại độc lập** — không có "một BaseEntity" thuần (chỉ createdAt mà không là User/Product). Nó là khuôn mẫu để kế thừa, không để tạo ra. `abstract` diễn đạt đúng ý: `new BaseEntity()` → compile error.

2. **`@MappedSuperclass` củng cố** — không có bảng `base_entity` → `persist(new BaseEntity())` sẽ throw runtime mơ hồ. `abstract` biến nó thành compile error rõ ràng.

3. **Nếu class thường** → 4 rủi ro: cho phép `new BaseEntity()` vô nghĩa; lỡ dùng như type thật; người đọc nhầm "có bảng"; lỡ tạo `JpaRepository<BaseEntity, Long>`. Không cái nào vỡ app ngay, nhưng đều là cửa mở cho bug.

→ **Nguyên tắc OOP chung:** class chỉ-để-kế-thừa, không bao giờ instantiate trực tiếp → để `abstract` (như `AbstractList`, `InputStream`). `abstract` giao tiếp ý định + compiler enforce giùm.

## 5.8 Q&A — Field chung cho mọi bảng đặt ở đâu? (+ soft delete)

### Câu hỏi học viên:
> Nếu muốn thêm 1 field vào tất cả bảng, thêm vào BaseEntity hay từng bảng? Ví dụ is_deleted default false có đúng practice không?

### Trả lời

**Field áp dụng MỌI bảng → thêm vào BaseEntity.** Đúng practice, đúng chỗ — thêm 1 chỗ, mọi entity tự có.

**NHƯNG `is_deleted` là soft delete — không phải field thường.** Thêm field chỉ là 5% công việc. 4 cạm bẫy:

**Cạm bẫy 1 — Mọi SELECT phải lọc `is_deleted=false`:**
- `findAll()`/`findById()` mặc định trả CẢ record đã xoá
- Giải pháp: `@SQLRestriction("is_deleted = false")` (Hibernate 6.3+, trước là `@Where`) — tự chèn vào mọi SELECT
- ⚠️ `@SQLRestriction` phải đặt trên TỪNG `@Entity`, KHÔNG đặt được ở `@MappedSuperclass`

**Cạm bẫy 2 — DELETE phải thành UPDATE:**
- `repository.delete()` mặc định xoá THẬT
- Giải pháp: `@SQLDelete(sql = "UPDATE users SET is_deleted = true WHERE id = ?")`

**Cạm bẫy 3 — Unique constraint vỡ:**
- `email` unique → soft delete user "a@x.com" → tạo lại "a@x.com" → UNIQUE violation (record cũ vẫn chiếm)
- Giải pháp phức tạp: partial unique index (PostgreSQL `WHERE is_deleted=false`, MySQL không hỗ trợ tốt), hoặc composite unique, hoặc append timestamp khi xoá → chưa có giải pháp đẹp universal

**Cạm bẫy 4 — Default value field-level vs DB-level:**
- `private Boolean isDeleted = false` đúng cho entity mới, nhưng row CŨ → cột `is_deleted = NULL`
- `@SQLRestriction("is_deleted = false")` → `NULL = false` trả false → record cũ biến mất khỏi mọi query
- Giải pháp: `@Column(nullable=false)` + migrate `UPDATE ... SET is_deleted=false WHERE is_deleted IS NULL`

### Tóm tắt

| Câu hỏi | Trả lời |
|---|---|
| Field chung mọi bảng → BaseEntity? | ✅ Đúng practice |
| Chỉ thêm field là xong? | ❌ Soft delete cần `@SQLRestriction` + `@SQLDelete` per-entity + xử lý unique + migrate NULL |

→ **Field đặt đúng chỗ (BaseEntity), nhưng soft delete là cả một feature, không phải một field.**

→ Lời khuyên: nếu chỉ vài bảng cần soft delete → tạo `SoftDeletableEntity extends BaseEntity` riêng, không nhét vào BaseEntity. Hoặc nếu chỉ cần lịch sử thay đổi → dùng Hibernate Envers (audit log table) thay vì soft delete.

## 5.9 Tổng kết Cụm 3

✅ **BaseEntity + JPA Auditing:**
- `BaseEntity` abstract, `@MappedSuperclass`, `@EntityListeners(AuditingEntityListener.class)`, 4 field audit
- `SpringSecurityAuditorAware` cung cấp user hiện tại (fallback SYSTEM)
- `@EnableJpaAuditing` ở `JpaAuditingConfig`
- 5 entity `extends BaseEntity`, xoá field timestamp cũ
- Service xoá `setCreatedAt`/`setUpdatedAt` thủ công
- Category + ProductVariant trước không có timestamp giờ tự có đủ 4 field audit

🎓 **Kiến thức đào sâu:**
- 3 cách auditing (Hibernate `@CreationTimestamp` / JPA `@PrePersist` / Spring Data `@CreatedDate`) + trade-off
- `AuditingEntityListener` hook vào lifecycle `@PrePersist`/`@PreUpdate`
- `@MappedSuperclass` vs 4 inheritance strategy — tại sao không tạo bảng riêng
- `AuditorAware` là cầu nối Auditing ↔ Security, entity không coupling Security
- `ddl-auto: update` chỉ THÊM không xoá/sửa → giữ tên field cũ thì an toàn
- Repeated column MappingException khi sót field trùng ở entity con
- Tại sao class chỉ-để-kế-thừa nên `abstract`
- Field chung → BaseEntity; soft delete là feature phức tạp (4 cạm bẫy), không chỉ là field

⏳ **Còn lại trong backlog:** Cụm 2.3 (@Version apply), Cụm 2.4 (TOCTOU), Cụm 4 (Security exception + @AuthenticationPrincipal), Cụm 5 (compensating tx). Production-grade thêm: Flyway/Liquibase thay `ddl-auto: update`.

# Phần 6 — Cụm 3.6: Logging với `@Slf4j` (2026-06-07)

## 6.1 Logging vs Auditing — khác gì nhau

| | Auditing (DB) | Logging (log file/console) |
|---|---|---|
| Lưu ở đâu | Cột `created_by`, `updated_at` trong bảng | File log / stdout / log aggregator (ELK, Loki) |
| Trả lời | "Record này hiện do ai tạo, sửa lúc nào?" | "Chuyện gì đã xảy ra theo thời gian? Vì sao lỗi?" |
| Tồn tại | Vĩnh viễn (đến khi record bị xoá) | Theo retention (vài ngày → vài tháng) |
| Dùng để | Hiển thị UI, audit trail nghiệp vụ | Debug, monitor, alert, trace lỗi |
| Lịch sử | Không (chỉ state hiện tại) | Có (mỗi event 1 dòng theo thời gian) |

→ Bổ sung nhau, không thay thế. Ví dụ user đổi role: auditing chỉ biết lần sửa CUỐI (`updated_by`), logging biết cả lịch sử mỗi lần đổi.

## 6.2 Kiến trúc logging Java — Facade vs Implementation

```
CODE: log.info("User created: {}", id);
   │
   ▼ FACADE (interface) — SLF4J  → chỉ là "mặt tiền", KHÔNG tự ghi
   │   (code chỉ phụ thuộc SLF4J, không phụ thuộc impl cụ thể)
   ▼ (runtime binding)
   IMPLEMENTATION (engine thật) — Logback (Spring Boot default)
       → thực sự ghi console/file, format, rotate
       Alternative: Log4j2, java.util.logging
```

→ Tách facade/impl giống JDBC: code dùng `java.sql.Connection`, đổi MySQL→PostgreSQL chỉ đổi driver. Logging: code dùng SLF4J, đổi Logback→Log4j2 không sửa code.

→ `@Slf4j` (Lombok) sinh sẵn:
```java
private static final org.slf4j.Logger log = org.slf4j.LoggerFactory.getLogger(YourClass.class);
```

## 6.3 Log Levels

```
TRACE  ← chi tiết nhất (hiếm dùng)
DEBUG  ← debug info (dev environment)
INFO   ← sự kiện nghiệp vụ bình thường ("user created", "order placed")
WARN   ← bất thường nhưng chưa lỗi ("retry lần 2", "cache miss cao")
ERROR  ← lỗi cần chú ý ("payment failed", "DB connection lost")
```

| Tình huống | Level |
|---|---|
| Tạo/sửa/xoá entity thành công | `INFO` |
| Business rule violation (duplicate, not found) | `WARN` (input sai, không phải bug) |
| Exception bất ngờ (NPE, DB down) | `ERROR` + stack trace |
| Chi tiết tham số debug | `DEBUG` |

→ Bẫy: log mọi thứ ở ERROR → log production đầy "ERROR" giả → alert noise → bỏ sót lỗi thật.

## 6.4 ⚠️ Quy tắc VÀNG — KHÔNG log dữ liệu nhạy cảm

```java
// ❌ TUYỆT ĐỐI KHÔNG
log.info("login: email={}, password={}", email, password);
log.info("Created user: {}", user);   // ← nếu user.toString() có password!
log.info("JWT token: {}", token);

// ✅ ĐÚNG
log.info("User login attempt: email={}", email);
log.info("User created: id={}, email={}", user.getId(), user.getEmail());
```

**Cấm log:** `password`, `token` (JWT/refresh), `creditCard`, `cvv`, `ssn`, `secret`, `apiKey`, OTP.

→ Lý do: log lưu plain text, đẩy lên aggregator (nhiều người xem), giữ lâu → rò rỉ bảo mật (GDPR/PCI-DSS).

→ **Cảnh báo `@Slf4j` + entity:** `log.info("user: {}", user)` gọi `user.toString()`. Nếu entity có Lombok `@ToString`/`@Data` → in cả `password`! Đây là lý do CLAUDE.md cấm `@Data` cho entity. → Luôn log field cụ thể, KHÔNG log cả object entity.

## 6.5 Performance — parameterized logging

```java
// ❌ String concat — luôn nối chuỗi DÙ level tắt
log.debug("Processing user " + userId + " with " + count);

// ✅ Parameterized — chỉ nối chuỗi KHI level bật
log.debug("Processing user {} with {}", userId, count);
```

→ Với `{}`, SLF4J chỉ thay placeholder SAU khi check level bật. Nếu DEBUG tắt (production) → không tốn CPU nối chuỗi. Với `+`, chuỗi nối TRƯỚC khi gọi → luôn tốn dù log bị bỏ. → Luôn dùng `{}`.

## 6.6 Pattern áp dụng + scope

Scope đã chốt với học viên:
- Log 5 service write (User, Category, Product, ProductVariant, ProductImage)
- Chỉ log **write operation** (create/update/delete) ở `INFO`
- KHÔNG log read, KHÔNG log business rule violation

Pattern mẫu (UserService):
```java
@Slf4j @Service
public class UserService {
    @Transactional
    public UserResponse createUser(CreateUserRequest request) {
        log.info("Creating user with email: {}", request.getEmail());  // bắt đầu (business key, KHÔNG password)
        // ... logic ...
        User createdUser = userRepository.save(user);
        log.info("User created successfully: id={}", createdUser.getId());  // success với id
        return userMapper.fromEntity(createdUser);
    }
    public UserResponse getUserById(Long id) { ... }  // read → KHÔNG log
}
```

## 6.7 Bug review — copy-paste log sai entity

`ProductVariantService.deleteVariant` ban đầu:
```java
log.info("Category delete successfully: id={}", variantId);  // ← "Category"?! Đây là Variant
```
→ Đã sửa thành `"Variant deleted successfully"`.

→ **Bài học:** Bug copy-paste trong log message là loại **âm thầm nguy hiểm** — code chạy đúng, test pass, nhưng production có sự cố thì log sai entity dắt người debug đi sai hướng hàng giờ. Log sai còn tệ hơn không log. → Khi copy log giữa các method, LUÔN đọc lại đổi đúng tên entity.

Ngoài ra: `setPrimary`/`deleteImage` (ProductImageService) ban đầu thiếu log (đều là write) → đã thêm cho nhất quán scope.

## 6.8 Tổng kết Cụm 3.6

✅ **Logging:**
- 5 service write thêm `@Slf4j` + log create/update/delete ở `INFO`
- Parameterized `{}`, log business key (slug/sku/id), KHÔNG log password/file content/object entity
- Fix bug copy-paste "Category" trong `deleteVariant`
- Thêm log thiếu cho `setPrimary`/`deleteImage`

🎓 **Kiến thức:**
- Logging vs Auditing (log file vs DB column, lịch sử vs state hiện tại)
- Kiến trúc Facade (SLF4J) vs Implementation (Logback) — tách như JDBC driver
- 5 log levels + khi nào dùng (WARN cho business violation, ERROR cho bug)
- Quy tắc vàng KHÔNG log sensitive + cảnh báo `@Slf4j` + entity `toString()`
- Parameterized `{}` vs `+` — performance khi level tắt

# Phần 7 — Cụm 4 (#8): Spring Security Exception Flow (2026-06-07)

## 7.1 Bức tranh lớn — 2 "thế giới" tách biệt

Tại sao 401 không xử lý được trong `GlobalExceptionHandler` như exception thường? Vì request đi qua 2 thế giới, exception mỗi thế giới xử lý bởi cơ chế khác.

```
                    HTTP Request
                         │
   ╔═════════════════════▼═════════════════════════════╗
   ║  THẾ GIỚI 1 — SERVLET FILTER CHAIN (Spring Security)║
   ║   [JwtAuthenticationFilter]  ← set authentication  ║
   ║   [ExceptionTranslationFilter] ← bắt security exc  ║
   ║   [AuthorizationFilter]  ← check quyền URL-level   ║
   ╚═════════════════════╪═════════════════════════════╝
                         ▼
   ╔═════════════════════════════════════════════════════╗
   ║  THẾ GIỚI 2 — SPRING MVC (DispatcherServlet)        ║
   ║   [DispatcherServlet]                               ║
   ║   [Controller] → @PreAuthorize check ở method        ║
   ║   [Service] → throw exception → @RestControllerAdvice║
   ╚═════════════════════════════════════════════════════╝
```

→ `@RestControllerAdvice`/`@ExceptionHandler` là cơ chế của **Spring MVC** (Thế giới 2), CHỈ bắt exception bên trong DispatcherServlet. Exception ở **Filter Chain (Thế giới 1)** — TRƯỚC khi chạm DispatcherServlet — `@RestControllerAdvice` KHÔNG bao giờ thấy.

→ So sánh Express: `app.use(authMiddleware)` chạy trước route. Nếu nó `res.status(401)` luôn → không tới error middleware cuối. Filter Spring cũng vậy.

## 7.2 Ba loại "từ chối truy cập" — KHÁC NHAU

| # | Tình huống | HTTP | Xảy ra ở đâu | Ai xử lý |
|---|---|---|---|---|
| 1 | Chưa xác thực (no token / token sai) | **401** | Filter Chain | `AuthenticationEntryPoint` |
| 2 | Đã auth, thiếu quyền — **URL level** (`.requestMatchers().hasRole()`) | **403** | Filter (`AuthorizationFilter`) | `AccessDeniedHandler` |
| 3 | Đã auth, thiếu quyền — **Method level** (`@PreAuthorize`) | **403** | DispatcherServlet (method AOP) | `@RestControllerAdvice` |

→ **401 vs 403 semantic:** 401 = "không biết bạn là ai" (đi login) · 403 = "biết rồi nhưng không đủ quyền" (login lại vô ích).

## 7.3 `ExceptionTranslationFilter` — error handler của Thế giới 1

```
[ExceptionTranslationFilter]
   try { filterChain.doFilter() }   ← chạy filter sau (gồm AuthorizationFilter)
   catch:
     - AuthenticationException (chưa auth) → authenticationEntryPoint.commence() → 401
     - AccessDeniedException (thiếu quyền):
         + user CHƯA auth → authenticationEntryPoint (401)
         + user ĐÃ auth   → accessDeniedHandler.handle() → 403
```

→ Đây là lý do `AuthenticationEntryPoint` + `AccessDeniedHandler` tồn tại — callback cho `ExceptionTranslationFilter`, nơi `@RestControllerAdvice` không với tới.

## 7.4 Tại sao `@PreAuthorize` LẠI tới được `@RestControllerAdvice`?

`@PreAuthorize` dùng AOP method interceptor — check quyền bên trong Thế giới 2 (lúc gọi method, đã qua DispatcherServlet):

```
[DispatcherServlet] → [Controller method]
   @PreAuthorize("hasRole('ADMIN')") ← AOP check
     fail → throw AuthorizationDeniedException
            → exception nằm TRONG DispatcherServlet → @RestControllerAdvice BẮT ĐƯỢC ✅
```

→ Khác biệt: URL-level authz → `AuthorizationFilter` (Thế giới 1) → cần `AccessDeniedHandler`. Method-level `@PreAuthorize` → method AOP (Thế giới 2) → `@RestControllerAdvice` bắt.

## 7.5 Code đã thêm

`JwtAccessDeniedHandler` (đối xứng `JwtAuthenticationEntryPoint`):
```java
@Component
@RequiredArgsConstructor
public class JwtAccessDeniedHandler implements AccessDeniedHandler {
    private final ObjectMapper objectMapper;
    @Override
    public void handle(HttpServletRequest request, HttpServletResponse response,
                       AccessDeniedException ex) throws IOException, ServletException {
        response.setStatus(HttpServletResponse.SC_FORBIDDEN);   // 403, KHÔNG phải 401
        response.setContentType(MediaType.APPLICATION_JSON_VALUE);
        response.setCharacterEncoding("UTF-8");
        BaseResponse<Void> body = BaseResponse.error("You do not have permission to access this resource");
        objectMapper.writeValue(response.getOutputStream(), body);
    }
}
```

Wire vào `SecurityConfig`:
```java
.exceptionHandling(ex -> ex
        .authenticationEntryPoint(jwtAuthenticationEntryPoint)   // 401
        .accessDeniedHandler(jwtAccessDeniedHandler)              // 403
)
```

## 7.6 Q&A check kiến thức — verbatim

### 3 câu hỏi:
1. Tại sao no-token trả 401 không phải 403?
2. CUSTOMER gọi `POST /api/categories` (cần ADMIN qua `@PreAuthorize`) → đi thế giới nào? Nếu URL-level thì ai xử lý?
3. AuthenticationEntryPoint vs AccessDeniedHandler — cái nào "chưa login", cái nào "thiếu quyền"?

### Trả lời học viên (verbatim):
> 1. Tại vì no-token nó cho thấy là người dùng chưa đăng nhập thể hiện no authentication nên sẽ trả lỗi 401 chứ không phải user không có quyển truy cập (lỗi 403)
> 2. Thế giới thứ 2 sẽ xử lý, nếu dủng URL-level thì thế giới 1 sẽ xử lý.
> 3. AuthenticationEntryPoint xử lý chưa login, AccessDeniedHandler xử lý chưa có quyền.

→ ✅ 3/3 đúng hết, kể cả điểm tinh tế nhất (method-level vs URL-level → 2 thế giới khác nhau).

## 7.7 Thí nghiệm của học viên — minh chứng sống "2 thế giới"

### Câu hỏi học viên:
> Tôi đã chạy đúng rồi, nhưng tôi muốn test lại nên đã comment phần accessDeniedHandler thì lại vẫn báo lỗi 403 như lúc chưa comment, vấn đề đang nằm ở đâu?

### Giải đáp — KHÔNG phải bug, là hành vi đúng

Endpoint test (`POST /api/categories`) dùng `@PreAuthorize` (method-level) → khi CUSTOMER gọi:
```
Token CUSTOMER hợp lệ
 → [Filter] JwtAuthFilter set auth OK; AuthorizationFilter .anyRequest().authenticated() PASS
   (KHÔNG có URL-level .hasRole() fail)
 → [DispatcherServlet] CategoryController.create()
   @PreAuthorize("hasRole('ADMIN')") → fail → throw AuthorizationDeniedException
   → @RestControllerAdvice (GlobalExceptionHandler) bắt → 403 JSON
```

→ 403 này đến từ `GlobalExceptionHandler` (Thế giới 2), KHÔNG phải `JwtAccessDeniedHandler` (Thế giới 1). Comment `.accessDeniedHandler()` không ảnh hưởng vì handler đó chỉ xử lý URL-level (filter), test không kích hoạt đường đó. `JwtAccessDeniedHandler` chưa từng được gọi trong test này.

### Cách chứng minh — nhìn message khác nhau

| Nguồn | Message |
|---|---|
| `GlobalExceptionHandler` (Thế giới 2) | "You do not have permission to **perform this action**" |
| `JwtAccessDeniedHandler` (Thế giới 1) | "You do not have permission to **access this resource**" |

→ Lúc comment, message là "perform this action" → xác nhận `GlobalExceptionHandler` xử lý (Thế giới 2).

### Cách test ĐÚNG JwtAccessDeniedHandler
Thêm tạm URL-level rule: `.requestMatchers(POST, "/api/categories").hasRole("ADMIN")` → CUSTOMER gọi → `AuthorizationFilter` (Thế giới 1) chặn → `JwtAccessDeniedHandler` → message "access this resource". Comment `.accessDeniedHandler()` lúc này MỚI thấy khác biệt (403 plain text thay JSON).

### Insight quan trọng
> Codebase dùng `@PreAuthorize` (method-level) cho mọi authz → `AccessDeniedHandler` gần như KHÔNG bao giờ được gọi. Nó là **lưới an toàn** cho: tương lai thêm URL-level rule; authz fail ở filter khác; đảm bảo KHÔNG đường nào trả 403 plain text — mọi 403 đều JSON consistent.

→ Tinh thần "defense in depth" lại xuất hiện: đường chính (method-level) đã xử lý, vẫn bịt nốt đường phụ (filter-level).

## 7.8 Tổng kết Cụm 4 (#8)

✅ **Security exception handlers:**
- `JwtAccessDeniedHandler implements AccessDeniedHandler` → 403 JSON consistent
- Wire vào `SecurityConfig.exceptionHandling`: entryPoint (401) + accessDeniedHandler (403)
- `JwtAuthenticationEntryPoint` đã có từ quick fix (401 JSON)

🎓 **Kiến thức đào sâu:**
- 2 thế giới: Filter Chain vs DispatcherServlet — `@RestControllerAdvice` chỉ thấy Thế giới 2
- 3 loại từ chối: 401 (chưa auth), 403 URL-level (filter), 403 method-level (`@PreAuthorize`)
- `ExceptionTranslationFilter` = error handler của Thế giới 1
- Tại sao `@PreAuthorize` (method AOP) tới được `@RestControllerAdvice` nhưng no-token thì không
- `AccessDeniedHandler` vs `@ExceptionHandler(AccessDeniedException)` — KHÔNG xung đột vì bắt ở 2 thế giới khác nhau
- Thí nghiệm thực: comment handler vẫn 403 (vì đi method-level) → minh chứng "2 thế giới"

## 8.4 Ba khái niệm cốt lõi

# Phần 9 — Cụm 2.3: Optimistic Locking với `@Version` (2026-06-08)

## 9.1 Vấn đề: Lost Update (mất cập nhật)

Race condition khi 2 transaction cùng đọc → cùng sửa → cùng ghi một row. TX đến sau ghi đè
update của TX trước mà **không có lỗi nào báo** — đó là điểm nguy hiểm.

```
t1  TX-A đọc Product(price=100, stock=5)
t2  TX-B đọc Product(price=100, stock=5)
t3  TX-A set price=200 → COMMIT (price=200, stock=5)
t4  TX-B set stock=10  → COMMIT (price=100, stock=10)  ❌ update price của A bốc hơi
```

## 9.2 Giải pháp: cột `version`

Hibernate tự đổi câu UPDATE khi entity có field `@Version`:

```sql
UPDATE products SET price=200, version=6 WHERE id=1 AND version=5;
```

Sau UPDATE, Hibernate đọc **số row affected**:
- `affected = 1` → OK, version trong context = 6
- `affected = 0` → có TX khác đã commit (version đã ≠ 5) → ném `OptimisticLockException`

→ Phát hiện xung đột bằng cách **đếm số row UPDATE**, không cần khoá, chi phí ~0.

## 9.3 Q&A — verbatim

**Hỏi:** *"ok đã hiểu, nhưng vẫn đề vẫn tồn tại mà, nếu như cả 2 user cùng lấy product về
có version là 5, như vậy thì khi cả 2 user này cùng update product đều thành công mà đúng không"*

**Đáp:** Không. Hiểu lầm nằm ở giả định "cả 2 cùng update nên cả 2 cùng thấy version=5 trong DB".
Dù 2 user bấm "cùng lúc" ở tầng app, khi xuống DB thì **DB tự serialize ghi vào cùng 1 row bằng
row-level write lock** — không có chuyện 2 lệnh UPDATE cùng row chạy song song thật. DB cho 1 thằng
vào trước, thằng kia chờ:

```
[1] A: UPDATE ... SET version=6 WHERE id=1 AND version=5  → row đang là 5 → KHỚP → 1 row ✅ → COMMIT, DB=6
[2] B: UPDATE ... SET version=6 WHERE id=1 AND version=5  → row GIỜ là 6 → KHÔNG KHỚP → 0 row ❌ → exception
```

Cả 2 cùng cầm `version=5` (ở app) là đúng — nhưng đó chỉ là ảnh chụp lúc đọc. Cái quyết định
thắng/thua là **giá trị version THẬT trong DB tại lúc UPDATE chạy**. Điều kiện `AND version=5` của B
được DB đánh giá lúc B ghi, mà lúc đó A đã commit version=6 → B không tìm thấy row → fail.

---

**Hỏi:** *"Chỉ cần thêm từ khoá @Version vào field version là được à, còn các việc còn lại thì spring
sẽ tự hỗ trợ mình đúng không. ví dụ như khi update cũng không cần thêm điều kiện and version = ? vào à"*

**Đáp:** Chính xác. Chỉ cần `@Version` là xong — Hibernate lo hết:
- INSERT: tự set `version=0` cho row mới
- SELECT: nhớ version hiện tại trong Persistence Context
- UPDATE: tự thêm `version=version+1` và `WHERE ... AND version=?`
- Tự đếm row affected, tự ném exception khi = 0

Đây là sức mạnh **declarative programming** của JPA: khai báo ý định bằng 1 annotation, framework lo cơ chế.
(Khác Node.js: nhiều ORM/raw query phải tự viết `WHERE version=?` + tự `version+1`.)

## 9.4 Quyết định kỹ thuật

1. **Đặt `@Version` ở `BaseEntity`** — 5 entity đã `extends BaseEntity` → viết 1 chỗ, tất cả có. DRY.
2. **Kiểu `Long` (wrapper), KHÔNG `long` (primitive):** entity mới có `version=null` → Hibernate biết
   "đây là INSERT". Nếu `long` → default `0` → Hibernate nhầm là entity đã tồn tại → có thể ném lỗi sai.
3. **Hibernate tự tăng version — TUYỆT ĐỐI không `setVersion()` thủ công.**
4. **Bắt `ObjectOptimisticLockingFailureException` (không phải `OptimisticLockException`):** Spring wrap
   exception đặc thù Hibernate thành cây `DataAccessException` thống nhất → code không phụ thuộc ORM.
   Trả **409 Conflict**, KHÔNG `log.error` (lỗi nghiệp vụ bình thường, không phải bug server).

## 9.5 Hệ quả với MapStruct (bug compile)

Thêm field vào `BaseEntity` làm vỡ các mapper **Request DTO → Entity**:

```
[ERROR] Unmapped target property: "version"
```

Vì `unmappedTargetPolicy = ERROR` (mặc định MapStruct): entity có `version` nhưng Request DTO không có
→ MapStruct báo lỗi. Sửa: thêm `@Mapping(target = "version", ignore = true)` vào `fromRequestDto` +
`updateEntity` của cả 4 mapper (Category/User/Product/ProductVariant). Method `fromEntity` (Entity→Response)
KHÔNG lỗi vì version là source thừa (MapStruct mặc định không báo lỗi source thừa) — đúng với quyết định
trước: **không đưa version vào Response DTO**.

→ Bài học: mỗi field BaseEntity mới đều bắt khai báo ý định rõ ràng ở mapper — an toàn hơn bỏ sót ngầm.

## 9.6 Xử lý data cũ `version = NULL`

Với `ddl-auto: update`, cột `version` thêm vào → row cũ có `version = NULL`. UPDATE row cũ sẽ
fail oan vì `WHERE ... AND version = NULL` không bao giờ true trong SQL.

→ Đã chọn **Cách A (backfill)**: `UPDATE <table> SET version=0 WHERE version IS NULL` cho 5 bảng.
(Production thật bắt buộc dùng cách này, kèm Flyway/Liquibase. Project học có thể reset DB cho nhanh.)

## 9.7 Cách test xung đột (verbatim gotcha)

Vì service tự `findById` mỗi lần, bắn 2 request **lần lượt** không bao giờ xung đột → phải cho 2
transaction **chồng lấn**. Kéo dài cửa sổ bằng `Thread.sleep` tạm thời **sau `findById`**:

```java
Product product = productRepository.findById(id).orElseThrow(...);
try { Thread.sleep(15000); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
```

Restart → mở 2 tab Postman → bắn tab 1, trong lúc treo bắn tab 2.

**Hỏi:** *"Tôi đã thử như bạn nhưng mà cả 2 đều update được là sao"*

**Đáp:** Log cho thấy 2 transaction CHỒNG LẤN đúng (2 dòng "Update product : 3" trước mọi "successfully"),
nhưng **chỉ có 1 câu `update products`** trong log → chỉ 1 transaction thực sự dirty. Nguyên nhân:

> `@Version` chỉ kích hoạt khi entity **thực sự dirty**. Request gửi body trùng giá trị đang có → Hibernate
> dirty-checking thấy không có gì đổi → **không sinh UPDATE** → không check version → trả 200 "ảo".

→ Sửa: mỗi tab đổi 1 field sang giá trị MỚI, KHÁC nhau (tab1 `stock=111`, tab2 `stock=222`). Khi đó log
có **2 câu UPDATE**, câu thứ 2 trả 0 row → 409. ✅ Tab 2 ra **409 Conflict** như mong đợi.

## 9.7b Q&A — Server biết version bằng cách nào khi client không gửi? (verbatim)

**Hỏi:** *"khi update tôi không đẩy bất cứ thông tin version nào từ client xuồng server vậy thì làm sao
server đang biết entity này có version là gì"*

**Đáp:** Server biết version qua `findById` — câu SELECT load cả cột `version` vào Persistence Context,
Hibernate ghi nhớ version vừa đọc (vd 5), đến lúc flush dùng nó trong `WHERE ... AND version=5`. Server
không cần client gửi vì nó tự đọc version mới nhất ngay trong transaction.

**NHƯNG đây là lỗ hổng tinh tế.** Vì server tự đọc version **mới nhất tại thời điểm update**, cách này
chỉ bảo vệ 1 trong 2 loại race:

**Loại 1 — 2 request CHỒNG LẤN thời gian (✅ được bảo vệ — chính là cái test bằng Thread.sleep):**

```
t1  Req A: findById → version=5 (giữ trong tx A)
t2  Req B: findById → version=5 (giữ trong tx B)   ← cả 2 còn đang chạy
t3  Req A: commit → version=6 ✅
t4  Req B: commit → WHERE version=5 → 0 row → 409 ✅
```

**Loại 2 — "think-time" / stale client (❌ KHÔNG được bảo vệ):**

```
09:00  User A mở form → thấy price=100 (version=5)
09:01  User B sửa price=200 → lưu → DB version=6
09:05  User A bấm Lưu (price=90)
        → server findById → đọc version=6 (MỚI NHẤT, không phải 5 mà A thấy)
        → UPDATE ... WHERE version=6 → 1 row ✅ → A GHI ĐÈ lên B
        → mất update của B  ❌
```

Đây mới đúng là "lost update" kinh điển trong web app (user nghĩ lâu giữa lúc load form và submit).
Server stateless, **không nhớ** 5 phút trước trả cho A version mấy → chỉ client mới biết nó đã xem version nào.

**Muốn bảo vệ Loại 2 → bắt buộc client gửi version:**

```
Client gửi: { price: 90, version: 5 }   // "tôi sửa trên nền version 5"
Server:
   product = findById(id)               // DB hiện là version 6
   if (!product.getVersion().equals(request.getVersion()))  // 6 != 5 → throw → 409
```

| | Bảo vệ gì | Client gửi version? |
|---|---|---|
| **Impl hiện tại** (findById tự đọc) | Chỉ race chồng lấn tích tắc (Loại 1) | Không |
| **Full optimistic lock** (client gửi version) | Cả think-time / stale client (Loại 2) | **Có** |

**Nghịch lý:** quyết định trước "không include version vào Response DTO" nhất quán với việc chỉ bảo vệ
Loại 1. Muốn bảo vệ Loại 2 phải ĐẢO LẠI: thêm version vào cả Response + Request DTO + so sánh thủ công
(hoặc `merge()` detached entity). → Đã ghi backlog (Cụm 2.3b), quyết sau khi cân nhắc app có đáng lo Loại 2 không.

## 9.8 Tổng kết Cụm 2.3

✅ **Optimistic Locking:**
- `@Version Long version` ở `BaseEntity` → 5 entity có optimistic lock
- Handler `ObjectOptimisticLockingFailureException` → 409 Conflict
- 4 mapper ignore `version` ở method Request→Entity
- Backfill `version=0` cho data cũ

🎓 **Kiến thức đào sâu:**
- Lost update + vì sao row-level write lock khiến "cả 2 cùng thắng" là bất khả
- Optimistic vs Pessimistic — khái niệm chung ngành, không riêng Spring
- `@Version` = declarative; Hibernate tự `version+1` + `WHERE version=?` + đếm row affected
- `Long` vs `long` cho `@Version` (null phân biệt new vs existing)
- Spring wrap exception ORM → `ObjectOptimisticLockingFailureException`
- Dirty-checking gotcha: no-op update không sinh SQL → không check version
- Cách test race bằng `Thread.sleep` mở rộng cửa sổ transaction
- **Loại 1 (chồng lấn) vs Loại 2 (think-time)** — impl hiện tại chỉ bảo vệ Loại 1; Loại 2 cần client gửi version

⏳ **Còn lại trong session:** Cụm 2.3b (full optimistic lock cho think-time), Cụm 2.4 (TOCTOU), Cụm 5 (compensating transaction).

# Phần 10 — Cụm 2.4: TOCTOU Race Condition (2026-06-09)

## 10.1 TOCTOU là gì

**TOCTOU = Time Of Check To Time Of Use** — bug khi check một điều kiện rồi hành động dựa trên nó,
nhưng giữa 2 bước có kẽ hở để thứ khác chen vào làm điều kiện hết đúng.

`UserService.createUser`:

```java
boolean existUser = userRepository.existsByEmail(request.getEmail());  // ① CHECK
if (existUser) throw new DuplicateResource("Email already exists");
User createdUser = userRepository.save(user);                          // ② USE
```

Race — 2 request đăng ký cùng email đồng thời:

```
t1  A: existsByEmail → false
t2  B: existsByEmail → false      ← cả 2 thấy "chưa tồn tại"
t3  A: save → INSERT ✅
t4  B: save → INSERT → DB unique constraint VIOLATED → DataIntegrityViolationException 💥
```

Trước fix: exception này rơi vào `handleException(Exception.class)` → **500** (sai, phải là **409**).

## 10.2 Tiền đề: unique constraint đã có sẵn ở DB

`User.email`, `Category.slug`, `Product.slug` + `sku`, `ProductVariant.sku` + composite
`(product_id, size, color)` — tất cả `unique = true`. DB là trọng tài cuối cùng.

## 10.3 Bài học cốt lõi (liên hệ Cụm 2.3)

```
Cụm 2.3 — @Version:   bảo vệ race khi UPDATE row ĐÃ tồn tại (so version)
Cụm 2.4 — TOCTOU:     bảo vệ race khi INSERT row CHƯA tồn tại (unique constraint)
```

> Mọi check ở tầng application (`existsBy`, version đọc lên) đều có kẽ hở race. **Chỉ DB mới là trọng
> tài cuối cùng** — qua unique constraint (INSERT) hoặc version check (UPDATE). App-check chỉ là tối ưu
> cho trường hợp thường + thông báo đẹp, KHÔNG phải đảm bảo đúng đắn.

## 10.4 Q&A — verbatim

**Hỏi (vì sao không bỏ luôn `existsBy`?):** *"Bởi vì không có nhiều trường hợp xảy 2 user cùng tạo với
1 email với khoảng thời gian như bạn, nên việc triển khai existByEmail sẽ validate được hầu hết các case,
xử lý trả về lỗi cho user dễ hiểu hơn."*

→ Đúng. `existsBy` xử lý ~99.9% case (không race) với message đẹp, đúng field. Giữ lại cho common case.

**Hỏi (catch ở đâu?):** *"Catch DataIntegrityViolationException ở trong các service thì hợp lý nhất, vì
nó sẽ tường mình hơn cho một số case cụ thể. nếu xử lý trong GlobalExceptionHandler thì cũng được, nhưng
sẽ báo lỗi chung chung thôi"*

→ Lập luận có cơ sở, nhưng 3 thực tế kỹ thuật đẩy về GlobalExceptionHandler:
1. **Không tường minh hơn:** `DataIntegrityViolationException` không nói field nào trùng (Product có cả
   slug + sku). Muốn biết phải parse tên constraint — fragile, phụ thuộc DB. Cái "tường minh" đã giao `existsBy`.
2. **DRY:** lặp try-catch ở 4-5 method create vs viết 1 handler global.
3. **Timing:** INSERT có thể hoãn tới commit (sau khi method return) → service try-catch bắt hụt. (Các
   entity ở đây dùng `IDENTITY` → INSERT chạy ngay lúc `save()`, nên trường hợp này bắt được; nhưng với
   `SEQUENCE` + batch thì flush hoãn tới commit. GlobalExceptionHandler bắt được mọi lúc.)

→ Kết: friendly message ở service (`existsBy`), correctness guarantee ở global (`DataIntegrityViolation` → 409).
**Đặt mỗi việc ở tầng phù hợp nhất.**

## 10.5 Code

Handler trong `GlobalExceptionHandler`:

```java
@ExceptionHandler(DataIntegrityViolationException.class)
public ResponseEntity<BaseResponse<Void>> handleDataIntegrityViolationException(DataIntegrityViolationException e) {
    log.warn("Data integrity violation (possible duplicate/constraint): {}", e.getMostSpecificCause().getMessage());
    return ResponseEntity.status(HttpStatus.CONFLICT)
            .body(BaseResponse.error("A record with the same unique value already exists."));
}
```

3 điểm review quan trọng:
- **Message:** KHÔNG copy message của optimistic lock ("modified by someone else") — sai ngữ cảnh. Đây là
  trùng dữ liệu lúc tạo mới → "already exists".
- **`getMostSpecificCause()`:** lấy câu lỗi tận cùng từ DB driver (gọn), không dump cả stack trace của
  `DataIntegrityViolationException` (nhiều lớp wrapper Spring).
- **KHÔNG trả `e.getMessage()` cho client:** message gốc lộ tên bảng/constraint/SQL → rò rỉ cấu trúc DB.
  Dùng message cố định.

## 10.6 Deep-teach: `warn` vs `error` (log level)

```
ERROR → "Có cái gì HỎNG, cần người vào xem ngay" → bug, DB sập, external chết → thường gắn alert
WARN  → "Bất thường nhưng hệ thống TỰ XỬ LÝ được, ghi lại để theo dõi" → race đã trả 409, retry OK
```

TOCTOU race dự kiến được + đã xử lý gọn (409) → `warn`. Để `error` → log đầy "lỗi giả" → **alert fatigue**
(ops chai lì, bỏ qua cả alert thật). Phân loại log level đúng = giữ tỉ lệ tín hiệu/nhiễu lành mạnh.

Áp dụng luôn: sửa `InvalidFileException` từ `log.error` → `log.warn` (400 client-error, không phải bug server).
Lưu ý phân biệt: message `InvalidFileException` do MÌNH viết → an toàn trả client; còn message
`DataIntegrityViolationException` từ DB driver → KHÔNG trả.

## 10.7 Tổng kết Cụm 2.4

✅ **TOCTOU race:**
- Handler `DataIntegrityViolationException` → 409 (lưới an toàn cho race INSERT)
- `InvalidFileException`: `log.error` → `log.warn` (nhất quán log level)

🎓 **Kiến thức đào sâu:**
- TOCTOU = kẽ hở check→use; DB constraint là trọng tài cuối cùng
- Liên hệ Cụm 2.3: optimistic lock (UPDATE) vs unique constraint (INSERT)
- Đặt việc đúng tầng: friendly message (service) vs correctness (global)
- `warn` vs `error` — tư duy ops, alert fatigue
- Không leak message DB-driver cho client; phân biệt message tự-viết vs message-từ-driver

📌 **Backlog phát sinh:** `LocalStorageService` dòng 60 ném `InvalidFileException` cho lỗi I/O ghi file
thật (server error) — đang gộp chung handler → trả 400. Nên tách exception riêng (500) sau.

⏳ **Còn lại trong session:** Cụm 2.3b (full optimistic lock), Cụm 5 (compensating transaction).
