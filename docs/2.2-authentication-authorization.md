## 1. Password Hashing với BCrypt

### Tại sao không dùng MD5/SHA-256?

| Cách | An toàn? | Lý do |
|---|---|---|
| MD5/SHA-256 | ❌ | Quá nhanh → dễ brute-force, dễ dùng rainbow table |
| BCrypt | ✅ | **Chậm có chủ đích** + tự thêm **salt** mỗi lần hash |

### BCrypt hoạt động thế nào?

```
$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
 │  │  │                                    │
 │  │  └── Salt (22 ký tự)                  └── Hash (31 ký tự)
 │  └── Cost factor (10 = 2^10 = 1024 rounds)
 └── Algorithm version
```

- **Salt** tự động random mỗi lần → cùng password "123456" nhưng 2 user sẽ có hash khác nhau
- **Cost factor** = 10 nghĩa là chạy 2^10 vòng → hash chậm ~100ms. Brute-force 1 triệu password mất ~28 giờ thay vì vài giây

### So sánh với Node.js

Bên Node dùng `bcrypt` hoặc `bcryptjs` cũng y hệt. Spring Boot dùng `BCryptPasswordEncoder` — cùng algorithm, chỉ khác API.

### Cách dùng trong Spring

```java
// Khi tạo user → hash password
String passwordHash = passwordEncoder.encode(request.getPassword());

// Khi login → so sánh
passwordEncoder.matches("raw_password", user.getPassword()); // true/false
```

**Lưu ý:** `encode()` gọi 2 lần với cùng input → ra 2 kết quả **khác nhau** (do salt random). Nên phải dùng `matches()` để so sánh, **không được** dùng `==` hay `.equals()`.

## 2. Authentication vs Authorization

| | Authentication (Xác thực) | Authorization (Phân quyền) |
|---|---|---|
| Câu hỏi | **Bạn là ai?** | **Bạn được làm gì?** |
| Ví dụ | Đăng nhập bằng email + password | ADMIN mới được xóa user |
| Thời điểm | Xảy ra **trước** | Xảy ra **sau** authentication |
| Bên Node.js | `passport.authenticate()` | `if (req.user.role === 'admin')` |
| Bên Spring | `AuthenticationManager` | `@PreAuthorize("hasRole('ADMIN')")` |

## 3. JWT (JSON Web Token)

### So sánh với Node.js

Bên Node dùng `jsonwebtoken` package (`jwt.sign()`, `jwt.verify()`). Concept hoàn toàn giống, chỉ khác thư viện. Bên Java dùng `jjwt` (io.jsonwebtoken).

### Cấu trúc JWT — 3 phần ngăn cách bởi dấu `.`

```
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyQGVtYWlsLmNvbSJ9.abc123signature
│                     │                                    │
└── Header            └── Payload                          └── Signature
```

- **Header:** algorithm + token type
- **Payload:** chứa thông tin user (email, role, thời hạn...) — **không mã hóa**, chỉ encode Base64 → **KHÔNG đặt thông tin nhạy cảm** ở đây
- **Signature:** `HMAC-SHA256(header + payload, SECRET_KEY)` — để verify token không bị sửa đổi

### Flow đăng nhập với JWT

```
1. Client gửi POST /api/auth/login { email, password }
2. Server verify password → đúng → tạo JWT → trả về client
3. Client lưu JWT (localStorage/cookie)
4. Mỗi request sau đó, client gửi header: Authorization: Bearer <token>
5. Server verify JWT → extract user info → xử lý request
```

### Access Token vs Refresh Token

| | Access Token | Refresh Token |
|---|---|---|
| Thời hạn | Ngắn (15-30 phút) | Dài (7-30 ngày) |
| Mục đích | Truy cập API | Lấy access token mới khi hết hạn |
| Nơi gửi | Header mỗi request | Chỉ gửi khi cần refresh |

**Tại sao cần 2 token?** Nếu access token bị lộ, hacker chỉ dùng được 15 phút. Refresh token lưu an toàn hơn và ít bị expose.

**Chiến lược refresh:**
- **Chỉ tạo AT mới**, giữ nguyên RT cũ (đơn giản — dùng cách này trước)
- **Tạo cả cặp mới** + vô hiệu RT cũ (an toàn hơn — gọi là **token rotation**)

## 4. Spring Security Filter Chain

### Khác biệt lớn nhất so với Node.js

Bên Express, viết middleware đơn giản:

```javascript
app.use('/api/admin', authMiddleware, adminMiddleware);
```

Bên Spring Security, mọi request đều đi qua một **chuỗi filter** (filter chain):

```
Request HTTP
    │
    ▼
┌─────────────────────────────┐
│   Spring Security Filters   │
│                             │
│  1. CorsFilter              │  ← Xử lý CORS
│  2. CsrfFilter              │  ← Xử lý CSRF (tắt cho REST API)
│  3. JwtAuthenticationFilter  │  ← ★ Filter custom của mình
│     - Lấy token từ header    │
│     - Verify token           │
│     - Set user vào context   │
│  4. AuthorizationFilter      │  ← Kiểm tra quyền truy cập
│                             │
└─────────────────────────────┘
    │
    ▼
  Controller (nếu pass hết filters)
```

**Ý tưởng chính:** Viết `JwtAuthenticationFilter` **chen vào trước** `UsernamePasswordAuthenticationFilter` trong filter chain:
1. Đọc header `Authorization: Bearer <token>`
2. Verify token bằng secret key
3. Nếu hợp lệ → set thông tin user vào `SecurityContext` (giống `req.user` bên Express)
4. Nếu không hợp lệ → bỏ qua, để các filter sau xử lý (trả 401)

## 5. Câu hỏi kiểm tra & trả lời

**Q: Tại sao dùng BCrypt mà không dùng SHA-256 để hash password?**

A: Vì BCrypt an toàn hơn — mỗi lần hash ra kết quả khác nhau (do salt random), và **chậm có chủ đích** (cost factor 2^10 rounds) nên chống brute-force tốt hơn. SHA-256 hash cực nhanh → hacker thử hàng tỷ password/giây.

**Q: Nếu access token hết hạn, client phải làm gì?**

A: Gọi API refresh token để lấy token mới. Server kiểm tra refresh token còn hạn không, nếu còn thì tạo access token mới trả về.

**Q: JwtAuthenticationFilter sẽ được đặt ở đâu trong flow xử lý request?**

A: Đặt **trước** `UsernamePasswordAuthenticationFilter` trong Security filter chain. Request đi qua JWT filter trước khi Spring Security xử lý authentication mặc định.

**Q: Dòng `Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8))` dùng để làm gì?**

A: Chuyển chuỗi secret (string) thành object `SecretKey` để dùng cho việc ký và verify JWT. Tách ra 3 bước: (1) `secret` là string từ `application.yaml`, (2) `.getBytes(UTF_8)` chuyển thành `byte[]` vì thuật toán mã hoá làm việc với bytes, chỉ định UTF_8 để kết quả giống nhau trên mọi máy, (3) `Keys.hmacShaKeyFor()` tạo `SecretKey` object + validate key đủ dài cho HS256. Bên Node.js chỉ cần truyền string thẳng vào `jwt.sign()`, còn Java yêu cầu kiểu dữ liệu chặt chẽ hơn.

**Q: So sánh build token giữa Node.js và Java? Tại sao cần field `subject` để xác định chủ token?**

A: Bên Node quăng hết vào object payload (`{ email, role }`), bên Java tách ra: field chuẩn JWT có method riêng (`.subject()`, `.issuedAt()`, `.expiration()`), field tuỳ chỉnh dùng `.claim()`. Kết quả ra cùng 1 JWT.

`subject` (`sub`) là field chuẩn theo đặc tả JWT (RFC 7519), trả lời câu hỏi "token này đại diện cho ai?". Bên Node cũng có thể dùng `sub` nhưng không bắt buộc. Lợi ích dùng `sub` theo chuẩn: (1) mọi hệ thống/ngôn ngữ đều hiểu, (2) thư viện hỗ trợ sẵn `getSubject()`, (3) không cần đọc doc để biết field nào chứa thông tin chủ token. Các field chuẩn khác: `iat` (tạo lúc nào), `exp` (hết hạn), `iss` (ai tạo), `aud` (token dành cho ai).

## OncePerRequestFilter

### Vấn đề nó giải quyết

Trong Spring, một request có thể **đi qua cùng một filter nhiều lần** — ví dụ khi có forward hoặc redirect nội bộ. Nếu dùng filter thường (`Filter` interface), JWT verify sẽ chạy 2-3 lần cho cùng 1 request → lãng phí.

`OncePerRequestFilter` đảm bảo **chỉ chạy đúng 1 lần mỗi request**, bất kể có bao nhiêu lần forward/redirect.

### So sánh với Express

```javascript
// Express — middleware tự nhiên chỉ chạy 1 lần
app.use((req, res, next) => {
  // verify JWT
  next();
});
```

Bên Express bạn không cần lo chuyện này vì middleware chỉ chạy 1 lần. Nhưng Spring phức tạp hơn (filter chain, forward, dispatch types...) nên cần class này để xử lý hộ.

### `doFilter` vs `doFilterInternal`

**`doFilter()`** — method gốc của `Filter` interface:

```java
// Filter interface — bạn phải TỰ xử lý mọi thứ
public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
    // Tự kiểm tra đã chạy chưa?
    // Tự cast ServletRequest → HttpServletRequest
    // Logic xử lý
    chain.doFilter(request, response);
}
```

**`doFilterInternal()`** — method của `OncePerRequestFilter`, đã xử lý sẵn 2 việc cho bạn:

```java
// OncePerRequestFilter — đã lo hộ
protected void doFilterInternal(HttpServletRequest request,   // ← đã cast sẵn
                                HttpServletResponse response,  // ← đã cast sẵn
                                FilterChain chain) {
    // Chỉ cần viết logic — không lo chạy trùng
}
```

### Bên trong `OncePerRequestFilter` làm gì?

Nó override `doFilter()` và làm thế này (đơn giản hoá):

```java
// Pseudocode bên trong OncePerRequestFilter
public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) {
    String flag = "ALREADY_FILTERED_" + filterName;

    if (request.getAttribute(flag) != null) {
        // Đã chạy rồi → skip, đi tiếp
        chain.doFilter(req, res);
        return;
    }

    request.setAttribute(flag, true);  // Đánh dấu đã chạy
    doFilterInternal((HttpServletRequest) req, (HttpServletResponse) res, chain);
}
```

Dùng **attribute trên request** như một cái flag — lần đầu set flag rồi chạy logic, lần sau thấy flag thì skip.

### Tóm lại

| | `Filter.doFilter()` | `OncePerRequestFilter.doFilterInternal()` |
|---|---|---|
| Chạy mấy lần? | Có thể nhiều lần | **Đúng 1 lần** |
| Cast request? | Tự cast | Đã cast sẵn `HttpServletRequest` |
| Khi nào dùng? | Hiếm khi | **Hầu hết các trường hợp** trong Spring Security |

Nói ngắn gọn: `doFilterInternal` = `doFilter` phiên bản **an toàn hơn + tiện hơn**. Trong Spring Security gần như luôn dùng `OncePerRequestFilter`.

## Bí mật: Không ai nhớ hết

Ngay cả senior Java cũng **không thuộc lòng** tất cả class của Spring Security. Cái họ có là **mental model** — hiểu bức tranh tổng thể, rồi khi cần class cụ thể thì tra.

## Cách tiếp cận đúng

### Bước 1: Hiểu bức tranh tổng thể trước

Spring Security chỉ xoay quanh **3 câu hỏi**:

```
Request đến → (1) Bạn là ai? → (2) Bạn được làm gì? → (3) Cho phép/từ chối
              Authentication     Authorization            Access Decision
```

Mọi class trong Spring Security đều phục vụ 1 trong 3 câu hỏi này. Khi thấy class lạ, tự hỏi: **"Class này trả lời câu hỏi nào trong 3 câu trên?"**

### Bước 2: Map sang Express (cái bạn đã biết)

```javascript
// Express — tất cả gói gọn trong 1 middleware
app.use((req, res, next) => {
    const token = req.headers.authorization;       // Lấy token
    const decoded = jwt.verify(token, SECRET);     // Verify
    req.user = decoded;                            // ← Gắn user vào request
    next();
});

// Sau đó check quyền
if (req.user.role === 'admin') { ... }
```

Spring Security làm **y hệt**, chỉ là tách ra nhiều class:

```
Express                          Spring Security
─────────────────────────────────────────────────────
req.headers.authorization   →    JwtAuthenticationFilter (lấy token)
jwt.verify(token)           →    JwtTokenProvider (verify token)
req.user = decoded          →    UsernamePasswordAuthenticationToken ← ★
                                 + SecurityContextHolder
if (req.user.role)          →    @PreAuthorize("hasRole('ADMIN')")
```

### Bước 3: `UsernamePasswordAuthenticationToken` là gì?

Cái tên dài và khó hiểu, nhưng bản chất nó chỉ là **một cái túi chứa thông tin user** — giống hệt `req.user` bên Express:

```javascript
// Express
req.user = { email: "test@test.com", role: "CUSTOMER" };
```

```java
// Spring — cùng ý nghĩa, chỉ là phải gói vào object
new UsernamePasswordAuthenticationToken(
    "test@test.com",           // principal (ai?) — giống req.user.email
    null,                      // credentials (password) — null vì đã verify xong
    List.of(new SimpleGrantedAuthority("ROLE_CUSTOMER"))  // quyền — giống req.user.role
);
```

**Tại sao tên là "UsernamePassword" mà mình dùng cho JWT?** Vì Spring Security thiết kế class này ban đầu cho đăng nhập bằng username/password. Nhưng nó đủ generic để **tái sử dụng cho JWT** — chỉ cần truyền email vào principal, password để null. Đây là convention của Spring Security, ai cũng dùng vậy.

### Bước 4: Cách tra cứu khi gặp class lạ

Khi gặp class Spring Security mà không hiểu, đây là thứ tự tra:

1. **Hỏi: "Nó là `req.` gì bên Express?"** — map về cái bạn biết trước
2. **Đọc JavaDoc dòng đầu tiên** — Spring doc dòng đầu luôn là tóm tắt
3. **Xem nó implement interface nào** — interface cho biết vai trò của nó
4. **Không cần hiểu sâu ngay** — dùng được trước, hiểu sau

Ví dụ thực tế:

```
Gặp: SecurityContextHolder
→ Hỏi: "Bên Express, thông tin user lưu ở đâu?" → req.user
→ Vậy SecurityContextHolder = nơi lưu req.user, nhưng ở cấp thread
→ Done. Đủ dùng.
```

**Người học: Spring Security là một chủ đề riêng của Spring hả? Khi dùng `SecurityContextHolder` có nghĩa là khi chạy dự án Spring Boot, chúng ta sẽ có một kho chứa security chung dùng cho tất cả các request đúng không? Kho chứa giữa các instance của ứng dụng sẽ khác nhau và sẽ mất khi ứng dụng dừng đúng không? Thêm nữa là chúng ta có thể set bất cứ thông tin đối tượng nào vào trong authentication này được không, vì nhỡ đâu ngoài email và role chúng ta còn những thông tin khác?**

### 1. Spring Security là chủ đề riêng?

Đúng. Spring Security là một **module riêng** trong hệ sinh thái Spring, giống như:

```
Spring Framework (core)
├── Spring Boot        — auto-config, chạy nhanh
├── Spring Data JPA    — làm việc với database
├── Spring Security    — ★ authentication & authorization
├── Spring Cloud       — microservices
└── ...
```

Mỗi module là một dependency riêng, bạn thêm vào khi cần. Đó là lý do mình phải thêm `spring-boot-starter-security` vào `pom.xml`.

### 2. SecurityContextHolder — "kho chứa" chung?

Đây là phần bạn hiểu **gần đúng** nhưng cần chỉnh lại một chút quan trọng.

**Không phải 1 kho chung cho tất cả request.** Mà là **mỗi request (mỗi thread) có ngăn riêng:**

```
Request A (Thread-1) → SecurityContext { user: "an@test.com", role: CUSTOMER }
Request B (Thread-2) → SecurityContext { user: "binh@test.com", role: ADMIN }
Request C (Thread-3) → SecurityContext { chưa set → anonymous }
```

Nó dùng `ThreadLocal` — một cơ chế Java cho phép mỗi thread giữ **bản sao riêng** của dữ liệu. Bạn đã học phần multi-threading ở bài 0.4, nhớ không?

```
SecurityContextHolder
├── Thread-1 → SecurityContext → Authentication { an@test.com }
├── Thread-2 → SecurityContext → Authentication { binh@test.com }
└── Thread-3 → SecurityContext → null (chưa đăng nhập)
```

So sánh bên Express:

```javascript
// Express — mỗi request có req riêng, không lẫn nhau
app.use((req, res, next) => {
    req.user = decoded;  // chỉ request này thấy
});
```

Spring Security cũng vậy, chỉ là thay vì gắn vào `req` thì gắn vào **ThreadLocal** của thread đang xử lý request đó.

**Và đúng — khi ứng dụng dừng thì mất hết**, vì nó chỉ nằm trong RAM. Giữa các instance cũng **hoàn toàn độc lập**. Đây không phải nơi lưu trữ lâu dài — nó chỉ sống trong **vòng đời 1 request**.

### 3. Có thể set thêm thông tin khác ngoài email, role không?

**Được!** Nhìn lại constructor:

```java
new UsernamePasswordAuthenticationToken(
    principal,       // ← Object — có thể là bất cứ gì
    credentials,     // ← Object
    authorities      // ← Collection<GrantedAuthority>
);
```

`principal` có kiểu **Object**, nên bạn có thể truyền cả một object phức tạp:

```java
// Cách 1: Chỉ truyền email (đang dùng — đủ cho project mình)
new UsernamePasswordAuthenticationToken("an@test.com", null, authorities);

// Cách 2: Truyền cả User entity
User user = userRepository.findByEmail(email);
new UsernamePasswordAuthenticationToken(user, null, authorities);

// Cách 3: Tạo custom object chứa mọi thứ bạn muốn
public class UserPrincipal {
    private Long id;
    private String email;
    private String fullName;
    private String role;
    private Long shopId;     // ← thông tin thêm
}
new UsernamePasswordAuthenticationToken(userPrincipal, null, authorities);
```

Rồi ở Controller lấy ra:

```java
// Cách 1: Lấy email
String email = SecurityContextHolder.getContext()
    .getAuthentication().getName();

// Cách 2/3: Lấy full object
UserPrincipal me = (UserPrincipal) SecurityContextHolder.getContext()
    .getAuthentication().getPrincipal();
me.getShopId();  // dùng bình thường
```

**Hiện tại project mình chỉ cần email + role nên cách 1 là đủ.** Sau này nếu cần thêm thông tin (ví dụ seller cần `shopId`), bạn refactor sang cách 3.

**Người học: Hiện tại mỗi request đến Spring Boot sẽ tạo ra một thread đúng không, sau đó thread này sẽ chứa thông tin cho request đó bao gồm cả security, sau khi trả về response thì thông tin trong thread này bao gồm cả security sẽ được giải phóng?**

Chính xác! Tóm lại flow:

```
Request đến → Tomcat tạo/lấy Thread từ pool
    → Thread giữ: SecurityContext, request data, ...
    → Xử lý xong → Trả response
    → SecurityContext bị xoá (clear)
    → Thread trả về pool, sẵn sàng cho request khác
```

Chỉ bổ sung một chi tiết nhỏ: Tomcat dùng **thread pool** — nó không tạo thread mới mỗi lần mà **tái sử dụng**. Đó là lý do Spring phải **clear SecurityContext sau mỗi request** — nếu không, thread được tái sử dụng cho request sau có thể còn dính thông tin user cũ.

Bên Express thì không cần lo vì mỗi `req` object là mới hoàn toàn, còn Spring phải dọn dẹp `ThreadLocal` vì thread được tái sử dụng.

## Giải thích chi tiết SecurityConfig

### 1. Các Annotation trên class

**`@Configuration`** — Bạn đã học cái này ở bài 0.1 — đánh dấu class này là **nơi khai báo Bean**. Spring sẽ scan class này khi khởi động và chạy các method có `@Bean` bên trong. So sánh Express: giống file `config.js` mà bạn export các setting ra.

**`@EnableWebSecurity`** — **Bật Spring Security cho toàn bộ ứng dụng.** Nếu không có annotation này, dù bạn thêm dependency `spring-boot-starter-security` vào `pom.xml`, các cấu hình security trong class này sẽ **không có tác dụng**. Nó làm 2 việc: đăng ký filter chain mặc định của Spring Security vào ứng dụng, và cho phép bạn **tuỳ chỉnh** lại cấu hình security thông qua `HttpSecurity`.

**`@EnableMethodSecurity`** — Bật tính năng **phân quyền trên từng method** — cho phép dùng `@PreAuthorize` trực tiếp trên Controller method:

```java
// Không có @EnableMethodSecurity → @PreAuthorize bị bỏ qua hoàn toàn, ai cũng gọi được
// Có @EnableMethodSecurity → Spring kiểm tra quyền trước khi chạy method

@PreAuthorize("hasRole('ADMIN')")
@DeleteMapping("/{id}")
public void deleteUser(@PathVariable Long id) { ... }
```

So sánh Express:
```javascript
// Express — check quyền bằng middleware trên từng route
router.delete('/:id', requireRole('admin'), deleteUser);

// Spring — check quyền bằng annotation trên method
// @PreAuthorize("hasRole('ADMIN')") ← tương đương requireRole('admin')
```

Tại sao tách thành 2 annotation (`@EnableWebSecurity` + `@EnableMethodSecurity`) mà không gộp? Vì không phải dự án nào cũng cần phân quyền trên method — có dự án chỉ cần phân quyền theo URL là đủ. Spring cho bạn **chọn bật/tắt riêng**.

### 2. Constructor Injection

```java
private final JwtAuthenticationFilter jwtAuthenticationFilter;

public SecurityConfig(JwtAuthenticationFilter jwtAuthenticationFilter) {
    this.jwtAuthenticationFilter = jwtAuthenticationFilter;
}
```

Đây là **Dependency Injection** qua constructor — bạn đã học ở bài 0.1. `JwtAuthenticationFilter` đã đánh `@Component` nên Spring tự động inject vào đây. Mình cần nó để đăng ký vào filter chain ở bước 4 bên dưới.

### 3. `securityFilterChain` — method quan trọng nhất

```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
```

Đây là nơi bạn **cấu hình toàn bộ security** cho ứng dụng. `HttpSecurity` là một **builder** — bạn gọi các method liên tiếp (chaining) để cấu hình từng phần.

So sánh Express: giống việc bạn setup tất cả middleware trong `app.js`:

```javascript
// Express — setup rải rác
app.use(cors());
app.use(helmet());
app.use('/api/auth', authRouter);       // public
app.use('/api', authMiddleware);         // protected

// Spring — setup TẬP TRUNG trong 1 method
http.csrf(...).sessionManagement(...).authorizeHttpRequests(...).addFilterBefore(...);
```

**Bước 1: Tắt CSRF**

```java
.csrf(csrf -> csrf.disable())
```

**CSRF (Cross-Site Request Forgery)** — kiểu tấn công mà hacker lừa user click link để gửi request giả mạo. CSRF protection cần cho **web app dùng cookie/session** (vì browser tự động gửi cookie). REST API dùng JWT thì **không cần** — vì token nằm trong header `Authorization`, browser không tự động gửi, hacker không thể lợi dụng.

**Bước 2: Stateless Session**

```java
.sessionManagement(session ->
    session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)
)
```

Nói với Spring: **đừng tạo session/cookie**. Mặc định Spring Security sẽ tạo `JSESSIONID` cookie (giống `express-session`) — nhưng mình dùng JWT nên không cần. `STATELESS` nghĩa là: mỗi request phải **tự mang theo token**, server không nhớ gì giữa các request. Đây đúng với triết lý REST.

**Bước 3: Phân quyền theo URL**

```java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/**").permitAll()     // public
    .requestMatchers("/api/users").permitAll()        // public (register)
    .anyRequest().authenticated()                     // còn lại cần token
)
```

Giống hệt bên Express khi bạn đặt middleware auth trên một số route:

```javascript
// Express
router.post('/api/auth/login', loginHandler);           // không cần auth
router.post('/api/auth/refresh', refreshHandler);       // không cần auth
router.post('/api/users', registerHandler);             // không cần auth
router.use('/api', authMiddleware);                     // còn lại cần auth
```

- `"/api/auth/**"` — dấu `**` nghĩa là match mọi path con: `/api/auth/login`, `/api/auth/refresh`, `/api/auth/xxx`...
- `permitAll()` — ai cũng truy cập được, không cần token
- `anyRequest().authenticated()` — **tất cả request còn lại** phải có token hợp lệ

**Thứ tự quan trọng!** Spring kiểm tra từ trên xuống, rule đầu tiên match sẽ được áp dụng. Nếu bạn đặt `anyRequest().authenticated()` lên trước thì `/api/auth/**` cũng bị chặn.

**Bước 4: Thêm JWT Filter**

```java
.addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)
```

Nói với Spring: **chèn `jwtAuthenticationFilter` của mình VÀO TRƯỚC `UsernamePasswordAuthenticationFilter`** trong chuỗi filter.

`UsernamePasswordAuthenticationFilter` là filter mặc định của Spring — xử lý đăng nhập bằng form (username + password). Mình không dùng form login, mình dùng JWT, nên cần filter của mình chạy trước để:
1. Đọc token từ header
2. Verify → set `SecurityContext`
3. Khi đến `UsernamePasswordAuthenticationFilter`, đã có authentication rồi → skip

```
Request → ... → [JwtAuthenticationFilter] → [UsernamePasswordAuthenticationFilter] → ... → Controller
                 ★ của mình (chèn trước)      mặc định của Spring (bị skip vì đã auth)
```

**Cuối cùng: `http.build()`** — Build ra `SecurityFilterChain` object từ tất cả cấu hình trên. `@Bean` sẽ đăng ký object này vào Spring container → Spring Security dùng nó để xử lý mọi request.

### 4. `authenticationManager` Bean

```java
@Bean
public AuthenticationManager authenticationManager(AuthenticationConfiguration config) throws Exception {
    return config.getAuthenticationManager();
}
```

`AuthenticationManager` là interface chính của Spring Security dùng để **xác thực user** (kiểm tra email + password đúng không). Tại sao cần khai báo Bean này? Vì mặc định Spring Security **không expose** `AuthenticationManager` ra ngoài. Nhưng mình cần dùng nó trong `AuthService` khi login — nên phải khai báo `@Bean` để có thể inject được.

Thực ra trong project mình, `AuthService` sẽ tự verify password bằng `passwordEncoder.matches()` nên có thể **không cần** Bean này. Nhưng để sẵn vì một số cách triển khai login dùng `authenticationManager.authenticate()`.

**Người học: 1. `@EnableWebSecurity` có nghĩa là ban đầu Spring Web không có filter chain, sau khi thêm annotation này thì sẽ có một filter chain bọc bên ngoài trước tất cả các request của Spring hả?**

**2. Cách hoạt động của file này đó là khi Spring hoạt động, IoC sẽ scan project, thấy file này được đánh dấu là `@Configuration` → sẽ tạo instance cho class này, sau đó mỗi khi có request đến sẽ inject instance này vào cho request sau đó gọi method có chứa Bean đúng không?**

**3. Nếu như có logic nào khác phát sinh thì liệu tôi có thể vào chain để kiểm tra được không, ví dụ như request header phải có trường `X-Header-Require: string` thì sẽ triển khai như nào?**

### Trả lời câu 1: `@EnableWebSecurity` — ban đầu không có filter chain?

**Gần đúng, nhưng cần chỉnh lại:**

Thực ra khi bạn thêm dependency `spring-boot-starter-security` vào `pom.xml`, Spring Boot **tự động tạo sẵn** một filter chain mặc định (nhờ auto-configuration). Filter chain mặc định này sẽ:
- Bật CSRF
- Tạo session
- Chặn **tất cả** request → yêu cầu đăng nhập bằng form
- Tạo user mặc định (username: `user`, password: in ra console)

`@EnableWebSecurity` không phải "tạo" filter chain, mà là **cho phép bạn tuỳ chỉnh lại** filter chain đó theo ý mình. Nếu không có annotation này, Spring dùng cấu hình mặc định — chặn hết, bắt login bằng form.

```
Không có @EnableWebSecurity → Filter chain MẶC ĐỊNH (chặn hết, form login)
Có @EnableWebSecurity       → Filter chain TUỲ CHỈNH (bạn quyết định mọi thứ)
```

So sánh Express:
```javascript
// Giống như Express mặc định không có middleware nào
// Khi bạn app.use(cors()), app.use(helmet()) → bạn đang "tuỳ chỉnh"
// @EnableWebSecurity = cho phép bạn tuỳ chỉnh pipeline của Spring Security
```

### Trả lời câu 2: Cách hoạt động — IoC scan rồi inject cho mỗi request?

**Phần đầu đúng, phần sau cần chỉnh:**

✅ Đúng: Spring khởi động → IoC scan → thấy `@Configuration` → tạo instance `SecurityConfig` → chạy các method `@Bean` → tạo `SecurityFilterChain` object + `AuthenticationManager` object → đăng ký vào container.

❌ Sai ở chỗ: **Không phải mỗi request đến mới inject và gọi `@Bean`**. Các method `@Bean` chỉ chạy **1 lần duy nhất khi khởi động**. `SecurityFilterChain` được tạo ra 1 lần và **tái sử dụng cho mọi request**.

```
Khởi động app (1 lần):
  IoC scan → @Configuration → gọi securityFilterChain() → tạo SecurityFilterChain object
                                                           ↓
                                                     Lưu trong container

Request 1 → đi qua SecurityFilterChain (object đã tạo sẵn)
Request 2 → đi qua SecurityFilterChain (cùng object đó)
Request 3 → đi qua SecurityFilterChain (cùng object đó)
...
```

So sánh Express:
```javascript
// Express — setup 1 lần khi start server
app.use(cors());           // chạy 1 lần khi khởi động
app.use(authMiddleware);   // chạy 1 lần khi khởi động
app.listen(3000);

// Sau đó mỗi request đi qua middleware đã đăng ký
// KHÔNG phải mỗi request lại app.use() lại

// Spring cũng y hệt — @Bean chạy 1 lần, tạo filter chain 1 lần
```

### Trả lời câu 3: Thêm logic custom vào filter chain — ví dụ kiểm tra `X-Header-Require`?

**Được!** Đây chính là sức mạnh của filter chain — bạn muốn thêm bao nhiêu filter cũng được. Cách triển khai:

**Bước 1: Tạo filter mới**

```java
@Component
public class RequiredHeaderFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) throws ServletException, IOException {

        String headerValue = request.getHeader("X-Header-Require");

        if (headerValue == null || headerValue.isBlank()) {
            // Thiếu header → trả 400 luôn, không đi tiếp
            response.setStatus(HttpServletResponse.SC_BAD_REQUEST);
            response.getWriter().write("Missing X-Header-Require header");
            return;  // KHÔNG gọi filterChain.doFilter() → request dừng ở đây
        }

        // Có header → cho đi tiếp
        filterChain.doFilter(request, response);
    }
}
```

**Bước 2: Đăng ký vào SecurityConfig**

```java
http
    .csrf(csrf -> csrf.disable())
    .sessionManagement(...)
    .authorizeHttpRequests(...)
    // Thêm filter mới — chạy TRƯỚC JWT filter
    .addFilterBefore(requiredHeaderFilter, JwtAuthenticationFilter.class)
    // JWT filter vẫn ở đây
    .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class);
```

**Flow lúc này:**

```
Request
  │
  ▼
RequiredHeaderFilter          ← ★ mới thêm
  │ có header? 
  │   Không → 400 Bad Request (dừng)
  │   Có → tiếp tục ↓
  ▼
JwtAuthenticationFilter       ← verify token
  │
  ▼
UsernamePasswordAuthenticationFilter
  │
  ▼
Controller
```

Ngoài ra bạn có thể đặt filter ở các vị trí khác:

```java
// Trước 1 filter cụ thể
.addFilterBefore(myFilter, SomeFilter.class)

// Sau 1 filter cụ thể
.addFilterAfter(myFilter, SomeFilter.class)

// Thay thế 1 filter (ít dùng)
.addFilterAt(myFilter, SomeFilter.class)
```

So sánh Express — cực kỳ giống:
```javascript
// Express — thứ tự middleware cũng quan trọng
app.use(requiredHeaderMiddleware);  // chạy trước
app.use(authMiddleware);            // chạy sau
```

**Khác biệt duy nhất:** Express đăng ký middleware bằng `app.use()` theo thứ tự code. Spring đăng ký bằng `addFilterBefore/After` — chỉ rõ vị trí tương đối với filter nào.

---

**Người học: `@RestControllerAdvice` cũng giống như filter chain, thay vì filter sẽ là ngõ đầu tiên mà các request phải đi qua để đến các resource thì `@RestControllerAdvice` là ngõ cuối cùng response đi qua để đến với user à?**

Câu hỏi rất hay, bạn đang cố map mental model — nhưng cần chỉnh lại chút.

## `@RestControllerAdvice` không nằm trong "đường đi" của request/response

Nó **không phải ngõ cuối** mà response đi qua. Nó hoạt động theo cơ chế hoàn toàn khác — **bắt exception**.

```
Filter Chain → Controller → Service → Repository
                  │
                  │ throw exception
                  ▼
          @RestControllerAdvice (bắt exception, tạo response lỗi)
                  │
                  ▼
              Response trả về client
```

Nếu **không có exception**, `@RestControllerAdvice` **hoàn toàn không chạy**. Nó không tham gia vào flow bình thường.

So sánh Express:

```javascript
// Middleware bình thường — LUÔN chạy (giống Filter)
app.use((req, res, next) => { next(); });

// Error middleware — CHỈ chạy khi có lỗi (giống @RestControllerAdvice)
app.use((err, req, res, next) => {
    res.status(500).json({ error: err.message });
});
```

Bạn biết bên Express, error middleware có **4 tham số** `(err, req, res, next)` — Express chỉ gọi nó khi có `next(err)` hoặc throw. `@RestControllerAdvice` cũng y hệt — Spring chỉ gọi nó khi có exception được throw ra.

## Flow đầy đủ

```
Request thành công (không lỗi):
  Filter Chain → Controller → Service → Response
                                         (RestControllerAdvice KHÔNG chạy)

Request có lỗi:
  Filter Chain → Controller → Service → throw Exception!
                                              │
                              RestControllerAdvice bắt lấy
                                              │
                                        Error Response
```

## Vậy "ngõ cuối" thực sự là gì?

Nếu bạn muốn logic chạy với **mọi response** (cả thành công lẫn lỗi), đó là **Filter** — vì filter chạy cả **trước và sau** Controller:

```java
public void doFilterInternal(...) {
    // Code ở đây chạy TRƯỚC controller (request đi vào)
    
    filterChain.doFilter(request, response);
    
    // Code ở đây chạy SAU controller (response đi ra) ← "ngõ cuối"
}
```

Giống Express:
```javascript
app.use((req, res, next) => {
    // Trước controller
    next();
    // Sau controller ← nhưng Express ít dùng pattern này
});
```

### Tóm lại

| | Filter Chain | @RestControllerAdvice |
|---|---|---|
| Khi nào chạy? | **Mọi request** | **Chỉ khi có exception** |
| Vai trò | Cổng vào/ra cho request/response | Bắt lỗi tập trung |
| Express tương đương | `app.use(middleware)` | `app.use((err, req, res, next) => {...})` |

---

**Người học: Nếu như mà response cần trả về một trường ví dụ như là thời gian server hiện tại trả về trong mỗi response thì sẽ xử lý trong filter luôn đúng không? Bạn có thể ví dụ cho tôi không?**

Đúng rồi! Vì cần chạy với **mọi response** nên dùng Filter. Ví dụ:

```java
@Component
public class ResponseTimestampFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) throws ServletException, IOException {

        // Thêm header vào response TRƯỚC khi đi tiếp
        response.setHeader("X-Server-Time", LocalDateTime.now().toString());

        // Cho request đi tiếp vào Controller
        filterChain.doFilter(request, response);
    }
}
```

Kết quả mọi response sẽ có:
```
HTTP/1.1 200 OK
X-Server-Time: 2026-04-12T15:30:45.123
Content-Type: application/json
...
```

Nhưng lưu ý — cách trên thêm vào **response header**. Nếu bạn muốn thêm vào **response body** (JSON) thì phức tạp hơn nhiều vì body đã được serialize trong Controller rồi. Lúc đó dùng Filter không hợp lý — thay vào đó nên đặt trường đó trong `BaseResponse`:

```java
public class BaseResponse<T> {
    private boolean success;
    private String message;
    private T data;
    private String serverTime = LocalDateTime.now().toString();  // ← tự động có trong mọi response
}
```

Cách này đơn giản hơn nhiều và đúng chỗ hơn — data thuộc về body thì để trong body, metadata thuộc về header thì để trong header.

---

**Người học: Java có cách nào tạo giá trị mặc định cho tham số giống Node không? Ví dụ khi throw exception bắt điền message, muốn có message mặc định, chỉ custom khi thực sự cần.**

Java **không hỗ trợ default parameter** như Node.js:

```javascript
// Node.js — được
function throwError(message = "Something went wrong") {
    throw new Error(message);
}
throwError();        // "Something went wrong"
throwError("Custom"); // "Custom"
```

```java
// Java — KHÔNG được viết thế này
public void throwError(String message = "Something went wrong") { } // ❌ Lỗi compile
```

### Cách giải quyết: Constructor Overloading

Tạo **nhiều constructor** — một cái không nhận tham số (dùng message mặc định), một cái nhận tham số (custom message):

```java
public class InvalidUserOrPassword extends RuntimeException {

    // Constructor không tham số — dùng message mặc định
    public InvalidUserOrPassword() {
        super("Username or password is incorrect");
    }

    // Constructor có tham số — custom message khi cần
    public InvalidUserOrPassword(String message) {
        super(message);
    }
}
```

Khi dùng:
```java
throw new InvalidUserOrPassword();                    // → "Username or password is incorrect"
throw new InvalidUserOrPassword("Email không tồn tại"); // → "Email không tồn tại"
```

### Dependencies (pom.xml)

```xml
<!-- Spring Security (thay thế spring-security-crypto cũ) -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>

<!-- JWT library (io.jsonwebtoken) — 3 module -->
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-api</artifactId>
    <version>0.12.6</version>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-impl</artifactId>
    <version>0.12.6</version>
    <scope>runtime</scope>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-jackson</artifactId>
    <version>0.12.6</version>
    <scope>runtime</scope>
</dependency>
```

**Giải thích:** `jjwt` tách thành 3 module theo pattern modular (giống SLF4J tách API và implementation). `jjwt-api` là API chính, `jjwt-impl` + `jjwt-jackson` là runtime implementation.

### Cấu hình JWT (application.yaml)

```yaml
jwt:
  secret: my-super-secret-key-for-jwt-must-be-at-least-256-bits-long-for-hs256
  access-token-expiration: 900000      # 15 phút (ms)
  refresh-token-expiration: 604800000  # 7 ngày (ms)
```

Secret key phải đủ dài (≥ 256 bits) cho thuật toán HS256. Trên production dùng environment variable, không hard-code.

### Cấu trúc file


### JwtTokenProvider — class tạo và verify JWT

```java
@Component
public class JwtTokenProvider {
    // Đọc config từ application.yaml qua @Value
    // secretKey: dùng Keys.hmacShaKeyFor() tạo SecretKey từ string

    // generateAccessToken(email, role) → JWT string chứa sub, role, iat, exp
    // generateRefreshToken(email) → JWT string đơn giản hơn, thời hạn dài
    // getEmailFromToken(token) → extract subject
    // getRoleFromToken(token) → extract custom claim "role"
    // validateToken(token) → true/false, catch JwtException
    // parseClaims(token) → parse + verify → Claims (payload)
}
```

Giống `jwt.sign()` / `jwt.verify()` bên Node.js nhưng gói thành Spring Bean.

### JwtAuthenticationFilter — filter xử lý JWT

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    // OncePerRequestFilter → đảm bảo chỉ chạy 1 lần mỗi request

    // doFilterInternal():
    //   1. extractToken() → lấy token từ header "Authorization: Bearer <token>"
    //   2. validateToken() → kiểm tra hợp lệ
    //   3. Tạo UsernamePasswordAuthenticationToken(email, null, authorities)
    //   4. Set vào SecurityContextHolder → từ đây các layer sau biết user là ai
    //   5. filterChain.doFilter() → tiếp tục chain (giống next() trong Express)
}
```

**Lưu ý quan trọng:** `SecurityContextHolder` là **thread-safe** — mỗi thread (mỗi request) có SecurityContext riêng. Đây là lý do Spring multi-threaded mà không bị xung đột thông tin user giữa các request.

**Authority format:** Spring Security yêu cầu prefix `ROLE_` → `ROLE_CUSTOMER`, `ROLE_ADMIN`, `ROLE_SELLER`.

## 7. Bài tập — ĐÃ HOÀN THÀNH ✅

### AuthService — 2 method:

**`login(LoginRequest request)` → `AuthResponse`**
- Tìm user theo email (dùng `UserRepository`)
- Nếu không tìm thấy → throw `ResourceNotFoundException`
- So sánh password bằng `passwordEncoder.matches(rawPassword, hashedPassword)` — **lưu ý thứ tự tham số: raw trước, hash sau**
- Nếu sai password → throw `InvalidUserOrPassword`
- Nếu đúng → dùng `JwtTokenProvider` tạo access token + refresh token → trả về `AuthResponse`
- Role lấy từ `user.getRole()`, không hard-code

**`refreshToken(String refreshToken)` → `AuthResponse`**
- Validate refresh token bằng `JwtTokenProvider`
- Nếu không hợp lệ → throw `InvalidToken`
- Lấy email từ token → tìm user trong DB để lấy role → tạo access token mới
- Trả về `AuthResponse` (access token mới, **giữ nguyên refresh token cũ**)

### AuthController — 2 endpoint:

- `POST /api/auth/login` — nhận `@Valid @RequestBody LoginRequest`, gọi `AuthService.login()`
- `POST /api/auth/refresh` — nhận `@Valid @RequestBody RefreshTokenRequest`, gọi `AuthService.refreshToken()`

### Các file bổ sung:

- `RefreshTokenRequest` — DTO chứa field `token` (@NotBlank)
- `InvalidUserOrPassword` — exception với constructor mặc định (overloading)
- `InvalidToken` — exception với message mặc định "Invalid Token"
- `GlobalExceptionHandler` — thêm handler cho `InvalidUserOrPassword` (401) và `InvalidToken` (401)
